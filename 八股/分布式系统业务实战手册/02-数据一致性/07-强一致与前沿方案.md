## 1. 强一致性方案

### 1.1 什么时候需要强一致

- 资金交易
- 库存扣减（部分场景）
- 配置变更
- 权限变更
- 临界资源操作

### 1.2 强一致实现方案

**方案 1：分布式锁 + 唯一数据源**

```
所有写操作通过分布式锁串行化
所有读操作直接读唯一数据源（无缓存）
适用：写并发不高，强一致要求高
```

**方案 2：2PC/XA 事务**

```
跨数据库的强一致事务
通过协调者统一提交/回滚
适用：同库跨表，少用
```

**方案 3：Paxos/Raft 共识算法**

```
多副本写入，多数节点确认才返回成功
如 etcd, Consul, ZooKeeper
适用：配置中心、元数据存储
详见 [02-共识算法](./02-共识算法与可调一致性.md)
```

**方案 4：线性一致性存储**

```
使用支持线性一致性的存储（如 etcd）
业务数据写入 etcd → 强一致
适用：少量关键数据
```

### 1.3 强一致的代价

```
性能：写延迟增加（多节点确认）、吞吐量下降
可用性：多数节点不可用则无法写入
复杂度：部署复杂、运维复杂、故障处理复杂
```

## 2. fencing token 与分布式锁的真正安全

> Martin Kleppmann 指出 Redis 锁（包括 RedLock）不安全的根本原因，并提出 fencing token 解法。

### 2.1 问题：GC 暂停导致双客户端持锁

```
时刻 T1: 客户端 1 获取锁（过期 10s）
时刻 T2: 客户端 1 GC Stop-The-World（暂停 15s）
时刻 T3: 锁过期（10s 到）
时刻 T4: 客户端 2 获取锁
时刻 T5: 客户端 1 GC 结束，仍认为自己持锁
时刻 T6: 客户端 1 和 2 都认为自己持锁 → 同时写存储

→ 即使 Redis 锁本身正确，GC/网络延迟仍可能导致双持锁
→ RedLock 也无法解决（Martin 的核心批评）
```

### 2.2 fencing token 解法

```
原理：
  - 锁服务每次发放锁时附带单调递增的 token
  - 客户端写存储时携带 token
  - 存储拒绝旧 token 的写

流程：
  1. 客户端 1 获取锁 → token=33
  2. 客户端 1 GC 暂停
  3. 锁过期，客户端 2 获取锁 → token=34
  4. 客户端 2 写存储（token=34）→ 成功
  5. 客户端 1 GC 结束，写存储（token=33）
  6. 存储检查：33 < 34 → 拒绝

→ 即使锁失效，旧 token 也写不进 → 真正安全

实现要求：
  - 锁服务能生成单调递增 token
  - 存储能校验 token（如用版本号字段）
```

### 2.3 Redis 锁 vs ZK 锁的真正区别

```
Redis 锁（含 RedLock）：
  - 无 fencing token 能力
  - GC/网络延迟下不安全
  - 适合"性能优先、偶尔不一致可接受"

ZK 锁：
  - 临时节点 + zxid（单调递增）
  - zxid 可作 fencing token
  - GC 延迟下仍安全
  - 适合"正确性优先"

结论：
  - 资金/库存等强一致 → ZK 锁（有 fencing）
  - 防重复提交等弱场景 → Redis 锁（够用）
```

## 3. CRDT 无冲突复制数据类型

> CRDT（Conflict-free Replicated Data Type）让最终一致系统自动解决冲突，无需人工干预。协同编辑、离线优先应用的核心。

### 3.1 问题：最终一致的冲突

```
传统最终一致：
  节点 A 设 X=1，节点 B 设 X=2
  同步时冲突 → 怎么办？
  - LWW（最后写胜出）：靠时间戳，但时钟可能不准
  - 业务规则：如"取较大值"
  - 人工解决

CRDT：
  数据结构本身保证合并无冲突
  → 自动收敛到一致状态
```

### 3.2 CRDT 类型

```
基于状态的 CRDT（CvRDT）：
  - 合并状态（merge 函数）
  - merge 满足交换律、结合律、幂等
  → 任意顺序合并结果一致

基于操作的 CRDT（CmRDT）：
  - 传播操作
  - 操作满足交换律
  → 任意顺序执行结果一致
```

### 3.3 常见 CRDT

| CRDT | 说明 | 应用 |
|------|------|------|
| G-Counter | 只增计数器 | PV/UV |
| PN-Counter | 可增可减计数器 | 余额、库存 |
| LWW-Register | 最后写胜出 | 配置 |
| OR-Set | 观察删除集合 | 标签 |
| LWW-Map | 最后写胜出 Map | KV |
| RGA | 复制图数组 | 协同文本 |

### 3.4 CRDT 示例：G-Counter

```
N 个节点，每个节点维护自己的计数器
  节点 A: [3, 0, 0]  → A 计数 3 次
  节点 B: [0, 5, 0]  → B 计数 5 次
  节点 C: [0, 0, 2]  → C 计数 2 次

合并（取各位最大值）：
  merge([3,0,0], [0,5,0]) = [3,5,0]
  merge([3,5,0], [0,0,2]) = [3,5,2]

总值 = 3+5+2 = 10

特性：
  - 交换律：merge(A,B) = merge(B,A)
  - 结合律：merge(merge(A,B),C) = merge(A,merge(B,C))
  - 幂等：merge(A,A) = A
  → 任意顺序、任意次数合并结果一致
```

### 3.5 CRDT 应用场景

```
协同编辑：Yjs、Automerge
  - 多人同时编辑文档
  - 离线编辑后合并
  - 无冲突自动收敛

分布式数据库：Riak
  - 自动冲突解决
  - 无需业务介入

即时通讯：状态同步
  - 已读未读状态
  - 多端同步
```

## 4. 向量时钟与冲突检测

> Dynamo 模型的核心，用向量时钟检测并发冲突。

### 4.1 为什么需要向量时钟

```
问题：判断两个事件是否并发

标量时间戳不够：
  - 时钟可能不准（时钟漂移）
  - 无法判断因果关系

向量时钟：
  - 每个节点维护一个向量
  - 可判断因果关系（A → B 还是并发）
```

### 4.2 向量时钟原理

```
向量时钟：[node1: count1, node2: count2, ...]

规则：
  1. 节点本地事件 → 自己的 count+1
  2. 发送消息 → 附带当前向量时钟
  3. 接收消息 → 自己的 count+1，与消息向量取各分量 max

因果判断：
  A → B（A 因果先于 B）：A 的所有分量 ≤ B 的对应分量
  并发：A 和 B 不可比较（不满足 A→B 也不满足 B→A）

示例：
  节点 A 写 X=1：VC_A = [1,0]
  同步到 B
  节点 B 写 X=2：VC_B = [1,2]（A 的 VC + B 自增）
  → VC_A ≤ VC_B → 因果序，无冲突

  节点 A 写 X=1：VC_A = [1,0]
  节点 B 写 X=2：VC_B = [0,1]（未同步）
  → 不可比较 → 并发冲突
```

### 4.3 冲突解决

```
检测到并发冲突后：
  1. LWW：时间戳最新者胜（可能丢数据）
  2. 业务规则：如合并字段
  3. 客户端解决：返回多个版本让客户端选
  4. CRDT：自动合并（见 §3）

应用：
  - DynamoDB / Riak：向量时钟 + 客户端解决
  - Git：类似向量时钟的版本图
```

## 5. 事件溯源与 CQRS

> 进阶架构模式，用不可变事件流保证一致性，读写的终极解耦。

### 5.1 传统 CRUD 的问题

```
传统：直接改状态
  UPDATE account SET balance = 100 WHERE id=1

问题：
  - 难审计（不知道改了几次、改成什么）
  - 难回滚（状态被覆盖）
  - 难对账（无历史）
  - 并发难（行锁竞争）
```

### 5.2 事件溯源（Event Sourcing）

```
事件溯源：存事件流，状态由事件回放得出

  不存：balance = 100
  存：[创建账户, 存入50, 存入30, 取出20, 存入40]
  → 回放：0+50+30-20+40 = 100

特点：
  - 事件不可变（只追加）
  - 天然审计（完整历史）
  - 任意时间点状态重建
  - 与 Saga 天然契合

问题：
  - 回放慢 → 用快照优化（定期存状态）
  - 模式变更难 → 版本化事件
  - 查询难 → 配合 CQRS
```

### 5.3 CQRS（命令查询职责分离）

```
CQRS：读模型和写模型分离

写模型（命令侧）：
  - 事件流（强一致）
  - 处理业务逻辑
  - 产出事件

读模型（查询侧）：
  - 物化视图（最终一致）
  - 按查询优化（如 ES、宽表）
  - 多个读模型并存

同步：
  写 → 事件 → 消费者 → 更新读模型

优点：
  - 读写独立优化
  - 读模型可多个（不同查询需求）
  - 写模型聚焦业务一致性

缺点：
  - 复杂度高
  - 最终一致（读模型有延迟）
  - 运维成本高
```

### 5.4 事件溯源 + CQRS 的应用

```
适用场景：
  - 金融账户（完整审计）
  - 订单状态机（状态流转可追溯）
  - 协同编辑（操作历史）
  - 复杂业务流程（与 Saga 结合）

不适用：
  - 简单 CRUD
  - 读多写少且查询简单
  - 团队不熟悉（学习成本高）

代表：
  - Axon Framework
  - EventStoreDB
  - Kafka 事件流
```

### 5.5 与传统方案的对比

| 维度 | 传统 CRUD | 事件溯源 + CQRS |
|------|-----------|------------------|
| 一致性 | 行锁保证 | 事件流保证 |
| 审计 | 难 | 天然支持 |
| 回滚 | 难 | 反向事件 |
| 查询 | 直接查 | 读模型 |
| 复杂度 | 低 | 高 |
| 性能 | 读写竞争 | 读写分离 |
