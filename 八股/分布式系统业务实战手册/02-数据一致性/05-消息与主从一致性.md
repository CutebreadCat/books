## 1. 消息消费一致性

### 1.1 问题

使用 MQ 异步处理业务时，如何保证：
- 消息不丢
- 消息不重复
- 消费顺序正确
- 消费失败能重试

### 1.2 消息不丢方案

```
生产端不丢：
  1. 开启 MQ 确认机制（Producer Confirm）
  2. 消息持久化（MQ 端）
  3. 失败重试 + 死信队列

MQ 端不丢：
  1. 消息持久化（磁盘）
  2. 集群多副本（如 Kafka min.insync.replicas）
  3. 同步刷盘

消费端不丢：
  1. 手动 ACK（处理成功后才确认）
  2. 消费失败重试
  3. 重试 N 次后进死信队列
  4. 死信队列人工处理
```

### 1.3 消息重复问题

```
重复原因：
  - 网络抖动，生产端重发
  - 消费端处理成功但 ACK 失败，MQ 重投
  - 消费端宕机，MQ 重新分配

解决：消费端幂等
  - 唯一 ID + 去重表
  - 状态机
  - 乐观锁
  - Redis SETNX 去重
```

### 1.4 消息顺序性

```
方案 1：单队列单消费者
  - 同一业务的消息发到同一队列
  - 队列只有一个消费者
  - 缺点：吞吐量低

方案 2：分片区 + 单消费者每片区
  - 按 user_id 分片区
  - 每个片区单消费者
  - Kafka 的 partition 机制

方案 3：业务层处理
  - 消息携带版本号/时间戳
  - 消费时检查版本，旧消息丢弃
```

### 1.5 消息堆积处理

```
原因：消费速度 < 生产速度

处理：
  1. 增加消费者数量（注意分片数限制）
  2. 批量消费（一次拉多条）
  3. 异步处理（消费后发到本地队列，多线程处理）
  4. 临时扩容
  5. 死信队列兜底
```

## 2. 主从复制一致性

### 2.1 主从延迟问题

```
主库写入 → Binlog 复制 → 从库应用 → 数据可读
延迟通常：0.1ms ~ 1s（正常）
异常延迟：秒级甚至分钟级

业务影响：
  - 用户刚写入的数据，立刻读可能读到旧值
  - 主从切换时可能丢数据
```

### 2.2 强制读主库方案

| 方案 | 说明 | 适用 |
|------|------|------|
| 关键操作读主库 | 用户自己写后立即读 → 主库 | 写后读场景 |
| 延迟感知路由 | 中间件监控延迟，延迟大则读主 | 通用 |
| 写后读标记 | Redis 标记 force_master TTL=1s | 精准控制 |

### 2.3 半同步复制

```
异步复制（默认）：
  主库写完 Binlog 立即返回，不等从库
  → 性能高，但主库宕机可能丢数据

半同步复制：
  主库写完 Binlog 后，等待至少一个从库确认收到
  → 性能略降，但数据更安全
  → 超时后降级为异步

全同步复制：
  主库等待所有从库确认
  → 性能差，少用
```

### 2.4 主从切换一致性

```
主库宕机 → 提升从库为新主库

风险：
  1. 异步复制下，从库可能落后主库
  2. 切换后，旧主库恢复，可能成为"脑裂"

保障：
  1. 半同步复制减少数据丢失
  2. 切换前等待从库追上（延迟 < 阈值）
  3. 旧主库恢复后，先与新主库对齐再接入
  4. 使用 MHA/MGR 等高可用方案
```

### 2.5 MySQL Group Replication（MGR）

```
MGR 特点：
  - 基于 Paxos 变种
  - 多主写入（冲突检测）
  - 强一致性（多数节点确认）

适用：
  - 强一致需求
  - 多机房部署
  - 自动故障转移
```

### 2.6 主从 + 缓存的三层不一致

```
请求 → Redis → 从库 → 主库

三层都可能有延迟：
  - Redis vs 从库：缓存未更新
  - 从库 vs 主库：复制延迟

最坏情况：
  写主库 → 从库还没同步 → 缓存读到旧从库值

应对：
  - 关键读走主库（绕过缓存）
  - 或读主库刷新缓存
  - 业务容忍短暂不一致
```
