## 1. 跨系统数据对账

### 1.1 为什么需要对账

分布式系统中，数据经过多步流转，可能因异常导致不一致。对账是发现和修复不一致的最后防线。

### 1.2 对账类型

| 类型 | 说明 | 频率 |
|------|------|------|
| 实时对账 | 业务操作完成后异步核对 | 秒级 |
| 定时对账 | 定时批量核对 | 小时/天 |
| 全量对账 | 全量数据核对 | 天/周 |
| 增量对账 | 只核对变化数据 | 小时 |

### 1.3 对账设计

```
定时对账任务：
  1. 从系统 A 导出业务数据
  2. 从系统 B 导出对应数据
  3. 比对关键字段（状态、金额、时间）
  4. 发现不一致 → 生成差异报告
  5. 自动修正（简单差异）或人工确认

实时对账：
  业务操作完成 → 发 MQ → 对账服务异步核对 → 告警 + 修正
```

### 1.4 差异处理策略

```
正向修正（补漏）：
  系统 A 已成功，系统 B 未执行 → 重试执行 B

反向修正（冲正）：
  系统 A 已执行但应该取消 → 发冲正指令

数据重建：
  严重不一致 → 从源头数据重建下游数据

人工介入：
  复杂差异 → 人工分析处理
```

## 2. 最终一致性保障机制

### 2.1 重试机制

```
同步重试：
  - 失败立即重试
  - 适用于网络抖动等瞬时故障
  - 限制重试次数

异步重试：
  - 失败后写入重试表
  - 定时任务扫描重试
  - 适用于持续故障

指数退避：
  - 重试间隔指数增长（1s, 2s, 4s, 8s...）
  - 避免重试风暴

死信队列：
  - 重试 N 次后进入死信队列
  - 人工处理
```

### 2.2 补偿机制

```
正向补偿：
  操作未完成 → 继续完成
  如：扣款成功但发货失败 → 重新发货

反向补偿：
  操作部分完成 → 回滚已完成部分
  如：扣款成功但发货失败 → 退款

业务补偿：
  无法技术回滚 → 业务层面处理
  如：发错短信 → 再发一条说明短信
```

### 2.3 状态机校验

```
业务流程设计为状态机：
  PENDING → PAID → SHIPPED → DELIVERED → COMPLETED

每次状态变更：
  1. 校验当前状态是否允许变更
  2. 状态变更必须原子（乐观锁）
  3. 记录状态变更日志

异常恢复：
  定时扫描卡在中间状态的任务 → 触发补偿
```

## 3. 一致性延迟 SLA（进阶）

> "最终一致"到底多久？无约束可能永远不一致。本节讲如何用 SLA 约束一致性延迟。

### 3.1 问题

```
最终一致性的"最终"是多久？

  无约束 → 可能几秒、几分钟、甚至永远不一致
  → 业务不可控
  → 用户投诉
```

### 3.2 一致性延迟 SLA 定义

```
定义：
  P99 一致性延迟 < X 秒
  即 99% 的数据在 X 秒内达到一致

示例：
  - 缓存一致性 SLA：P99 < 1s
  - ES 同步 SLA：P99 < 10s
  - 跨系统对账 SLA：P99 < 5min
```

### 3.3 监控实现

```
1. 实时对账采样
   - 按比例（如 1%）实时对账
   - 记录数据变更到一致的时间差

2. 延迟分布统计
   - P50 / P90 / P99 延迟
   - 监控趋势

3. 告警
   - 延迟超 SLA → 告警
   - 延迟突增 → 预警
```

### 3.4 降级策略

```
延迟过大时降级：
  1. 延迟 > 阈值 → 关键读走主库（绕过缓存）
  2. 延迟 > 阈值 → 暂停写入（避免不一致扩大）
  3. 延迟 > 阈值 → 切换强一致模式

例：
  缓存一致性延迟 > 5s
  → 读请求绕过缓存直查 DB
  → 写请求双写（DB + 缓存）
  → 延迟恢复后切回
```

### 3.5 SLA 与业务的对齐

```
不同业务的 SLA 要求：
  - 金融：P99 < 0s（强一致，不允许延迟）
  - 电商库存：P99 < 1s
  - 商品缓存：P99 < 10s
  - 搜索同步：P99 < 30s
  - 数据统计：P99 < 1h

对齐方法：
  - 与业务方明确 SLA
  - 监控实测延迟
  - 不达标优化或降级
```
