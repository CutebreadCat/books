## 1. 为什么需要异地多活

### 1.1 单机房的局限

```
单机房风险：
  - 机房级故障（断电、网络、火灾）→ 全站不可用
  - 区域性网络问题 → 影响全国用户
  - 法律合规（数据本地化）

异地多活价值：
  - 容灾：任一机房故障，其他机房继续服务
  - 就近接入：用户访问最近机房，降低延迟
  - 容量：多机房分担流量
  - 合规：数据存本地
```

### 1.2 异地多活的层级

```
同城双活：
  - 同城两个机房
  - 低延迟同步（< 1ms）
  - 一机房故障快速切换

两地三中心：
  - 两个城市、三个机房
  - 主机房 + 同城备 + 异地灾备
  - 平衡成本与容灾

异地多活：
  - 多地多机房都对外服务
  - 用户分流到最近机房
  - 最高级容灾，也最复杂
```

## 2. 单元化架构（LDC）

> LDC（Logic Data Center）是蚂蚁金服提出的单元化架构，是异地多活的主流方案。

### 2.1 核心思想

```
按用户分片到单元：
  - 每个单元（机房）服务一部分用户
  - 单元内闭环（数据、计算都在本单元）
  - 跨单元访问最小化

例：
  单元 A：服务 user_id % 4 == 0 的用户
  单元 B：服务 user_id % 4 == 1 的用户
  单元 C：服务 user_id % 4 == 2 的用户
  单元 D：服务 user_id % 4 == 3 的用户
```

### 2.2 单元化架构组件

```
接入层：
  - 按用户路由到对应单元
  - GSLB（全局负载均衡）+ 单元路由

服务层：
  - 服务按单元部署
  - 同单元服务调用优先
  - 跨单元调用走专线

数据层：
  - 数据按用户分片
  - 每个单元存自己的用户数据
  - 跨单元数据同步

全局数据：
  - 非用户维度的数据（如商品、配置）
  - 多单元共享，单向同步
```

### 2.3 单元路由

```
路由流程：
  1. 用户请求 → GSLB
  2. GSLB 解析用户 ID
  3. 路由到对应单元
  4. 单元内闭环处理

异常路由：
  - 用户 ID 未知 → 默认单元
  - 单元故障 → 切换到备单元（需数据同步）
```

### 2.4 单元化的一致性优势

```
单元内强一致：
  - 用户数据在本单元
  - 本地事务保证强一致
  - 无跨单元事务

跨单元最终一致：
  - 异步同步（Binlog 复制）
  - 容忍短暂不一致

→ 既保证用户体验，又避免分布式事务
```

## 3. 跨地域数据同步

### 3.1 同步内容分类

```
按数据类型：
  1. 用户维度数据（订单、账户）
     - 单元内闭环，无需跨单元
     - 仅同步到备单元（容灾）

  2. 全局数据（商品、配置）
     - 主单元写入，单向同步到其他单元
     - 多读一写

  3. 共享数据（如跨用户的关系）
     - 复杂，需特殊处理
```

### 3.2 同步方案

```
方案 1：Binlog 同步
  - Canal/Debezium 订阅 Binlog
  - 跨地域传输
  - 延迟：秒级到分钟级（看网络）

方案 2：数据库原生同步
  - MySQL 异步复制
  - OB/OceanBase 异地同步
  - TiDB Placement Driver

方案 3：应用层同步
  - 业务变更发 MQ
  - 跨地域 MQ 传输
  - 消费者写入对端

延迟：
  - 同城：< 10ms
  - 跨城：10-100ms
  - 跨国：100-300ms
```

### 3.3 同步的一致性问题

```
单向同步（主→备）：
  - 简单，无冲突
  - 主挂切备，可能丢未同步数据

双向同步（多活）：
  - 双方都可写
  - 冲突处理复杂
  - 需向量时钟/CRDT/业务规则
```

## 4. 冲突处理

### 4.1 冲突场景

```
场景：同一用户在两个单元同时操作

  单元 A：用户余额 +100
  单元 B：用户余额 -50
  同步时合并 → ？

冲突类型：
  - 更新冲突（同字段）
  - 唯一冲突（如用户名）
  - 删除冲突（一删一改）
```

### 4.2 冲突解决策略

```
1. 避免冲突（推荐）
   - 单元化：同一用户只在一个单元写
   - 根本无冲突

2. LWW（最后写胜出）
   - 时间戳最新者胜
   - 简单但可能丢数据
   - 时钟需同步（NTP/TrueTime）

3. 业务规则
   - 余额：取较小值（保守）
   - 计数：累加
   - 状态：按状态机

4. 向量时钟 + 客户端解决
   - 检测冲突返回客户端
   - 客户端决定
   - 详见 [07-强一致与前沿方案](./07-强一致与前沿方案.md)

5. CRDT
   - 数据结构自动合并
   - 详见 [07-强一致与前沿方案](./07-强一致与前沿方案.md)
```

### 4.3 单元化如何避免冲突

```
单元化的核心优势：同一用户只在一个单元写

  用户 U 的所有操作 → 都路由到单元 A
  → 单元 A 强一致
  → 同步到备单元 B（仅容灾）
  → 无双向写冲突

切换时的处理：
  单元 A 故障 → 用户切到单元 B
  → 单元 B 数据可能滞后
  → 切换前等同步追上
  → 或容忍少量数据丢失
```

## 5. 强一致的代价（Spanner）

### 5.1 Spanner 的全球强一致

```
Google Spanner：
  - 全球部署的分布式数据库
  - 线性一致性
  - 跨洲强一致

实现：TrueTime API
  - 原子钟 + GPS 同步时钟
  - 时钟误差 < 7ms
  - 用 TrueTime 打时间戳
  - 等待"最大误差"后提交
  → 保证全局顺序

代价：
  - 跨洲写延迟高（100ms+）
  - TrueTime 硬件成本
  - 复杂度高
```

### 5.2 CockroachDB / TiDB

```
开源 NewSQL 的全球一致：
  CockroachDB：
    - 基于 Raft
    - 全球强一致
    - 无 TrueTime，用 HLC（混合逻辑时钟）

  TiDB：
    - 基于 Raft
    - Placement Driver 控制副本位置
    - 可配置地域分布

代价：
  - 跨地域写延迟高
  - 性能不如单机房
```

### 5.3 强一致 vs 最终一致的取舍

```
强一致（Spanner/CockroachDB）：
  - 数据绝对一致
  - 跨地域延迟高
  - 适用：金融全球账户

最终一致（单元化 + 异步同步）：
  - 延迟低
  - 短暂不一致
  - 适用：电商、社交

实践：
  - 大多数业务用最终一致 + 单元化
  - 极少数金融核心用强一致
```

## 6. 金融 vs 社交异地多活

### 6.1 金融异地多活

```
特点：
  - 不能丢数据
  - 强一致要求
  - 金额绝对准确

方案：
  1. 两地三中心
     - 主机房写，同城备同步，异地灾备
     - 半同步复制保证不丢
  2. 单元化（按用户）
     - 用户数据单元内强一致
     - 跨单元异步同步（容灾）
  3. 对账兜底
     - 多维度实时对账
     - 差异立即处理

关键：
  - 切换前必须等同步追上
  - 宁可不可用也不能丢数据（CP）
```

### 6.2 社交异地多活

```
特点：
  - 可容忍短暂不一致
  - 高可用优先
  - 海量用户

方案：
  1. 单元化（按用户）
     - 用户就近接入
     - 单元内闭环
  2. 异步同步
     - 非用户维度数据单向同步
     - 容忍秒级延迟
  3. 读己之写
     - 用户自己写后立即读走本单元
  4. 最终一致
     - 跨单元数据最终一致
     - 不强求实时

关键：
  - 高可用优先（AP）
  - 用户体验优先
```

### 6.3 对比

| 维度 | 金融 | 社交 |
|------|------|------|
| 一致性 | 强一致（CP） | 最终一致（AP） |
| 切换 | 等同步追上 | 直接切，容忍丢 |
| 延迟 | 高（多节点确认） | 低（异步） |
| 对账 | 实时 + 严格 | 定时 + 宽松 |
| 复杂度 | 高 | 中 |
