# 批量操作的一致性

> 批量扣库存、批量发券、批量退款、批量导入……这些场景在真实业务中极其高频。但与单条操作不同，批量操作天然面临"部分成功、部分失败"的复杂局面。本篇给出工程级的解决方案。

## 1. 问题背景

```
典型业务场景：

场景1：批量扣减库存
  用户购物车有 5 件商品，下单时一次性扣减 5 个 SKU 的库存
  → 其中 3 个成功，2 个库存不足
  → 怎么处理？回滚前 3 个？还是允许部分成功？

场景2：批量发放优惠券
  运营给 10000 个用户批量发券
  → 其中 500 个用户券包已满，发不了
  → 这 500 个怎么处理？重试？跳过？通知？

场景3：批量退款
  用户申请了 3 件商品的退款
  → 第 1 件退款成功，第 2 件退款中，第 3 件退款失败
  → 用户看到的最终状态是什么？

场景4：批量数据导入
  HR 批量导入 1000 条员工信息
  → 其中 10 条格式错误，5 条重复，其余成功
  → 怎么给用户反馈？怎么保证不重复导入？
```

## 2. 问题本质

```
单条操作的一致性：
  BEGIN;
  UPDATE stock SET ...;   -- 要么成功，要么失败
  COMMIT;

批量操作的一致性：
  操作1: 成功
  操作2: 成功
  操作3: 失败（库存不足）
  操作4: 成功
  操作5: 失败（网络超时）

核心矛盾：
  1. 是否要求原子性？→ 全成功或全失败 vs 允许部分成功
  2. 如何处理失败项？→ 回滚、补偿、跳过、重试
  3. 如何给调用方返回结果？→ 统一成功 vs 部分结果列表
```

**与分布式事务的区别**：
- 分布式事务：多个服务/数据库之间的跨节点一致性
- 批量操作：同一节点内的多条记录操作，但结果需要分项处理

---

## 3. 方案对比

| 方案 | 原子性 | 适用场景 | 复杂度 | 推荐 |
|------|--------|----------|--------|------|
| 全量事务 | 全成功/全失败 | 订单内多个 SKU（强关联） | 低 | ✅ 订单场景 |
| 部分成功+回滚 | 成功项保留，失败项回滚 | 关联性弱的批量操作 | 中 | ✅ 库存扣减 |
| 部分成功+记录失败 | 成功项保留，失败项记录待处理 | 批量发券/导入 | 中 | ✅ 运营场景 |
| 异步批处理 | 全部异步，结果轮询 | 大批量（万级+） | 高 | ✅ 大数据量 |

---

## 4. 方案详解

### 4.1 方案一：全量事务（All-or-Nothing）

**核心思路**：把所有操作包在一个大事务里，要么全部成功，要么全部回滚。

```
示例：购物车下单（5 件商品）

BEGIN;
  INSERT orders(...);           -- 创建订单
  INSERT order_items(...);      -- 5 条明细
  UPDATE stock SET ... WHERE sku_id=1 AND stock >= qty;  -- 扣库存1
  UPDATE stock SET ... WHERE sku_id=2 AND stock >= qty;  -- 扣库存2
  UPDATE stock SET ... WHERE sku_id=3 AND stock >= qty;  -- 扣库存3
  UPDATE stock SET ... WHERE sku_id=4 AND stock >= qty;  -- 扣库存4
  UPDATE stock SET ... WHERE sku_id=5 AND stock >= qty;  -- 扣库存5
COMMIT;

关键点：
  - 每条 UPDATE 都带 WHERE stock >= qty 条件（乐观锁）
  - 如果任意一条库存不足，整个事务回滚
  - 用户感知："下单失败，部分商品库存不足"
```

**优点**：
- 简单，不需要额外补偿逻辑
- 用户不会买到"部分商品"的尴尬情况

**缺点**：
- 事务时间长，锁持有时间久（5 个 SKU 的库存行锁同时持有）
- 事务内行数越多，死锁概率越高
- 高并发下性能差

**适用**：
- 订单内多个 SKU 必须一起购买（如套装、套餐）
- 强关联的批量操作

**优化**：
```
减少锁持有时间：
  1. 先查询所有 SKU 库存是否充足（不加锁）
  2. 如果充足，再开启事务统一扣减
  3. 如果不足，直接返回失败，不走事务

注意：
  步骤1的查询和步骤2的扣减之间，库存可能被其他事务扣减
  → 步骤2的 UPDATE 仍需要带 WHERE stock >= qty 条件兜底
```

### 4.2 方案二：部分成功 + 回滚（Tentative + Compensation）

**核心思路**：逐条执行，成功项保留，遇到失败项时回滚已成功的项。

```
示例：购物车下单（允许部分购买）

逻辑：
  1. 逐条扣减库存
     - SKU 1: 成功（扣减成功）
     - SKU 2: 成功
     - SKU 3: 失败（库存不足）
  2. 遇到失败，回滚已成功的项
     - SKU 1: 回滚（库存加回）
     - SKU 2: 回滚（库存加回）
  3. 返回失败结果，告知用户哪些商品库存不足
  4. 用户可以选择：修改数量后继续 / 移除缺货商品后重试

实现：
  BEGIN;
    -- 记录已成功的项
    success_list = []
    for sku in sku_list:
        if deduct_stock(sku):       -- 逐条扣减
            success_list.add(sku)
        else:
            -- 回滚所有已成功的项
            for s in success_list:
                rollback_stock(s)
            return FAILURE
    COMMIT;
```

**优点**：
- 用户不会买到部分商品（要么全买，要么不买）
- 比全量事务更灵活（可以提前终止，减少锁持有）

**缺点**：
- 回滚逻辑复杂
- 回滚过程中又失败怎么办？→ 需要补偿机制兜底
- 高并发下回滚可能导致竞争

**适用**：
- 购物车下单，但允许用户移除缺货商品后重试
- 强关联但不想持有长事务的场景

### 4.3 方案三：部分成功 + 记录失败（最通用）

**核心思路**：逐条执行，成功项保留，失败项记录到失败列表，最终返回"部分成功"的结果。

```
示例：批量发券（10000 个用户）

处理流程：
  1. 将 10000 个用户分成小批（如每批 100 个）
  2. 逐批处理，每批内逐条发券

  批次处理：
    success_list = []   -- 发券成功
    fail_list = []      -- 发券失败（记录原因）

    for user in batch:
        try:
            send_coupon(user)
            success_list.add(user)
        except Exception as e:
            fail_list.add({user, reason: e.message})

  3. 全部处理完成后：
     - 成功：9850 个
     - 失败：150 个（其中 120 个券包已满，30 个用户不存在）
  4. 返回详细结果给调用方（运营后台）
  5. 运营可以选择：
     - 对失败项重试（如券包已满的用户扩容后重试）
     - 导出失败列表人工处理

结果格式：
  {
    "total": 10000,
    "success": 9850,
    "fail": 150,
    "fail_details": [
      { "user_id": "U001", "reason": "COUPON_LIMIT_REACHED", "suggest": "扩容后重试" },
      { "user_id": "U002", "reason": "USER_NOT_FOUND", "suggest": "人工核实" }
    ]
  }
```

**关键设计**：

1. **幂等性**
   ```
   每条发券操作必须幂等：
     - 基于 user_id + coupon_id + batch_id 建立唯一索引
     - 重复发券 → 直接返回成功（幂等）
   ```

2. **分批处理**
   ```
   大批量必须分批：
     - 每批 100 ~ 500 条（根据单条操作耗时调整）
     - 批次之间短暂停顿（避免数据库瞬时压力）
     - 每批独立事务，失败只影响本批
   ```

3. **进度可查询**
   ```
   批量任务表：
     batch_id, status, total, success, fail, create_time, finish_time

   用户可查询进度：
     "已完成 9850/10000，预计剩余 30 秒"
   ```

**优点**：
- 最灵活，真实业务中最常用
- 成功项不浪费，失败项可重试
- 用户体验好（知道哪些成功、哪些失败、为什么失败）

**缺点**：
- 需要设计失败原因码和重试策略
- 需要进度查询和结果导出能力
- 需要幂等性保障

**适用**：
- 批量发券、批量发消息、批量导入、批量退款
- 运营后台类操作
- 成功项和失败项之间无强关联的场景

---

### 4.4 方案四：异步批处理 + 结果轮询

**核心思路**：大批量操作（万级~百万级）走异步任务，用户提交任务后通过轮询/回调获取结果。

```
示例：批量退款（10 万笔订单）

流程：
  1. 用户提交批量退款任务
     → 系统接收请求，生成 task_id
     → 返回：{ "task_id": "T123", "status": "PROCESSING" }

  2. 后台异步执行
     → 任务调度器分批处理（每批 100 笔）
     → 每批独立事务，记录成功/失败明细
     → 进度写入任务表

  3. 用户查询进度
     GET /api/batch_tasks/T123
     → 返回：{ "status": "PROCESSING", "completed": 45000, "total": 100000 }

  4. 任务完成后
     → 通知用户（IM/邮件）
     → 提供结果下载（CSV：成功列表 + 失败列表 + 原因）

任务表设计：
  CREATE TABLE batch_task (
    task_id BIGINT PRIMARY KEY,
    task_type VARCHAR(32),       -- REFUND / COUPON / IMPORT
    status VARCHAR(16),          -- PENDING / PROCESSING / SUCCESS / PARTIAL / FAIL
    total_count INT,
    success_count INT DEFAULT 0,
    fail_count INT DEFAULT 0,
    fail_details TEXT,           -- JSON 数组，失败明细
    create_time DATETIME,
    finish_time DATETIME,
    operator_id BIGINT
  );
```

**优点**：
- 处理量大，不受接口超时限制
- 系统压力可控（分批+限流）
- 结果可持久化、可审计

**缺点**：
- 用户无法立即看到结果
- 需要任务调度系统（如 XXL-Job）
- 需要结果查询/下载能力

**适用**：
- 数据量 > 1000 的批量操作
- 运营后台批量处理
- 对实时性要求不高的场景

## 5. 补偿机制

批量操作的失败项必须可补偿：

```
补偿策略矩阵：

| 失败原因 | 是否可重试 | 补偿方式 | 示例 |
|----------|------------|----------|------|
| 库存不足 | 否 | 跳过/通知用户 | 商品卖完了 |
| 网络超时 | 是 | 自动重试3次 | 远程调用失败 |
| 数据库连接满 | 是 | 延迟重试 | 数据库瞬时压力 |
| 用户状态异常 | 否 | 记录失败+人工处理 | 用户已注销 |
| 幂等冲突 | 否 | 视为成功 | 重复提交 |

重试策略：
  - 固定间隔：失败 1s 后重试
  - 指数退避：1s → 2s → 4s → 8s
  - 最大重试：3 次
  - 重试仍失败 → 进入死信队列 / 人工处理
```

## 6. 业务选型决策

```
决策树：

Q1: 这批操作是否必须全部成功？（强关联）
  是 → 方案一：全量事务（如订单内多 SKU 套装）
  否 → Q2

Q2: 操作量是否 > 1000？
  是 → 方案四：异步批处理 + 结果轮询
  否 → Q3

Q3: 成功项和失败项之间是否有关联？
  是（如库存，买了3件不能留2件）→ 方案二：部分成功+回滚
  否（如发券，发10个成功、2个失败互不干扰）→ 方案三：部分成功+记录失败

Q4: 是否需要立即给用户反馈？
  是 → 方案三（同步返回部分结果）
  否 → 方案四（异步任务）
```

| 业务场景 | 推荐方案 | 说明 |
|----------|----------|------|
| 购物车多 SKU 下单 | 方案一（全量事务）或方案二（回滚） | 用户感知"要么全买要么不买" |
| 批量发券 | 方案三（部分成功） | 成功即发，失败记录原因 |
| 批量退款 | 方案三（部分成功） | 每笔退款独立，互不影响 |
| 批量数据导入 | 方案三 + 方案四 | 小批量同步，大批量异步 |
| 订单批量发货 | 方案四（异步） | 量大，不需要实时反馈 |
| 批量调整库存 | 方案三（部分成功） | 记录失败明细，运营人工核实 |

## 7. 踩坑与反模式

### 坑 1：大事务导致死锁
```
问题：全量事务中 5 个 SKU 库存同时加锁，顺序不一致导致死锁
  事务A: 锁 SKU1 → 等 SKU2
  事务B: 锁 SKU2 → 等 SKU1
解决：统一加锁顺序（如按 SKU ID 升序）
```

### 坑 2：回滚过程中又失败
```
问题：方案二回滚已扣库存时，该库存又被其他事务扣减了
  回滚失败 → 数据不一致
解决：回滚操作也要用乐观锁（WHERE stock + qty <= max_stock）
      回滚失败 → 进入补偿队列人工处理
```

### 坑 3：失败原因不明
```
问题：返回"部分失败"，但失败原因只有一个笼统的"系统错误"
  运营/用户不知道怎么办
解决：失败原因必须分类、可枚举、有处理建议
      COUPON_LIMIT_REACHED → "用户券包已满，建议扩容"
      USER_NOT_FOUND → "用户不存在，建议核实用户ID"
```

### 坑 4：缺少幂等导致重复
```
问题：用户看到"批量发券失败"，点击重试
  实际第一次已经成功 8000 个，重试又发了 8000 个
解决：每条操作必须幂等（唯一索引/去重表）
      重试时幂等返回成功，不会重复执行
```

### 坑 5：异步任务没有监控
```
问题：提交了批量任务，系统挂了就不知道执行到哪了
解决：任务状态必须持久化，每批执行后更新进度
      任务执行异常时告警，支持人工触发重试/回滚
```

### 坑 6：批量操作没有限流
```
问题：运营一次性导入 100 万条数据，直接把数据库打挂
解决：批量操作必须有限流：
  - 单用户提交上限（如每次最多 1000 条）
  - 系统级并发上限（如同时最多 5 个批量任务在执行）
  - 分批执行间隔（如每批之间 sleep 100ms）
```

## 8. 面试追问应对

**Q1：购物车有 5 件商品，下单时怎么保证一致性？**
> 如果这 5 件商品是强关联的（如套装），我会用一个数据库事务全量扣减，每条 UPDATE 带 WHERE stock >= qty 乐观锁条件，任意一条失败则全事务回滚。如果是普通购物车，允许多买少买，我会先逐条扣减，遇到失败时回滚已扣减的库存，返回具体哪些商品库存不足，让用户移除缺货商品后重试。

**Q2：批量发券部分失败怎么处理？**
> 我会采用"部分成功 + 记录失败"的方案。首先把 1 万个用户分成小批（每批 100 个），逐条发券，成功的记录成功数，失败的记录失败原因（如券包已满、用户不存在）。处理完成后返回详细结果：成功 9850 个，失败 150 个（其中 120 个券包已满，30 个用户不存在）。运营可以按失败原因分类处理：券包已满的扩容后重试，用户不存在的导出人工核实。每条发券操作基于 user_id + coupon_id + batch_id 做唯一索引保证幂等，防止重试时重复发券。

**Q3：大批量操作（如 10 万笔退款）怎么做？**
> 走异步批处理。用户提交退款请求后，系统立即返回任务 ID 和"处理中"状态。后台通过定时任务（如 XXL-Job）分批处理，每批 100 笔，每笔独立事务。每批处理完后更新任务进度表。任务完成后通知用户，并提供结果下载（成功列表 + 失败列表 + 失败原因）。这样可以避免接口超时，也保护数据库不被瞬时压力打满。

**Q4：批量操作怎么防止死锁？**
> 如果必须用全量事务，我会统一加锁顺序，比如按 SKU ID 升序加锁，确保所有事务按相同顺序获取锁，避免循环等待。更好的做法是避免大事务，改用逐条处理 + 失败回滚的方案，或者异步分批，每批独立事务，减少同时持有锁的数量。
