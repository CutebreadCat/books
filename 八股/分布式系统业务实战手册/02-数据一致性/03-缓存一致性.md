## 1. 问题背景

使用 Redis 缓存 MySQL 数据时，如何保证两者数据一致？

```
读流程：
  请求 → Redis → 命中返回
              ↓ 未命中
              MySQL → 写入 Redis → 返回

写流程：
  请求 → ??? (先操作 Redis 还是 MySQL？)
```

---

## 2. 四种方案对比

| 方案 | 并发问题 | 一致性 | 推荐 |
|------|----------|--------|------|
| 先更新缓存再更新 DB | 缓存成功 DB 失败 → 数据丢失 | 最差 | ❌ 绝不推荐 |
| 先更新 DB 再更新缓存 | 并发写导致缓存是旧值 | 中 | ⚠️ 不推荐 |
| 先更新 DB 再删除缓存 | 极端时序：删缓存后读 DB 旧值再写缓存 | 较好 | ✅ 推荐 |
| 先删缓存再更新 DB | 并发读写：删缓存后读 DB 旧值写缓存 | 差 | ❌ 不推荐 |

### 2.1 为什么"先删缓存再更新 DB"危险

```
时刻 T1: 线程 A 删除缓存
时刻 T2: 线程 B 读缓存 → 未命中 → 读 DB（旧值）
时刻 T3: 线程 A 更新 DB（新值）
时刻 T4: 线程 B 把旧值写入缓存

结果：DB 是新值，缓存是旧值 → 数据不一致
      且会持续到缓存过期或下次更新
```

### 2.2 为什么"先更新 DB 再删除缓存"较好

```
正常流程：
  线程 A: 更新 DB → 删除缓存
  线程 B: 读缓存 → 未命中 → 读 DB（新值）→ 写入缓存

极端并发场景（出现概率极低）：
  时刻 T1: 缓存恰好失效
  时刻 T2: 线程 B 读 DB（旧值）
  时刻 T3: 线程 A 更新 DB（新值）
  时刻 T4: 线程 A 删除缓存
  时刻 T5: 线程 B 把旧值写入缓存

  → 需满足：缓存恰好失效 + 读 DB 早于写 DB + 写缓存晚于删缓存
  → 概率极低，但理论上存在
```

## 3. 推荐方案：延迟双删

```
写操作流程：
1. 删除 Redis 缓存
2. 更新 MySQL
3. 延迟 500ms 后再次删除 Redis 缓存

为什么需要第二次删除？
  步骤 1 删缓存后，可能有读请求：
  → 读请求发现缓存空 → 读 MySQL（此时可能还是旧值） → 写入缓存
  → 步骤 2 更新 MySQL
  → 缓存里仍然是旧值！
  → 步骤 3 延迟后再删缓存，清除这个脏缓存
```

**延迟时间设置**：
- 一般 500ms - 1s
- 应大于一次读操作的耗时（读 DB + 写缓存）
- 可根据业务 QPS 调整

## 4. 进阶方案：MQ 异步删缓存

```
写操作流程：
1. 更新 MySQL
2. 发送消息到 MQ（消息内容：删除缓存 key）
3. 消费者收到消息后删除 Redis 缓存

优点：
  - MQ 保证消息不丢
  - 失败可重试
  - 异步不阻塞主流程

缺点：
  - 引入 MQ 增加复杂度
  - 消息消费有延迟
```

---

## 5. 终极方案：Binlog 订阅

```
方案：Canal/Debezium 订阅 MySQL Binlog

流程：
  1. 业务代码只更新 MySQL（不操作缓存）
  2. Canal 模拟 MySQL 从库，订阅 Binlog
  3. Binlog 变更 → Canal 推送到 MQ
  4. 消费者收到变更事件 → 删除/更新 Redis 缓存

优点：
  - 完全解耦业务代码与缓存维护
  - 数据最终一致
  - 延迟通常 < 1 秒
  - 不会遗漏任何变更

缺点：
  - 增加运维复杂度
  - 需要保证 Canal 高可用
  - 消费者需幂等
```

## 6. 保障措施

```
1. 设置合理的过期时间
   - 即使所有方案失效，过期后下次读会从 DB 加载最新值
   - 兜底机制

2. 延迟双删 + Binlog 订阅
   - 双保险：业务层延迟双删 + Binlog 订阅兜底

3. 定期全量对账
   - 定时任务扫描，对比 Redis 和 MySQL 数据
   - 发现不一致则修复

4. 监控告警
   - 监控缓存命中率异常
   - 监控不一致数量
```

## 7. 业务决策

| 场景 | 方案 | 说明 |
|------|------|------|
| 低并发普通业务 | 先更新 DB 再删缓存 | 简单够用 |
| 中等并发 | 延迟双删 | 成本低效果好 |
| 高并发核心数据 | MQ 异步删缓存 | 可靠性高 |
| 强一致要求 | Binlog 订阅 + 读写都走 DB | 读时查缓存 + DB 双校验 |
| 极致一致 | 不用缓存（只读 DB） | 牺牲性能换一致 |

### 7.1 多级缓存的一致性

```
请求 → 本地缓存 → Redis → MySQL

多级缓存一致性更复杂：
  - 各级 TTL 不同（本地 10s，Redis 60s）
  - 数据变更需逐级清理

方案：
  1. TTL 短 + 容忍短暂不一致
  2. 消息广播清理本地缓存
  3. 版本号控制（key 含版本号，变更即换 key）
```
