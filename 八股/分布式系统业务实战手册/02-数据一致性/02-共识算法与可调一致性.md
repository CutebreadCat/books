## 1. 为什么需要共识算法

### 1.1 问题背景

```
场景：3 个节点的复制
  写入 → 3 个节点都写成功 → 强一致
  但如果要求"都成功"，任一节点挂了就不可用

矛盾：
  - 全部确认 → 强一致但可用性差
  - 部分确认 → 可用但可能不一致

共识算法解决：
  - 多数派确认即可（2/3）
  - 保证多数派一致 → 强一致
  - 少数派可宕机 → 高可用
```

### 1.2 共识算法要解决的问题

1. **如何选主**：谁是 Leader
2. **如何复制日志**：Leader 如何把变更同步到 Followers
3. **如何保证安全**：已提交的数据不会丢
4. **如何处理故障**：节点宕机/恢复/网络分区
5. **如何成员变更**：增减节点

## 2. Raft 详解

Raft 是为**可理解性**设计的共识算法（etcd/Consul 用）。

### 2.1 核心思想

```
Raft 三分解：
  1. Leader 选举：选一个 Leader
  2. 日志复制：Leader 接收写请求，复制到 Followers
  3. 安全性：保证已提交的日志不丢
```

### 2.2 节点状态

```
Follower（跟随者）：
  - 默认状态
  - 被动接收 Leader 的请求

Candidate（候选人）：
  - Follower 超时未收到心跳 → 变 Candidate
  - 发起选举

Leader（领导者）：
  - 处理所有写请求
  - 发心跳维持地位
  - 复制日志到 Followers

转换：
  Follower →（选举超时）→ Candidate →（获多数票）→ Leader
  Candidate →（获多数票）→ Leader
  Candidate →（发现更高 term）→ Follower
  Leader →（发现更高 term）→ Follower
```

### 2.3 Leader 选举

```
term（任期）：
  - 单调递增的整数
  - 每次选举 term+1
  - 防止旧 Leader 干扰

选举流程：
  1. Follower 超时未收到心跳（150-300ms 随机）
  2. 变 Candidate，term+1
  3. 投自己一票
  4. 发 RequestVote RPC 给其他节点
  5. 收到多数票 → 变 Leader
  6. 发心跳维持

防分裂投票：
  - 随机超时时间（150-300ms）
  - 先超时的先发起选举
  - 减少多个 Candidate 同时竞选
```

### 2.4 日志复制

```
写流程：
  1. 客户端写请求 → Leader
  2. Leader 写本地日志（未提交）
  3. Leader 发 AppendEntries RPC 给所有 Followers
  4. Follower 写本地日志，回复 ACK
  5. Leader 收到多数 ACK → 提交（commit）
  6. Leader 返回客户端成功
  7. Leader 下一条 AppendEntries 携带 commitIndex
  8. Follower 应用已提交日志到状态机

关键：
  - 多数 ACK 即可提交（不要求全部）
  - 提交后不可撤回（安全性）
  - Leader 故障，新 Leader 必须包含已提交日志
```

### 2.5 安全性保证

```
Raft 安全性约束：
  1. 选举限制：Candidate 的日志必须至少和多数节点一样新
     → 保证新 Leader 包含所有已提交日志
  2. Leader 只追加：Leader 不会覆盖自己的日志
  3. 日志匹配：相同 index+term 的日志内容相同
  4. 状态机安全：已应用的日志不会回滚

脑裂处理：
  - 网络分区 → 可能有旧 Leader
  - 旧 Leader 在少数派分区 → 无法提交（缺多数）
  - 多数派分区选出新 Leader（更高 term）
  - 分区恢复 → 旧 Leader 发现更高 term → 降级
```

### 2.6 成员变更

```
问题：直接加节点可能导致两个多数派（脑裂）

单节点变更（简单）：
  - 一次只加/减一个节点
  - 不会产生两个不相交的多数派

联合一致（复杂）：
  - Cold → Cold,new → Cnew
  - 两阶段变更
```

### 2.7 Raft 的工程优化

```
1. 日志压缩（Snapshot）
   - 定期快照，删除已应用日志
   - 减少日志体积

2. 批量与管道
   - 批量 AppendEntries
   - 管道化发送（不等 ACK 发下一条）

3. PreVote
   - 选举前先探测，避免无谓选举
   - 防止网络抖动触发频繁选举

4. ReadIndex / Lease Read
   - 读不走完整 Raft 流程
   - 提升读性能
```

## 3. Paxos 详解

Paxos 是 Leslie Lamport 提出的经典共识算法，**正确但难懂**。

### 3.1 Basic Paxos

```
角色：
  - Proposer：提议者
  - Acceptor：接受者
  - Learner：学习者

两阶段：
  阶段一（Prepare）：
    1. Proposer 选提案号 N，发 Prepare(N) 给多数 Acceptor
    2. Acceptor 收到 Prepare(N)：
       - 若 N > 已承诺的最大号 → 承诺不再接受 < N 的提案
       - 若已接受过提案 → 返回已接受的提案
       - 否则拒绝

  阶段二（Accept）：
    1. Proposer 收到多数承诺 → 发 Accept(N, value)
       - value：如果 Acceptor 返回了已接受值，用最大号已接受值
       - 否则用自己的 value
    2. Acceptor 收到 Accept → 接受（除非承诺了更大号）
    3. Learner 学习已选定的值

关键：提案号全局唯一且单调递增
```

### 3.2 Multi-Paxos

```
Basic Paxos 每次都要两阶段，开销大

Multi-Paxos 优化：
  - 选一个稳定的 Leader
  - Leader 直接发 Accept（跳过 Prepare）
  - 只有 Leader 变更时才重新 Prepare

  → 大幅减少开销
  → 但 Multi-Paxos 没规定 Leader 选举细节（留给实现）
```

### 3.3 为什么 Paxos 难

```
1. 论文抽象，缺工程细节
2. 没规定 Leader 选举、日志复制等工程问题
3. 角色和流程复杂
4. 边界条件多

Raft 的贡献：
  - 把 Paxos 的正确性保留
  - 工程细节明确（Leader/日志/快照）
  - 为可理解性设计
```

## 4. Raft vs Paxos vs ZAB

| 维度 | Raft | Paxos | ZAB（ZooKeeper） |
|------|------|-------|-------------------|
| 设计目标 | 可理解性 | 通用性 | 顺序一致性 |
| Leader | 强 Leader | 可无 Leader | 有 Leader |
| 日志 | 顺序日志 | 不规定 | 顺序日志（ZXID） |
| 选举 | term + 随机超时 | 无明确规定 | epoch + 事务ID |
| 复杂度 | 中 | 高 | 中 |
| 工程实现 | etcd/Consul | 少（Chubby） | ZooKeeper |
| 读性能 | ReadIndex 优化 | - | 直接读（顺序一致） |

**选型**：
- 新项目 → Raft（etcd/Consul）
- 已有 ZK 生态 → ZAB
- 学术研究 → Paxos

## 5. QUORUM 读写与可调一致性

### 5.1 QUORUM 原理

```
N 个副本，写成功 W 个，读成功 R 个

关键公式：
  W + R > N → 读和写必有交集 → 强一致
  W + R ≤ N → 可能读到旧值 → 最终一致

例（N=3）：
  W=2, R=2 → W+R=4 > 3 → 强一致
  W=1, R=1 → W+R=2 < 3 → 最终一致（高可用高性能）
  W=3, R=1 → W+R=4 > 3 → 强一致（写慢读快）
  W=1, R=3 → W+R=4 > 3 → 强一致（写快读慢）
```

### 5.2 可调一致性（Cassandra/DynamoDB）

```
业务按需选一致性级别：

  ONE：只等 1 个副本 → 快但可能不一致
  QUORUM：等多数 → 强一致但慢
  ALL：等所有 → 最强但最慢
  LOCAL_QUORUM：本地数据中心多数
  EACH_QUORUM：每个数据中心多数

业务选型：
  - 日志写入：ONE（性能优先）
  - 用户资料：QUORUM（一致优先）
  - 金融数据：ALL（绝对一致）
```

### 5.3 QUORUM 的数学保证

```
为什么 W+R>N 保证强一致？

  N=3, W=2, R=2
  写成功 2 个（集合 A）
  读成功 2 个（集合 B）
  A 和 B 在 3 个节点中必有交集（鸽巢原理）
  → 读到至少 1 个最新值
  → 用版本号判断最新 → 强一致

  反例 W=1, R=1：
  A 和 B 可能无交集
  → 读到旧值 → 最终一致
```

### 5.4 QUORUM 的局限

```
1. 不保证线性一致
   - QUORUM 保证读最新已提交
   - 但跨操作的全局顺序不保证
   - 需要额外机制（如版本号/时间戳）

2. 网络分区时
   - 多数派分区可继续
   - 少数派分区不可用
   - 这是 CP 的代价

3. 写性能
   - W=QUORUM 比写一个慢
   - 高写场景需权衡
```

## 6. 工程实现对比

| 系统 | 算法 | 一致性 | 适用 |
|------|------|--------|------|
| etcd | Raft | 线性一致 | 配置/元数据 |
| Consul | Raft | 线性一致 | 服务发现 |
| ZooKeeper | ZAB | 顺序一致 | 协调服务 |
| Cassandra | QUORUM | 可调 | NoSQL |
| DynamoDB | QUORUM | 可调 | NoSQL（云） |
| Spanner | Paxos | 线性一致（全局） | 全球数据库 |
| TiDB | Raft | 线性一致 | NewSQL |

### 6.1 etcd vs ZooKeeper（高频面试）

| 维度 | etcd | ZooKeeper |
|------|------|-----------|
| 算法 | Raft | ZAB |
| 语言 | Go | Java |
| 一致性 | 线性一致 | 顺序一致 |
| API | gRPC + REST | 自定义协议 |
| 生态 | K8s | Hadoop/Spark |
| 数据模型 | KV | 树形节点 |
| Watch | 流式 | 一次性（需重连） |
| 学习成本 | 低 | 中 |

**选型**：K8s 生态选 etcd，传统大数据选 ZK。
