# 读写分离与会话一致性

> 读写分离是 MySQL 最常见的扩展手段，但它引入了一个高频面试陷阱：**用户刚写完数据，立刻读却读不到**。这就是"读己之写"（Read-Your-Writes）问题。本篇从工程落地角度给出完整方案。

## 1. 问题背景

```
典型架构：
  写请求 → MySQL 主库
  读请求 → MySQL 从库（1-N 个）

问题：主从复制存在延迟（毫秒 ~ 秒级）
  主库写入 → Binlog 同步 → 从库重放 → 从库可读

用户场景：
  T1: 用户发布了一条评论（写主库）
  T2: 用户刷新页面查看评论列表（读从库）
  → 从库还没同步完，列表里没有刚发的评论
  → 用户以为"发布失败"，反复提交
```

**影响范围**：
- 用户发表内容后立刻刷新（评论、帖子、评价）
- 用户修改资料后查看个人页
- 管理员上架商品后查看商品列表
- 订单创建后查询订单详情

## 2. 问题本质：读己之写

```
一致性模型光谱中的位置：

  线性一致 ──→ 顺序一致 ──→ 因果一致 ──→ 读己之写 ──→ 单调读 ──→ 最终一致
                                                      ↑
                                                  本节聚焦

读己之写（Read-Your-Writes）：
  如果一个进程写入了一个值，那么该进程后续读取一定能读到这个值（或更新的值）。

读写分离天然破坏了这个保证：
  写发生在主库，读发生在从库，从库可能还没收到写操作。
```

**与缓存一致性的区别**：
- 缓存一致性：Redis vs MySQL 之间的不一致
- 会话一致性：主库 vs 从库之间的不一致，且发生在**同一个用户的同一次会话**中

---

## 3. 方案对比

| 方案 | 原理 | 一致性 | 复杂度 | 性能 | 推荐 |
|------|------|--------|--------|------|------|
| 强制走主库 | 写后一定时间内读主库 | 强 | 低 | 中 | ✅ 最常用 |
| 缓存前置 | 写后读缓存而非从库 | 强 | 中 | 高 | ✅ 读多场景 |
| 会话粘性 | 同一会话绑定主库读 | 强 | 中 | 低 | ⚠️ 主库压力大 |
| 延迟窗口 | 写后等待 N 毫秒再读 | 中 | 低 | 低 | ❌ 不推荐 |
| 复制位点 | 写后记录 Binlog 位点，从库追赶到位点后才读 | 强 | 高 | 低 | ⚠️ 极少用 |

---

## 4. 方案详解

### 4.1 方案一：强制走主库（最常用）

**核心思路**：写操作后的短暂时间内，相关读请求强制路由到主库。

```
实现方式：

1. 标记法（推荐）
   用户写操作成功后，在响应中返回一个标记（如 token 或时间戳）。
   前端在随后的读请求中携带该标记，后端识别到标记后强制走主库。

   示例：
     写：POST /api/comment → 返回 { "comment_id": 123, "fresh_token": "abc" }
     读：GET /api/comments?fresh_token=abc → 路由到主库

2. 会话标记法
   用户会话中记录"最近写操作时间戳"。
   读请求时，如果当前时间 - 写时间戳 < 阈值（如 500ms），则读主库。

   示例：
     session.last_write_time = now()
     读请求时：
       if (now() - session.last_write_time < 500ms):
           read_from_master()
       else:
           read_from_slave()
```

**关键参数**：
- 强制主库时间窗口：一般 500ms ~ 2s（根据主从延迟调整）
- 窗口内只影响当前用户的读，不影响其他用户

**优点**：
- 简单，无需改存储结构
- 精确控制哪些读需要强一致

**缺点**：
- 主库读压力增加（但只在写后的短暂窗口内）
- 需要前端配合传标记

---

### 4.2 方案二：缓存前置（读多场景推荐）

**核心思路**：写操作后，把数据写入缓存；读请求先读缓存，缓存命中则直接返回，不再读从库。

```
架构：
  写请求：
    → 写 MySQL 主库
    → 同时/异步写入 Redis（key = 用户ID + 数据类型）
    → 设置短 TTL（如 5s）

  读请求：
    → 先读 Redis（按用户ID + 数据类型）
    → 命中 → 直接返回
    → 未命中 → 读从库

示例：用户评论列表
  写：发布评论 → 写主库 + 删除/更新 Redis `user:comments:user_id`
  读：GET `user:comments:user_id` → 命中则直接返回（含最新评论）
```

**为什么是"前置"而非普通缓存？**
- 普通缓存：按业务 key 缓存，如 `comment:comment_id`
- 缓存前置：按**用户维度**缓存聚合数据，如 `user:comments:user_id`

**优点**：
- 主库读压力不增加
- 性能高（Redis 读延迟 < 1ms）

**缺点**：
- 需要维护用户维度的缓存
- 缓存与主库数据一致性需要兜底（TTL 短 + 延迟双删）

---

### 4.3 方案三：会话粘性（谨慎使用）

**核心思路**：同一会话内的所有读写请求都路由到主库。

```
实现：
  用户登录后建立会话 → 标记该会话为"写活跃"
  会话期间所有读请求 → 走主库
  会话结束/超时后 → 恢复从库读

适用：
  后台管理系统（管理员操作后立即查看结果）
  编辑/发布类系统（编辑中频繁保存和预览）
```

**严重缺点**：
- 如果用户会话很长（如 30 分钟），主库读压力极大
- 违背了读写分离的初衷

**建议**：
- 仅用于后台管理类低并发场景
- 生产环境高并发业务不推荐

---

### 4.4 方案四：延迟窗口（不推荐）

**核心思路**：写操作后，线程 sleep 几百毫秒再读。

```
示例：
  写评论 → sleep(500ms) → 读评论列表

问题：
  1. 用户体验差（白白等待）
  2. 延迟不可控（主从延迟可能 > sleep 时间）
  3. 高并发下线程sleep浪费资源
```

**反模式**：在接口里主动 sleep 是严重的代码坏味道。

---

### 4.5 方案五：复制位点（理论优雅，工程复杂）

**核心思路**：MySQL 主库执行事务后记录 GTID/Binlog 位点，读从库时等待从库追赶到该位点后再返回。

```
流程：
  1. 写主库 → 获取执行后的 GTID（如 gtid = 'uuid:123'）
  2. 读从库时 → 执行 SELECT WAIT_FOR_EXECUTED_GTID_SET('uuid:123', timeout)
  3. 从库追赶到位点后 → 执行读操作
  4. 超时未追上 → 降级读主库

MySQL 支持：
  WAIT_FOR_EXECUTED_GTID_SET() — 等待从库追上指定 GTID
  或者 session_track_gtids = OWN_GTID
```

**优点**：理论完美，保证读己之写
**缺点**：
- 需要 MySQL 5.7+ 且开启 GTID
- 等待操作阻塞线程，性能差
- 工程实现复杂，大多数公司不用

## 5. 业务选型决策

```
决策树：

Q1: 用户写后是否一定会立刻读？
  否 → 不需要特殊处理，最终一致即可
  是 → Q2

Q2: 该数据是否是聚合列表（如评论列表、订单列表）？
  是 → 方案二：缓存前置（按用户维度缓存聚合结果）
  否 → Q3

Q3: 读请求量是否很大？
  是 → 方案二：缓存前置（保护从库和主库）
  否 → 方案一：强制走主库（简单直接）

Q4: 是否是后台管理/编辑类场景？
  是 → 方案三：会话粘性（但限制会话时长）
  否 → 方案一：强制走主库

特殊场景：
  支付/订单状态 → 方案一强制走主库（不能容忍任何延迟）
  社交 Feed → 方案二缓存前置（用户只看自己的聚合列表）
  商品库存 → 直接读 Redis（不走从库）
```

| 业务场景 | 推荐方案 | 说明 |
|----------|----------|------|
| 用户发布评论后刷新列表 | 缓存前置 | 评论列表按用户聚合，缓存命中率高 |
| 用户修改昵称后查看个人页 | 强制走主库 | 单条记录，读一次主库成本低 |
| 订单创建后查询订单详情 | 强制走主库 | 涉及资金，不容忍延迟 |
| 商家上架商品后查看商品列表 | 强制走主库（500ms窗口） | 后台操作，并发低 |
| 用户点赞后查看点赞数 | 缓存前置 | 计数型数据，Redis 原生支持 |

## 6. 踩坑与反模式

### 坑 1：强制主库窗口时间过长
```
问题：窗口设为 5s，主库读压力暴增
解决：根据主从延迟监控动态调整，一般 500ms 足够
      设置上限（如最大 2s），超过则从库读
```

### 坑 2：缓存前置没有设置 TTL
```
问题：缓存永远不失效，数据不一致后无法修复
解决：TTL 必须设置（5s ~ 60s），作为兜底
```

### 坑 3：强制主库后没有降级
```
问题：主库故障时，强制主库的读请求全部失败
解决：主库不可用时，降级读从库（接受短暂不一致）
```

### 坑 4：所有读都走主库
```
问题："为了安全"，所有读都强制走主库
解决：只有"写后立刻读同一条/同一批数据"的场景才需要
      其他读请求正常走从库
```

### 坑 5：忽略跨会话一致性
```
问题：用户 A 写了一条数据，用户 B 立刻读不到
      （这不是读己之写，是跨用户最终一致）
解决：区分"会话内一致"和"全局最终一致"
      读己之写只解决同一会话，跨用户靠最终一致 + 缓存 TTL
```

## 7. 面试追问应对

**Q1：读写分离后，用户发表了评论刷新看不到怎么办？**
> 这是典型的读己之写问题。我的方案是：写操作后，在响应中返回一个 fresh_token，前端在随后 500ms 内的读请求携带该 token，后端识别后强制走主库。窗口过后恢复从库读。同时配合用户维度的缓存前置，进一步降低主库压力。

**Q2：强制走主库会不会把主库打挂？**
> 不会。强制走主库只发生在写后的短暂窗口（500ms），且只影响当前用户的相关读请求。通过监控主从延迟动态调整窗口，如果主库压力大，可以缩短窗口或降级到缓存前置方案。

**Q3：缓存前置和延迟双删有什么区别？**
> 延迟双删解决的是缓存与数据库之间的不一致；缓存前置解决的是读写分离中从库延迟导致的读不到新数据。缓存前置的思路是"写后把聚合结果放在用户维度的缓存里，读时直接从缓存拿，绕过从库"。

**Q4：MySQL GTID 方案了解吗？**
> 了解。MySQL 5.7+ 支持 WAIT_FOR_EXECUTED_GTID_SET()，可以等待从库追赶到指定 GTID 后再读。这是理论上最优雅的方案，但生产环境极少用，因为阻塞等待性能差，且实现复杂。大多数场景用"强制主库 + 缓存前置"更务实。
