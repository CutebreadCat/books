## 1. 为什么需要量化

```
面试中的典型对话：

面试官：你们系统的一致性怎么保证？
候选人：我们用 MQ 异步同步，保证最终一致。
面试官：最终是多久？
候选人：……一般很快。
面试官：如果 1 小时还没一致呢？
候选人：……

问题：
  "最终一致"是一个没有边界的承诺。
  没有度量就没有管理，没有管理就没有可靠性。

量化的意义：
  1. 定义"正常"：99.9% 的数据在 5 秒内达到一致
  2. 定义"异常"：超过 1 分钟不一致 → 告警
  3. 驱动优化：监控到延迟主要集中在 Binlog 同步环节 → 优化 Canal 配置
```

## 2. 核心指标体系

### 2.1 一致性率（Consistency Rate）

```
定义：在统计周期内，数据一致的比例。

计算方式：
  一致性率 = (1 - 不一致记录数 / 总记录数) × 100%

示例：
  对账扫描了 100 万条订单数据，发现 100 条 Redis 和 MySQL 不一致
  一致性率 = (1 - 100/1000000) × 100% = 99.99%

分级：
  - 核心数据（订单、支付）：一致性率 > 99.999%（5 个 9）
  - 重要数据（库存、积分）：一致性率 > 99.99%（4 个 9）
  - 普通数据（缓存、统计）：一致性率 > 99.9%（3 个 9）
```

### 2.2 不一致修复 MTTR（Mean Time To Repair）

```
定义：从不一致发现到修复完成的平均时间。

计算方式：
  MTTR = Σ(修复完成时间 - 发现时间) / 不一致事件数

示例：
  事件1：T0 发现，T0+30s 修复 → 30s
  事件2：T0 发现，T0+5min 修复 → 5min
  事件3：T0 发现，T0+2min 修复 → 2min
  MTTR = (30s + 5min + 2min) / 3 ≈ 2.5min

分级：
  - 核心数据：MTTR < 1 分钟
  - 重要数据：MTTR < 5 分钟
  - 普通数据：MTTR < 30 分钟
```

### 2.3 同步延迟（Replication Lag）

```
定义：主库数据变更到从库/缓存/ES 同步完成的延迟。

计算方式：
  同步延迟 = 数据在目标端可见时间 - 数据在主库提交时间

监控项：
  - MySQL 主从延迟：Seconds_Behind_Master
  - Canal → Kafka 延迟：Binlog 解析时间 - Binlog 产生时间
  - Kafka → ES 延迟：ES 写入时间 - Kafka 生产时间
  - 端到端延迟：用户读缓存时数据已更新 - 用户写 MySQL 时间

告警阈值：
  - 警告：延迟 > 1 秒
  - 严重：延迟 > 5 秒
  - 紧急：延迟 > 30 秒
```

### 2.4 不一致趋势（Inconsistency Trend）

```
定义：不一致数量/率随时间的变化趋势。

用途：
  - 发现一致性退化：昨天的不一致率是 99.99%，今天是 99.9% → 查原因
  - 评估发布影响：发布新版本后不一致率上升 → 回滚或修复
  - 容量规划：数据量增长后同步延迟增加 → 扩容同步链路

监控方式：
  - 每小时/每天统计不一致数量
  - 绘制不一致率趋势图
  - 与发布、促销、流量峰值关联分析
```

## 3. 对账监控体系

### 3.1 对账频率设计

```
按数据重要性分级对账：

| 数据类型 | 对账频率 | 抽样比例 | 说明 |
|----------|----------|----------|------|
| 订单/支付 | 实时 + 每 5 分钟全量 | 100% | 资损风险最高 |
| 库存 | 每 10 分钟 | 100% | 超卖/少卖影响大 |
| 积分/优惠券 | 每小时 | 100% | 用户感知强 |
| 缓存数据 | 每 30 分钟 | 10% 抽样 | 最终一致，可容忍 |
| 统计/报表 | 每天 | 1% 抽样 | 非核心数据 |

对账方式：
  1. 实时对账：关键操作（如支付成功）立即触发对账
  2. 定时对账：定时任务扫描全量或增量数据
  3. 抽样对账：大数据量时按哈希抽样（如 user_id % 100 = 0）
```

### 3.2 对账结果处理

```
对账结果分类：

  一致：无需处理
  不一致：
    ├── 可自动修复：
    │     - 缓存旧数据 → 自动删除缓存
    │     - 从库延迟 → 等待或强制同步
    │     - 计数偏差 → 以主库为准重新计算
    │
    └── 需人工介入：
          - 金额差异 → 人工核实（可能涉及资损）
          - 状态冲突 → 人工判断正确状态
          - 数据丢失 → 人工补单或数据修复

自动修复流程：
  1. 对账发现不一致
  2. 判断是否可以自动修复（规则引擎）
  3. 自动修复 → 记录修复日志
  4. 无法自动修复 → 生成工单 → 人工处理
  5. 修复后再次对账 → 确认一致
```

### 3.3 对账表设计

```sql
CREATE TABLE reconciliation_log (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  task_id VARCHAR(64),           -- 对账任务ID
  data_type VARCHAR(32),         -- 数据类型：ORDER/STOCK/COUPON
  primary_key VARCHAR(64),     -- 业务主键：订单号/用户ID
  source_value TEXT,             -- 源端数据（如MySQL）
  target_value TEXT,             -- 目标端数据（如Redis/ES）
  diff_fields TEXT,              -- 不一致字段（JSON）
  status VARCHAR(16),            -- PENDING / AUTO_FIXED / MANUAL / IGNORED
  fix_method VARCHAR(64),        -- 修复方式
  fix_time DATETIME,             -- 修复时间
  fix_operator VARCHAR(64),      -- 修复人（自动修复为system）
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX idx_task_type (task_id, data_type),
  INDEX idx_status (status),
  INDEX idx_create_time (create_time)
);
```

## 4. 延迟 SLA 设计

```
SLA（Service Level Agreement）：一致性延迟的服务等级协议

核心数据延迟 SLA：

  ┌─────────────────────────────────────────┐
  │  订单创建 → 缓存可见      < 500ms  │ 99.9%
  │  订单创建 → 从库可读      < 1s    │ 99.9%
  │  订单创建 → ES 可搜索     < 5s    │ 99.5%
  │  库存扣减 → 缓存更新      < 200ms │ 99.99%
  │  支付成功 → 状态同步      < 1s    │ 99.99%
  │  退款完成 → 库存回补      < 30s   │ 99.9%
  └─────────────────────────────────────────┘

SLA 未达成时的处理：
  1. 告警：SLA 违约触发 P1 告警
  2. 降级：超过 SLA 时，读请求强制走主库
  3. 补偿：定时对账修复不一致数据
  4. 复盘：SLA 违约记录用于故障复盘
```

## 5. 监控大盘与告警

### 5.1 一致性监控大盘

```
第一层：全局视图
  - 今日一致性率（核心业务）
  - 不一致数量趋势（近 7 天）
  - 同步延迟 P99（各链路）
  - 修复 MTTR（近 7 天）

第二层：按链路拆分
  - MySQL → Redis 一致性率
  - MySQL → ES 一致性率
  - MySQL → 从库 一致性率
  - 主库 → MQ → 消费端 一致性率

第三层：按数据类型拆分
  - 订单数据一致性率
  - 库存数据一致性率
  - 用户数据一致性率
  - 优惠券数据一致性率

第四层：异常明细
  - 最新不一致记录（Top 100）
  - 未修复不一致列表
  - 自动修复成功率
  - 人工处理工单列表
```

### 5.2 告警规则

```
告警分级：

P0（电话）：
  - 核心数据一致性率 < 99.9%
  - 同步延迟 > 30 秒（核心链路）
  - 发现资损级别不一致（金额差异）

P1（IM+短信）：
  - 核心数据一致性率 < 99.99%
  - 同步延迟 > 5 秒
  - 未修复不一致数量 > 1000

P2（IM）：
  - 普通数据一致性率 < 99.9%
  - 同步延迟 > 1 秒
  - 修复 MTTR > 10 分钟

P3（邮件）：
  - 不一致率趋势上升（周环比 +50%）
  - 对账任务执行失败
```

## 6. 一致性度量工具链

```
工具链组合：

  数据采集：
    - MySQL：SHOW SLAVE STATUS（Seconds_Behind_Master）
    - Canal：解析延迟监控（JMX 指标）
    - Kafka：Consumer Lag（kafka-consumer-groups.sh）
    - Redis：INFO replication（master_sync_last_io_seconds_ago）

  数据存储：
    - Prometheus：时序数据存储（延迟、一致性率）
    - MySQL：对账明细、修复记录

  可视化：
    - Grafana：一致性监控大盘
    - 自建 Web：对账明细查询、工单处理

  告警：
    - Alertmanager：基于 Prometheus 指标告警
    - 企业微信/钉钉：通知渠道

示例：PromQL 查询
  # 主从延迟
  mysql_slave_lag_seconds{instance="mysql-master"}

  # 不一致率
  (1 - reconciliation_inconsistent_count / reconciliation_total_count) * 100

  # 同步延迟 P99
  histogram_quantile(0.99, rate(sync_delay_bucket[5m]))
```

## 7. 面试追问应对

**Q1：你们怎么度量最终一致性？**
> 我们把"最终一致"量化为三个指标：一致性率、不一致修复 MTTR、同步延迟 P99。比如订单数据的一致性率要求 99.999%，同步延迟 P99 要小于 500ms。通过定时对账系统每小时扫描核心数据，发现不一致就记录到对账表，能自动修复的（如缓存旧数据）自动修复，不能自动修复的（如金额差异）生成工单人工处理。所有指标都接入 Prometheus + Grafana，延迟超标或不一致率下降时自动告警。

**Q2：如果主从延迟突然增大，怎么处理？**
> 首先看监控定位原因：是主库写入压力突增、从库 IO 瓶颈、还是网络抖动？然后：
> 1. 紧急：读请求强制走主库（避免用户读到旧数据）；
> 2. 短期：扩容从库 IO、调整 Binlog 格式（ROW→MIXED 降低网络开销）、优化大事务；
> 3. 长期：增加从库数量、引入 Canal+Kafka 异步解耦、或者考虑读流量切到缓存。同时告警通知 DBA 和开发负责人。

**Q3：对账发现不一致，但不知道以哪边为准？**
> 这是最难处理的情况。我们的规则是：
> 1. 有明确"主数据源"的，以主数据源为准（如 MySQL 是主，Redis 是缓存，以 MySQL 为准）；
> 2. 两边都是业务系统，没有明确主从的 → 人工介入，结合业务日志、操作记录、第三方凭证判断；
> 3. 涉及金额差异的 → 必须人工核实，不能自动修复。修复后要记录到对账表，并复盘为什么会不一致，避免再次发生。
