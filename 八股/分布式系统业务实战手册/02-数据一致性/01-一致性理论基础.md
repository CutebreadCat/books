## 1. 一致性问题全景

### 1.1 什么是一致性问题

在分布式系统中，同一份数据可能存在于多个位置（多个数据库、缓存、搜索引擎、消息队列等），当数据发生变化时，需要保证所有副本的数据保持一致，这就是一致性问题。

### 1.2 一致性问题的分类

```
按数据存储位置：
├── 缓存一致性       (Redis vs MySQL)
├── 主从一致性       (MySQL Master vs Slave)
├── 跨服务一致性     (订单服务 vs 库存服务 vs 支付服务)
├── 多存储一致性     (MySQL vs Elasticsearch vs MongoDB)
└── 消息一致性       (业务操作 vs 消息发送)

按一致性强度：
├── 强一致性         (写完立刻读到最新值)
├── 最终一致性       (写完后经过一段时间最终一致)
├── 因果一致性       (有因果关系的操作保证顺序)
├── 读己之写一致性   (自己写的自己能立刻读到)
└── 会话一致性       (同一会话内一致)
```

### 1.3 一致性问题的根源

1. **网络不可靠**：节点间通信可能延迟、丢失、乱序
2. **节点故障**：节点可能宕机、重启
3. **并发操作**：多个操作同时修改同一数据
4. **分布式事务**：跨节点事务无法用本地事务保证
5. **异步复制**：数据同步有延迟

## 2. 一致性模型光谱（进阶）

> 这是理解一致性的**理论框架**。工程上常说"强一致 vs 最终一致"太粗，实际一致性是一个光谱，不同业务需要不同强度。

### 2.1 光谱全景

```
强 ←─────────────────────────────────────→ 弱

线性一致性 → 顺序一致性 → 因果一致性 → 读己之写 → 单调读 → 最终一致
Linearizability  Sequential    Causal      Read-Your-Writes  Monotonic Read  Eventual
```

### 2.2 线性一致性（Linearizability）

**定义**：所有操作有全局顺序，看起来像单机原子执行。操作的真实时间顺序被尊重。

```
特点：
  - 最严格的一致性
  - 操作有全局总序
  - 看起来像单机原子执行
  - 满足"实时性"（real-time order）

实现：
  - etcd / Consul（Raft）
  - ZooKeeper（ZAB）
  - Spanner（TrueTime）

代价：
  - 性能低（需多数节点确认）
  - 可用性受影响（分区时不可用）
```

### 2.3 顺序一致性（Sequential Consistency）

**定义**：所有节点看到操作的全局顺序一致，但顺序不需要按真实时间。

```
与线性一致性的区别：
  - 线性一致：全局顺序 + 按真实时间
  - 顺序一致：全局顺序，但可重排（不按真实时间）

特点：
  - 比 linearizability 弱
  - 不要求实时性
  - ZK 满足顺序一致性

示例：
  真实时间：A写1 → B写2 → C读
  顺序一致允许：C读到2或1（只要所有节点看到相同顺序）
  线性一致要求：C必须读到2（按真实时间顺序）
```

### 2.4 因果一致性（Causal Consistency）

**定义**：有因果关系的操作保证顺序一致，无因果关系的可乱序。

```
因果关系：
  - 写 A 后读 A 再写 B → B 因果依赖 A
  - 必须先看到 A 再看到 B

无因果关系：
  - 两个独立用户的写
  - 可乱序

实现：
  - 向量时钟检测因果
  - 社交系统常用（如 Feed 流）

应用：
  - 评论回复（先看到评论才能看到回复）
  - 消息系统（先发消息才能回复）
```

### 2.5 读己之写一致性（Read-Your-Writes）

**定义**：自己写的自己能立刻读到。

```
问题场景：
  用户改了头像 → 刷新 → 看到旧头像 → 再刷新 → 新头像
  → 体验差

实现方案：
  1. 写后读强制读主（写哪个节点，读哪个节点）
  2. 写后设标记（Redis 标记 force_master TTL=Ns）
  3. 会话粘性（同一会话路由到同一节点）

应用：
  - 社交（改资料后立即看到）
  - 电商（下单后立即看到订单）
```

### 2.6 单调读一致性（Monotonic Read）

**定义**：一旦读到新值，不会再读到旧值。

```
问题场景（数据倒退幻觉）：
  用户刷新 → 看到余额 100（新值）
  再刷新 → 看到余额 80（旧值，从延迟的从库读）
  → 用户困惑

实现方案：
  1. 记录用户最后读的时间戳/版本
  2. 后续读必须 ≥ 该版本
  3. 不满足则等待或读主

应用：
  - 余额查询
  - 消息列表（不能消息越来越少）
```

### 2.7 最终一致性（Eventual Consistency）

**定义**：停止写入后，经过一段时间，所有副本最终一致。

```
特点：
  - 最弱保证
  - 中间状态任意
  - 只保证"最终"

问题：
  - "最终"是多久？无约束
  - 需要配合延迟 SLA（见 06）

应用：
  - 缓存（Redis vs MySQL）
  - ES 同步
  - 社交流
```

### 2.8 业务选型对照

| 业务 | 推荐模型 | 说明 |
|------|----------|------|
| 银行转账 | 线性一致 | 钱不能错 |
| 配置中心 | 线性一致 | 配置必须全局一致 |
| 社交 Feed | 读己之写 + 单调读 | 用户体验 |
| 评论回复 | 因果一致 | 回复必须在评论后 |
| 搜索 | 最终一致 | 短暂不一致可接受 |
| 缓存 | 最终一致 + 读己之写 | 性能优先 |

### 2.9 一致性模型的可组合性

```
重要：一致性模型可组合
  - 读己之写 + 单调读 + 最终一致 = 社交场景常用组合
  - 不需要上线性一致性即可满足体验

选型思路：
  - 不是越强越好
  - 用最弱的一致性满足业务需求
  - 组合多个弱一致性模型往往比单一强一致性更优
```

## 3. CAP 与 BASE 理论

### 3.1 CAP 理论

| 要素 | 含义 |
|------|------|
| C（一致性） | 所有节点看到相同数据 |
| A（可用性） | 每个请求都得到响应 |
| P（分区容错） | 网络分区时系统仍运行 |

**核心结论**：分布式系统中 P 必须保证，因此只能在 C 和 A 之间选择。

**CAP 的现实选择**：

```
CP 系统：优先一致性
  - ZooKeeper / etcd / HBase / MongoDB

AP 系统：优先可用性
  - Cassandra / DynamoDB / Eureka / Redis Cluster
```

### 3.2 BASE 理论

| 要素 | 含义 |
|------|------|
| BA（基本可用） | 允许响应时间增加或功能降级 |
| S（软状态） | 允许中间状态（数据不一致） |
| E（最终一致） | 经过一段时间后数据一致 |

### 3.3 PACELC 理论（CAP 的扩展）

```
CAP 只考虑分区时，PACELC 还考虑无分区时：

  P（分区）时：在 A 和 C 之间选
  E（无分区）时：在 L（延迟）和 C（一致性）之间选

  例：
    Cassandra：PA + EL（分区选A，平时选L低延迟）
    Spanner：PC + EC（分区选C，平时也选C）
    DynamoDB：PA + EL

→ 提醒：平时（无分区）也要在延迟和一致性间取舍
```

## 4. 一致性 vs 共识 vs 共识算法

容易混淆的概念，明确区分：

```
一致性（Consistency）：
  - 数据的多副本表现
  - 是"结果"（数据是否一致）
  - 是属性，不是算法

共识（Consensus）：
  - 多节点就某个值达成一致
  - 是"过程"（如何达成一致）
  - 是一致性的一种实现手段

共识算法（Consensus Algorithm）：
  - 实现共识的具体算法
  - Paxos / Raft / ZAB
  - 解决"如何在不可靠节点间达成共识"

关系：
  共识算法 → 达成共识 → 保证线性一致性
  但一致性不一定需要共识（如最终一致不需要）
```
