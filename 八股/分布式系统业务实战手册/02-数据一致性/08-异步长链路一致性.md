# 异步高并发长链路一致性（关键进阶）

> 本篇解决一个高频且棘手的实战问题：在异步高并发的长调用链路中，如何既做到"提前返回"（快速响应用户），又保证"数据不丢、最终一致"。
>
> 这是秒杀、抢购、大促等场景的核心挑战，也是面试的高阶考点。

## 1. 问题背景

```
典型长链路（下单）：
  用户下单
    → 创建订单
    → 扣减库存
    → 核销优惠券
    → 增加积分
    → 发送通知
    → 风控检查
    → 数据统计

链路长度：6-8 个服务串联
单服务 RT：50-100ms
同步总 RT：300-800ms

高并发场景（秒杀 10万 QPS）：
  - 同步链路 → 服务串联等待 → 响应慢 → 用户流失
  - 任一服务慢/挂 → 整个链路雪崩
  - DB/MQ 扛不住瞬时压力
```

**核心矛盾**：长链路本身慢，高并发要求快；要快就得异步化（提前返回）；异步化后，如何保证数据不丢、最终一致？

---

## 2. 同步长链路的困境

```
1. 响应时间长
   总 RT = Σ 各服务 RT，用户等待 800ms 体验差

2. 雪崩风险
   任一服务故障 → 整个请求失败，下游被上游拖垮

3. 资源占用
   请求期间线程、连接被占用，高并发下线程池打满

4. 吞吐量受限
   同步阻塞 → 单机 QPS 低，需要更多机器
```

## 3. 提前返回的异步化设计

**核心思路**：主流程只做最小必要操作，其余异步化，快速响应用户。

```
同步流程（慢）：
  下单 → 创建订单 → 扣库存 → 核销券 → 加积分 → 通知 → 返回（800ms）

异步流程（快）：
  下单 → 创建订单 + 发 MQ → 返回（< 50ms）
  ↓ 消费者异步处理：
  → 扣库存
  → 核销券
  → 加积分
  → 通知

用户感知：下单成功，后续异步完成
```

**关键问题**：提前返回后，异步操作失败了怎么办？数据丢了怎么办？

## 4. 保证不丢数据的四道防线

这是本篇的核心。提前返回 ≠ 数据可以丢，必须有可靠的保障机制。

### 4.1 第一道防线：本地消息表

```
下单服务：
  BEGIN;
  INSERT orders(...);              -- 业务操作
  INSERT message_table(            -- 消息记录
    msg_id, content, status='NEW'
  );
  COMMIT;

→ 业务和消息在同一事务，要么都成功要么都失败
→ 不会出现"业务成功但消息丢失"
```

### 4.2 第二道防线：可靠消息投递

```
后台任务扫描 message_table：
  → status='NEW' 的消息 → 投递到 MQ
  → 投递成功 → status='SENT'
  → 投递失败 → 重试（指数退避）
  → 多次失败 → 告警 + 人工介入

→ MQ 持久化 + 生产确认，保证消息不丢
```

### 4.3 第三道防线：幂等消费

```
消费者收到消息：
  → 检查是否已处理（去重表/状态机/唯一键）
  → 已处理 → ACK（幂等）
  → 未处理 → 执行业务 → ACK
  → 处理失败 → 不 ACK，MQ 重投

→ 即使消息重复投递，也不会重复处理
```

### 4.4 第四道防线：对账兜底

```
定时任务（如每 10 分钟）：
  → 扫描 message_table 中 status='SENT' 但超时未确认的消息
  → 重新投递或人工核查
  → 跨系统对账（订单 vs 库存 vs 积分）

→ 即使前三道都漏了，对账能兜底
```

## 5. 长链路一致性保障

长链路（多跳）的一致性，关键是"每一跳都可靠"。

```
链路：订单服务 →(MQ)→ 库存服务 →(MQ)→ 积分服务 →(MQ)→ 通知服务

每一跳的保障：
  1. 发送方：本地消息表保证业务+消息原子
  2. MQ：持久化 + 确认机制
  3. 接收方：幂等消费
  4. 每一跳都有重试 + 死信队列

链路级保障：
  - 分布式 TraceID 贯穿全链路
  - 每一跳记录处理状态
  - 对账系统核对链路两端数据
  - 异常告警（某跳卡住超时）

示例：
  订单创建成功 → 库存扣减成功 → 积分增加失败
  → 积分服务重试
  → 仍失败 → 死信队列 → 人工处理
  → 对账发现"订单已支付但积分未加" → 补偿
```

## 6. 实战案例：秒杀下单链路

```
场景：10 万人抢 100 件商品

第一层：Redis 预扣减（瞬时流量拦截）
  Lua 原子操作：DECR stock + 记录用户
  → 库存耗尽（100个成功）→ 其余快速失败
  → 10万请求 50ms 内处理完

第二层：MQ 异步下单（成功请求排队）
  Redis 预扣成功的 100 个请求 → 发 MQ
  → 用户立即返回"排队中"（提前返回）
  → 订单服务按能力消费（如 1000 QPS）

第三层：订单服务本地事务
  消费 MQ：
    BEGIN;
    INSERT orders(...);
    INSERT message_table(msg='扣实际库存');
    INSERT message_table(msg='核销券');
    INSERT message_table(msg='加积分');
    COMMIT;

第四层：各消费者异步处理
  库存服务：扣实际库存（MySQL 乐观锁）
  优惠券服务：核销优惠券
  积分服务：增加积分
  通知服务：发送通知

第五层：对账兜底
  - 订单 vs 库存（已支付订单是否都扣了库存）
  - 订单 vs 积分（已支付订单是否都加了积分）
  - 异常告警 + 自动补偿

用户感知：
  下单 → "排队中" → 几秒后 → "下单成功，去支付"
```

## 7. 提前返回的边界

不是所有操作都能异步化，必须明确边界：

```
必须同步（用户依赖结果）：
  ✓ 创建订单（用户要拿订单号去支付）
  ✓ 支付（用户要确认支付成功）
  ✓ 库存预扣减（防止超卖，必须立即判断）
  ✓ 风控拦截（高危请求必须立即拦）
  ✓ 唯一性校验（防重复提交）

可以异步（后续可补偿）：
  ✓ 扣实际库存（预扣已保证不超卖）
  ✓ 核销优惠券（订单已锁定券）
  ✓ 增加积分（可后补）
  ✓ 发送通知（可延迟）
  ✓ 数据统计（可延迟）
  ✓ 搜索索引更新（可延迟）

判断原则：
  - 用户是否依赖这个结果？依赖 → 同步
  - 失败是否可补偿？可补偿 → 异步
  - 是否影响核心数据一致性？影响 → 谨慎异步
  - 是否有对账兜底？有 → 可放心异步
```

## 8. 一致性风险与应对

```
风险 1：用户收到成功，但异步处理失败
  应对：重试机制 + 对账兜底 + 人工介入入口

风险 2：用户查不到异步结果
  场景：下单返回成功，但查订单列表没有
  应对：订单创建是同步的，异步只是后续处理

风险 3：异步处理顺序错乱
  应对：不依赖顺序的并行；依赖顺序用 MQ 分区有序

风险 4：异步堆积
  应对：监控堆积量 + 自动扩容 + 堆积超限限流上游
```

## 9. 方案选型总结

| 场景 | 同步/异步 | 一致性方案 |
|------|-----------|------------|
| 创建订单 | 同步 | 本地事务 |
| 扣库存 | 同步预扣 + 异步实扣 | Redis 预扣 + MQ + 乐观锁 |
| 支付 | 同步 | 第三方回调 + 状态机 |
| 加积分 | 异步 | 本地消息表 + MQ |
| 发通知 | 异步 | MQ（可容忍丢失） |
| 数据统计 | 异步 | MQ / Binlog 同步 |

### 9.1 面试要点

**Q：如何在高并发下既快速响应用户，又保证数据不丢？**

回答框架：
1. 同步做最小必要操作（创建订单 + Redis 预扣）
2. 提前返回，异步处理其余
3. 四道防线保证不丢：本地消息表 + 可靠投递 + 幂等消费 + 对账兜底
4. 明确同步/异步边界
5. 对账兜底是最后防线
