# 消息可靠性保障

> MQ 是分布式系统的"神经系统"，消息一旦丢失、重复或积压，轻则数据不一致，重则资损。本篇从生产端、Broker、消费端三端出发，给出消息可靠性的工程级保障方案。

## 1. 问题背景：三要素

```
消息可靠性三要素：

  不丢（At-Least-Once）  →  消息至少被消费一次
  不重（Exactly-Once）   →  消息恰好被消费一次（依赖幂等）
  不积压（Low Latency）  →  消息从生产到消费的延迟可控

现实：
  大多数业务要求 At-Least-Once + 幂等 = 效果上的 Exactly-Once
  金融场景可能要求真正的 Exactly-Once（事务消息）
```

**面试陷阱**：
- 面试官问"如何保证消息不丢失"——不能只回答"开启持久化"，要从三端（生产、Broker、消费）分别回答。
- 面试官问"消息重复了怎么办"——不能只回答"幂等"，要回答"怎么幂等、幂等键怎么设计、幂等表怎么存"。

## 2. 消息不丢：三端保障

### 2.1 生产端：确保消息到达 Broker

```
风险：
  1. 消息发送后，网络抖动导致 Broker 没收到
  2. Producer 异步发送时，消息还在缓冲区就崩溃
  3. Producer 发送成功但本地事务回滚了（业务数据没写，消息已发）

方案：

  方案 A：同步发送 + 重试（最稳妥）
    result = producer.send(message, timeout=3s)
    if result.is_success:
        // 成功，继续
    else:
        // 失败，重试 3 次（指数退避）
        retry_with_backoff()

  方案 B：异步发送 + 回调确认
    producer.send(message, callback={
        on_success: () => { /* 确认成功 */ }
        on_fail: () => { /* 记录失败日志，人工介入 */ }
    })

  方案 C：本地消息表（最终一致性，最可靠）
    BEGIN;
      INSERT business_data(...);           -- 业务写库
      INSERT message_table(msg, status='NEW');  -- 消息记录
    COMMIT;

    后台任务扫描 message_table：
      status='NEW' → 发送 MQ → 成功 → status='SENT'
      status='SENT' → 等待 ACK → 收到 ACK → status='ACKED'
```

**关键**：
- 同步发送适合关键业务（订单、支付）
- 异步发送适合高吞吐场景（日志、埋点）
- 本地消息表解决"业务数据写成功了，但消息发不出去"的问题

---

### 2.2 Broker 端：确保消息持久化

```
风险：
  1. Broker 收到消息后还没写入磁盘就宕机
  2. Broker 主节点宕机，从节点还没同步完

方案（Kafka 为例）：

  1. 消息持久化（必须开启）
     log.flush.interval.messages = 10000  -- 每 1 万条刷盘
     log.flush.interval.ms = 1000         -- 每 1 秒刷盘

  2. 副本机制（ACK = all）
     producer.acks = all  -- 等待所有 ISR 副本确认
     min.insync.replicas = 2  -- 至少 2 个副本同步才算成功

  3. 禁用自动提交 Offset
     consumer.enable.auto.commit = false
     手动提交 Offset：
       - 先处理业务 → 再提交 Offset
       - 避免"Offset 提交了但业务没处理完"的丢消息

方案（RocketMQ 为例）：

  1. 同步刷盘（SYNC_FLUSH）
     flushDiskType = SYNC_FLUSH  -- 每条消息同步刷盘
     （性能低，但可靠性高；异步刷盘 ASYNC_FLUSH 性能高，但可能丢消息）

  2. 主从同步（SYNC_MASTER）
     brokerRole = SYNC_MASTER  -- 主从同步复制
     （ASYNC_MASTER 主从异步，性能高但可能丢消息）

  3. 事务消息（两阶段提交）
     半消息 → 本地事务执行 → 确认提交/回滚
     解决"业务写成功了但消息没发出去"和"消息发出去了但业务回滚了"的问题
```

**关键 trade-off**：
- SYNC_FLUSH + SYNC_MASTER：可靠性最高，性能最低（适合订单、支付）
- ASYNC_FLUSH + ASYNC_MASTER：性能最高，可能丢消息（适合日志、埋点）

### 2.3 消费端：确保消息被处理

```
风险：
  1. 消息拉取后，消费者还没处理完就宕机
  2. 消费者抛异常，但 Offset 已经自动提交了
  3. 消费者处理成功但提交 Offset 失败

方案：

  1. 先处理业务，再提交 Offset（手动提交）
     流程：
       poll() → 获取消息
       执行业务逻辑（如扣减库存）
       业务成功 → commitSync() 提交 Offset
       业务失败 → 不提交 Offset，消息会被重新消费

  2. 业务处理 + 幂等
     即使消息重复消费，也不会导致数据错误

  3. 异常处理策略
     try:
       process(message)
       commitSync()
     except BusinessException:
       // 业务异常（如库存不足）→ 记录失败，提交 Offset（避免无限重试）
       log_fail(message)
       commitSync()
     except SystemException:
       // 系统异常（如 DB 连接断开）→ 不提交 Offset，让 MQ 重试
       raise

  4. 消费超时控制
     消费者设置 max.poll.interval.ms（Kafka）
     如果处理时间超过阈值，协调器认为消费者已死，触发重平衡
     → 可能导致消息被重复消费 → 必须幂等
```

## 3. 消息不重：幂等设计

```
为什么 MQ 一定会重复？
  1. Producer 重试：第一次发送网络超时，实际 Broker 已收到，重试又发一条
  2. Consumer 重试：业务处理成功但提交 Offset 失败，MQ 重投
  3. 分区重平衡：消费者组重平衡时，消息被重新分配

结论：MQ 的重复消费是**必然事件**，不是偶然事件。必须设计幂等。
```

### 3.1 幂等键设计

```
核心原则：每条业务操作必须有唯一标识

  方案 1：业务唯一键（推荐）
    操作类型 + 业务实体 ID
    示例：
      订单创建：order_id 唯一
      库存扣减：order_id + sku_id 唯一
      优惠券核销：order_id + coupon_id 唯一

  方案 2：消息唯一 ID（兜底）
    Producer 生成 message_id（如 UUID 或 Snowflake）
    消费者用 message_id 去重

  方案 3：组合键（最可靠）
    业务唯一键 + 消息 ID
    先按业务键去重（保证业务正确），再按消息 ID 去重（防止极端 case）
```

### 3.2 幂等实现方式

```
方式 1：数据库唯一索引（最常用）
  CREATE TABLE order_item (
    order_id BIGINT,
    sku_id BIGINT,
    ...
    UNIQUE KEY uk_order_sku (order_id, sku_id)
  );

  消费消息：
    INSERT INTO order_item(...) VALUES(...)
    → 若 DuplicateKeyException → 幂等，忽略

方式 2：幂等表（显式控制）
  CREATE TABLE idempotent_log (
    idempotent_key VARCHAR(64) PRIMARY KEY,  -- 业务唯一键或消息ID
    status VARCHAR(16),                     -- SUCCESS / FAIL
    create_time DATETIME
  );

  消费消息：
    1. SELECT * FROM idempotent_log WHERE idempotent_key = ?
    2. 若存在且 status='SUCCESS' → 直接返回（幂等）
    3. 若不存在 → 执行业务 → INSERT idempotent_log(key, 'SUCCESS')
    4. 若业务失败 → INSERT idempotent_log(key, 'FAIL')

  优点：可追踪、可审计、可重试失败项
  缺点：多一次查询，性能略低

方式 3：状态机（天然幂等）
  订单状态：CREATED → PAID → SHIPPED → COMPLETED
  同一消息多次消费：
    若当前状态 >= 目标状态 → 忽略（幂等）
    若当前状态 < 目标状态 → 推进状态机

  示例：
    消息："订单 123 支付成功"
    当前状态：PAID
    再次收到消息：状态已经是 PAID → 忽略

  优点：不需要额外的幂等表
  缺点：只适用于有状态机的业务场景

方式 4：Redis SETNX（轻量级）
  SET idempotent:msg_id "1" EX 86400 NX
  → 设置成功 → 首次消费，执行业务
  → 设置失败 → 已消费过，直接返回

  适用：不需要长期审计、允许 TTL 后重复的场景
  缺点：Redis 宕机可能丢失幂等记录；TTL 过期后可能重复消费
```

### 3.3 幂等失效的场景

```
坑 1：幂等键设计不当
  错误：用 "user_id" 作为发券幂等键
  问题：同一用户多次发券被拦截
  正确："user_id + coupon_id + batch_id"

坑 2：幂等表和业务表不在同一个事务
  错误：先写幂等表 → 再执行业务（两个事务）
  问题：幂等表写成功但业务失败，下次认为已处理，跳过
  正确：幂等检查 + 业务操作 + 幂等记录写入，在同一个本地事务

坑 3：幂等记录不清理
  问题：幂等表无限增长，查询越来越慢
  解决：定期归档（如保留 30 天，冷数据迁移）
```

## 4. 消息不积压：监控与治理

```
消息积压的危害：
  1. 延迟：订单超时取消的消息积压在队列里，用户迟迟收不到通知
  2. 内存：Broker 消息堆积过多，内存/磁盘耗尽
  3. 过期：消息超过 TTL 被丢弃，数据丢失
  4. 雪崩：消费者追不上生产者，问题越来越大
```

### 4.1 积压监控

```
核心指标：

  1. 消息堆积量（Lag）
     Lag = 生产者 Offset - 消费者 Offset
     告警阈值：
       - 业务队列：Lag > 1000 告警
       - 延迟敏感队列：Lag > 100 告警
       - 死信队列：Lag > 0 立即告警

  2. 消息消费延迟
     消息从生产到消费的时间差
     告警阈值：
       - 普通业务：> 5 分钟
       - 延迟敏感（如订单超时）：> 30 秒

  3. 消费速率 vs 生产速率
     生产速率 > 消费速率 → 必然积压
     需要提前扩容或限流
```

### 4.2 积压治理

```
发现积压后的处理步骤：

  步骤 1：定位原因
    1. 消费者宕机了？→ 重启消费者
    2. 消费者抛异常了？→ 查日志修复 Bug
    3. 消费太慢了？→ 优化消费逻辑或扩容
    4. 生产者突发流量？→ 上游限流

  步骤 2：紧急处理

    方案 A：扩容消费者（最快）
      - 增加消费者实例
      - 注意：Kafka 一个分区只能被一个消费者消费
        → 扩容消费者前先看分区数是否足够
        → 分区数 < 消费者数时，多出来的消费者空闲

    方案 B：跳过非关键消息（兜底）
      - 场景：日志队列积压，可以丢弃部分旧日志
      - 操作：调整消费者 Offset 到最新位置（会丢消息，谨慎）

    方案 C：降低消费粒度（优化）
      - 原来是单条处理 → 改为批量消费（如每次消费 100 条）
      - 原来是同步写 DB → 改为批量写或异步写

    方案 D：临时队列分流
      - 原队列：topic_order（积压）
      - 新队列：topic_order_urgent（紧急订单）
      - 生产者将新订单发到紧急队列，优先处理
      - 旧队列慢慢消费

  步骤 3：长期优化
    - 消费逻辑异步化（如消费者只写本地队列，后台线程处理）
    - 数据库批量写入（如每 100 条合并一次 INSERT）
    - 缓存预热（减少消费时的 DB 查询）
    - 分区数预先评估（根据峰值流量设计分区数）
```

## 5. 消息顺序：乱序与保序

```
为什么消息会乱序？
  1. Producer 重试：第一条消息超时重试，第二条消息先到 Broker
  2. 多分区：消息分散到多个分区，各分区消费速度不同
  3. 多消费者：同一分区的消息被不同消费者并发处理

大部分业务不需要全局有序，只需要"业务有序"
```

### 5.1 分区有序（Kafka/RocketMQ）

```
原理：
  同一业务标识的消息发到同一个分区
  一个分区内的消息天然有序（FIFO）

实现：
  Producer 发送时按 key 分区：
    partition = hash(order_id) % partition_count

  示例：
    订单 123 的所有消息（创建、支付、发货）→ 同一个分区
    订单 456 的所有消息 → 另一个分区

  注意：
    消费者必须单线程消费该分区，否则多线程处理仍会乱序
    Kafka 中一个分区只能被一个消费者消费，天然保证单线程

  限制：
    热点 Key 会导致单个分区成为热点（如所有订单都按用户ID，某用户是超级大V）
    → 需要设计好分区 key，避免热点
```

### 5.2 业务层排序（兜底）

```
场景：无法保证 MQ 有序，或消息乱序已经发生

方案：
  消息体中携带业务时间戳或版本号
  消费者收到消息后，先放入内存排序队列
  按业务时间戳/版本号排序后再处理

示例：
  消息1：{ order_id: 123, event: 'PAID', version: 2 }
  消息2：{ order_id: 123, event: 'CREATED', version: 1 }
  消息3：{ order_id: 123, event: 'SHIPPED', version: 3 }

  消费者收到后按 version 排序：
    1. CREATED (v1) → 处理
    2. PAID (v2) → 处理
    3. SHIPPED (v3) → 处理

  如果先收到 v2，发现 v1 还没到 → 暂存，等待 v1
  如果等待超时 → 直接处理 v2（假设 v1 丢失了）

  适用：状态机推进类场景（订单状态、库存状态）
```

## 6. 死信队列：最终兜底

```
什么是死信队列（DLQ）？
  消息多次消费失败（超过最大重试次数）后，被移入死信队列
  死信队列中的消息不会自动消费，需要人工介入或特殊处理

为什么需要死信队列？
  防止无限重试：某条消息一直消费失败，阻塞整个队列
  隔离问题消息：把坏消息隔离出来，不影响正常消息消费
  人工处理：运营/开发可以查看死信消息，排查问题后手动处理

配置（RocketMQ 示例）：
  maxReconsumeTimes = 16  -- 最大重试 16 次
  重试间隔：1m, 2m, 3m, 4m, 5m, 6m, 7m, 8m, 9m, 10m, 20m, 30m, 1h, 2h, 3h
  16 次后进入 %DLQ%topic_name

处理策略：
  1. 告警：死信队列有消息时立即告警
  2. 查看：通过管理后台查看死信消息内容和失败原因
  3. 重试：修复问题后，手动将死信消息重新发回原队列
  4. 丢弃：确认消息无意义后，直接丢弃
  5. 补偿：如果消息代表的业务操作未完成，走补偿流程（如定时任务对账）
```

## 7. 业务选型决策

| 场景 | 可靠性要求 | 推荐方案 |
|------|------------|----------|
| 订单创建/支付 | 最高 | 事务消息（RocketMQ）+ 本地消息表 + 手动提交 Offset + 幂等 |
| 库存扣减 | 高 | 本地消息表 + 同步发送 + 幂等（业务唯一键） |
| 通知/短信 | 中 | 异步发送 + 失败重试 + 幂等（允许短暂丢失） |
| 日志/埋点 | 低 | 异步发送 + 批量发送 + 允许少量丢失 |
| 订单状态变更 | 高 | 分区有序（按 order_id）+ 状态机幂等 |
| 数据同步 | 中 | 分区有序 + 业务层排序（版本号） |

## 8. 踩坑与反模式

### 坑 1：自动提交 Offset + 先提交后处理
```
错误配置：
  enable.auto.commit = true
  auto.commit.interval.ms = 5000

问题：
  每 5 秒自动提交一次 Offset，不管业务是否处理成功
  → 业务处理失败，但 Offset 已提交，消息丢失

正确：
  enable.auto.commit = false
  业务处理成功后手动 commitSync()
```

### 坑 2：消费异常后没有区分异常类型
```
错误：
  try { process() } catch (Exception e) { log.error(e) }
  所有异常都吞掉，继续消费下一条

问题：
  系统异常（如 DB 挂了）应该重试，但代码直接吞掉，消息丢失

正确：
  区分 BusinessException（业务异常，提交 Offset）和 SystemException（系统异常，不提交 Offset）
```

### 坑 3：幂等键没有包含业务维度
```
错误：
  用消息 ID 作为唯一幂等键
  问题：不同业务的消息 ID 不同，但同一业务可能发多次消息

正确：
  业务唯一键为主（如 order_id），消息 ID 为辅（兜底）
```

### 坑 4：死信队列没有监控
```
问题：
  消息进入死信队列，系统没有任何感知
  等用户投诉时才发现数据不一致

正确：
  死信队列 Lag > 0 立即告警（P0 级别）
  死信消息必须有处理 SOP（查看→重试/补偿→关闭）
```

### 坑 5：消息 TTL 设置过短
```
问题：
  消息 TTL 设为 1 小时，消费者积压 2 小时
  → 消息过期被丢弃，数据丢失

正确：
  关键业务消息不设 TTL 或设得很长（如 7 天）
  监控积压量，防止消息过期
```

## 9. 面试追问应对

**Q1：怎么保证消息不丢失？**
> 从三端保障：
> 1. 生产端：同步发送 + 重试，或本地消息表保证业务写库和消息记录原子；
> 2. Broker 端：开启同步刷盘（RocketMQ）或 ACK=all + min.insync.replicas（Kafka），确保消息持久化到多副本；
> 3. 消费端：手动提交 Offset，先处理业务再 commit，避免自动提交导致的消息丢失。关键业务用本地消息表 + 事务消息双重保障。

**Q2：消息重复消费了怎么办？**
> 消息重复是 MQ 的必然事件，必须设计幂等。我的方案是：
> 1. 业务唯一键为主键或唯一索引（如 order_id + sku_id），重复消费时触发 DuplicateKeyException，直接忽略；
> 2. 有状态的业务用状态机幂等（如订单已经是 PAID 状态，再次收到支付消息直接忽略）；
> 3. 兜底用幂等表显式记录处理状态，避免极端 case。

**Q3：消息积压了怎么处理？**
> 先定位原因：是消费者宕机、消费太慢、还是生产突发流量？然后：
> 1. 紧急扩容消费者（注意 Kafka 分区数要够）；
> 2. 优化消费逻辑（批量消费、批量写 DB、异步化）；
> 3. 如果是非关键队列（如日志），可以临时跳过旧消息；
> 4. 长期预防：合理评估分区数、监控 Lag 和消费延迟、设置告警阈值。

**Q4：RocketMQ 事务消息的原理？**
> 两阶段提交：
> 1. Producer 发送半消息（对消费者不可见）；
> 2. Producer 执行本地事务（如写订单表）；
> 3. 本地事务成功 → 发送确认提交，半消息变为可见；
> 4. 本地事务失败 → 发送回滚，半消息被删除；
> 5. 如果 Producer 没发确认，Broker 会回查 Producer 的本地事务状态，根据状态决定提交或回滚。
> 这样就保证了"本地事务成功，消息一定发出去；本地事务失败，消息一定不发"。

**Q5：Kafka 和 RocketMQ 在可靠性上有什么区别？**
> Kafka 追求高吞吐，默认异步刷盘、异步复制，可靠性靠配置调优（ACK=all、同步刷盘）。RocketMQ 设计更偏向业务可靠性，原生支持事务消息、同步刷盘、定时消息、死信队列，更适合金融/订单类场景。但 Kafka 的吞吐和生态更强，适合日志/大数据场景。
