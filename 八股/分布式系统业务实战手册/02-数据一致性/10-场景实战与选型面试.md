## 1. 一致性场景实战

### 1.1 订单系统一致性

```
方案：本地消息表 + Saga 补偿

  订单服务：
    BEGIN;
    INSERT orders (status='PENDING');
    INSERT message_table (msg='deduct_stock', status='NEW');
    COMMIT;
    → 后台扫描消息表，发送到 MQ

  库存服务：
    消费 MQ → 扣减库存
    → 成功：ACK + 发"扣减成功"消息
    → 失败：发"扣减失败"消息

  订单服务：
    消费"扣减结果"消息
    → 成功：订单状态 → STOCK_DEDUCTED
    → 失败：订单状态 → CANCELLED + 释放优惠券
```

### 1.2 支付系统一致性

```
方案：TCC

  Try：
    支付：检查支付能力
    订单：状态 PENDING → PAYING
    账户：冻结金额
    积分：无操作

  Confirm：
    支付：确认扣款
    订单：状态 PAYING → PAID
    账户：扣减冻结
    积分：增加积分

  Cancel：
    支付：取消支付
    订单：状态 PAYING → CANCELLED
    账户：解冻
```

### 1.3 缓存与数据库一致性

```
方案：Binlog 订阅 + 延迟双删

写流程：
  1. 更新 MySQL
  2. 业务层删除 Redis（延迟双删第一次）
  3. Canal 订阅 Binlog → 发 MQ
  4. 消费者删除 Redis（兜底）

读流程：
  1. 读 Redis
  2. 未命中 → 读 MySQL → 写 Redis（设 TTL）
```

### 1.4 搜索与数据库一致性

```
方案：Binlog 同步 ES

  写：MySQL → Canal → ES
  读：精确查询 MySQL，搜索查询 ES
  保障：Binlog 顺序保证 + 消费者幂等 + 定时全量重建
```

## 2. 数据迁移期间的一致性切换（进阶）

> 数据迁移（如分库分表）期间的一致性是工程难点，本节深入。

### 2.1 双写期间的读一致性

```
问题：双写期间，读老库还是新库？

方案 1：读老库（推荐）
  - 老库数据更全
  - 新库在追赶
  - 用户无感知

方案 2：读新库（灰度）
  - 少量流量读新库验证
  - 发现问题快速切回

方案 3：双读对比
  - 读新老库对比
  - 不一致告警
  - 仅用于验证阶段
```

### 2.2 一致性切换点

```
如何判断可以切换读？

  1. 数据对账零差异（持续 N 小时）
  2. 双写稳定（无失败）
  3. 灰度读新库无异常

切换流程：
  老库读 100% → 新库读 1% → 10% → 50% → 100%
  每步观察：响应时间、错误率、业务指标
```

### 2.3 回滚时的反向同步

```
问题：切到新库后发现有问题，要回滚

风险：新库已写入数据，老库没有

方案：
  1. 迁移前建反向同步（新→老）
  2. 切换期间双写 + 反向同步
  3. 回滚时老库数据是最新的
  4. 回滚后关闭反向同步

或：
  1. 切换后只读新库，不写
  2. 观察无问题再写新库
  3. 出问题回滚到老库（数据未变）
```

### 2.4 灰度用户的"读切换"

```
问题：灰度切换时，用户可能读到"半新半旧"

场景：
  用户 A 的订单在老库
  切换后部分流量到新库
  用户 A 查订单 → 新库没有 → 404

方案：
  1. 按用户灰度（同用户要么全老要么全新）
  2. 新库查不到 → 回退查老库（兜底）
  3. 灰度用户列表明确控制
```

## 3. 一致性选型决策树

```
是否需要强一致？
├── 是
│   ├── 跨数据库 → 2PC/XA（性能差，少用）
│   ├── 跨服务 → TCC（复杂，但强一致）
│   └── 配置/元数据 → Paxos/Raft（etcd/ZK）
│
└── 否（最终一致）
    ├── 业务操作 + 消息发送
    │   ├── 同步 → 本地消息表
    │   └── 异步 → MQ 事务消息
    │
    ├── 长流程业务 → Saga
    │
    ├── 缓存一致性
    │   ├── 低并发 → 先更新 DB 再删缓存
    │   ├── 中并发 → 延迟双删
    │   ├── 高并发 → MQ 异步删缓存
    │   └── 极致 → Binlog 订阅
    │
    ├── 主从一致性
    │   ├── 关键读 → 强制读主
    │   └── 一般读 → 半同步复制 + 延迟监控
    │
    └── 异地多活
        ├── 金融 → 单元化 + 强一致
        └── 社交 → 单元化 + 最终一致
```

## 4. 一致性面试高频问题

### Q1: 如何保证缓存与数据库一致性？

**要点**：四种方案对比 → 推荐先更新 DB 再删缓存（+延迟双删）→ 进阶 Binlog 订阅 → 兜底 TTL + 对账

详见 [03-缓存一致性](./03-缓存一致性.md)

### Q2: 分布式事务有哪些方案？

**要点**：2PC（强一致慢）/ TCC（资金类）/ Saga（长流程）/ 本地消息表（最实用）/ MQ 事务消息

详见 [04-分布式事务与Seata](./04-分布式事务与Seata.md)

### Q3: 如何保证消息不丢？

**要点**：生产确认 + 持久化 + 手动 ACK + 死信队列

详见 [05-消息与主从一致性](./05-消息与主从一致性.md)

### Q4: 如何保证消息不重复？

**要点**：消费端幂等（唯一 ID / 状态机 / 乐观锁）

详见 [05-消息与主从一致性](./05-消息与主从一致性.md)

### Q5: 主从延迟导致不一致怎么办？

**要点**：强制读主 / 半同步复制 / 写后读标记 / 业务容忍

详见 [05-消息与主从一致性](./05-消息与主从一致性.md)

### Q6: TCC 的三个关键问题？

**要点**：幂等性 / 空回滚 / 悬挂控制

详见 [04-分布式事务与Seata](./04-分布式事务与Seata.md)

### Q7: 一致性模型有哪些？（进阶）

**要点**：线性一致 / 顺序一致 / 因果一致 / 读己之写 / 单调读 / 最终一致

详见 [01-一致性理论基础](./01-一致性理论基础.md)

### Q8: Raft 如何保证一致性？（进阶）

**要点**：Leader 选举 + 日志复制 + 安全性（已提交不丢）+ 脑裂处理

详见 [02-共识算法与可调一致性](./02-共识算法与可调一致性.md)

### Q9: fencing token 解决什么问题？（进阶）

**要点**：GC 暂停导致双持锁 → fencing token 单调递增 → 存储拒绝旧 token

详见 [07-强一致与前沿方案](./07-强一致与前沿方案.md)

### Q10: 异地多活如何保证一致性？（进阶）

**要点**：单元化（同用户单单元写）+ 异步同步 + 冲突处理 + 对账兜底

详见 [09-异地多活一致性](./09-异地多活一致性.md)

### Q11: CRDT 是什么？（前沿）

**要点**：无冲突复制数据类型，自动合并，协同编辑/离线优先

详见 [07-强一致与前沿方案](./07-强一致与前沿方案.md)

### Q12: 高并发下如何提前返回又不丢数据？（关键进阶）

**要点**：同步做最小操作 → 异步处理 → 四道防线（本地消息表+可靠投递+幂等消费+对账兜底）

详见 [08-异步长链路一致性](./08-异步长链路一致性.md)
