# 库存分段与流水表一致性

> 秒杀场景下，单 Key 库存成为 Redis 热点，性能瓶颈明显。"库存分段"是面试中的高频追问，但分段后又引入了新的一致性问题。本篇从分段原理到流水表设计，给出完整方案。

## 1. 问题背景：单 Key 热点

```
问题：秒杀场景下，100 万用户同时扣减同一个商品的库存

单 Key 方案：
  Redis Key: stock:1001 = 100
  100 万请求同时 DECR stock:1001 → 单 Key 热点

  问题：
    1. Redis 单节点 QPS 有上限（纯内存操作约 10万 QPS）
    2. 所有请求打到同一个 Redis 节点，网络带宽和 CPU 成为瓶颈
    3. 虽然 DECR 是原子操作，但并发量过大时 Redis 仍可能卡顿

  面试追问：
    "如果库存只有 100 件，但 QPS 10 万，怎么进一步提高并发度？"
    → 答"单 Key 原子 DECR"是不够的，面试官要的是"分段"
```

---

## 2. 库存分段原理

```
核心思路：把一个商品的库存分成多段，分散到多个 Key/多个 Redis 节点上

分段前：
  stock:1001 = 100
  → 100 万请求竞争 1 个 Key

分段后：
  stock:1001:0 = 20   (Redis 节点 A)
  stock:1001:1 = 20   (Redis 节点 B)
  stock:1001:2 = 20   (Redis 节点 C)
  stock:1001:3 = 20   (Redis 节点 D)
  stock:1001:4 = 20   (Redis 节点 E)
  → 100 万请求分散到 5 个 Key × 5 个节点

关键问题：
  1. 用户请求路由到哪个段？
  2. 某段卖光了，但总库存还有，怎么办？
  3. 怎么保证总库存不超卖？
  4. 分段后怎么对账？
```

## 3. 库存分段方案详解

### 3.1 方案一：随机分段（最简）

```
原理：
  每个请求随机选一个段扣减。

实现：
  segment = random(0, N-1)  -- N 为段数
  result = redis.decr("stock:1001:" + segment)
  if result >= 0:
      return "扣减成功"
  else:
      -- 这一段卖光了，但其他段可能还有
      -- 尝试其他段？还是直接返回失败？

问题：
  1. 某段卖光后，随机可能再次命中这个段，浪费请求
  2. 不均匀：有的段先卖光，有的段还剩很多，但用户随机选中已卖光的段
  3. 库存只剩 1 件，但 5 个段同时被扣减到 -1 → 超卖风险

适用：
  演示/面试初步回答，生产不推荐
```

### 3.2 方案二：轮询分段（推荐）

```
原理：
  每个请求按顺序轮询各段，直到找到有库存的段。

实现：
  客户端维护一个轮询索引（或基于请求序号取模）：
    segment = request_index % N

  或者服务端轮询：
    for i in range(N):
        segment = (start + i) % N
        result = redis.decr("stock:1001:" + segment)
        if result >= 0:
            return "扣减成功，段号：" + segment
    return "库存不足"

改进：
  记录每个段的剩余库存（本地缓存或 Redis Hash）：
    stock:1001:meta = {
      "0": 20, "1": 15, "2": 20, "3": 0, "4": 5
    }
  扣减前先查 meta，只访问有库存的段，减少无效请求

优点：
  - 相对均匀，不会集中打某一个段
  - 某段卖光后自动跳过

缺点：
  - 轮询需要多次 Redis 访问（如果前面几个段都卖光了）
  - 极端情况：只剩最后一段有 1 个库存，前面 N-1 次都是无效访问
  - 多段同时扣减到 -1 的超卖风险（需要 Lua 原子操作）
```

### 3.3 方案三：中心化协调（不推荐）

```
原理：
  引入一个协调者，维护各段剩余库存，统一分配。

问题：
  1. 协调者成为单点瓶颈（面试官明确说"协调者仍然是串行的"）
  2. 协调者和 Redis 之间有一次网络往返，增加延迟
  3. 协调者宕机后如何恢复？

结论：
  违背了分段的初衷（去中心化），面试中说出这个方案会被扣分
```

### 3.4 方案四：自适应分段（生产级）

```
原理：
  1. 预分配：活动开始前，把库存均匀分配到 N 个段
  2. 本地缓存：应用层缓存各段剩余库存（定期同步或监听变更）
  3. 智能路由：优先访问剩余库存多的段
  4. 兜底：Lua 原子操作保证单段不超卖

实现细节：

  步骤 1：预热
    活动开始前，管理员设置：
      总库存：100
      段数：5
    系统自动初始化：
      for i in range(5):
          redis.set("stock:1001:" + i, 20)
      redis.set("stock:1001:total", 100)  -- 总库存冗余记录

  步骤 2：扣减（Lua 原子操作）
    -- 先尝试指定段
    local segment = tonumber(ARGV[1])
    local key = KEYS[1] .. ":" .. segment
    local stock = tonumber(redis.call('GET', key) or '0')
    if stock > 0 then
        redis.call('DECR', key)
        redis.call('DECR', KEYS[1] .. ":total")
        return {1, segment}  -- 成功，返回段号
    end
    return {0, -1}  -- 该段库存不足

  步骤 3：段级轮询（如果首选段失败）
    如果第一次扣减失败，轮询其他段
    但注意：轮询本身不是原子操作，可能轮询过程中其他请求扣减了库存
    → 所以每段扣减都用 Lua 原子操作，失败就继续下一段

  步骤 4：总库存兜底
    所有段都扣减失败后，再检查总库存
    local total = tonumber(redis.call('GET', "stock:1001:total") or '0')
    if total <= 0 then
        return "库存不足"

优化点：
  1. 本地缓存段库存：应用层缓存各段剩余量（TTL 1s），减少 Redis 访问
  2. 动态调整段数：如果某商品库存从 100 增加到 10000，动态增加段数
  3. 段数与并发量匹配：段数 ≈ 预估并发量 / 单段安全 QPS（如 5000）
```

## 4. 库存流水表设计

```
为什么需要流水表？
  1. 幂等：同一订单重复扣减时，通过流水表去重
  2. 对账：Redis 预扣 vs MySQL 实扣 的核对依据
  3. 回滚：订单取消时，根据流水表精确回补哪一段的库存
  4. 审计：追踪库存变更历史，排查问题

表结构：
```sql
CREATE TABLE stock_flow (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  sku_id BIGINT NOT NULL,              -- 商品SKU
  order_id VARCHAR(64) NOT NULL,       -- 订单号
  user_id BIGINT NOT NULL,             -- 用户ID
  segment INT,                         -- 段号（分段库存时）
  action VARCHAR(16),                  -- 动作：PRE_DEDUCT / DEDUCT / RELEASE / REFUND
  quantity INT NOT NULL,               -- 数量
  pre_deduct_id BIGINT,                -- 关联预扣记录（实扣时关联）
  status VARCHAR(16),                  -- SUCCESS / FAIL / PENDING
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uk_order_sku_action (order_id, sku_id, action),  -- 幂等
  INDEX idx_sku_time (sku_id, create_time),
  INDEX idx_order (order_id)
);
```

**流水类型详解**：

| 动作 | 说明 | 场景 |
|------|------|------|
| PRE_DEDUCT | 预扣 | 下单时，Redis 预扣减 |
| DEDUCT | 实扣 | 支付成功，MySQL 实际扣减 |
| RELEASE | 释放 | 订单取消/超时，释放预扣 |
| REFUND | 退款回补 | 退款后，库存加回 |

**流水表与 Redis/MySQL 的一致性**：

```
预扣流程：
  1. Redis Lua 预扣：DECR stock:1001:segment
  2. 写流水表：INSERT stock_flow (order_id, sku_id, segment, action='PRE_DEDUCT', ...)
  3. 如果步骤 1 成功但步骤 2 失败 → 定时对账发现 Redis 扣了但无流水 → 补偿回补 Redis

实扣流程：
  1. 支付成功触发
  2. MySQL 实扣：UPDATE stock SET physical = physical - qty WHERE sku_id=1001 AND physical >= qty
  3. 写流水表：INSERT stock_flow (..., action='DEDUCT', pre_deduct_id=xxx)
  4. 删除 Redis 预扣记录（或等待超时释放）

释放流程：
  1. 订单超时/取消
  2. 查流水表：SELECT * FROM stock_flow WHERE order_id='O123' AND action='PRE_DEDUCT'
  3. 根据 segment 和 quantity 回补 Redis：INCR stock:1001:segment
  4. 写流水表：INSERT stock_flow (..., action='RELEASE')
```

**流水表的对账价值**：

```
对账维度：
  1. Redis 预扣 vs 流水表 PRE_DEDUCT 记录数
  2. MySQL 实扣 vs 流水表 DEDUCT 记录数
  3. 订单状态 vs 流水表（已支付订单是否都有 DEDUCT 记录）
  4. 已取消订单是否都有 RELEASE 记录

不一致场景：
  - Redis 预扣了，但流水表没有 PRE_DEDUCT →  Redis 多扣，需回补
  - 流水表有 PRE_DEDUCT，但 Redis 没扣 →  流水表误记，需标记
  - 订单已支付，流水表无 DEDUCT →  漏扣，需补扣
  - 订单已取消，流水表无 RELEASE →  漏释放，需补释放
```

## 5. 分段后的一致性保障

### 5.1 不超卖的三重保障

```
第一层：Redis 单段 Lua 原子扣减
  Lua 脚本保证 DECR 和检查 >=0 是原子的
  → 单段不会超卖

第二层：总库存冗余校验
  所有段都失败后，检查总库存
  → 防止段级不一致导致的多扣

第三层：MySQL 乐观锁兜底
  实扣时：UPDATE stock SET physical = physical - qty WHERE sku_id=? AND physical >= qty
  → 最终数据以 MySQL 为准，Redis 只是预扣

注意：
  Redis 分段扣减 + MySQL 实扣 的模型中，
  Redis 可能多扣（某段扣到 -1），但 MySQL 不会超卖。
  多扣的 Redis 段通过定时对账 + 流水表修复。
```

### 5.2 段级库存与总库存的一致性

```
问题：
  各段库存之和 ≠ 总库存（Redis 记录的总库存）
  原因：某段扣减时网络异常，Lua 执行了但返回丢失，客户端以为失败又扣了另一段

解决：
  1. 定时对账：每小时扫描各段库存之和 vs 总库存
  2. 偏差修复：
     - 如果各段之和 < 总库存：以各段之和为准，修正总库存
     - 如果各段之和 > 总库存：说明有段多扣了，找到具体段，修复并回补
  3. 初始化校验：活动开始前，强制校验 各段之和 = 总库存
```

### 5.3 分段与不分段的对比

| 维度 | 单 Key | 分段 |
|------|--------|------|
| 并发能力 | 单节点瓶颈 | 分散到多 Key/多节点 |
| 超卖风险 | 原子 DECR，不会超卖 | 需 Lua + 总库存兜底 + MySQL 乐观锁 |
| 复杂度 | 低 | 高（需段号管理、流水表、对账） |
| 库存利用率 | 100% | 可能不均匀（某段剩很多，某段已卖光） |
| 适用场景 | 库存 < 1000，并发 < 1万 QPS | 库存 > 1000，并发 > 10万 QPS |

## 6. 业务选型决策

```
是否需要分段？

Q1: 预估并发量是否 > 10万 QPS？
  否 → 单 Key + Lua 原子扣减，简单够用
  是 → Q2

Q2: 单 Redis 节点是否会成为瓶颈？
  是 → 分段，分散到多个 Key/多个节点
  否 → 单 Key 足够

Q3: 库存是否充足（> 1000）？
  是 → 分段有意义（每段有足够的量）
  否 → 分段收益不大，单 Key 即可

段数设计：
  段数 = 预估并发量 / 单段安全 QPS
  单段安全 QPS（经验值）：Redis 单 Key 约 5万 ~ 10万 QPS
  示例：预估 50万 QPS → 段数 = 50/5 = 10 段

注意：
  段数不是越多越好，过多会增加轮询开销和管理复杂度
  一般段数：5 ~ 20 段
```

| 场景 | 推荐方案 | 说明 |
|------|----------|------|
| 普通商品库存（日销 < 1000） | 单 Key + Lua | 简单，无分段复杂度 |
| 秒杀活动（QPS 10万+，库存 100~1000） | 分段 5~10 段 + 流水表 | 分散热点 |
| 大促爆品（QPS 100万+，库存 10000+） | 分段 20 段 + 本地缓存段路由 + 流水表 | 极致并发 |
| 库存 < 100 的限量商品 | 单 Key + 队列削峰 | 分段反而增加复杂度 |

## 7. 踩坑与反模式

### 坑 1：分段后没有总库存兜底
```
问题：只分段，不维护总库存
  → 各段之和可能不等于真实库存
  → 超卖或漏卖
解决：维护总库存（冗余但必要），定时对账
```

### 坑 2：轮询段时没有用 Lua 原子操作
```
问题：先 GET 段库存，再 DECR
  → GET 时库存 1，DECR 时已经被别人扣成 0 了
  → 扣成 -1，超卖
解决：每段扣减必须用 Lua 脚本，检查 + 扣减原子执行
```

### 坑 3：段数设计不合理
```
问题：100 件库存分 100 段，每段 1 件
  → 段数太多，轮询开销大
  → 大部分段瞬间卖光，后续请求大量无效轮询
解决：段数与库存量匹配，每段至少 10 ~ 50 件
```

### 坑 4：订单取消时不知道回补哪一段
```
问题：没有流水表记录扣减的是哪一段
  → 订单取消只能随机回补或全部段都尝试回补
解决：流水表必须记录 segment 字段，精确回滚
```

### 坑 5：本地缓存段库存与 Redis 不一致
```
问题：应用层缓存各段剩余量，但 Redis 实际已变化
  → 路由到已卖光的段，浪费请求
解决：本地缓存 TTL 短（如 1s），或监听 Redis 变更主动更新
      允许少量不一致，最终靠轮询兜底
```

### 坑 6：预热时各段库存不一致
```
问题：预热脚本执行中断，部分段初始化成功，部分失败
  → 各段之和 ≠ 总库存
解决：预热脚本必须幂等，支持重试
      预热完成后强制校验：SUM(各段) = 总库存
```

## 8. 面试追问应对

**Q1：库存 100 件，QPS 100 万，怎么设计？**
> 我会用库存分段。把 100 件库存分成 10 段，每段 10 件，分散到 10 个 Redis Key。用户请求通过轮询方式访问各段，每段扣减用 Lua 原子脚本保证不超卖。同时维护一个总库存作为兜底。如果所有段都扣减失败，再检查总库存确认是否真正售罄。配合库存流水表记录每笔扣减的段号和数量，用于订单取消时的精确回补和后续对账。MySQL 实扣时用乐观锁兜底，确保最终数据不会超卖。

**Q2：分段后，如果某段卖光了但总库存还有，怎么办？**
> 轮询机制会跳过已卖光的段，继续尝试其他段。为了优化，应用层可以缓存各段剩余量（TTL 1s），优先访问剩余量多的段。极端情况下，如果轮询所有段都失败，最后检查总库存冗余记录。如果总库存也显示为 0，则返回"库存不足"；如果总库存 > 0 但各段都失败，说明段级不一致，触发对账修复。

**Q3：分段后怎么保证不超卖？**
> 三重保障：第一，Redis 每段扣减用 Lua 原子操作，单段不会超卖；第二，维护总库存，所有段都失败后做最终校验；第三，MySQL 实扣时用乐观锁（WHERE stock >= qty），这是最终兜底。即使 Redis 层某段多扣了（如网络异常导致），MySQL 层也不会超卖。多扣的段通过对账系统发现后回补。

**Q4：库存流水表有什么用？**
> 四个作用：1. 幂等——同一订单重复扣减时，通过唯一索引去重；2. 对账——核对 Redis 预扣、MySQL 实扣、订单状态三者是否一致；3. 回滚——订单取消时，根据流水表精确知道扣减了哪一段，精确回补；4. 审计——追踪库存变更历史，排查超卖或库存异常的原因。

**Q5：分段和本地缓存（Caffeine）多级缓存有什么区别？**
> 分段解决的是"单 Key 热点"问题，把同一商品的库存分散到多个 Redis Key。多级缓存解决的是"Redis 单机热点"问题，在应用层加本地缓存减少 Redis 访问。两者可以结合：分段解决 Redis 内的热点，本地缓存解决应用层到 Redis 的网络热点。但多级缓存会引入本地缓存和 Redis 之间的不一致，需要消息广播清理，复杂度更高。
