## 1. 为什么时间在分布式系统中是难题

```
单机时间：
  - 一个时钟，全局有序
  - 时间戳比较即可排序

分布式时间：
  - 每台机器自己的时钟
  - 时钟不同步（时钟漂移）
  - 网络延迟不确定

导致的问题：
  1. 事件顺序难定
     - A 在 10:00 发生，B 在 10:01 发生
     - 但 B 的机器时钟慢了 2 分钟
     - 实际 B 比 A 早

  2. LWW（最后写胜出）出错
     - 靠时间戳判断先后
     - 时钟不准 → 旧值覆盖新值

  3. 一致性难保证
     - 线性一致需要全局时间序
     - 没有准确时间 → 难
```

## 2. 物理时钟的问题

### 2.1 时钟漂移

```
问题：机器时钟不准

原因：
  - 石英晶体有误差（每天漂移秒级）
  - 温度影响
  - 虚拟化时钟漂移更严重

NTP 同步：
  - 与时间服务器同步
  - 但有网络延迟误差
  - 可能调整时钟（前跳/后跳）

时钟跳变的问题：
  - 时钟回拨 → ID 重复（Snowflake）
  - 时钟前跳 → 超时误判
  - 跨节点时钟差 → 事件顺序错乱
```

### 2.2 物理时钟不可靠的后果

```
场景：LWW（最后写胜出）

  节点 A（时钟 10:00）：写 X=1
  节点 B（时钟 09:59，慢了1分钟）：写 X=2
  
  同步时按时间戳：
  X=1 时间戳 10:00 > X=2 时间戳 09:59
  → 保留 X=1（后写的反而被覆盖）

→ 数据丢失！
```

## 3. Lamport 逻辑时钟

> Leslie Lamport 1978 年提出，用逻辑序代替物理时间。

### 3.1 核心思想

```
不关心物理时间，关心事件的因果顺序

规则：
  1. 每个进程维护一个计数器 C
  2. 本地事件：C = C + 1
  3. 发送消息：C = C + 1，消息携带 C
  4. 接收消息：C = max(C, 消息C) + 1

  → 每个事件有一个逻辑时间戳

性质：
  - 如果 A → B（因果），则 C(A) < C(B)
  - 但 C(A) < C(B) 不一定 A → B（可能并发）
  → 偏序，非全序
```

### 3.2 示例

```
进程 P1：A(C=1) → 发送(C=2) → B(C=3)
进程 P2：接收(C=max(0,2)+1=3) → C(C=4)

  A(1) → 发送(2) → B(3)
              ↓
           接收(3) → C(4)

因果：A → B → 接收 → C
逻辑时间戳：1 < 2 < 3 < 4 ✓
```

### 3.3 局限

```
Lamport 时钟只能判断因果序，不能判断并发

  A(1)     B(1)
   ↓        ↓
  C(2)     D(2)

  C 和 D 的时间戳都是 2
  → 无法判断谁先谁后（实际是并发）

→ 需要向量时钟解决并发判断
```

## 4. 向量时钟

> 已在 [07-强一致与前沿方案](./07-强一致与前沿方案.md) 详述，这里补充与时间的关系。

### 4.1 与 Lamport 的区别

```
Lamport：标量（单值）
  - 能判断因果序
  - 不能判断并发

向量时钟：向量（多值）
  - 能判断因果序
  - 能判断并发
  → 详见 [07 §4](./07-强一致与前沿方案.md)
```

### 4.2 用于冲突检测

```
DynamoDB/Riak 用向量时钟：
  - 检测并发冲突
  - 返回多个版本让客户端解决
  - 或配合 CRDT 自动解决
```

## 5. TrueTime（Spanner）

> Google Spanner 的创新：用原子钟实现全球准确时间。

### 5.1 问题

```
线性一致性需要全局时间序
但物理时钟不准
→ Spanner 的解法：让时钟足够准
```

### 5.2 TrueTime API

```
TrueTime 不返回具体时间，返回区间：

  TT.now() → TTInterval { earliest, latest }

  - earliest ≤ 真实时间 ≤ latest
  - 区间宽度 < 7ms（原子钟 + GPS）

关键 API：
  TT.after(t)  ：时间是否已过 t
  TT.before(t) ：时间是否未到 t
```

### 5.3 Commit Wait

```
Spanner 写流程：
  1. 选时间戳 s = TT.now().latest
  2. 等待直到 TT.after(s) 为真（确保 s 已过）
  3. 提交

  → 等待 = Commit Wait
  → 保证 s 之后的写时间戳 > s
  → 全局有序

代价：
  - 等待 7ms（最大误差）
  - 跨洲延迟高
  - 需要原子钟硬件
```

### 5.4 TrueTime 的意义

```
  - 首次实现全球线性一致
  - 证明"用准确时钟换强一致"可行
  - 但成本极高（硬件 + 等待）
  → 大多数系统用不起
```

## 6. HLC 混合逻辑时钟

> CockroachDB 用 HLC（Hybrid Logical Clock），兼顾物理时钟的可读性和逻辑时钟的可靠性。

### 6.1 思想

```
  - 结合物理时钟 + 逻辑时钟
  - 用物理时间作为主体
  - 用逻辑计数器处理同物理时间的并发
  - 不需要原子钟
```

### 6.2 结构

```
HLC = (physical_time, logical_counter)

  - physical_time：从本地时钟取
  - logical_counter：同物理时间时递增

规则：
  1. 本地事件：
     p = max(本地时钟, 上次p)
     若 p == 上次p → l = l + 1
     否则 l = 0
  
  2. 接收消息：
     p = max(本地时钟, 上次p, 消息p)
     若 p == 上次p == 消息p → l = max(l, 消息l) + 1
     若 p == 上次p → l = max(l, 消息l) + 1
     若 p == 消息p → l = 消息l + 1
     否则 l = 0
```

### 6.3 优势

```
vs 物理时钟：
  - 不受时钟漂移影响
  - 保证因果序

vs 逻辑时钟：
  - 接近物理时间（可读）
  - 可与外部系统对齐

vs TrueTime：
  - 不需要原子钟
  - 但不保证线性一致（只保证因果一致）
```

## 7. 对比与选型

| 方案 | 全序 | 因果序 | 并发检测 | 硬件要求 | 适用 |
|------|------|--------|----------|----------|------|
| 物理时钟 | ✗（不准） | ✗ | ✗ | 无 | 简单场景 |
| Lamport | ✗（偏序） | ✓ | ✗ | 无 | 消息排序 |
| 向量时钟 | ✗（偏序） | ✓ | ✓ | 无 | 冲突检测 |
| TrueTime | ✓ | ✓ | ✓ | 原子钟 | Spanner |
| HLC | 部分 | ✓ | ✓ | 无 | CockroachDB |

### 7.1 业务选型

```
不需要严格顺序：
  → 物理时钟 + NTP（够用）

需要因果序（如消息系统）：
  → Lamport / 向量时钟

需要冲突检测（如协同编辑）：
  → 向量时钟 + CRDT

需要全球线性一致（如金融）：
  → TrueTime（Spanner）

需要因果一致但无原子钟：
  → HLC（CockroachDB）
```

### 7.2 与一致性模型的关系

```
详见 [01-一致性理论基础](./01-一致性理论基础.md)

  线性一致 → 需要 TrueTime / 共识算法
  顺序一致 → 需要共识算法
  因果一致 → 逻辑时钟/向量时钟/HLC
  最终一致 → 无需时钟保证
```
