## 1. 分布式 ID 生成

### 1.1 业务痛点

分库分表后自增 ID 不唯一；分布式系统需要全局唯一标识符。

### 1.2 方案对比

| 方案 | 有序性 | 性能 | 依赖 | 适用 |
|------|--------|------|------|------|
| UUID | 无序 | 高 | 无 | 非业务 ID（如日志 ID） |
| MySQL 自增 | 有序 | 低 | DB | 单库 |
| MySQL 号段模式 | 有序 | 中 | DB | 中规模 |
| Redis INCR | 有序 | 高 | Redis | 高性能但依赖 Redis |
| Snowflake | 有序 | 高 | 时钟 | 大规模通用 |
| Leaf（美团） | 有序 | 高 | DB+ZK | 生产级 |
| Tinyid（滴滴） | 有序 | 高 | DB | 号段模式增强 |

### 1.3 Snowflake 详解

```
64 位 ID 结构：
  0 | 0000...0000(41 位时间戳) | 0000...0000(10 位机器 ID) | 0000...0000(12 位序列号)

  时间戳：毫秒级，可用约 69 年
  机器 ID：5 位数据中心 + 5 位机器 → 1024 台机器
  序列号：单机每毫秒 4096 个 ID

优点：
  - 趋势递增（对索引友好）
  - 不依赖外部存储
  - 高性能（本地生成）

注意：
  - 时钟回拨问题 → 回拨时拒绝生成或借用未来时间
  - 机器 ID 分配 → 需要统一分配机制（ZK/配置中心）
  - 不是严格递增 → 跨机器 ID 可能不连续
```

**时钟回拨问题**：

```
问题：服务器时钟回拨（NTP 同步、人工调整）
  → 可能生成重复 ID

解决方案：
  1. 回拨时间 < 5ms → 等待回拨时间后再生成
  2. 回拨时间 > 5ms → 报错拒绝生成
  3. 借用未来时间（如自增逻辑时钟）
  4. 使用百度 UidGenerator 的方式（自定义 epoch）
```

### 1.4 号段模式（Leaf-Segment）

```
原理：
  1. 从 MySQL 获取一个号段（step=1000）
     UPDATE leaf_alloc SET max_id = max_id + step WHERE tag = ?
     SELECT max_id FROM leaf_alloc WHERE tag = ?
  2. 本地缓存的号段内直接分配（无 IO）
  3. 号段用到一定比例时异步预加载下一号段

优点：
  - 高性能（本地分配，偶尔 DB 交互）
  - 趋势递增
  - DB 宕机可继续分配（本地有缓存号段）

缺点：
  - DB 是单点（需要高可用）
  - 号段浪费（服务重启未用完的号段丢失）
  - ID 可预测（不安全场景慎用）
```

### 1.5 Leaf-Snowflake（美团改进）

```
改进点：
  1. 机器 ID 通过 ZK 自动分配
  2. 解决时钟回拨问题
  3. 监控告警

流程：
  1. 服务启动 → ZK 注册 → 获取唯一 workerID
  2. 启动定时任务上报时间戳到 ZK
  3. 生成 ID 前检查时钟回拨
```

### 1.6 选型建议

| 场景 | 推荐方案 |
|------|----------|
| 小规模、单库 | MySQL 自增 |
| 中规模、可控 | 号段模式 |
| 大规模、高性能 | Snowflake/Leaf |
| 简单场景、不依赖外部 | UUID（不推荐做主键） |
| 严格递增 | 号段模式 |

## 2. 一致性哈希

### 2.1 业务场景

- 缓存集群节点分布
- 分库分表路由
- 分布式存储数据分布
- 负载均衡

### 2.2 为什么不用简单取模？

```
简单取模 hash(key) % N：
  问题：节点增减时，几乎所有数据都需要重新分布
  例：3 节点 → 4 节点，约 75% 的 key 需要迁移

一致性哈希：
  优化：节点增减时，只影响相邻节点的数据
  例：3 节点 → 4 节点，约 25% 的 key 需要迁移
```

### 2.3 一致性哈希原理

```
1. 把 0 ~ 2^32 的值空间组成一个哈希环
2. 每个节点用 hash(IP) 计算在环上的位置
3. 每个 key 用 hash(key) 计算在环上的位置
4. key 顺时针找到的第一个节点就是它的归属节点

节点增减影响：
  - 增加节点 N：只影响 N 逆时针方向到上一个节点之间的 key
  - 删除节点 N：只影响 N 上的 key，重新分配到顺时针下一节点
```

### 2.4 虚拟节点解决数据倾斜

```
问题：少量物理节点时，数据分布可能严重不均匀

解决：每个物理节点映射多个虚拟节点
  node_A → VA_0, VA_1, VA_2, ... VA_150
  node_B → VB_0, VB_1, VB_2, ... VB_150

  虚拟节点均匀分布到哈希环上
  → 数据按虚拟节点分布 → 自然均匀到物理节点

虚拟节点数：
  - 一般 150-200 个/物理节点
  - 越多越均匀，但内存开销增大
```

### 2.5 应用

- Redis Cluster：使用 16384 个 slot（变体）
- Memcached：一致性哈希 + 虚拟节点
- Cassandra：一致性哈希
- 自研分库分表中间件

## 3. CAP 与 BASE 理论

### 3.1 CAP 理论

| 要素 | 含义 |
|------|------|
| C（一致性） | 所有节点看到相同数据 |
| A（可用性） | 每个请求都得到响应 |
| P（分区容错） | 网络分区时系统仍运行 |

**核心结论**：分布式系统中 P 必须保证，因此只能在 C 和 A 之间选择。

### 3.2 CAP 的现实选择

```
CP 系统：优先一致性
  - ZooKeeper
  - etcd
  - HBase
  - MongoDB（默认）

AP 系统：优先可用性
  - Cassandra
  - DynamoDB
  - Eureka
  - Redis Cluster

CA 系统：单机系统（不存在分布式 CA）
```

### 3.3 BASE 理论

| 要素 | 含义 |
|------|------|
| BA（基本可用） | 允许响应时间增加或功能降级 |
| S（软状态） | 允许中间状态（数据不一致） |
| E（最终一致） | 经过一段时间后数据一致 |

### 3.4 业务选型决策

| 业务 | CP/AP | 说明 |
|------|--------|------|
| 银行转账 | CP | 宁可不可用也不能数据不一致 |
| 电商库存 | AP + 最终一致 | 短暂超卖可接受，事后修正 |
| 社交动态 | AP | 不一致几秒无所谓 |
| 配置中心 | CP | 配置不一致比不可用更危险 |
| 用户认证 | CP | 认证错误是安全问题 |
| 搜索推荐 | AP | 结果略有差异可接受 |
| 注册中心 | AP | Eureka 选 AP，ZK 选 CP |

## 4. 服务降级与熔断

### 4.1 业务痛点

- 依赖服务故障导致连锁反应（雪崩效应）
- 系统过载时需要保护核心功能
- 非核心功能可以暂时牺牲

### 4.2 熔断器模式

```
三种状态：
  Closed（正常）：请求正常通过，统计失败率
  Open（熔断）：请求直接拒绝/走降级逻辑
  Half-Open（半开）：少量请求试探，成功则恢复

触发条件：
  - 时间窗口内失败率 > 阈值（如 50%）
  - 时间窗口内慢调用率 > 阈值
  - 连续失败次数 > 阈值

恢复机制：
  - 熔断后等待一段时间（如 5s）→ 半开
  - 半开状态放少量请求试探
  - 成功 → 恢复 Closed
  - 失败 → 继续 Open
```

### 4.3 降级策略

```
第一层：非核心服务降级
  - 评论区 → "评论功能暂时维护中"
  - 推荐系统 → 返回热门榜单代替个性化推荐

第二层：核心服务简化
  - 搜索 → 从 ES 降级到 MySQL 简单查询
  - 图片 → 从原图降级到缩略图

第三层：兜底数据
  - Redis 缓存 → 返回上次缓存数据（允许过期）
  - 本地文件缓存 → 最近的静态数据

第四层：限流保护
  - 降级 + 限流组合 → 先限流后降级
```

### 4.4 降级触发条件

| 触发条件 | 说明 |
|----------|------|
| 异常比例 | 异常率 > 50% 触发降级 |
| 慢调用比例 | 慢调用率 > 30% 触发降级 |
| 资源使用率 | CPU > 80%、内存 > 90% 触发降级 |
| 依赖故障 | 下游服务不可用触发降级 |
| 手动降级 | 运维人工触发 |

### 4.5 实现工具

- **Hystrix**（Netflix，已停止维护）
- **Resilience4j**（Hystrix 替代）
- **Sentinel**（阿里，推荐）
- **Spring Cloud Circuit Breaker**

## 5. 灰度发布

### 5.1 业务痛点

新功能上线影响所有用户，出 bug 则全量故障。

### 5.2 灰度策略

| 策略 | 维度 | 粒度 | 适用场景 |
|------|------|------|----------|
| 按用户比例 | 随机 5% → 10% → 50% → 100% | 粗 | 通用功能 |
| 按用户标签 | VIP/新用户/地区 | 细 | 定向功能 |
| 按功能模块 | 某接口/某页面 | 中 | 接口变更 |
| A/B 测试 | 对比新旧效果 | 精 | 产品决策 |
| 蓝绿发布 | 两套环境切换 | 粗 | 整体发布 |
| 金丝雀发布 | 少量实例先上 | 中 | 风险控制 |

### 5.3 实现方式

```
1. 配置中心控制
   灰度规则存配置中心 → 网关/服务读取规则路由

2. 网关层路由
   Nginx/网关根据灰度规则分流到新旧版本服务

3. 功能开关
   代码层 if(feature_flag.enabled) → 新逻辑 else → 旧逻辑

4. 数据隔离
   新版本用新表/新分片 → 出问题可快速回滚数据
```

### 5.4 灰度观测指标

```
必须监控：
  - 错误率对比（新 vs 旧）
  - 响应时间对比
  - 业务指标对比（转化率、下单率等）
  - 日志异常对比

回滚条件：
  - 新版本错误率 > 旧版本 2 倍 → 立即回滚
  - 响应时间退化 > 30% → 回滚
  - 业务指标退化 → 评估后决定
```

## 6. 限流设计

### 6.1 限流算法对比

| 算法 | 特点 | 临界问题 | 适用场景 |
|------|------|----------|----------|
| 固定窗口计数 | 简单 | 窗口边界突发 2 倍流量 | 低精度要求 |
| 滑动窗口计数 | 平滑 | 内存开销稍大 | 中精度要求 |
| 滑动窗口日志 | 精确 | 内存开销大 | 精确限流 |
| 令牌桶 | 允许突发 | 实现稍复杂 | API 限流（允许短突发） |
| 漏桶 | 匀速输出 | 不允许突发 | 流量整形（匀速消费） |

### 6.2 限流维度

```
单机限流：
  - 单机 QPS 限制
  - 保护单机资源

分布式限流：
  - 全局 QPS 限制
  - Redis + Lua 实现

多维度限流：
  - 全局限流（保护系统）
  - 用户限流（防刷）
  - IP 限流（防爬虫）
  - 接口限流（保护慢接口）
```

### 6.3 限流位置

```
第一级：网关层限流
  - Nginx limit_req
  - Spring Cloud Gateway

第二级：应用层限流
  - Sentinel
  - Resilience4j

第三级：资源层限流
  - 数据库连接池
  - 线程池
```

### 6.4 限流后的处理

| 处理方式 | 说明 |
|----------|------|
| 直接拒绝 | 返回 429 Too Many Requests |
| 排队等待 | 请求进入队列，慢慢处理 |
| 降级返回 | 返回简化结果或缓存 |
| 异步处理 | 转为异步任务，稍后通知 |

## 7. 服务注册与发现

### 7.1 业务痛点

微服务架构下，服务实例动态变化（扩容、缩容、故障），如何让调用方知道目标服务的地址？

### 7.2 方案对比

| 方案 | CAP | 特点 | 适用 |
|------|-----|------|------|
| ZooKeeper | CP | 强一致，复杂 | 强一致需求 |
| etcd | CP | 强一致，K8s 生态 | K8s 环境 |
| Eureka | AP | 高可用，最终一致 | Spring Cloud |
| Nacos | AP/CP | 支持切换 | 阿里系 |
| Consul | CP | 多功能 | HashiCorp 生态 |

### 7.3 服务发现流程

```
服务提供者：
  1. 启动时注册到注册中心
  2. 定时发送心跳
  3. 优雅关闭时注销

服务消费者：
  1. 启动时从注册中心拉取服务列表
  2. 本地缓存服务列表
  3. 定时更新
  4. 调用时从本地缓存选择实例

注册中心：
  1. 维护服务列表
  2. 心跳检测
  3. 故障实例剔除
  4. 通知消费者更新
```

### 7.4 健康检查

```
主动检查：
  - 注册中心定时调用健康检查接口
  - 失败则标记为不健康

被动检查：
  - 服务定时发送心跳
  - 超时未收到心跳则标记为不健康

调用方检查：
  - 调用失败累计达到阈值 → 标记为不健康
  - 半开试探恢复
```

## 8. 配置中心

### 8.1 业务痛点

- 多环境配置管理（dev/test/staging/prod）
- 多节点配置同步
- 配置动态更新不重启
- 配置变更审计

### 8.2 方案对比

| 方案 | 特点 | 适用 |
|------|------|------|
| Apollo（携程） | 功能全，UI 好 | 大型企业 |
| Nacos（阿里） | 注册 + 配置一体 | 阿里系 |
| Spring Cloud Config | Spring 原生 | Spring Cloud |

### 8.3 配置中心设计要点

```
1. 配置版本管理
   - 每次变更记录版本
   - 支持回滚

2. 配置灰度发布
   - 按实例/集群灰度
   - 避免全量故障

3. 配置变更通知
   - 长轮询 / 推送
   - 客户端监听变更

4. 配置加密
   - 敏感信息加密存储
   - 权限控制

5. 配置审计
   - 谁改了什么
   - 何时改的
```

## 9. 链路追踪

### 9.1 业务痛点

微服务调用链长（10+ 服务），一个请求失败需要追踪整个链路找根因。

### 9.2 核心设计

```
TraceID：全链路唯一标识（请求入口生成）
SpanID：单次调用标识
ParentSpanID：上游调用标识

数据采集：
  每个服务在调用前后记录：
  - TraceID/SpanID/ParentSpanID
  - 调用目标服务名
  - 调用耗时
  - 调用结果（成功/失败/异常）

数据存储：
  - 实时查看：Jaeger/Zipkin UI
  - 长期存储：ES/Cassandra
```

### 9.3 实际排查流程

```
1. 用户报错 → 获取请求时间 + 用户信息
2. 查 Trace 系统 → 找到该用户最近的 TraceID
3. 看调用链 → 找到耗时最长/失败的 Span
4. 深入该 Span → 查日志、查监控
5. 定位根因 → 修复
```

### 9.4 工具选型

| 工具 | 特点 |
|------|------|
| Jaeger | CNCF，云原生 |
| Zipkin | 老牌，简单 |
| SkyWalking | 国产，APM 全功能 |
| Pinpoint | 韩国，无侵入 |

## 11. 微服务网关设计

网关是微服务架构的"前门"，所有外部请求都经过它。理解网关的边界，才能理解服务治理的全貌。

### 11.1 网关的职责边界

```
网关该做的（核心职责）：
  ✓ 路由转发：请求转发到后端服务
  ✓ 鉴权认证：统一身份认证
  ✓ 限流熔断：保护后端服务
  ✓ 灰度分流：按规则路由到不同版本
  ✓ 协议转换：HTTP ↔ gRPC、WebSocket
  ✓ 日志监控：统一请求日志
  ✓ 压缩加密：响应压缩、TLS 卸载

网关不该做的（避免职责膨胀）：
  ✗ 业务逻辑：业务逻辑应在服务内
  ✗ 数据聚合：聚合应在 BFF 层
  ✗ 复杂计算：网关要轻量快速
```

### 11.2 网关选型

不脱离业务讲选型：

| 场景需求 | 推荐 | 理由 |
|----------|------|------|
| Spring Cloud 生态、Java 团队 | Spring Cloud Gateway | 与生态无缝集成、过滤器链灵活 |
| 高性能、多语言、K8s 环境 | APISIX / Kong | 基于 Nginx/OpenResty，性能极高 |
| 传统企业、SOAP/REST 混合 | Kong | 插件生态丰富 |
| 极致简单、小项目 | Nginx + Lua | 够用就行 |
| 阿里系、需要全链路 | Higress（阿里开源） | 与 Sentinel/MSE 集成 |

### 11.3 网关高可用

```
网关单点 = 系统单点，必须高可用：

方案 1：多实例 + LB
  客户端 → SLB/LVS → 网关集群（多实例）→ 后端服务
  → 任一网关实例宕机不影响

方案 2：网关集群 + 注册中心
  网关从注册中心拉取后端服务列表
  → 后端服务实例变更自动感知
```

### 11.4 网关常见踩坑

1. **网关变重**：业务逻辑往网关堆 → 保持网关轻量
2. **网关单点**：只部署一个实例 → 必须多实例 + LB
3. **跨域处理**：CORS 在网关统一处理，别散落到各服务
4. **链路追踪**：网关入口生成 TraceID 透传到下游

## 12. 监控告警体系

"没有监控的系统 = 黑盒"。监控不是上线后才补，而是架构设计的一部分。

### 12.1 监控分层

```
业务监控（最重要）：
  - 下单量、支付成功率、转化率
  - 异常订单数、投诉量
  - 业务指标异常 = 直接影响用户

应用监控：
  - QPS、响应时间、错误率
  - JVM、线程池、连接池
  - 接口级指标

中间件监控：
  - MySQL：慢查询、连接数、复制延迟
  - Redis：内存、命中率、大 key
  - MQ：堆积量、消费速度

基础设施监控：
  - CPU、内存、磁盘、网络
  - 容器指标
```

### 12.2 指标设计原则

**USE 原则**（资源类）：Utilization（使用率）、Saturation（饱和度）、Errors（错误）
**RED 原则**（服务类）：Rate（请求率）、Errors（错误率）、Duration（耗时）
**SLI/SLO**：定义服务等级指标和目标，如"99.9% 请求 < 200ms"

### 12.3 告警设计

告警不是越多越好，**告警疲劳**比没有告警更危险：

```
告警分级：
  P0（电话+IM）：核心业务中断、数据丢失
  P1（IM+短信）：重要功能异常、错误率突增
  P2（IM）：次要功能异常、性能下降
  P3（邮件）：仅记录，不主动通知

告警原则：
  - 可执行：告警来了能知道做什么
  - 低噪音：避免频繁误报
  - 有上下文：附带排查线索
  - 可收敛：问题解决后告警自动恢复
```

### 12.4 工具栈

```
指标监控：Prometheus + Grafana
日志：ELK / Loki
链路：SkyWalking / Jaeger
告警：Alertmanager / 钉钉/企微机器人
统一：Datadog / 阿里云 ARMS（商业）
```

### 12.5 故障排查思路

```
1. 现象确认：什么时间、什么功能、影响范围
2. 监控定位：看业务监控、应用监控、中间件监控
3. 链路追踪：根据 TraceID 找到慢/失败的节点
4. 日志分析：查对应时间段的错误日志
5. 根因定位：结合代码、配置、依赖
6. 应急处理：降级、回滚、扩容
7. 复盘改进：写故障报告、补监控、改流程
```

## 13. 容量评估方法论

"我需要多少台机器？"这是架构师必答题。容量评估不是拍脑袋，有方法论。

### 13.1 评估流程

```
1. 估算业务量
   - 日活用户、PV、订单量
   - 峰值倍数（一般日均的 5-10 倍）

2. 拆解到 QPS
   - 总 PV / 86400 × 峰值倍数 = 平均 QPS
   - 峰值 QPS = 平均 QPS × 峰值倍数

3. 评估资源
   - 单机 QPS 能力（压测得到）
   - 所需实例数 = 峰值 QPS / 单机 QPS × 冗余系数(1.5)

4. 评估存储
   - 日增量 × 保留天数 × 副本数
   - 加索引、日志、临时表的膨胀系数(2-3)

5. 评估带宽
   - 平均请求大小 × QPS
   - 考虑内网 + 外网
```

### 13.2 实战案例：电商系统容量评估

```
假设：
  - 日活 100 万
  - 人均浏览 20 页 → 日 PV 2000 万
  - 下单转化率 5% → 日订单 100 万
  - 峰值倍数 8

QPS 估算：
  - 浏览 QPS：2000万/86400 × 8 ≈ 1852 QPS
  - 下单 QPS：100万/86400 × 8 ≈ 93 QPS
  - 支付 QPS：约 50 QPS

资源估算（假设单机 QPS 1000）：
  - 商品服务：1852/1000 × 1.5 ≈ 3 台
  - 订单服务：93/1000 × 1.5 ≈ 1 台（但要做高可用，至少 2 台）
  - 支付服务：50/1000 × 1.5 ≈ 1 台（同上 2 台）

存储估算：
  - 订单表：100万/天 × 365天 × 1KB × 2(索引) ≈ 730GB/年
  - → 一年后分库分表

Redis 估算：
  - 商品缓存：10万商品 × 10KB = 1GB
  - 用户 session：100万 × 1KB = 1GB
  - → 8GB 实例够用
```

### 13.3 压测验证

容量评估是估算，**必须压测验证**：

```
压测方式：
  - 基准压测：单接口逐步加压，找单机极限
  - 链路压测：模拟真实链路，找系统瓶颈
  - 全链路压测：生产环境或影子库，最真实

压测工具：
  - JMeter：通用
  - wrk：轻量高性能
  - Gatling：高性能、脚本化
  - 全链路压测平台：自研（大厂才有）
```

### 13.4 容量规划原则

1. **留有余量**：资源利用率不超过 70%
2. **弹性扩容**：云上按需扩缩容
3. **降级预案**：容量不足时优先保核心
4. **持续评估**：业务增长要提前规划

## 14. 微服务通信与 RPC

微服务拆开后，服务间怎么通信是基础问题。通信方式选错，性能、可维护性、跨语言能力都会受影响。

### 14.1 通信方式分类

```
同步通信：
  - REST (HTTP/JSON)
  - RPC (gRPC/Dubbo/Thrift)

异步通信：
  - 消息队列 (Kafka/RocketMQ/RabbitMQ)
  - 事件驱动

选型维度：
  - 是否需要同步响应
  - 性能要求（延迟、吞吐）
  - 跨语言需求
  - 团队技术栈
  - 服务治理需求（熔断、限流、链路追踪）
```

### 14.2 REST vs RPC 对比

| 维度 | REST (HTTP/JSON) | RPC (gRPC/Dubbo) |
|------|------------------|-------------------|
| **协议** | HTTP/1.1 或 HTTP/2 | TCP（Dubbo）/ HTTP/2（gRPC） |
| **序列化** | JSON（文本、可读） | Protobuf/Hessian（二进制、紧凑） |
| **性能** | 中等 | 高（二进制+连接复用） |
| **可读性** | 好（人能读） | 差（二进制） |
| **跨语言** | 天然支持 | 需 IDL 生成代码 |
| **浏览器友好** | 好 | 差（gRPC 需 gRPC-Web） |
| **服务治理** | 依赖框架 | 内置（Dubbo 强） |
| **适用** | 对外 API、跨系统 | 内部服务间调用 |

### 14.3 主流 RPC 框架选型

不脱离业务讲选型：

| 场景 | 推荐 | 理由 |
|------|------|------|
| Java 单语言、强治理需求 | **Dubbo** | 中文文档好、治理能力强、阿里生态 |
| 多语言、云原生、高性能 | **gRPC** | 跨语言、HTTP/2、Google 背书 |
| Spring Cloud 生态 | **OpenFeign** | 与 Spring Cloud 无缝集成 |
| 对外 API、第三方对接 | **REST** | 通用、易调试 |
| 老系统、Thrift 已有 | **Thrift** | 不折腾 |

**Dubbo vs gRPC 深度对比**（高频面试题）：

| 维度 | Dubbo | gRPC |
|------|-------|------|
| 语言 | Java 为主（已支持多语言） | 跨语言 |
| 协议 | Dubbo 协议（TCP） | HTTP/2 |
| 序列化 | Hessian2/Protobuf/JSON | Protobuf |
| 服务治理 | 强（注册、路由、负载均衡、熔断内置） | 弱（需配合 Istio 等） |
| 生态 | 国内成熟 | 全球通用 |
| 学习成本 | 中 | 低（IDL 简单） |
| 适用 | 国内 Java 项目 | 多语言/云原生 |

### 14.4 序列化选型

```
JSON：
  - 可读、跨语言、调试友好
  - 体积大、性能中
  - 对外 API、配置、日志

Protobuf：
  - 二进制、紧凑、快
  - 需 IDL、不可读
  - gRPC、内部高性能通信

Hessian2：
  - 二进制、Java 友好
  - Dubbo 默认
  - Java 内部通信

Kryo/FST：
  - Java 专用、极快
  - 不跨语言
  - 纯 Java 内部
```

### 14.5 Service Mesh（服务网格）

当微服务数量多、治理代码（熔断/限流/重试/链路）侵入业务时，Service Mesh 把这些能力下沉到基础设施层。

**传统微服务（SDK 模式）**：
```
业务代码 + 治理 SDK（Hystrix/Sentinel/链路追踪）
  → 治理能力耦合在业务里
  → 升级治理要改业务代码、重新部署
  → 多语言要重复实现
```

**Service Mesh（Sidecar 模式）**：
```
业务代码 ←→ Sidecar（Envoy） ←→ Sidecar（Envoy） ←→ 业务代码
  → 治理能力在 Sidecar，业务无感知
  → 升级 Sidecar 不动业务
  → 多语言统一治理

代表：Istio + Envoy、Linkerd
```

**何时用 Service Mesh**：
- 微服务数量多（50+）
- 多语言环境
- 治理需求复杂、频繁变更
- 团队能驾驭 Istio 复杂度

**何时不用**：
- 微服务少、Java 单语言 → Dubbo/Spring Cloud 够用
- 团队小、运维能力有限 → Istio 学习成本高
- 性能极敏感 → Sidecar 有额外开销

### 14.6 通信可靠性保障

```
超时控制：
  - 必须设置超时（连接超时 + 读超时）
  - 超时时间根据 SLA 设定（如 P99 < 200ms）

重试：
  - 幂等接口才重试
  - 限制重试次数（1-3 次）
  - 指数退避

熔断：
  - 失败率高时熔断
  - 详见 §4

降级：
  - 依赖服务不可用时返回兜底
  - 详见 §4
```

## 15. 分布式安全设计

安全是分布式系统的红线。鉴权做不好，所有努力归零。本节聚焦业务侧的安全设计，不讲渗透测试。

### 15.1 认证 vs 授权

```
认证（Authentication）：你是谁？
  - 账号密码、短信验证码、生物识别
  - JWT、Session、Token

授权（Authorization）：你能做什么？
  - RBAC（角色-权限）
  - ABAC（属性-权限）
  - ACL（访问控制列表）
```

### 15.2 JWT 详解

JWT（JSON Web Token）是分布式系统常用的无状态认证方案。

```
JWT 结构：Header.Payload.Signature

Header：{"alg": "HS256", "typ": "JWT"}
Payload：{"userId": 123, "role": "admin", "exp": 1690000000}
Signature：HMACSHA256(base64(header) + "." + base64(payload), secret)

特点：
  - 无状态：服务端不存 session，token 自带信息
  - 自包含：payload 可放业务信息
  - 防篡改：签名保护
  - 不可加密：payload 默认 base64（可读），敏感信息别放
```

**JWT vs Session 对比**：

| 维度 | Session | JWT |
|------|---------|-----|
| 状态 | 有状态（服务端存） | 无状态（token 自带） |
| 扩展性 | 需共享 session（Redis） | 天然分布式友好 |
| 注销 | 简单（删 session） | 难（token 未过期前有效） |
| 续期 | 简单 | 难（需 refresh token） |
| 安全 | 较高 | 中（需注意存储） |

**JWT 的坑**：
1. **无法主动注销** → 解决：维护黑名单（Redis 存未过期但已注销的 token）
2. **续期难** → 解决：双 token（access_token 短期 + refresh_token 长期）
3. **payload 别放敏感信息** → 默认 base64 可读
4. **secret 要保护好** → 泄露等于全部沦陷

### 15.3 OAuth2 与 SSO

**OAuth2**：第三方授权（如"用微信登录"）

```
四种授权模式：
  - 授权码模式（最常用、最安全）
  - 简化模式
  - 密码模式
  - 客户端凭证模式

授权码模式流程：
  1. 用户 → 客户端：我要用微信登录
  2. 客户端 → 微信：请求授权码
  3. 微信 → 用户：是否授权？
  4. 用户 → 微信：同意
  5. 微信 → 客户端：返回授权码
  6. 客户端 → 微信：用授权码换 token
  7. 微信 → 客户端：返回 access_token
  8. 客户端 → 微信：用 token 获取用户信息
```

**SSO（单点登录）**：一次登录，多系统通用

```
方案：基于 JWT + 网关
  1. 用户登录认证中心 → 拿到全局 JWT
  2. 访问各子系统 → 携带 JWT
  3. 网关验证 JWT → 透传用户信息到子系统
  4. 子系统信任网关透传的用户信息

详见 [01-Redis §8 会话管理](../01-基础存储与中间件/00-Redis业务场景实战.md#8-会话管理--登录态与token)
```

### 15.4 API 安全要点

```
1. HTTPS 必备
   - 全站 HTTPS，禁用 HTTP

2. 防重放
   - 请求带时间戳 + nonce
   - 服务端校验时间窗口 + nonce 去重

3. 防篡改
   - 请求签名（参数 + 时间戳 + secret → MD5/HMAC）
   - 服务端验签

4. 限流
   - 防暴力破解、防爬虫
   - 详见 §6

5. 越权防护
   - 水平越权：用户 A 查用户 B 的数据 → 校验数据归属
   - 垂直越权：普通用户调管理员接口 → 校验角色权限

6. 敏感数据保护
   - 手机号、身份证脱敏显示
   - 密码 bcrypt 加盐存储
   - 数据传输加密（HTTPS/字段加密）
```

### 15.5 数据加密

```
传输加密：HTTPS / TLS
存储加密：
  - 密码：bcrypt / scrypt / argon2（不可逆）
  - 敏感字段：AES（可逆，密钥管理）
  - 数据库整库加密：TDE（透明数据加密）

密钥管理：
  - 密钥不要硬编码
  - 用 KMS（密钥管理服务）
  - 密钥轮换
```

### 15.6 幂等性安全

详见 [02-MySQL §4 幂等设计](../01-基础存储与中间件/01-MySQL业务场景实战.md#4-幂等设计--防重复处理)，安全角度补充：

```
支付场景：
  - 必须幂等（防重复扣款）
  - 唯一订单号 + 状态机
  - 第三方回调必须验签 + 幂等

转账场景：
  - 必须幂等（防重复转账）
  - 流水号唯一 + 乐观锁
```

## 16. 性能优化方法论

性能优化不是"加个缓存"这么简单，而是系统性的方法论。

### 16.1 性能优化的层次

```
从上到下：
  1. 业务层：业务逻辑是否合理？能否减少不必要的操作？
  2. 架构层：缓存、异步、分片、读写分离
  3. 应用层：代码效率、连接池、线程池
  4. 中间件层：DB 索引、Redis 优化、MQ 调优
  5. 系统层：JVM 调优、操作系统参数
  6. 硬件层：CPU、内存、磁盘、网络

原则：从上层往下优化，上层优化收益大、成本低
```

### 16.2 性能瓶颈定位

```
1. 监控先行
   - 响应时间（P50/P90/P99）
   - 吞吐量（QPS/TPS）
   - 资源使用率（CPU/内存/IO/网络）
   - 中间件指标

2. 链路追踪
   - 找到慢节点
   - 详见 §9

3. 压测
   - 找到系统瓶颈点
   - 单接口压测 + 全链路压测

4. 性能分析工具
   - Java：arthas、async-profiler、jstack、jmap
   - MySQL：EXPLAIN、慢查询日志
   - Redis：SLOWLOG、bigkeys
```

### 16.3 通用优化手段

**减少**：
```
- 减少请求：合并接口、批量操作、缓存
- 减少数据：只查需要的字段、分页
- 减少计算：预计算、异步化
- 减少网络：本地缓存、连接复用
```

**并行**：
```
- 多线程处理
- 异步化（MQ）
- 批量并行
```

**分片**：
```
- 数据分片（分库分表）
- 任务分片（定时任务分片）
- 请求分片（CDN、负载均衡）
```

### 16.4 典型优化案例

**慢查询优化**：
```
1. EXPLAIN 看执行计划
2. 加索引 / 改 SQL / 拆大查询
3. 详见 [02-MySQL §8](../01-基础存储与中间件/01-MySQL业务场景实战.md#8-sql-优化实战)
```

**接口优化**：
```
场景：商品详情页，需查 商品信息 + 评论 + 推荐 + 库存

串行（慢）：
  商品 → 评论 → 推荐 → 库存 → 返回（4 次串行 RT）

并行（快）：
  并行查 商品 / 评论 / 推荐 / 库存 → 合并返回（1 次 RT）

缓存（更快）：
  商品信息缓存 → 其余并行查 → 合并
```

**写优化**：
```
场景：批量导入 10 万条数据

错误做法：循环单条 INSERT → 10 万次 DB 交互
优化：
  1. 批量 INSERT（INSERT INTO ... VALUES (), (), ...）
  2. 分批（每批 1000 条）
  3. 关闭自动提交、手动事务
  4. LOAD DATA INFILE（更快）
  5. 异步化（MQ + 消费者批量写）
```

### 16.5 性能优化的原则

1. **数据驱动**：先监控、再优化，别拍脑袋
2. **二八原则**：20% 的代码影响 80% 的性能，找热点
3. **不过度优化**：满足业务需求即可，过度优化增加复杂度
4. **可回滚**：优化要能回滚，避免引入新问题
5. **持续优化**：性能是动态的，业务增长会带来新瓶颈
