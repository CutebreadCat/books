## 1. 单体架构

```
特点：
  - 所有功能在一个应用里
  - 一个进程
  - 共享数据库
  - 统一部署

典型：Spring Boot 单体应用
  Controller + Service + DAO 全在一起
  打一个 jar/war 部署
```

### 1.1 单体的优点

```
1. 简单
   - 开发简单：一个项目
   - 部署简单：一个包
   - 测试简单：无跨服务

2. 高效
   - 方法调用（无网络开销）
   - 事务简单（本地事务）
   - 无分布式问题

3. 低成本
   - 无需中间件
   - 运维简单
   - 团队小即可

4. 易调试
   - 单进程 debug
   - 日志在一个文件
   - 无链路追踪需求
```

## 2. 分布式架构

```
特点：
  - 功能拆分到多个服务
  - 多进程、多机器
  - 独立数据库
  - 独立部署
  - 网络通信

典型：微服务架构
  订单服务 + 库存服务 + 支付服务 + ...
  各自独立部署，通过 HTTP/RPC/MQ 通信
```

### 2.1 分布式的优点

```
1. 可扩展
   - 按需扩容单个服务
   - 不必整体扩容

2. 解耦
   - 服务独立迭代
   - 故障隔离
   - 团队分治

3. 技术栈灵活
   - 不同服务可用不同语言
   - 新技术试点

4. 高可用
   - 部分服务故障不影响整体
   - 多副本、多机房
```

## 3. 单体的缺点深度解析

### 3.1 规模问题

```
1. 代码膨胀
   - 百万行代码在一个项目
   - 编译慢、启动慢
   - IDE 卡顿

2. 团队协作冲突
   - 多团队改同一代码库
   - 合并冲突频繁
   - 发布互相阻塞

3. 耦合严重
   - 模块间依赖混乱
   - 改一处影响多处
   - "蝴蝶效应"

4. 技术债积累
   - 老代码不敢动
   - "屎山"形成
```

### 3.2 扩展问题

```
1. 无法局部扩容
   - 下单量大 → 整个应用扩容
   - 但其他模块不需要 → 浪费

2. 单机瓶颈
   - 单机 CPU/内存有上限
   - 无法突破单机性能

3. 数据库瓶颈
   - 所有模块共享 DB
   - 一个慢查询拖垮全局
```

### 3.3 可靠性问题

```
1. 一处故障全局故障
   - 内存泄漏 → 整个应用挂
   - 一个模块 OOM → 全站不可用

2. 发布风险大
   - 一次发布全量上线
   - 一个 bug 影响所有功能

3. 无法独立降级
   - 想关掉非核心功能保核心？做不到
```

### 3.4 技术演进问题

```
1. 技术栈锁定
   - 想引入新语言？不行
   - 想升级框架？全量升级

2. 创新受限
   - 新功能要兼容老代码
   - "祖宗之法不可变"
```

## 4. 分布式的缺点深度解析

> 分布式不是银弹，它引入了大量复杂性。

### 4.1 一致性问题（最严重）

```
单体：本地事务保证 ACID
分布式：跨服务无法本地事务

问题：
  订单服务创建订单 + 库存服务扣库存
  → 如何保证两者同时成功或同时失败？

解法：分布式事务（2PC/TCC/Saga/本地消息表）
  → 极其复杂
  → 详见 [数据一致性子目录](../02-数据一致性/00-数据一致性专题.md)

代价：
  - 开发复杂度 10 倍
  - 调试困难
  - 性能下降
```

### 4.2 网络问题

```
单体：方法调用（纳秒级，可靠）
分布式：网络调用（毫秒级，不可靠）

问题：
  1. 延迟
     - 方法调用 1ns → 网络 1ms（1000 倍）
     - 串联调用 → 延迟叠加

  2. 不可靠
     - 超时
     - 丢包
     - 网络分区

  3. 超时难题
     - 超时了，对方到底执行了没？
     - 重试可能导致重复
     → 幂等设计必不可少
```

### 4.3 运维复杂度

```
单体：1 个应用 × N 台机器
分布式：M 个服务 × N 台机器 = M×N 个实例

问题：
  1. 部署复杂
     - 多个服务独立部署
     - 版本管理
     - 依赖协调

  2. 监控复杂
     - 每个服务都要监控
     - 链路追踪必备
     - 详见 [15-链路追踪](../05-稳定性与可观测/01-链路追踪深度实战.md)

  3. 排障困难
     - 一个请求跨多服务
     - 日志分散
     - 需要链路追踪 + 日志聚合

  4. 运维成本
     - 中间件多（注册中心/配置中心/MQ...）
     - 需要专业 SRE
```

### 4.4 数据一致性问题

```
单体：共享数据库，强一致
分布式：各自数据库，数据分散

问题：
  1. 跨库 JOIN 做不到
     → 应用层组装 or 宽表冗余

  2. 跨库事务
     → 分布式事务（复杂）

  3. 数据冗余
     → 多份数据一致性
     [数据一致性子目录](../02-数据一致性/00-数据一致性专题.md)读.md)

  4. 分库分表
     → 跨片查询难
     → 详见 [02 §2](../01-基础存储与中间件/01-MySQL业务场景实战.md)
```

### 4.5 测试困难

```
单体：单进程测试，环境简单
分布式：多服务联调，环境复杂

问题：
  1. 联调环境
     - 需要所有服务可用
     - 环境搭建成本高

  2. 依赖管理
     - 服务间版本依赖
     - "我的服务依赖你的服务"

  3. 故障注入
     - 模拟下游故障
     - 需要专门工具
```

### 4.6 性能问题

```
1. 网络开销
   - 每次调用都是网络 IO
   - 串联调用延迟叠加

2. 序列化
   - 数据需序列化/反序列化
   - CPU 开销

3. 中间件开销
   - 注册中心、配置中心、网关
   - 每一跳都有开销
```

### 4.7 团队问题

```
1. 接口协调
   - 跨团队接口定义
   - 版本兼容

2. 职责边界
   - 服务边界划分
   - "这个功能该谁做"

3. DevOps 要求
   - 每个团队要懂运维
   - 否则依赖运维团队
```

## 5. 何时该上分布式

### 5.1 不要为了微服务而微服务

```
错误理由：
  ✗ "微服务是趋势"
  ✗ "大厂都用微服务"
  ✗ "显得技术先进"

正确理由：
  ✓ 团队规模大（> 10 人），需要分治
  ✓ 业务模块边界清晰
  ✓ 单体已成为瓶颈（性能/团队/迭代）
  ✓ 不同模块有不同的扩展需求
```

### 5.2 上分布式的信号

```
团队信号：
  - 团队 > 10 人
  - 多团队改同一代码库
  - 发布互相阻塞

技术信号：
  - 代码量 > 百万行
  - 编译 > 10 分钟
  - 启动 > 5 分钟
  - 单机扛不住流量

业务信号：
  - 模块边界清晰
  - 模块迭代速度差异大
  - 模块可用性要求不同
```

### 5.3 什么时候保持单体

```
  - 创业初期，验证业务
  - 团队小（< 10 人）
  - 业务简单
  - 流量小
  → 单体足够，不要过度设计
```

## 6. 架构演进路径

```
不要一步到位，循序渐进：

阶段 1：单体
  - 快速验证业务
  - 一个应用

阶段 2：单体 + 读写分离
  - 读流量大 → 加从库
  - 仍单体

阶段 3：单体 + 缓存
  - 加 Redis 缓存
  - 仍单体

阶段 4：垂直拆分
  - 按业务拆分（如前端 + 后端 API）
  - 2-3 个大服务

阶段 5：水平拆分（微服务）
  - 按领域拆细
  - 每个服务独立

阶段 6：多机房/多活
  - 容灾需求
  - 详见 [数据一致性/09](../02-数据一致性/09-异地多活一致性.md)

原则：
  - 每次演进解决明确问题
  - 不要跳级
  - 过度架构 = 找麻烦
```

## 7. 选型决策框架

```
团队规模？
├── < 10 人 → 单体（不要折腾）
└── > 10 人
    │
    流量规模？
    ├── < 1万 QPS → 单体 + 缓存 + 读写分离
    └── > 1万 QPS
        │
        业务复杂度？
        ├── 简单 → 单体 + 分库分表
        └── 复杂（多领域）
            │
            是否需要独立迭代？
            ├── 否 → 垂直拆分（2-3 大服务）
            └── 是 → 微服务
                │
                是否需要容灾？
                ├── 否 → 单机房微服务
                └── 是 → 多机房/多活
```

### 7.1 决策表

| 场景 | 推荐 | 理由 |
|------|------|------|
| 创业初期 | 单体 | 快速验证 |
| 小团队小流量 | 单体 + 缓存 | 够用 |
| 中等规模 | 垂直拆分 | 平衡 |
| 大团队多业务 | 微服务 | 团队分治 |
| 超大规模 | 微服务 + 多活 | 容灾 |

### 7.2 架构选型的反模式

```
1. 过度架构
   - 小项目上微服务
   - 创业公司上 K8s + Istio
   → 成本远大于收益

2. 跟风架构
   - "XX 大厂用了 XX"
   - 不考虑自身场景

3. 一步到位
   - 想直接上微服务
   - 跳过单体阶段

4. 技术驱动
   - 为了用新技术而改架构
   - 不是业务驱动
```
