## 1. 为什么营销规则是最脏的代码

```
面试常见回答：
  "优惠券就是一张表，用户领券后，下单时检查是否满足条件，满足就减钱。"

真实业务：
  - 用户有 3 张优惠券：满 100 减 10（全品类）、满 200 减 30（3C 专属）、5 元无门槛
  - 商品参加了满 200 减 50 的跨店满减
  - 用户是 VIP，享受 95 折
  - 今天有秒杀价，秒杀商品不能用优惠券
  - 订单里同时有秒杀商品和普通商品

问题：
  1. 哪些优惠可以叠加？哪些互斥？
  2. 优惠的计算顺序是什么？
  3. 优惠金额怎么分摊到每个商品？
  4. 部分退款时，退多少？
  5. 如果优惠规则改了，历史订单怎么兼容？
```

## 2. 营销活动模型抽象

### 2.1 活动类型抽象

```
所有营销活动可以抽象为统一的模型：

活动（promotion）：
  - 类型： coupon（优惠券）/ full_reduction（满减）/ discount（折扣）/ seckill（秒杀）/ vip_discount（会员价）
  - 作用范围： order（订单级）/ item（商品级）/ merchant（店铺级）/ platform（平台级）
  - 触发条件： 满 X 元 / 指定商品 / 指定品类 / 指定用户 / 无门槛
  - 优惠形式： 减金额 / 打折扣 / 固定价
  - 优先级： 数字越小优先级越高
  - 互斥组： 同一组内的活动互斥
  - 可叠加组： 可与其他组的活动叠加

示例：
  秒杀活动：
    type=SECKILL, scope=ITEM, trigger=指定商品, form=固定价, priority=1, mutex_group=PRICE, stack_group=NONE

  会员折扣：
    type=VIP_DISCOUNT, scope=ORDER, trigger=VIP用户, form=95折, priority=10, mutex_group=DISCOUNT, stack_group=COUPON+FULL_REDUCTION

  优惠券：
    type=COUPON, scope=ORDER, trigger=满100元, form=减10元, priority=20, mutex_group=COUPON, stack_group=DISCOUNT+FULL_REDUCTION

  跨店满减：
    type=FULL_REDUCTION, scope=ORDER, trigger=满200元, form=减50元, priority=15, mutex_group=FULL_REDUCTION, stack_group=COUPON+DISCOUNT
```

### 2.2 规则配置表设计

```sql
-- 活动主表
CREATE TABLE promotion (
    promotion_id BIGINT PRIMARY KEY,
    promotion_type VARCHAR(32),       -- SECKILL / COUPON / FULL_REDUCTION / DISCOUNT / VIP
    name VARCHAR(128),
    start_time DATETIME,
    end_time DATETIME,
    status ENUM('ACTIVE', 'PAUSED', 'ENDED'),
    priority INT,                     -- 优先级，数字越小越优先
    mutex_group VARCHAR(32),          -- 互斥组标识
    stack_group VARCHAR(128),         -- 可叠加组（逗号分隔）
    create_time DATETIME
);

-- 活动规则表
CREATE TABLE promotion_rule (
    rule_id BIGINT PRIMARY KEY,
    promotion_id BIGINT,
    rule_type ENUM('TRIGGER', 'TARGET', 'LIMIT'),  -- 触发条件/作用对象/限制条件
    rule_key VARCHAR(64),             -- 如：min_amount / category / user_level / max_discount
    rule_value VARCHAR(256),          -- 如：100 / 3C / VIP / 50
    operator ENUM('EQ', 'GT', 'LT', 'IN', 'NOT_IN'),  -- 等于/大于/小于/包含/不包含
    create_time DATETIME
);

-- 活动与商品关联表
CREATE TABLE promotion_sku (
    id BIGINT PRIMARY KEY,
    promotion_id BIGINT,
    sku_id BIGINT,
    create_time DATETIME
);
```

## 3. 优惠计算引擎

### 3.1 计算流程

```
优惠计算引擎（ Pricing Engine ）：

输入：
  - 订单商品列表（sku_id, quantity, unit_price, category, merchant_id）
  - 用户身份（user_id, vip_level）
  - 用户持有的优惠券列表
  - 当前生效的活动列表

步骤：
  1. 筛选适用活动
     遍历所有活动，根据规则表判断每个活动是否适用于当前订单

  2. 分组与互斥
     按 mutex_group 分组，每组内只选最优的一个活动
     示例：
       COUPON 组：用户有 3 张券，选面额最大的那张
       FULL_REDUCTION 组：可能多个满减活动，选减得最多的

  3. 计算叠加
     按 priority 从低到高（数字小优先）依次应用优惠
     注意：优惠的应用顺序会影响最终价格
     示例：先打 95 折再减 10 元， vs 先减 10 元再打 95 折，结果不同

  4. 输出优惠明细
     返回每个活动减免了多少钱，以及最终应付金额

输出：
  {
    "original_amount": 250.00,
    "discount_details": [
      { "promotion_id": 1, "type": "SECKILL", "discount": 50.00, "reason": "秒杀价" },
      { "promotion_id": 2, "type": "VIP_DISCOUNT", "discount": 10.00, "reason": "VIP 95折" },
      { "promotion_id": 3, "type": "COUPON", "discount": 10.00, "reason": "满100减10" }
    ],
    "final_amount": 180.00
  }
```

### 3.2 优先级设计

```
优先级数字越小越优先（先应用）：

  1  - 秒杀价（固定价，最优先）
  5  - 会员价（固定价，会员专享）
  10 - 平台折扣（如全场 9 折）
  15 - 跨店满减（订单级满减）
  20 - 店铺优惠券（店铺级）
  25 - 平台优惠券（平台级）
  30 - 积分抵扣（最后应用）

为什么顺序重要？
  先应用固定价（秒杀/会员），因为固定价不参与后续计算
  再应用折扣（百分比），因为折扣基数大时减得多
  最后应用优惠券（固定减金额），因为优惠券通常有门槛

反例：如果先减 10 元再打 9 折，vs 先打 9 折再减 10 元
  原价 100：先减 10 = 90，再 9 折 = 81
  原价 100：先 9 折 = 90，再减 10 = 80
  → 顺序不同，结果不同
```

## 4. 叠加与互斥规则

### 4.1 叠加规则矩阵

```
常见叠加规则（业务规则，可配置）：

                    秒杀价  会员价  折扣   满减   优惠券  积分
秒杀价               ❌     ❌     ❌     ❌     ❌     ❌
会员价               ❌     ❌     ✅     ✅     ✅     ✅
折扣                 ❌     ✅     ❌     ✅     ✅     ✅
满减                 ❌     ✅     ✅     ❌     ✅     ✅
优惠券               ❌     ✅     ✅     ✅     ❌     ✅
积分                 ❌     ✅     ✅     ✅     ✅     ❌

解释：
  ❌ = 互斥（不能同时享受）
  ✅ = 可叠加（可以同时享受）

  秒杀价：最优先，不与任何其他优惠叠加
  会员价：可与除秒杀外的所有优惠叠加
  优惠券之间：通常互斥（选最优的一张），但部分业务允许叠加（如店铺券+平台券）
  满减之间：通常互斥（选最优的一个）
```

### 4.2 互斥规则实现

```
算法：
  1. 按 mutex_group 分组，每组内选最优
  2. 遍历所有组，检查是否可叠加（stack_group 包含关系）

示例：
  活动A：秒杀价，mutex_group=PRICE, stack_group=NONE
  活动B：会员折扣，mutex_group=DISCOUNT, stack_group=COUPON+FULL_REDUCTION
  活动C：优惠券，mutex_group=COUPON, stack_group=DISCOUNT+FULL_REDUCTION
  活动D：满减，mutex_group=FULL_REDUCTION, stack_group=COUPON+DISCOUNT

处理：
  1. 秒杀价（A）最优先，且不可叠加 → 如果订单有秒杀商品，直接选 A，其他都忽略
  2. 非秒杀商品：
     - DISCOUNT 组（B）和 COUPON 组（C）可叠加（B.stack_group 包含 COUPON）
     - DISCOUNT 组（B）和 FULL_REDUCTION 组（D）可叠加（B.stack_group 包含 FULL_REDUCTION）
     - COUPON 组（C）和 FULL_REDUCTION 组（D）可叠加（C.stack_group 包含 FULL_REDUCTION）
     → 最终：B + C + D 同时生效
```

### 4.3 多优惠券的叠加

```
场景：用户有 2 张券，是否都能用？

类型1：互斥（选最优一张）
  用户有：满 100 减 10、满 200 减 30
  订单金额：250
  → 选满 200 减 30（更优），放弃另一张

类型2：可叠加（店铺券 + 平台券）
  用户有：店铺券（满 100 减 10）、平台券（满 200 减 20）
  订单金额：250
  → 先应用店铺券（250 - 10 = 240），再应用平台券（240 >= 200 → 再减 20）
  → 最终：250 - 10 - 20 = 220

注意：
  叠加后，平台券的门槛是否重新计算？
  方案A：按原价计算门槛（250 >= 200，满足）→ 推荐，简单
  方案B：按折后价计算门槛（240 >= 200，满足）→ 复杂，可能用户不满
```

## 5. 优惠分摊算法

### 5.1 为什么要分摊

```
问题：
  订单有 3 个商品，总价 250 元
  优惠：满 200 减 50
  用户退了其中 1 个商品（价格 80 元），退多少钱？

如果直接退 80 元：
  → 用户实付 250 - 50 = 200，退了 80，还剩 120
  → 但剩余 2 个商品原价 170，按优惠比例应该付 170 - 50*(170/250) = 136
  → 用户多占了便宜（付了 120，应该付 136）

解决：优惠必须按商品金额比例分摊到每个商品
```

### 5.2 分摊算法

```
算法：按商品金额比例分摊

示例：
  商品A：100 元
  商品B：80 元
  商品C：70 元
  总价：250 元
  优惠：满 200 减 50

分摊：
  商品A分摊 = 50 * (100 / 250) = 20 元
  商品B分摊 = 50 * (80 / 250) = 16 元
  商品C分摊 = 50 * (70 / 250) = 14 元
  合计：20 + 16 + 14 = 50 元 ✓

商品实付：
  商品A实付 = 100 - 20 = 80 元
  商品B实付 = 80 - 16 = 64 元
  商品C实付 = 70 - 14 = 56 元

退款计算：
  退商品A：退款 = 80 元（不是原价 100 元）
  退商品B：退款 = 64 元
  退商品C：退款 = 56 元

精度问题：
  分摊后可能出现小数（如 14.666...）
  处理：最后一件商品用减法兜底（总优惠 - 前面已分摊 = 最后一项）
  避免四舍五入导致总和不等于优惠金额
```

### 5.3 分摊表设计

```sql
CREATE TABLE order_discount_apportion (
    id BIGINT PRIMARY KEY,
    order_id VARCHAR(64),
    item_id VARCHAR(64),           -- 订单明细ID
    promotion_id BIGINT,             -- 活动ID
    discount_amount DECIMAL(18,2),  -- 该商品分摊到的优惠金额
    original_amount DECIMAL(18,2),  -- 商品原价
    final_amount DECIMAL(18,2),    -- 商品实付价
    create_time DATETIME,
    UNIQUE KEY uk_order_item_promo (order_id, item_id, promotion_id)
);
```

## 6. 秒杀与普通促销的冲突

```
场景：
  商品A：原价 100 元，秒杀价 50 元（库存 10 件）
  用户把商品A加入购物车，同时还有其他普通商品
  结算时：
    1. 商品A是否可以用优惠券？
    2. 商品A是否参与满减？
    3. 商品A是否参与 VIP 折扣？

通用规则：
  秒杀商品通常不参与任何其他优惠（最优先级，独占）
  原因：
    1. 秒杀价已经是最低价，叠加优惠会导致资损
    2. 秒杀库存有限，如果还能用优惠券，黄牛会疯狂薅

例外情况：
  平台可能允许"秒杀商品 + 运费券"（运费券不影响商品价）
  或者"秒杀商品 + 积分抵扣"（积分是用户资产，不影响商家收入）

计算引擎处理：
  1. 识别订单中的秒杀商品
  2. 秒杀商品单独计算：固定价，不参与任何优惠
  3. 非秒杀商品正常计算优惠
  4. 两者金额相加为最终订单金额

示例：
  秒杀商品A：50 元（不参与任何优惠）
  普通商品B：100 元
  普通商品C：80 元
  优惠券：满 100 减 10（只能用于普通商品）
  满减：满 200 减 50（只能用于普通商品）

  普通商品总价：180 元
  满足满 100 减 10 → 减 10 元
  不满足满 200 减 50 → 不减
  普通商品实付：170 元
  订单总价：50 + 170 = 220 元
```

## 7. 踩坑与反模式

### 坑 1：优惠计算顺序没定义
```
问题：先打 9 折再减 10 元 vs 先减 10 元再打 9 折，结果不同
  → 同样的商品和优惠，不同用户看到不同价格
解决：明确优先级，固化计算顺序（如固定价 > 折扣 > 满减 > 优惠券）
```

### 坑 2：互斥规则遗漏
```
问题：新上线了一个活动，忘记配置互斥组
  → 用户同时享受了秒杀价 + 优惠券，导致资损
解决：新活动上线必须配置完整的互斥/叠加规则，经评审后生效
```

### 坑 3：优惠分摊后退款算错
```
问题：用户部分退款时，退的是原价，导致用户多占便宜
  → 或者退的是分摊价，但分摊精度问题导致差额
解决：退款必须按分摊后的实付价退，分摊表是退款计算的唯一依据
```

### 坑 4：优惠规则修改后历史订单不兼容
```
问题：活动规则改了，但用户已下单的订单按新规则计算
  → 价格变了，用户投诉
解决：订单创建时快照保存当时的优惠规则，后续计算以快照为准
```

### 坑 5：活动预热期提前泄露
```
问题：明天的活动，今天接口就返回了活动信息
  → 用户提前知道了价格，观望不买，或者传播到社交平台
解决：活动信息分阶段返回，预热期只显示"有活动"，不显示具体优惠力度
```

### 坑 6：活动并发冲突
```
问题：两个运营同时创建活动，同一商品有两个互斥的活动
  → 用户可能同时享受两个本来互斥的活动
解决：活动创建时检查冲突（同一商品 + 时间重叠 + 互斥组冲突）
```

## 8. 面试追问应对

**Q1：用户有多张优惠券，怎么选最优？**
> 我设计了一个优惠计算引擎。首先按 mutex_group 分组，每组内只选最优的一个活动。比如 COUPON 组内，用户有 3 张券，分别计算每张券能减多少钱（注意门槛条件），选减得最多的那张。然后检查不同组之间是否可以叠加（通过 stack_group 判断），如果可以叠加，就同时生效。最后按优先级从低到高依次应用，优先级数字小的先应用（如秒杀价最优先）。

**Q2：优惠分摊怎么设计？**
> 按商品金额比例分摊。例如订单总价 250，优惠 50，商品A 100、商品B 80、商品C 70，那么商品A分摊 50*(100/250)=20，商品B分摊 16，商品C分摊 14。每件商品分摊后记录到分摊表（order_discount_apportion），退款时按分摊后的实付价退，不是按原价退。精度问题用最后一件兜底减法处理，确保分摊总额等于优惠总额。

**Q3：秒杀商品和普通商品在同一个订单，优惠怎么算？**
> 秒杀商品单独计算，不参与任何其他优惠（秒杀价最优先，且独占）。非秒杀商品正常计算优惠。两者金额相加为最终订单金额。例如秒杀商品A 50元（固定价），普通商品B和C总价180元，满足满100减10，那么普通商品实付170元，订单总价 50+170=220 元。优惠券和满减只能用于非秒杀商品。

**Q4：优惠券和满减可以叠加吗？**
> 看业务规则，通常是可叠加的。我的设计是通过 stack_group 配置：优惠券组的 stack_group 包含 FULL_REDUCTION，满减组的 stack_group 包含 COUPON，说明两者可叠加。计算顺序上，先满减后优惠券（或反之，但必须固定顺序），门槛计算建议按原价算，避免折后价导致门槛判断复杂化。如果是店铺券+平台券，也可以叠加，门槛各自独立判断。

**Q5：退款时怎么算退多少钱？**
> 退款金额 = 商品分摊后的实付价。不能退原价，因为优惠是订单级别的。分摊表记录了每个商品分摊到的具体优惠金额，退款时查询分摊表即可。如果是整单退款，退订单实付总额。如果是部分退款，退该商品的分摊后实付价。退款后如果剩余商品不满足优惠门槛，不退已经享受的优惠（用户已经占了便宜，退款只退该商品的钱）。
