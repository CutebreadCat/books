## 1. 风控系统设计

风控是业务的安全护栏，防黄牛、防刷单、防欺诈、防薅羊毛。

### 1.1 风控场景

| 场景 | 风险 | 检测点 |
|------|------|--------|
| 注册 | 批量注册养号 | 手机号、IP、设备 |
| 登录 | 撞库、爆破 | 失败频率、异地 |
| 下单 | 黄牛抢购、刷单 | 频率、设备、行为 |
| 支付 | 盗刷、洗钱 | 金额、频率、来源 |
| 退款 | 薅羊毛、欺诈 | 退款频率、金额 |
| 评价 | 刷好评/差评 | 频率、内容相似度 |
| 领券 | 批量薅羊毛 | 频率、设备、账号 |

### 1.2 风控架构

```
风控系统架构：

事件采集 → 规则引擎 → 决策 → 处置 → 反馈

  事件采集：
    - 业务事件（注册/登录/下单）
    - 用户行为（点击/浏览）
    - 设备信息（指纹/IP）

  规则引擎：
    - 规则匹配（频率/阈值/黑名单）
    - 模型评分（ML 模型）
    - 实时计算（Flink）

  决策：
    - 放行
    - 拦截
    - 验证（验证码/短信）
    - 人工审核

  处置：
    - 拦截请求
    - 限制操作
    - 加入黑名单
    - 告警

  反馈：
    - 记录处置结果
    - 模型训练数据
```

### 1.3 规则引擎

```
规则类型：

1. 频率规则
   - 同 IP 注册 > 10 次/天 → 拦截
   - 同设备下单 > 5 次/小时 → 验证

2. 阈值规则
   - 单笔支付 > 1 万 → 人工审核
   - 退款率 > 30% → 告警

3. 名单规则
   - 黑名单 IP/手机号/设备 → 拦截
   - 灰名单 → 加强验证

4. 关联规则
   - 同设备多账号 → 风险
   - 同 IP 多订单 → 风险

5. 行为规则
   - 注册后立即下单 → 可疑
   - 浏览路径异常 → 机器人

实现：
  - 简单规则：Redis + Lua 实时判断
  - 复杂规则：规则引擎（Drools/自研）
  - 模型规则：ML 模型评分
```

### 1.4 设备指纹

```
设备指纹：唯一标识设备

采集信息：
  - 硬件：CPU、内存、屏幕
  - 软件：浏览器、UA、插件
  - 网络：IP、代理检测
  - 行为：鼠标轨迹、打字节奏

生成：
  - 采集信息 → Hash → 设备指纹 ID
  - 存 Redis（device:fingerprint:<id> → 用户列表）

应用：
  - 同设备多账号 → 关联风险
  - 设备黑名单 → 拦截
  - 异地登录 → 设备变化检测
```

### 1.5 黑名单管理

```
黑名单类型：
  - IP 黑名单
  - 手机号黑名单
  - 设备黑名单
  - 用户黑名单
  - 卡号黑名单（支付）

存储：
  - Redis Set（实时查询）
  - MySQL（持久化）

  查询：SISMEMBER blacklist:ip <ip>

加入黑名单：
  - 自动：风控规则触发
  - 人工：运营手动加
  - 申诉：用户可申诉解除

  TTL：
  - 临时黑名单：1-7 天
  - 永久黑名单：无过期
```

### 1.6 实时风控

```
实时风控（低延迟）：

  要求：< 50ms 完成风控判断

  实现：
  1. 规则预计算（规则编译为 Redis 操作）
  2. 热数据 Redis 缓存
  3. 同步调用风控接口

  流程：
  请求 → 风控 SDK → Redis 规则判断 → 放行/拦截

  Redis 数据：
  - 频率计数：freq:<user>:<action>:<window>
  - 黑名单：blacklist:<type>
  - 设备指纹：device:<fingerprint>
```

### 1.7 离线风控

```
离线风控（深度分析）：

  场景：发现已发生的风险，建立模型

  实现：
  1. 数据同步到数据仓库
  2. 离线分析（Spark/Hive）
  3. 模型训练
  4. 产出黑名单/规则
  5. 同步到实时风控

  周期：T+1 或小时级
```

### 1.8 踩坑

1. **规则过严** → 误伤正常用户，需灰度 + 白名单
2. **规则过松** → 黄牛绕过，需持续迭代
3. **实时风控慢** → 阻塞业务，规则要精简
4. **无申诉渠道** → 用户被误封无法恢复
5. **黑名单不更新** → 过期黑名单未清理

## 2. 数据导出与报表系统

数据导出和报表是运营后台的高频需求，大数据量场景下技术含量不低。

### 2.1 业务场景

```
导出场景：
  - 订单明细导出（运营对账）
  - 用户列表导出（营销触达）
  - 商品报表导出
  - 财务报表

报表场景：
  - 日报/周报/月报
  - 实时大屏
  - 自定义维度分析
```

### 2.2 导出方案对比

| 方案 | 数据量 | 实时性 | 复杂度 |
|------|--------|--------|--------|
| 同步导出 | 小（< 1万） | 实时 | 低 |
| 异步导出 | 大（百万级） | 延迟 | 中 |
| 预计算 | 大 | 准实时 | 高 |

### 2.3 同步导出（小数据量）

```
适用：< 1 万行

流程：
  1. 用户点导出
  2. 查询数据
  3. 生成 Excel/CSV
  4. 返回下载

问题：
  - 数据量大 → 超时、OOM
  - 大量数据查询 → 慢

优化：
  - 流式查询（MyBatis Cursor）
  - 流式写入 Excel（SXSSF）
  - 限制导出行数
```

### 2.4 异步导出（大数据量，推荐）

```
适用：百万级行

流程：
  1. 用户点导出 → 创建导出任务（PENDING）
  2. 返回任务 ID
  3. 后台任务消费：
     a. 分页查询数据（每页 1000-5000）
     b. 流式写入临时文件
     c. 上传到对象存储
     d. 更新任务状态 → COMPLETED
  4. 用户下载（通知或轮询）

关键点：
  - 分页查询避免 OOM
  - 流式写入避免内存堆积
  - 任务状态可查
  - 文件有效期管理

  详见 [07 定时任务](../05-稳定性与可观测/00-分布式定时任务与调度.md)
```

### 2.5 报表系统

```
报表类型：

1. 实时报表
   - 数据量小，实时查询
   - Redis 计数器 + MySQL 查询
   - 大屏展示

2. 预计算报表（T+1）
   - 离线计算 → 汇总表
   - 查询汇总表
   - 适合历史数据

3. 实时大屏
   - Flink 实时计算
   - 写入 ClickHouse/Redis
   - 前端轮询/WebSocket

预计算流程：
  1. 定时任务（凌晨）扫描当日数据
  2. 聚合计算（按维度）
  3. 写入报表汇总表
  4. 前端查询汇总表
```

### 2.6 导出格式

```
Excel（POI/SXSSF）：
  - 适合结构化数据
  - SXSSF 流式写入（避免 OOM）
  - 支持样式、公式

CSV：
  - 适合大数据量
  - 体积小、写入快
  - 无样式

PDF：
  - 适合正式报表
  - 模板渲染
  - 体积较大
```

### 2.7 踩坑

1. **同步导出大数据** → OOM/超时，必须异步
2. **不分页查询** → 一次性查百万行，OOM
3. **文件不过期** → 存储爆炸，设 TTL
4. **无进度反馈** → 用户不知道导出到哪了
5. **并发导出** → 同用户多任务，资源浪费

## 3. 批量操作设计

批量操作是运营高频需求：批量上架、批量改价、批量发券、批量删除。

### 3.1 批量操作类型

| 类型 | 说明 | 风险 |
|------|------|------|
| 批量更新 | 批量改状态/字段 | 误操作影响大 |
| 批量删除 | 批量删数据 | 不可恢复 |
| 批量导入 | Excel 批量导入 | 数据校验 |
| 批量发放 | 批量发券/发消息 | 资源消耗 |
| 批量上下架 | 商品批量操作 | 影响售卖 |

### 3.2 批量更新

```
方案对比：

1. 一条条更新
   - 简单但慢
   - 不推荐

2. 批量 UPDATE
   UPDATE product SET status=1 WHERE id IN (1,2,3,...)
   - 一次 DB 交互
   - 推荐小批量

3. 异步任务
   - 大批量拆分 → MQ → 多消费者并行
   - 适合超大批量

  详见 [04 §16 性能优化](../03-分布式架构与设计/00-分布式系统核心设计模式.md#16-性能优化方法论)
```

### 3.3 批量导入

```
批量导入流程：

1. 上传 Excel 文件
2. 解析文件（POI）
3. 数据校验
   - 格式校验
   - 业务校验（如商品是否存在）
   - 唯一性校验
4. 分批写入
   - 每批 1000 条
   - 事务保证每批原子
5. 记录成功/失败
6. 返回导入结果

关键点：
  - 文件大小限制（如 10MB）
  - 分批处理（避免大事务）
  - 错误处理（部分成功 + 错误报告）
  - 幂等（重复导入处理）
```

### 3.4 批量操作的事务策略

```
问题：批量操作 1 万条，要不要一个事务？

方案 1：整批一个事务
  - 全成功或全失败
  - 大事务锁表、undo log 膨胀
  - 不推荐大批量

方案 2：分批小事务
  - 每批 100-1000 条一个小事务
  - 部分成功部分失败
  - 记录失败批次，可重试

方案 3：先校验后执行
  1. 全量校验（不写库）
  2. 校验通过 → 分批写入
  3. 校验失败 → 返回错误，不执行
```

### 3.5 批量操作的并发控制

```
问题：运营同时发起多个批量操作 → 资源冲突

方案：
  1. 操作互斥（分布式锁）
     - 同类型操作串行
     - 防止冲突
  
  2. 资源限流
     - 限制并发批量操作数
     - 保护 DB
  
  3. 操作日志
     - 记录谁、何时、改了什么
     - 可追溯、可回滚
```

### 3.6 踩坑

1. **大事务** → 锁表，分批小事务
2. **无校验** → 脏数据导入
3. **无进度** → 用户不知道执行到哪
4. **不可回滚** → 批量删错无法恢复，需软删除 + 审计
5. **并发冲突** → 多人同时改，需互斥

## 4. 异步任务管理

异步任务（非定时任务）是业务系统常见需求：导入导出、批量发券、数据同步。

### 4.1 异步任务 vs 定时任务

| 维度 | 异步任务 | 定时任务 |
|------|----------|----------|
| 触发 | 用户/业务触发 | 时间触发 |
| 频率 | 不固定 | 固定 |
| 数据 | 特定数据 | 批量数据 |
| 管理 | 需进度/取消 | 自动执行 |
| 工具 | 自研/MQ | XXL-Job 等 |

详见 [07 分布式定时任务](../05-稳定性与可观测/00-分布式定时任务与调度.md) 了解定时任务。

### 4.2 异步任务模型

```
异步任务（async_task）
  - task_id：任务 ID
  - task_type：类型（导入/导出/发券...）
  - user_id：发起人
  - status：状态（PENDING/RUNNING/SUCCESS/FAILED/CANCELLED）
  - progress：进度（0-100）
  - params：任务参数（JSON）
  - result：结果（JSON，如文件 URL）
  - error_msg：错误信息
  - create_time / start_time / end_time
  - expire_time：任务过期时间
```

### 4.3 任务状态机

```
PENDING（待执行）
  ↓ 消费者拉取
RUNNING（执行中）
  ↓ 执行完成
SUCCESS（成功）
  ↓ 执行失败
FAILED（失败）→ 可重试
  ↓ 用户取消
CANCELLED（已取消）

超时处理：
  - RUNNING 超过阈值 → 标记超时 → 可重试或失败
```

### 4.4 任务执行架构

```
架构：
  业务 → 创建任务 → MQ → 消费者执行 → 更新状态

  1. 业务创建任务（写 DB，状态 PENDING）
  2. 发 MQ 消息
  3. 消费者接收 → 状态 RUNNING
  4. 执行任务
     - 分批处理
     - 更新进度
  5. 完成 → 状态 SUCCESS + 结果
  6. 失败 → 状态 FAILED + 错误

任务分配：
  - 按任务类型分队列
  - 多消费者并行
  - 任务幂等（防重复执行）
```

### 4.5 进度反馈

```
进度反馈方案：

1. 轮询（简单）
   - 前端定时查任务状态
   - 每 2-3 秒一次
   - 适合大部分场景

2. WebSocket（实时）
   - 服务端推送进度
   - 适合需要实时反馈的场景

3. SSE（Server-Sent Events）
   - 单向推送
   - 比 WebSocket 简单

进度计算：
  - progress = processed / total * 100
  - 更新频率：每处理 10% 更新一次（避免频繁写 DB）
```

### 4.6 任务取消

```
取消流程：
  1. 用户点取消 → 状态 CANCEL_REQUESTED
  2. 消费者检查取消标记
     - 定期检查（每处理 N 条检查一次）
     - 发现取消 → 停止处理
  3. 状态 → CANCELLED
  4. 清理已产生的部分结果

关键：
  - 消费者要主动检查取消标记
  - 不能强行 kill（资源泄漏）
  - 已处理的数据需考虑回滚/保留
```

### 4.7 任务超时

```
超时处理：
  1. 任务设置最大执行时间
  2. 定时任务扫描超时任务
  3. 超时 → 标记 TIMEOUT
  4. 可重试或失败

实现：
  - 创建任务时设 expire_time
  - 定时任务扫描 RUNNING 且超过 expire_time 的任务
  - 更新状态为 TIMEOUT
```

### 4.8 任务重试

```
重试策略：
  - 自动重试：失败后自动重试（限 3 次）
  - 手动重试：用户点重试

重试注意：
  - 任务必须幂等
  - 重试间隔指数退避
  - 重试次数限制
  - 死信队列兜底
```

### 4.9 踩坑

1. **任务不幂等** → 重试导致重复执行
2. **无进度反馈** → 用户不知道状态
3. **无法取消** → 长任务只能干等
4. **任务丢失** → MQ 丢消息，需 DB 兜底
5. **资源泄漏** → 任务取消但资源未释放
6. **无限重试** → 重试无限制，需设上限

## 5. 监控指标体系

### 5.1 风控系统监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 风控判断 RT | 风控接口响应时间 | > 50ms 告警 |
| 拦截率 | 被拦截/总请求 | 异常波动告警 |
| 误判投诉量 | 用户申诉数 | > 阈值评估规则 |
| 黑名单命中率 | 命中黑名单/总请求 | 监控趋势 |
| 规则触发 TOP | 触发最多的规则 | 定期 review |
| 风控可用性 | 风控服务可用率 | < 99.9% 告警 |

### 5.2 数据导出/报表监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 导出成功率 | 导出完成/导出请求 | < 95% 告警 |
| 导出耗时 | 单任务耗时 | > 30min 告警 |
| 报表生成延迟 | 报表数据延迟 | T+1 报表 > 8h 告警 |
| 导出任务积压 | 待处理任务数 | > 100 告警 |
| 文件存储用量 | 导出文件占用 | > 80% 预警 |

### 5.3 批量操作监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 批量操作成功率 | 成功/总操作 | < 99% 告警 |
| 批量操作耗时 | 单批次耗时 | 超预期告警 |
| 部分失败率 | 失败条数/总条数 | > 1% 告警 |
| 并发批量数 | 同时执行的任务数 | > 阈值限流 |

### 5.4 异步任务监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 任务成功率 | 成功/总任务 | < 95% 告警 |
| 任务执行时长 | 平均/P99 耗时 | 超预期告警 |
| 任务积压量 | PENDING 状态数 | > 1000 告警 |
| 任务超时数 | 超时任务数 | > 0 告警 |
| 重试次数 | 平均重试次数 | > 2 评估 |
| 死信任务数 | 进入死信的任务 | > 0 告警 |
