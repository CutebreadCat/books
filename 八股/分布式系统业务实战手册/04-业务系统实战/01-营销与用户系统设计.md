## 1. 优惠券系统设计

优惠券是营销的核心工具，看似简单实则陷阱众多。

### 1.1 业务模型

```
优惠券模板（coupon_template）
  - template_id：模板 ID
  - name：优惠券名
  - type：类型（满减/折扣/立减/兑换券）
  - denomination：面额（满减10元 / 折扣8折）
  - threshold：使用门槛（满 100 元可用）
  - scope：适用范围（全场/指定商品/指定分类）
  - total_count：发放总量
  - issued_count：已发放量
  - start_time / end_time：有效期
  - status：状态（未开始/进行中/已结束）

用户优惠券（user_coupon）
  - id
  - user_id
  - template_id
  - coupon_code：券码（唯一）
  - status：状态（未使用/已使用/已过期/已锁定）
  - get_time：领取时间
  - used_time：使用时间
  - order_id：使用的订单号
```

**优惠券类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 满减 | 满X减Y | 满100减20 |
| 折扣 | 打X折 | 8折优惠 |
| 立减 | 无门槛减 | 立减5元 |
| 兑换券 | 兑换商品 | 兑换1杯咖啡 |
| 运费券 | 免运费 | 包邮券 |

### 1.2 优惠券状态机

```
UNUSED（未使用）
  ↓ 锁定（下单时）
LOCKED（已锁定）
  ↓ 支付成功
USED（已使用）
  ↓ 订单取消/超时
UNUSED（回退）
  ↓ 过期
EXPIRED（已过期）

关键：
  - 锁定防并发使用
  - 订单取消回退优惠券
  - 过期不可用
```

### 1.3 优惠券发放

```
发放方式：
  - 主动领取：用户在活动页领取
  - 系统发放：满赠/新人券自动发
  - 兑换码：输入码兑换

防超发（高并发领取）：
  Redis 原子操作（Lua）：
    1. 检查已发放量 < 总量
    2. 检查用户领取次数限制
    3. INCR issued_count
    4. 记录用户领取

  限制策略：
    - 总量限制
    - 每人限领 N 张
    - 每日限领 N 张

异步落库：
  Redis 领取成功 → 发 MQ → 异步写 MySQL
```

### 1.4 优惠券核销

```
下单时核销流程：
  1. 校验优惠券
     - 属于当前用户
     - 状态为 UNUSED
     - 在有效期内
     - 满足使用门槛
     - 适用范围匹配
  2. 锁定优惠券 → 状态 LOCKED
  3. 计算优惠金额
  4. 关联订单号

订单取消时：
  - 优惠券回退 → 状态 UNUSED
  - 清除订单关联

核销幂等：
  - 优惠券状态机保证
  - UPDATE user_coupon SET status='USED' WHERE id=? AND status='LOCKED'
```

### 1.5 优惠金额计算

```
叠加规则（复杂）：
  - 互斥：同类优惠券不可叠加
  - 叠加：不同类可叠加（如店铺券+平台券）
  - 上限：优惠总额不超过商品金额

计算顺序：
  1. 商品级优惠（单品折扣）
  2. 店铺级优惠（店铺满减）
  3. 平台级优惠（平台券）
  4. 运费优惠（运费券）

分摊（拆单场景）：
  - 优惠按比例分摊到各子订单
  - 退款时按分摊金额退
  - 精度处理（分，整数）

示例：
  订单 200 元，优惠券满 200 减 50
  拆 2 单：A 120 元，B 80 元
  分摊：A 减 30，B 减 20
  A 退款 → 退 120-30=90
```

### 1.6 适用范围控制

```
范围类型：
  - 全场通用
  - 指定分类（如服装类）
  - 指定商品（指定 SPU/SKU）
  - 指定店铺
  - 排除商品（全场 except 某些）

实现：
  - 模板配置 scope_type + scope_ids
  - 下单时校验商品是否在范围内
  - 用 Redis Set 缓存范围，加速校验
```

### 1.7 踩坑

1. **超发** → 并发领取未原子控制
2. **重复使用** → 状态机不严格，并发下单用同一券
3. **优惠计算错** → 叠加/分摊逻辑错
4. **回退丢失** → 订单取消但优惠券没回退
5. **范围校验漏** → 不满足范围的商品用了券
6. **过期时间错** → 时区/边界问题

## 2. 营销活动系统设计

营销活动是促单拉新的核心手段，类型多样、规则复杂。

### 2.1 活动类型

| 类型 | 说明 | 技术难点 |
|------|------|----------|
| 秒杀 | 限时限量抢购 | 超高并发、防超卖 |
| 拼团 | 凑人数享低价 | 拼团状态、超时退款 |
| 砍价 | 砍到0元拿 | 防刷、进度计算 |
| 预售 | 定金+尾款 | 两阶段支付 |
| 限时购 | 限时特价 | 时间切换、库存预热 |
| 满赠 | 满X送Y | 赠品库存 |
| 新人专享 | 首单优惠 | 新人判断 |

### 2.2 拼团系统设计

```
拼团模型：
  团（group）
    - group_id
    - activity_id：活动
    - leader_id：团长
    - target_count：成团人数
    - current_count：当前人数
    - status：待成团/已成团/已失败
    - expire_time：成团截止

  团成员（group_member）
    - group_id
    - user_id
    - order_id：参团订单
    - join_time

拼团流程：
  1. 团长开团 → 创建团（待成团）
  2. 团长支付
  3. 邀请好友参团
  4. 成员支付
  5. 人满成团 → 通知发货
  6. 超时未满 → 自动退款

状态流转：
  PENDING（待成团）→ SUCCESS（已成团）
                 → FAILED（超时失败）→ 退款

关键技术：
  - 参团原子性（防超员）：Lua 原子 INCR
  - 超时退款：延迟队列 + 定时任务
  - 并发开团：分布式锁
```

### 2.3 预售系统设计

```
预售两阶段：
  定金阶段：
    1. 用户付定金（如 100 元）
    2. 记录预售订单
    3. 定金不退（或按规则退）

  尾款阶段：
    1. 尾款期开始 → 通知用户付尾款
    2. 用户付尾款
    3. 发货
    4. 尾款期未付 → 关闭订单（定金不退）

时间节点：
  - 定金开始/结束
  - 尾款开始/结束
  - 用定时任务触发状态变更
```

### 2.4 活动库存管理

```
活动库存独立于普通库存：
  - 活动商品有独立活动库存
  - 活动结束未售 → 回到普通库存
  - 活动库存预热（开始前同步到 Redis）

预热流程：
  1. 活动开始前 10 分钟
  2. 从 MySQL 同步活动库存到 Redis
  3. 停止下单（过渡期）
  4. 活动开始 → 开放下单

防超卖：
  - Redis 原子扣减
  - 详见 [01 §9](../01-基础存储与中间件/00-Redis业务场景实战.md#9-库存扣减--高并发抢购)
```

### 2.5 防刷设计

```
营销活动是黄牛重灾区：

防刷手段：
  1. 设备指纹：识别同一设备多账号
  2. 行为分析：异常频率（如 1 秒领 10 张）
  3. 限制策略：IP/设备/用户多维度限频
  4. 验证码：高风险操作加验证
  5. 黑名单：已知黄牛账号
  6. 风控系统：详见 [12 风控系统](./03-风控与数据系统设计.md)

实现：
  - 限频：Redis 滑动窗口
  - 设备指纹：前端采集 + 后端校验
  - 黑名单：Redis Set
```

### 2.6 踩坑

1. **活动库存与普通库存混淆** → 超卖或漏卖
2. **拼团超员** → 参团未原子控制
3. **预售尾款未付不退款** → 规则未明确告知用户
4. **防刷过严** → 误伤正常用户
5. **活动切换时间错** → 活动未开始就生效，或结束后仍可下单

## 3. 用户系统设计

用户系统是身份基石，涉及注册、登录、画像、关系链。

### 3.1 用户模型

```
用户主表（user）
  - user_id：用户 ID
  - nickname：昵称
  - avatar：头像
  - phone：手机号（加密存储）
  - email：邮箱
  - password：密码（bcrypt 加盐）
  - status：状态（正常/冻结/注销）
  - register_time：注册时间
  - register_source：注册来源

用户第三方账号（user_oauth）
  - user_id
  - oauth_type：weixin/qq/weibo
  - oauth_id：第三方唯一 ID
  - unionid：微信 unionid

用户信息扩展（user_profile）
  - user_id
  - gender：性别
  - birthday：生日
  - region：地区
  - tags：标签（JSON）
```

### 3.2 注册流程

```
手机号注册：
  1. 获取验证码
     - 校验手机号格式
     - 限频（同手机号 60s 一次，同 IP 每日 10 次）
     - 生成验证码存 Redis（5 分钟过期）
     - 调用短信服务发送
  2. 提交注册
     - 校验验证码
     - 检查手机号是否已注册
     - 创建用户（手机号唯一索引兜底）
     - 发新手券/新人任务

第三方注册/登录：
  1. 前端获取第三方授权码
  2. 后端用授权码换 access_token
  3. 用 access_token 获取用户信息
  4. 查 user_oauth
     - 已绑定 → 直接登录
     - 未绑定 → 首次需绑定手机号

防刷注册：
  - 验证码限频
  - 同 IP 注册限频
  - 设备指纹
  - 详见 §2.5 防刷
```

### 3.3 登录设计

```
登录方式：
  - 密码登录
  - 验证码登录
  - 第三方登录
  - 扫码登录

登录流程（详见 [01 §8 会话管理](../01-基础存储与中间件/00-Redis业务场景实战.md#8-会话管理--登录态与token)）：
  1. 校验账号密码
  2. 生成 Token（JWT）
  3. 存 Redis（user:token:<user_id>）
  4. 返回 Token

安全设计（详见 [04 §15](../03-分布式架构与设计/00-分布式系统核心设计模式.md#15-分布式安全设计)）：
  - 密码 bcrypt 加盐
  - 登录失败限频（防爆破）
  - 异地登录提醒
  - Token 双 token 续期
```

### 3.4 用户画像与标签

```
标签来源：
  - 基础标签：性别、年龄、地区
  - 行为标签：购买偏好、浏览历史
  - 计算标签：RFM（最近购买、频率、金额）

标签存储：
  - 用户标签表（user_id, tag_key, tag_value）
  - 或 Redis Hash（user:tags:<user_id>）

应用场景：
  - 个性化推荐
  - 精准营销
  - 用户分群
```

### 3.5 用户关系链

```
关系类型：
  - 关注（单向）：A 关注 B
  - 好友（双向）：A 和 B 互为好友

存储：
  关注关系：following:<user_id>（Set，存关注的人）
  粉丝关系：follower:<user_id>（Set，存粉丝）

  或 MySQL：
    user_relation(user_id, target_id, type, create_time)
    索引：(user_id, type), (target_id, type)

查询：
  - 关注列表：SMEMBERS following:<user_id>
  - 粉丝列表：SMEMBERS follower:<user_id>
  - 是否互关：SISMEMBER + SISMEMBER
  - 共同关注：SINTER following:<A> following:<B>

大 V 粉丝量：
  - 千万级粉丝 → Set 内存大
  - 方案：只存关注关系（粉丝用反向查询或离线计算）
```

### 3.6 踩坑

1. **手机号未加密** → 隐私泄露，需加密存储
2. **密码明文** → 必须 bcrypt 加盐
3. **Token 无法注销** → 黑名单机制
4. **第三方登录绑定错** → 同一第三方账号绑到多个用户
5. **验证码可爆破** → 限频 + 一次性
6. **关系链存储膨胀** → 大 V 粉丝用 Set 内存爆炸

## 4. 消息通知系统设计

消息通知是用户触达的关键通道，看似简单，多通道 + 频控 + 降噪是难点。

### 4.1 通知类型

| 通道 | 说明 | 适用场景 |
|------|------|----------|
| 站内信 | App 内消息 | 订单状态、系统公告 |
| Push | 手机推送 | 营销活动、订单提醒 |
| 短信 | 手机短信 | 验证码、重要通知 |
| 邮件 | 电子邮件 | 报表、正式通知 |
| 微信 | 公众号/小程序消息 | 服务通知 |
| 企微 | 企业微信通知 | B 端通知 |

### 4.2 通知模型

```
通知模板（notify_template）
  - template_id
  - channel：通道
  - code：模板编码（如 order_paid）
  - title：标题
  - content：内容（含变量占位）
  - variables：变量定义

通知记录（notify_record）
  - record_id
  - user_id
  - template_id
  - channel
  - content：实际内容
  - status：待发送/已发送/失败
  - send_time
  - create_time
```

### 4.3 发送流程

```
发送流程：
  1. 业务触发通知 → 发 MQ
  2. 通知服务消费 MQ
  3. 渲染模板（填充变量）
  4. 频控检查（见 §4.4）
  5. 降噪检查（见 §4.5）
  6. 调用通道接口发送
  7. 记录发送结果
  8. 失败重试

异步发送：
  - 全程异步（MQ 解耦）
  - 业务不阻塞
  - 失败不影响业务
```

### 4.4 频控设计

```
频控维度：
  - 用户维度：同用户某通道每日上限
  - 模板维度：同模板同用户每日上限
  - 全局维度：防止短信轰炸

实现（Redis 滑动窗口）：
  - key: freq:<user_id>:<channel>:<date>
  - INCR + EXPIRE
  - 超限则不发

示例规则：
  - 营销短信：同用户每日 3 条
  - Push：同用户每日 10 条
  - 订单通知：不限制（重要通知）
```

### 4.5 降噪设计

```
问题：短时间内大量同类通知打扰用户

降噪策略：
  1. 合并：同类型通知合并
     - 5 分钟内 3 条评论 → 合并为"有3条新评论"
  2. 聚合：按时间窗口聚合
     - 1 小时内的点赞聚合成一条
  3. 智能抑制：
     - 用户未读过多 → 降低发送频率
     - 用户已读 → 恢复频率

实现：
  - 通知进入"待聚合"队列
  - 定时任务聚合后发送
```

### 4.6 通道选择与降级

```
通道选择：
  - 按通知重要性选通道
  - 重要通知：短信 + Push
  - 一般通知：Push + 站内信
  - 营销通知：Push（避免短信成本）

降级策略：
  - Push 发送失败 → 短信兜底
  - 短信发送失败 → 站内信
  - 通道不可用 → 切换备用通道

可达性管理：
  - Push：维护设备 token，失效则清理
  - 短信：维护号码有效性
  - 微信：维护 openid
```

### 4.7 踩坑

1. **无频控** → 用户被骚扰、投诉
2. **同步发送** → 阻塞业务，必须异步
3. **无降级** → 通道故障通知丢失
4. **模板硬编码** → 改文案要改代码，用模板
5. **发送结果不记录** → 无法排查"为什么没收到"
6. **验证码通道混用** → 营销短信拉低验证码到达率

## 5. 监控指标体系

### 5.1 优惠券系统监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 券领取成功率 | 成功领取/尝试领取 | < 95% 告警 |
| 超发次数 | 发放量超过总量 | > 0 立即告警 |
| 核销成功率 | 核销成功/核销尝试 | < 99% 告警 |
| 优惠金额异常 | 优惠金额计算错误 | > 0 告警 |
| 券回退失败率 | 订单取消券未回退 | > 0.1% 告警 |

### 5.2 营销活动监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 活动库存超卖 | 活动库存 < 0 | > 0 立即告警 |
| 拼团成功率 | 成团/开团数 | < 50% 评估 |
| 活动接口 QPS | 活动接口流量 | 超容量告警 |
| 风控拦截率 | 被拦截/总请求 | > 10% 评估规则 |

### 5.3 用户系统监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 注册成功率 | 成功注册/尝试 | < 90% 告警 |
| 登录成功率 | 成功登录/尝试 | < 95% 告警 |
| 登录 RT | 登录响应时间 | > 500ms 告警 |
| 异常登录次数 | 撞库/爆破尝试 | > 阈值告警 |
| Token 验证失败率 | 验证失败/总验证 | > 1% 告警 |

### 5.4 消息通知监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
