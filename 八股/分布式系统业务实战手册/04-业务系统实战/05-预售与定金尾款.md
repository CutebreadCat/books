# 预售与定金尾款

> 预售是电商大促的标志性玩法（双 11/618 核心），与现货交易有本质差异：库存预占从"分钟级"扩展到"天级"，资金流转从"一次性"变成"两阶段"。面试中如果只有普通订单设计，没有预售设计，会被认为缺少真实业务深度。

## 1. 预售的业务本质

```
预售 vs 现货的核心差异：

维度          现货                    预售
------------ ----------------------  ----------------------
库存行为      即时扣减（下单即扣）     预占（先锁，发货时才扣）
资金流转      一次性支付全款          定金 + 尾款 两阶段
价格确定      下单时价格确定           预售价（可能低于现货价）
发货时间      付款后 1-3 天发货        约定时间统一发货（如 15 天后）
退款规则      常规退款                定金规则特殊（通常不可单独退）
超卖风险      下单时扣减，直接防超卖    预占≠实扣，需双段管理

为什么要有预售？
  1. 商家侧：提前锁定订单量，按需生产/采购，降低库存风险
  2. 平台侧：制造大促氛围，拉长活动周期
  3. 用户侧：更低价格，但需等待
```

## 2. 预售模型与状态机

### 2.1 预售订单模型

```
预售订单（pre_order）独立建表，还是与普通订单共用？

方案对比：
  独立表：预售订单单独 pre_order 表
    - 优点：字段清晰，查询快，预售规则独立演进
    - 缺点：需要维护两套订单逻辑

  统一表：订单表增加 is_pre_sale 标记
    - 优点：一套订单逻辑，查询方便
    - 缺点：预售特有的字段（如尾款支付时间）为空，表结构膨胀

推荐：统一表 + 预售扩展表（1:1）
  主订单表（order）保存通用字段
  预售扩展表（pre_order_ext）保存预售特有字段

主订单表（order）：
  order_id, user_id, status, total_amount, pay_amount, is_pre_sale, ...

预售扩展表（pre_order_ext）：
  order_id（主键，关联 order）
  pre_sale_type: 定金预售 / 全款预售
  deposit_amount: 定金金额
  final_payment_amount: 尾款金额（= total - deposit）
  deposit_pay_time: 定金支付截止时间
  final_payment_start_time: 尾款开始时间
  final_payment_end_time: 尾款截止时间
  estimated_ship_time: 预计发货时间
  deposit_status: 待支付/已支付/已取消
  final_payment_status: 未开始/待支付/已支付/已超时
```

### 2.2 预售状态机（定金预售）

```
完整状态流转：

【阶段一：定金】
INIT（预售下单）
  ↓ 用户支付定金
DEPOSIT_PAID（定金已付）
  ↓ 到达尾款开始时间，用户支付尾款
FULLY_PAID（全款付清）
  ↓ 商家发货
SHIPPED（已发货）
  ↓ 用户确认收货
COMPLETED（已完成）

异常分支：
  ↓ 定金超时未支付
DEPOSIT_CANCELLED（定金订单取消）
  ↓ 尾款超时未支付
FINAL_PAYMENT_TIMEOUT（尾款超时）→ 退款处理
  ↓ 用户申请退款
REFUNDING（退款中）→ REFUNDED（已退款）

与普通订单状态的区别：
  普通订单：PENDING_PAY → PAID → SHIPPED → COMPLETED
  预售订单：INIT → DEPOSIT_PAID → FULLY_PAID → SHIPPED → COMPLETED
  多了一个"定金支付"的中间状态
```

## 3. 核心流程设计

### 3.1 定金下单流程

```
流程：
  1. 用户浏览预售商品 → 显示"预售价"、"定金 XX 元"、"尾款支付时间"
  2. 用户提交定金订单
     - 创建主订单（order），status = INIT，is_pre_sale = true
     - 创建预售扩展表（pre_order_ext）
     - 预占库存（Redis 预占，但有效期长：如 3 天）
  3. 用户支付定金
     - 创建支付单（定金支付）
     - 调用第三方支付（微信/支付宝）
     - 支付回调 → 更新 order.status = DEPOSIT_PAID
     - 更新 pre_order_ext.deposit_status = PAID
  4. 写延迟队列（定金超时检查：如 30 分钟后）

注意：
  - 定金支付时，尾款支付时间尚未到达（可能几天后）
  - 用户可能支付了定金，但忘记支付尾款
```

### 3.2 尾款支付流程

```
流程：
  1. 到达尾款开始时间（如 11 月 10 日 20:00）
  2. 系统发通知：提醒用户支付尾款（短信/ Push/ 微信）
  3. 用户进入订单列表，点击"支付尾款"
     - 校验：当前时间是否在尾款支付窗口内
     - 校验：订单状态是否为 DEPOSIT_PAID
  4. 创建尾款支付单
     - 支付金额 = final_payment_amount
     - 支付方式：用户可选（支持定金支付方式外的其他方式）
  5. 支付回调
     - 更新 order.status = FULLY_PAID
     - 更新 pre_order_ext.final_payment_status = PAID
     - 触发发货流程（按预售发货时间统一发货）

特殊场景：
  1. 定金支付后，商品涨价了怎么办？
     - 预售价是锁定的，尾款按预售价计算，不受现货价格变化影响
  2. 定金支付后，商品降价了怎么办？
     - 预售价同样锁定，但平台可能补偿优惠券
  3. 定金支付后，用户不想要了
     - 定金规则：通常定金不可退（如"定金膨胀，定金不退"）
     - 但尾款未支付时，整体订单可取消
```

### 3.3 定金超时处理

```
定时任务：
  每 5 分钟扫描：
    SELECT * FROM order
    WHERE is_pre_sale = true
      AND status = INIT
      AND create_time < now() - 30分钟

处理：
  1. 取消订单：status = DEPOSIT_CANCELLED
  2. 释放预占库存：Redis INCR + 预占记录删除
  3. 记录取消原因：定金超时未支付

注意：
  定金超时时间通常较短（30 分钟 ~ 2 小时），制造紧迫感
```

### 3.4 尾款超时处理

```
定时任务：
  每 10 分钟扫描：
    SELECT * FROM order o JOIN pre_order_ext p ON o.order_id = p.order_id
    WHERE p.final_payment_status = 'NOT_STARTED'  -- 或 PENDING
      AND p.final_payment_end_time < now()

处理：
  1. 标记尾款超时：p.final_payment_status = TIMEOUT
  2. 订单取消：o.status = CANCELLED
  3. 释放预占库存
  4. 定金退款：虽然规则是"定金不退"，但尾款超时后通常整单退款
     - 退定金金额到用户原支付账户
     - 退款原因：尾款超时未支付
```

## 4. 库存预占的时间维度

```
预售库存与普通库存的关键区别：预占时间跨度长。

现货库存预占：
  下单 → 预占（30 分钟）→ 支付 → 实扣
  → 30 分钟不支付，自动释放

预售库存预占：
  付定金 → 预占（3 天）→ 付尾款 → 发货时实扣
  → 预占时间从 30 分钟变成 3 天，库存管理复杂度大增

库存模型设计：

  可售库存（sellable）= 实际库存（physical） - 现货预占（locked_spot） - 预售预占（locked_pre） - 安全库存

  预占表设计：
    CREATE TABLE stock_lock (
      lock_id BIGINT PRIMARY KEY,
      sku_id BIGINT,
      order_id VARCHAR(64),
      lock_type ENUM('SPOT', 'PRE_SALE'),  -- 现货预占 vs 预售预占
      quantity INT,
      expire_time DATETIME,               -- 预占到期时间
      status ENUM('LOCKED', 'RELEASED', 'CONVERTED'),
      create_time DATETIME
    );

  状态含义：
    LOCKED：已预占，未释放
    RELEASED：已释放（超时取消/主动取消）
    CONVERTED：已转为实扣（发货时）

预售预占 vs 现货预占的对比：

  | 维度 | 现货预占 | 预售预占 |
  |------|----------|----------|
  | 触发时机 | 下单时 | 付定金时 |
  | 有效期 | 30 分钟 | 3 天（定金到尾款） |
  | 释放条件 | 超时未支付 | 尾款超时 / 主动取消 |
  | 转实扣 | 支付成功时 | 发货时 |
  | 库存占用 | 短，影响小 | 长，需单独管理 |
```

**预售预占的库存管理策略**：

```
1. 预售库存单独分配
   总库存 1000 件：
     - 预售分配：500 件（预售期间可售）
     - 现货分配：500 件（现货期间可售）
     - 预售结束后，剩余预售库存转现货库存

2. 预售库存的 Redis 预占
   付定金时：DECR pre_stock:sku_id（预售库存 Key）
   发货时：预占转实扣（MySQL 实际扣减）
   注意：预售库存和现货库存用不同的 Redis Key，避免混淆

3. 预占过期释放
   预售预占有效期长（3 天），不能用 Redis 简单 TTL
   方案：
     - 预占记录表 + 定时任务扫描过期预占
     - 或者 Redis TTL 设为 3 天，但需兜底定时任务（Redis 持久化风险）
```

## 5. 定金与尾款的资金流转

```
资金流转的复杂性：定金和尾款是两次独立的支付，但属于同一笔订单。

支付流水设计：
  pay_flow 表需要关联订单 + 支付阶段：
    flow_id, pay_id, order_id, pay_stage（DEPOSIT / FINAL）, amount, ...

退款规则：
  1. 定金阶段取消（定金未支付）：
     - 无资金操作，直接取消
  2. 定金已付，尾款未开始：
     - 用户取消 → 退定金（全额退）
  3. 定金已付，尾款已付，未发货：
     - 用户退款 → 退全款（定金 + 尾款）
  4. 尾款超时：
     - 退定金（虽然规则是定金不退，但尾款超时属于平台责任，通常全退）
  5. 已发货：
     - 按正常退款流程（退货退款 / 仅退款）

"定金膨胀"的账务处理：
  场景：定金 10 元膨胀为 30 元，尾款只需支付 70 元（原价 100 元）
  用户视角：付定金 10 元 + 尾款 70 元 = 实付 80 元
  商家视角：订单金额 100 元，但用户只付了 80 元，平台补贴 20 元？
  实际：定金膨胀是营销优惠，膨胀部分由平台或商家承担
  账务记录：
    - 定金：10 元（用户实付）
    - 尾款：70 元（用户实付）
    - 优惠：20 元（平台/商家补贴）
    - 订单总价：100 元
```

## 6. 预售与普通现货的冲突处理

### 6.1 预售价与现货价的冲突

```
场景：
  用户 A 预付定金时，预售价 100 元（定金 10 + 尾款 90）
  用户 B 直接购买现货，现货价 120 元
  商品库存总共 100 件，预售分配 50 件，现货分配 50 件

冲突：
  1. 预售库存卖完了，但现货还有 → 用户 A 预占了库存，用户 B 可买现货
  2. 现货库存卖完了，用户想参加预售但预售也满了 → 显示"已售罄"
  3. 预售结束，预售库存没卖完 → 自动转现货库存

处理：
  库存分配：
    上架时：设置预售库存数（如 50）和现货库存数（如 50）
    预售期间：预售库存和现货库存独立扣减
    预售结束：预售剩余库存 → 加入现货库存
```

### 6.2 预售商品与普通商品在购物车共存

```
问题：购物车里有 1 个预售商品 + 2 个现货商品，用户点击结算

方案：
  1. 分单结算：预售商品和现货商品分开结算
     - 预售商品走预售流程（先付定金）
     - 现货商品走现货流程（直接支付全款）
  2. 合并结算：统一支付（不推荐，流程差异大）

推荐：分单结算
  购物车结算时：
    - 按商品类型拆分为多个子订单
    - 预售商品：生成预售订单，引导支付定金
    - 现货商品：生成普通订单，直接支付全款
  用户感知：
    - 购物车点击结算 → 看到两个订单：预售订单 + 现货订单
    - 可以只支付现货订单，预售订单单独处理
```

### 6.3 预售尾款支付时库存不足

```
极端场景：
  用户付了定金，预占了库存
  但商家操作失误，实际库存不足（如仓库盘点发现缺货）
  用户支付尾款时，发现无法发货

处理：
  1. 平台兜底：如果是因为商家原因，平台强制商家补货或赔偿
  2. 用户选择：
     - 等待补货（商家承诺发货时间）
     - 取消订单，全额退款（定金 + 尾款）+ 补偿优惠券
     - 更换同款商品（如果有库存）
  3. 系统机制：
     - 发货前再次校验库存（如果库存不足，拦截发货）
     - 触发退款流程 + 补偿流程
```

## 7. 踩坑与反模式

### 坑 1：预售库存和现货库存没隔离
```
问题：预售和现货共用同一个库存 Key
  → 预售预占了 50 件，现货用户看到还有 50 件，以为库存充足
  → 实际总库存 100，预售占了 50，现货只有 50
解决：Redis 分 Key：stock:spot:sku_id 和 stock:pre:sku_id
```

### 坑 2：定金超时释放后，没通知用户
```
问题：用户付了定金，但系统超时取消了，用户不知情
  → 用户几天后想付尾款，发现订单没了
解决：定金超时取消必须发通知（短信/ Push），告知原因和补救方式
```

### 坑 3：尾款支付时间窗口设计不合理
```
问题：尾款支付时间只有 1 小时，大量用户同时涌入支付
  → 系统被打挂，支付超时
解决：尾款支付窗口要足够长（24 小时），且支持分时段提醒
```

### 坑 4：定金膨胀的账务算不清
```
问题：定金 10 元膨胀为 30 元，用户退款时退 10 元还是 30 元？
  → 用户认为退 30 元（我付了 30 元的价值）
  → 商家认为退 10 元（我只收了 10 元）
解决：明确退款规则，定金膨胀部分属于优惠，退款只退实付金额
```

### 坑 5：预售订单状态机遗漏分支
```
问题：只设计了正常流程，没考虑"定金已付但尾款开始前取消"
  → 状态机没有对应的流转路径
解决：状态机必须覆盖所有分支，包括：
  定金超时、尾款超时、付定金后取消、付尾款后取消、已发货退款
```

### 坑 6：预售预占库存长期不释放
```
问题：Redis 设置了 3 天 TTL，但 Redis 宕机后数据丢失
  → 预占记录没了，但 MySQL 里还有预占记录
  → 库存不一致，实际超卖
解决：Redis TTL 只是第一层，必须配合定时任务扫描预占表兜底
```

## 8. 面试追问应对

**Q1：预售和普通订单有什么区别？**
> 核心区别在三个维度：1. 库存行为不同，预售是"长周期预占"（付定金时锁库存，发货时才实扣），现货是"即时扣减"；2. 资金流转不同，预售是"定金+尾款"两阶段，现货是一次性全款；3. 状态机不同，预售多了"定金已付"中间状态，尾款有独立的时间窗口。库存管理上，预售预占有效期长达几天，需要独立的预占表管理，不能简单用 Redis TTL。

**Q2：预售的库存怎么设计？**
> 预售库存和现货库存要分开管理。上架时给商品分配预售库存数（如 50）和现货库存数（如 50），各自独立扣减。预售期间，付定金时扣预售库存，用独立的 Redis Key（如 pre_stock:sku_id）。预售结束后，未售完的预售库存自动转现货库存。同时要维护预占表（stock_lock），记录每笔预占的到期时间，定时任务扫描释放过期预占。

**Q3：定金付了但尾款没付，定金退不退？**
> 看规则设计。通常有两种规则：1. "定金不退"（定金作为违约金，尾款超时只退尾款），但这用户体验差；2. "尾款超时全退"（定金+尾款都退），这是主流做法。从平台角度，尾款超时通常全退，因为用户无过错，平台要维护口碑。但定金主动取消（用户反悔）可以不退定金。这个规则必须在下单时明确告知用户。

**Q4：定金膨胀怎么账务处理？**
> 定金膨胀是营销优惠。用户付定金 10 元，膨胀为 30 元，尾款付 70 元，商品总价 100 元。账务上：用户实付 80 元（10+70），优惠 20 元由平台或商家承担。退款时，只退用户实付金额（80 元），优惠部分不退。膨胀金额在订单表里作为 discount_amount 记录，方便后续对账和结算。

**Q5：预售和现货在同一个购物车，怎么结算？**
> 分单结算。购物车结算时，按商品类型拆分为多个子订单：预售商品走预售流程（生成预售订单，先付定金），现货商品走现货流程（直接支付全款）。用户可以选择只支付现货订单，预售订单单独处理。不能合并为一个订单，因为支付流程和发货时间完全不同。
