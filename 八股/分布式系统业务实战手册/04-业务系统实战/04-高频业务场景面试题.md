## 场景1：秒杀系统设计

### 核心挑战

瞬时超高并发 + 库存有限 + 不能超卖 + 用户体验

### 完整方案

```
前端层：
  - 页面静态化 + CDN
  - 按钮防抖 + 答题验证
  - 前端随机丢弃部分请求（如随机 10% 直接返回"排队中"）

网关层：
  - 限流（令牌桶，全局 QPS 上限）
  - 黑名单过滤
  - 灰度分流

服务层：
  - Redis 预扣库存（Lua 原子操作）
  - MQ 异步下单（请求排队）
  - 本地缓存商品信息

数据库层：
  - 乐观锁兜底防超卖
  - 分库分表分散压力
  - 读写分离

降级预案：
  - Redis 不可用 → 纯 DB 乐观锁（慢但安全）
  - MQ 不可用 → 同步下单（慢但可用）
  - 流量超极限 → 静态页面"售罄"
```

### 关键数字参考

```
秒杀商品：100 件
预估并发：10 万 QPS
Redis 预扣：10 万次 DECR → 约 50ms 处理完
MQ 队列：10 万条消息 → 消费者逐步下单
最终下单：约 100 个成功订单（库存耗尽后快速失败）
```

### 加分点

- 提到前端层面的优化（CDN、防抖、丢弃）
- 提到限流的多层级（网关 + 应用 + 资源）
- 提到降级预案（Redis/MQ 故障时的兜底）
- 提到监控告警
- 提到防黄牛（设备指纹、行为分析）

## 场景2：点赞系统设计

### 核心挑战

- 高并发写入
- 防重复点赞
- 实时统计点赞数
- 查询点赞列表

### 完整方案

```
数据模型：
  MySQL: like_table(user_id, target_id, target_type, create_time)
         唯一索引：(user_id, target_id, target_type)
  Redis: 
    - like:count:<target_id> → 点赞数（STRING）
    - like:users:<target_id> → 点赞用户列表（ZSET，score=时间戳）

点赞流程：
  1. 检查是否已点赞（Redis SISMEMBER）
  2. 未点赞：
     a. MySQL 插入点赞记录（唯一索引兜底）
     b. Redis 点赞数 +1（INCR）
     c. Redis 用户列表添加（ZADD）
  3. 已点赞：取消点赞（反向操作）

查询点赞数：
  Redis GET like:count:<target_id>
  → 缓存未命中 → MySQL COUNT → 写 Redis

查询点赞列表：
  Redis ZREVRANGE like:users:<target_id> 0 20
```

### 高并发优化

```
写入优化：
  - 异步写入 MySQL（Redis 先记录，定时刷盘）
  - 批量写入（攒一批再写）

防重复：
  - Redis SISMEMBER 快速判断
  - MySQL 唯一索引兜底

数据一致性：
  - Redis 为主，MySQL 兜底
  - 定时对账
```

### 加分点

- 提到防重复点赞（唯一索引 + Redis 判断）
- 提到异步落库
- 提到热门内容的缓存策略
- 提到数据量大的分片方案

## 场景3：IM 系统设计

### 核心挑战

- 实时消息推送
- 消息有序
- 消息不丢
- 离线消息
- 群聊广播

### 完整方案

```
架构：
  客户端 ←WebSocket→ 接入层 → 消息服务 → 存储

消息发送：
  1. 客户端发消息到接入层
  2. 接入层路由到消息服务
  3. 消息服务：
     a. 生成消息 ID（全局唯一 + 递增）
     b. 写入存储（MySQL + Redis）
     c. 推送给在线接收方
     d. 离线方暂存

消息推送：
  - 在线：WebSocket 长连接推送
  - 离线：消息暂存 → 上线后拉取 + Push 通知

消息有序：
  - 单聊：按消息 ID 排序
  - 群聊：群内消息 ID 连续

消息不丢：
  - 客户端 ACK 机制
  - 服务端未收到 ACK 则重推
  - 客户端去重（消息 ID）
```

### 存储设计

```
MySQL：
  - 消息表（id, from_uid, to_uid, content, timestamp）
  - 按用户分库分表（如 user_id % 1024）

Redis：
  - 在线状态：user:online:<uid>
  - 未读数：user:unread:<uid>
  - 最近消息：user:recent:<uid>（List）
  - 消息 ID 生成器：INCR
```

### 加分点

- 提到消息可靠投递（ACK + 重传）
- 提到消息有序性
- 提到群聊的优化（写扩散 vs 读扩散）
- 提到消息加密
- 提到多端同步

## 场景4：短链系统设计

### 核心挑战

- 短码生成（短、唯一）
- 短链跳转（高并发读）
- 短链过期

### 完整方案

```
短码生成：
  方案 1：自增 ID + Base62 编码
    - ID = 1 → 短码 = "1"
    - ID = 10000 → 短码 = "2Bi"
    - 优点：短、有序
    - 缺点：可预测、爬虫友好

  方案 2：哈希算法（MurmurHash）
    - 长链 → MurmurHash → Base62
    - 优点：不可预测
    - 缺点：可能冲突（需查重）

  方案 3：预生成短码池
    - 提前生成一批短码存数据库
    - 用时取一个
    - 优点：高性能
    - 缺点：维护复杂

存储：
  MySQL: short_url(short_code, long_url, expire_time)
  Redis: 缓存 short_code → long_url

跳转流程：
  1. 用户访问短链
  2. Redis 查 long_url
  3. 未命中 → MySQL 查 → 写 Redis
  4. 302 重定向到 long_url
```

### 高并发优化

```
缓存：
  - Redis 缓存短码映射
  - 本地缓存热点短链

预生成：
  - 短码池预生成，避免实时生成

CDN：
  - 静态短链页面走 CDN
```

### 加分点

- 提到短码生成方案的对比
- 提到 301 vs 302 的选择（302 利于统计）
- 提到防滥用（黑名单、限流）
- 提到短链过期机制

- 订单状态流转
- 高并发下单
- 订单查询
- 订单超时处理
- 数据一致性

### 完整方案

```
订单状态机：
  PENDING → PAID → SHIPPED → DELIVERED → COMPLETED
                ↓
             CANCELLED

下单流程：
  1. 检查商品/库存/用户状态
  2. 创建订单（PENDING）+ 延迟队列（30 分钟超时）
  3. 扣减库存（Redis 预扣 + MySQL 兜底）
  4. 返回订单号

支付流程：
  1. 调用支付
  2. 支付回调 → 更新订单状态 PAID
  3. 发 MQ（支付成功事件）
  4. 异步：加积分、发通知、扣实际库存

超时处理：
  - 延迟队列 30 分钟
  - 未支付 → CANCELLED + 释放库存

查询优化：
  - 用户查订单：按 user_id 分片
  - 按订单号查：全局唯一
  - 列表分页：游标分页
```

## 场景6：抢红包系统设计

### 核心挑战

- 高并发拆红包
- 红包金额随机分配
- 防超发
- 防并发重复抢

### 完整方案

```
发红包：
  1. 计算总金额 + 数量
  2. 预分配金额（二倍均值法）
  3. 写入 Redis（红包列表 + 剩余数量）
  4. 写入 MySQL

拆红包：
  1. 检查是否已抢（Redis SET）
  2. 原子操作：DECR 剩余数量 + LPOP 金额
  3. 抢到金额 → 记录到 MySQL
  4. 抢完 → 异步通知发送者

二倍均值法：
  每次抢的金额 = 随机[0.01, (剩余金额/剩余人数)*2]
  → 保证每个人最少 0.01
  → 期望均等
```

### 关键代码

```lua
-- 抢红包 Lua 原子操作
local red_packet_key = KEYS[1]       -- 红包金额列表
local remain_count_key = KEYS[2]     -- 剩余数量
local user_set_key = KEYS[3]         -- 已抢用户集合
local user_id = ARGV[1]

-- 检查是否已抢
if redis.call('SISMEMBER', user_set_key, user_id) == 1 then
    return -2  -- 已抢过
end

-- 检查剩余数量
local remain = redis.call('GET', remain_count_key)
if tonumber(remain) <= 0 then
    return -1  -- 已抢完
end

-- 原子操作：扣减数量 + 取金额 + 记录用户
redis.call('DECR', remain_count_key)
local amount = redis.call('LPOP', red_packet_key)
redis.call('SADD', user_set_key, user_id)

return amount
```

### 加分点

- 提到金额分配算法（二倍均值法）
- 提到防并发重复抢
- 提到红包过期退回
- 提到异步记账

## 场景8：Feed 流系统设计

### 核心挑战

- 实时推送新内容
- 大量粉丝的大 V 发布
- 内容排序

### 完整方案

```
两种模式：

推模式（写扩散）：
  发布者发布 → 推送到所有粉丝的 Feed 列表
  优点：读快（直接读自己的 Feed）
  缺点：写慢（大 V 粉丝多）

拉模式（读扩散）：
  用户读 Feed → 拉取关注人的最新内容 → 合并排序
  优点：写快
  缺点：读慢（关注人多时）

推拉结合：
  普通用户 → 推模式
  大 V → 拉模式（粉丝读时拉取大 V 最新内容）
```

### 存储设计

```
Redis：
  - Feed 列表：feed:<uid>（ZSET，score=时间戳）
  - 关注关系：following:<uid>（SET）

MySQL：
  - 内容表
  - 关注关系表（分库分表）
```

### 加分点

- 提到推模式 vs 拉模式的取舍
- 提到大 V 的特殊处理
- 提到内容排序算法
- 提到缓存策略

### 完整方案

详见 [01-Redis §2 分布式锁](../01-基础存储与中间件/00-Redis业务场景实战.md#2-分布式锁--防并发重复操作) 和 [04 §14 微服务通信](../03-分布式架构与设计/00-分布式系统核心设计模式.md#14-微服务通信与-rpc)

```

3. 可重入锁（Hash 结构）
   - key=锁名, field=客户端ID, value=重入次数

4. RedLock 争议
   - Martin Kleppmann：时钟漂移问题
   - Antirez：实际可控
   - 实践：单节点 Redisson 够用，强一致用 ZK
```

- 提到 RedLock 的争议分析

## 场景19：JWT 与 SSO 设计

### 核心挑战

- 无状态认证
- Token 注销
- 多系统登录

### 完整方案

详见 [04 §15 分布式安全设计](../03-分布式架构与设计/00-分布式系统核心设计模式.md#15-分布式安全设计) 和 [01 §8 会话管理](../01-基础存储与中间件/00-Redis业务场景实战.md#8-会话管理--登录态与token)

### 核心要点

```
JWT 三段式：Header.Payload.Signature
  - 无状态、自包含、防篡改

JWT 的坑：
  1. 无法主动注销 → 黑名单（Redis）
  2. 续期难 → 双 token（access + refresh）
  3. payload 别放敏感信息

SSO 方案：
  认证中心发 JWT → 网关验证 → 透传子系统
```

### 加分点

- JWT vs Session 对比
- OAuth2 授权码模式流程
- 越权防护（水平/垂直）
