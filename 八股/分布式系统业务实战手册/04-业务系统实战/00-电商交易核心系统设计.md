## 1. 支付系统设计

支付是电商的"钱袋子"，出错就是资损。本节讲支付系统的完整设计。

### 1.1 业务模型

**核心实体**：

```
支付订单（pay_order）
  - pay_id：支付单号（全局唯一）
  - order_id：业务订单号
  - user_id：用户
  - amount：支付金额
  - channel：支付渠道（微信/支付宝/银联/余额）
  - status：支付状态
  - trade_no：第三方交易号
  - pay_time：支付时间
  - expire_time：过期时间

支付流水（pay_flow）
  - flow_id：流水号
  - pay_id：关联支付单
  - action：动作（PAY/REFUND/QUERY）
  - amount：金额
  - result：结果
  - create_time：时间
```

**支付状态机**：

```
PENDING（待支付）
  ↓ 支付成功
PAID（已支付）
  ↓ 申请退款
REFUNDING（退款中）
  ↓ 退款成功
REFUNDED（已退款）
  ↓ 超时未支付
CLOSED（已关闭）

异常状态：
  PAY_FAILED（支付失败）
  REFUND_FAILED（退款失败）
```

### 1.2 核心流程

**支付下单流程**：

```
1. 用户提交支付 → 创建支付单（PENDING）
2. 调用第三方支付（微信/支付宝下单接口）
3. 返回支付参数给前端
4. 用户完成支付（在第三方侧）
5. 第三方回调通知支付结果
6. 更新支付单状态 → 通知业务订单
```

**支付回调流程（重点）**：

```
第三方回调 → 支付系统：
  1. 验签（防止伪造回调）
  2. 幂等检查（同一支付单只处理一次）
  3. 更新支付单状态 PENDING → PAID
  4. 发 MQ 消息（支付成功事件）
  5. 返回 SUCCESS 给第三方

  关键：回调可能重复发送 → 必须幂等
        验签失败 → 拒绝，不能更新状态
        状态已是 PAID → 直接返回 SUCCESS（幂等）

业务订单服务消费 MQ：
  1. 更新订单状态 → PAID
  2. 触发后续业务（发货、加积分、发券）
```

**主动查询兜底**：

```
问题：第三方回调可能丢失或延迟

兜底：定时任务扫描 PENDING 状态的支付单
  1. 查询第三方支付状态
  2. 已支付 → 更新状态 + 发 MQ
  3. 未支付 → 继续等待
  4. 超时 → 关闭支付单

频率：每 1 分钟扫描一次
```

### 1.3 退款流程

```
退款流程：
  1. 业务发起退款 → 创建退款单（REFUNDING）
  2. 调用第三方退款接口
  3. 第三方回调退款结果
  4. 更新退款单状态 → REFUNDED
  5. 发 MQ（退款成功事件）
  6. 业务订单消费 → 订单状态 → REFUNDED

部分退款：
  - 一笔支付可多次部分退款
  - 累计退款金额 ≤ 支付金额
  - 每次退款独立单号

退款幂等：
  - 退款单号唯一
  - 第三方按退款单号去重
```

### 1.4 防重复支付

```
场景：用户在支付宝支付后，又用微信支付

方案：
  1. 支付单状态机
     UPDATE pay_order SET status='PAID' WHERE pay_id=? AND status='PENDING'
     → 只有一个支付能成功

  2. 前端控制
     支付中禁用其他支付方式
     查询支付状态，已支付则跳转

  3. 第三方预下单
     每次支付都创建新支付单，避免复用
```

### 1.5 支付路由

多支付渠道场景下的路由策略：

```
路由维度：
  - 渠道可用性（微信/支付宝是否可用）
  - 渠道费率（不同渠道手续费不同）
  - 用户偏好（用户常用支付方式）
  - 金额限制（如余额支付限额）
  - 风控规则（大额走银联）

实现：
  1. 维护渠道可用状态表（定时健康检查）
  2. 路由引擎按规则排序可用渠道
  3. 降级：首选渠道不可用 → 备选渠道
```

### 1.6 对账机制

```
T+1 对账：
  1. 下载第三方账单（微信/支付宝）
  2. 与本地支付单比对
     - 本地有第三方无 → 长款（本地多记）
     - 本地无第三方有 → 短款（本地漏记）
     - 金额不一致 → 差异
  3. 差异处理
     - 补单 / 冲正 / 人工确认

实时对账：
  关键金额（如单笔 > 1万）实时核对
```

### 1.7 一致性保障

| 场景 | 方案 | 详见 |
|------|------|------|
| 支付单与订单一致 | 本地消息表 + MQ | [03 §3.6](../02-数据一致性/00-数据一致性专题.md) |
| 支付回调幂等 | 状态机 | [02 §4](../01-基础存储与中间件/01-MySQL业务场景实战.md) |
| 支付与积分一致 | MQ + 幂等消费 | [03 §4](../02-数据一致性/00-数据一致性专题.md) |
| 对账修复 | 定时对账任务 | [03 §6](../02-数据一致性/00-数据一致性专题.md) |

### 1.8 踩坑

1. **回调验签缺失** → 被伪造回调攻击，必须验签
2. **回调不幂等** → 重复扣款/重复发货
3. **金额用浮点** → 精度问题，金额一律用分（整数）或 Decimal
4. **超时关闭不及时** → 用户重新支付但原单未关，重复支付
5. **退款不查余额** → 第三方余额不足退款失败，需预检查
6. **异步通知不可靠** → 只依赖回调，不做主动查询兜底

## 2. 订单系统设计

订单是电商的中枢，串联商品、库存、支付、物流。本节讲订单系统完整设计。

### 2.1 业务模型

**订单实体**：

```
主订单（order）
  - order_id：订单号（全局唯一）
  - user_id：用户
  - status：订单状态
  - total_amount：总金额
  - pay_amount：实付金额
  - discount_amount：优惠金额
  - freight：运费
  - pay_type：支付方式
  - pay_time：支付时间
  - ship_time：发货时间
  - create_time：下单时间

子订单（sub_order）—— 拆单后
  - sub_order_id：子订单号
  - order_id：父订单号
  - store_id：店铺
  - warehouse_id：仓库
  - amount：子单金额

订单明细（order_item）
  - item_id
  - order_id / sub_order_id
  - sku_id
  - quantity
  - unit_price
  - total_price
```

### 2.2 订单状态机

```
完整状态流转：

PENDING_PAY（待付款）
  ↓ 支付成功
PAID（已付款/待发货）
  ↓ 发货
SHIPPED（已发货）
  ↓ 确认收货
DELIVERED（已签收）
  ↓ 评价完成 / 自动完成
COMPLETED（已完成）

异常分支：
  ↓ 取消（待付款时）
CANCELLED（已取消）
  ↓ 申请退款（已付款后）
REFUND_APPLIED（退款申请中）
  ↓ 退款成功
REFUNDED（已退款）
  ↓ 退款拒绝
REFUND_REJECTED（退款拒绝）→ 回到 PAID

状态机要点：
  - 每次状态变更必须校验当前状态
  - 用乐观锁保证原子（WHERE status=?）
  - 记录状态变更日志
```

### 2.3 下单流程

```
下单完整流程：

1. 参数校验
   - 商品有效性（是否上架、是否在售）
   - 库存充足性
   - 价格校验（防止前端篡改价格）
   - 优惠券有效性
   - 用户风控检查

2. 计算价格
   - 商品金额 = Σ(sku_price × quantity)
   - 优惠金额 = 优惠券 + 满减 + 活动优惠
   - 运费 = 物流规则计算
   - 实付 = 商品金额 - 优惠 + 运费

3. 预占库存
   - Redis 预扣减（防超卖）
   - 详见 §3 库存系统

4. 创建订单
   - 写订单主表 + 明细表（本地事务）
   - 状态 = PENDING_PAY
   - 写延迟队列（30 分钟超时）

5. 优惠券核销
   - 标记优惠券已使用
   - 失败则回滚订单

6. 返回订单信息
   - 跳转支付

异常处理：
  - 任一步失败 → 回滚（释放库存、退优惠券）
  - 用 Saga 或本地消息表保证最终一致
```

### 2.4 拆单逻辑

```
拆单场景：一个订单含多店铺/多仓库商品

拆单维度：
  - 按店铺拆（不同店铺独立发货）
  - 按仓库拆（不同仓库独立发货）
  - 按商品类型拆（普通商品 vs 预售商品）

拆单流程：
  1. 下单时按拆单维度分组商品
  2. 每组生成一个子订单
  3. 父订单关联所有子订单
  4. 支付金额按比例分摊到子订单
  5. 各子订单独立发货/退款

金额分摊：
  - 优惠按比例分摊
  - 运费按店铺分摊
  - 注意精度（分摊后总和必须等于原金额）
```

### 2.5 订单超时处理

```
超时取消流程：
  1. 下单时写延迟队列（30 分钟）
  2. 30 分钟后消费消息
  3. 查订单状态
     - 仍 PENDING_PAY → 取消 + 释放库存 + 退优惠券
     - 已支付 → 忽略

多重保障：
  - 延迟队列（Redis ZSet / RocketMQ）
  - 定时任务兜底（每 5 分钟扫描超时订单）
  - 详见 [01 §6 延迟队列](../01-基础存储与中间件/00-Redis业务场景实战.md#6-延迟队列--超时处理) 和 [07 定时任务](../05-稳定性与可观测/00-分布式定时任务与调度.md)
```

### 2.6 退款流程

```
退款类型：
  - 仅退款（未发货）
  - 退货退款（已发货/已签收）
  - 部分退款（部分商品）

退款流程：
  1. 用户申请退款 → 状态 REFUND_APPLIED
  2. 商家审核
     - 同意 → 调用支付系统退款
     - 拒绝 → 状态回退
  3. 退款成功 → 状态 REFUNDED
  4. 退货退款需物流回寄
  5. 退款金额回退到原支付渠道

部分退款：
  - 按商品退款
  - 优惠分摊需重新计算
  - 退款金额 = 商品金额 - 分摊优惠
```

### 2.7 订单查询优化

```
查询场景与优化：

1. 用户查"我的订单"
   - 按 user_id 分片（分库分表）
   - 索引：idx_user_status_time(user_id, status, create_time)

2. 按订单号查详情
   - 全局唯一订单号
   - 通过订单号路由到分片

3. 运营后台按条件查
   - 复杂条件 → ES（Binlog 同步）
   - 详见 [05 §3 MySQL+ES](../01-基础存储与中间件/03-中间件组合架构实战.md#3-mysql--elasticsearch-搜索架构)

4. 分页查询
   - 游标分页（避免深分页）
   - 详见 [02 §8.3](../01-基础存储与中间件/01-MySQL业务场景实战.md#8-sql-优化实战)
```

### 2.8 订单数据归档

```
订单数据量大（亿级）→ 冷热分离

  热数据（近 3 个月）：主库
  冷数据（3 个月以上）：归档库

  详见 [02 §12 数据生命周期](../01-基础存储与中间件/01-MySQL业务场景实战.md#12-数据生命周期管理)
```

### 2.9 踩坑

1. **价格前端传** → 篡改风险，价格必须后端算
2. **库存预占不释放** → 订单取消但库存没回，超卖假象
3. **拆单金额不平** → 分摊精度问题，总和 ≠ 原金额
4. **状态机不严格** → 跳过状态（如待付款直接到已发货）
5. **退款金额错** → 部分退款时优惠分摊计算错
6. **并发改订单** → 状态覆盖，需乐观锁

## 3. 库存系统设计

库存不只是"扣减"，而是完整的库存管理体系。本节讲库存系统全貌。

### 3.1 业务模型

```
库存维度：
  - 可售库存（sellable）：可售卖的数量
  - 预占库存（locked）：已下单未支付，临时锁定
  - 实际库存（physical）：仓库真实库存
  - 在途库存（in_transit）：采购在途
  - 安全库存（safety）：最低保有量

关系：
  可售库存 = 实际库存 - 预占库存 - 安全库存

库存层级：
  - 总库存（SKU 级）
  - 仓库库存（SKU + 仓库）
  - 货位库存（SKU + 仓库 + 货位）
```

### 3.2 库存操作类型

| 操作 | 说明 | 库存影响 |
|------|------|----------|
| 预占 | 下单时锁定 | locked +, sellable - |
| 预占释放 | 订单取消 | locked -, sellable + |
| 实扣 | 支付成功 | physical -, locked - |
| 退货入库 | 退货完成 | physical +, sellable + |
| 调拨 | 仓库间调拨 | A 仓 -, B 仓 + |
| 盘点 | 盘点调整 | physical 调整 |
| 采购入库 | 采购到货 | physical +, sellable + |

### 3.3 库存扣减方案（高并发）

```
多层防线（详见 [01 §9 库存扣减](../01-基础存储与中间件/00-Redis业务场景实战.md#9-库存扣减--高并发抢购)）：

第一层：Redis 预扣减
  - Lua 原子操作：检查 + 扣减
  - 高并发下挡住大部分请求

第二层：MQ 异步落库
  - 预扣成功 → 发 MQ → 异步扣减 DB

第三层：MySQL 乐观锁兜底
  UPDATE stock SET stock = stock - 1 WHERE sku_id=? AND stock > 0

第四层：库存对账
  - Redis vs MySQL 定时对账
  - 修复差异
```

### 3.4 预占与释放

```
预占（下单时）：
  Redis: DECR stock:locked:<sku_id>（或预占库存）
  记录：预占记录表（order_id, sku_id, quantity, expire_time）

释放（订单取消/超时）：
  Redis: INCR stock:locked:<sku_id>
  删除：预占记录

超时自动释放：
  - 预占时设置过期时间（如 30 分钟）
  - 定时任务扫描过期预占 → 释放

实扣（支付成功）：
  Redis: DECR stock:physical:<sku_id>
  MySQL: UPDATE stock SET physical=physical-qty WHERE sku_id=?
  删除预占记录
```

### 3.5 多仓库库存

```
场景：商品在多仓库有库存

库存分配策略：
  - 就近原则：选离用户最近的仓库
  - 充足原则：选库存最充足的仓库
  - 成本原则：选物流成本最低的仓库

实现：
  1. 查询商品在各仓库的库存
  2. 按策略排序
  3. 分配库存（可能拆分到多仓库）
  4. 生成多子订单（按仓库拆单）

库存不足处理：
  - 部分有货：拆单（有货的先发）
  - 全部缺货：提示用户
```

### 3.6 安全库存与预警

```
安全库存：
  - 每个SKU 设最低保有量
  - 可售库存 = 实际 - 预占 - 安全库存
  - 避免卖到 0 导致突发缺货

库存预警：
  - 可售库存 < 阈值 → 预警
  - 预警方式：IM 通知采购、自动触发采购流程
  - 预警分级：紧急（< 安全库存）、警告（< 2倍安全库存）、提醒
```

### 3.7 库存盘点

```
盘点类型：
  - 全盘：定期全量盘点
  - 抽盘：随机抽查
  - 循环盘：按计划分批盘点

盘点流程：
  1. 冻结库存（盘点期间不操作）
  2. 实物盘点
  3. 对比系统库存
  4. 差异处理
     - 盘盈：physical +
     - 盘亏：physical -
  5. 记录盘点单

实现：
  - 盘点任务用定时任务调度
  - 详见 [07 定时任务](../05-稳定性与可观测/00-分布式定时任务与调度.md)
```

### 3.8 踩坑

1. **预占不释放** → 订单取消但库存未回，需定时清理过期预占
2. **Redis 与 DB 不一致** → 需对账修复
3. **并发超卖** → 必须原子操作（Lua/乐观锁）
4. **多仓库存分配错** → 拆单逻辑有误
5. **库存负数** → 扣减时未校验，允许负数库存

## 4. 商品系统设计

商品是电商的基石，商品系统管理商品的完整生命周期。

### 4.1 SPU 与 SKU 模型

```
SPU（Standard Product Unit）：标准化产品单元
  - 商品聚合，如"iPhone 15"
  - 属性：品牌、型号、描述
  - 一款商品一个 SPU

SKU（Stock Keeping Unit）：库存单元
  - 可售卖的最小单元，如"iPhone 15 256G 黑色"
  - 属性：颜色、容量、价格、库存
  - 一个 SPU 多个 SKU

关系：SPU 1:N SKU

示例：
  SPU: iPhone 15
    SKU 1: iPhone 15 128G 黑色
    SKU 2: iPhone 15 256G 黑色
    SKU 3: iPhone 15 128G 白色
```

### 4.2 商品数据模型

```
商品主表（product / SPU）
  - spu_id
  - name：商品名
  - category_id：分类
  - brand_id：品牌
  - description：描述
  - status：状态（上架/下架）

商品规格表（product_spec）
  - spu_id
  - spec_name：规格名（颜色/容量）
  - spec_value：规格值（黑色/256G）

SKU 表（product_sku）
  - sku_id
  - spu_id
  - spec_json：{"颜色":"黑色","容量":"256G"}
  - price：价格
  - stock：库存（或关联库存系统）
  - sku_code：SKU 编码

商品图片表（product_image）
  - spu_id / sku_id
  - image_url
  - sort
```

### 4.3 商品上下架

```
上架流程：
  1. 商品信息完善（标题、图片、价格、库存）
  2. 审核通过
  3. 上架 → status = ONLINE
  4. 同步到搜索（ES）
  5. 清除商品缓存

下架流程：
  1. 下架 → status = OFFLINE
  2. 从搜索移除
  3. 清除商品缓存
  4. 已下单的不影响，新用户不可见

批量上下架：
  - 批量操作
  - 异步处理（MQ）
  - 进度可查
```

### 4.4 价格管理

```
价格类型：
  - 原价（original_price）
  - 售价（price）：当前售价
  - 活动价（activity_price）：活动期间价格
  - 会员价（member_price）：会员专属

价格优先级：
  活动价 > 会员价 > 售价 > 原价

价格变更：
  - 记录价格变更历史
  - 价格审计
  - 防止恶意改价（权限控制）

下单时价格锁定：
  - 下单时记录价格快照
  - 支付前价格变动不影响订单
  - 防止"下单低价、支付高价"
```

### 4.5 商品搜索

```
搜索架构（详见 [05 §3](../01-基础存储与中间件/03-中间件组合架构实战.md#3-mysql--elasticsearch-搜索架构)）：

MySQL（主存储）→ Canal → ES（搜索）

ES 索引设计：
  - 商品名（text，分词）
  - 分类（keyword）
  - 品牌（keyword）
  - 价格（double range）
  - 销量（long sort）
  - 属性（nested，规格筛选）

搜索流程：
  1. 用户搜索 → ES 查询（关键词 + 过滤 + 排序 + 聚合）
  2. 返回 SKU 列表
  3. 从 Redis/MySQL 查详情
  4. 组装返回
```

### 4.6 商品缓存设计

```
缓存策略（详见 [01 §13](../01-基础存储与中间件/00-Redis业务场景实战.md#13-缓存三大问题--穿透雪崩击穿)）：

  SPU 信息：Redis 缓存，TTL 1 小时
  SKU 价格：Redis 缓存，TTL 5 分钟（价格变动频繁）
  商品库存：Redis 实时（详见库存系统）

缓存更新：
  - 商品变更 → 删除缓存（Cache Aside）
  - Binlog 订阅兜底

热商品：
  - 本地缓存（Caffeine）
  - 多级缓存
```

### 4.7 踩坑

1. **SPU/SKU 混淆** → 数据模型设计错，库存错乱
2. **价格前端传** → 篡改风险，后端查价
3. **上下架不同步搜索** → 下架商品仍可搜到
4. **缓存不一致** → 商品改了缓存没更新
5. **规格组合爆炸** → SKU 过多（如 5 规格各 5 值 = 3125 SKU）

## 5. 购物车系统设计

购物车是下单前的"暂存区"，设计看似简单，实则有不少门道。

### 5.1 业务模型

```
购物车项（cart_item）
  - user_id：用户
  - sku_id：商品 SKU
  - quantity：数量
  - selected：是否选中
  - add_time：加入时间
  - store_id：店铺（用于按店铺分组）

特点：
  - 数据量可能大（用户加很多商品）
  - 读写频繁
  - 不需要强一致（临时数据）
```

### 5.2 存储选型

```
方案对比（不脱离业务讲）：

Redis（推荐）：
  - 优点：高性能、TTL 自动清理
  - 缺点：内存有限、宕机丢失
  - 适用：大部分电商

Redis + MySQL：
  - Redis 为主（实时读写）
  - MySQL 兜底（持久化）
  - 适用：重视数据不丢的场景

纯 MySQL：
  - 优点：持久可靠
  - 缺点：性能差
  - 适用：低频场景

选型建议：
  - 日活 < 100 万：Redis 足够
  - 日活 > 100 万 + 要求数据不丢：Redis + MySQL
```

### 5.3 核心操作

```
加入购物车：
  1. 检查商品有效性（是否上架、是否在售）
  2. 检查库存
  3. 已存在 → 数量累加
  4. 不存在 → 新增
  5. 限制购物车商品数（如 100 件）

  Redis: HSET cart:<user_id> <sku_id> <quantity>

修改数量：
  HSET cart:<user_id> <sku_id> <new_quantity>

删除：
  HDEL cart:<user_id> <sku_id>

查询购物车：
  HGETALL cart:<user_id> → 组装商品信息

选中/取消选中：
  单独维护 selected set
  SADD/SREM cart:selected:<user_id> <sku_id>
```

### 5.4 结算流程

```
结算流程：
  1. 获取选中的购物车商品
  2. 校验商品（上架、库存、价格）
  3. 计算价格（商品金额、优惠、运费）
  4. 生成订单（详见 §2 订单系统）
  5. 清除已下单的购物车项

价格计算：
  - 实时查价（不信任购物车里的价格）
  - 优惠分摊（跨店铺优惠）
```

### 5.5 多端同步

```
场景：用户在 App 加购物车，Web 端也要看到

方案：
  - 购物车存 Redis（共享）
  - 各端读同一份数据
  - 登录后合并游客购物车与登录购物车

游客购物车：
  - 未登录时存本地（LocalStorage）
  - 登录后上传合并到服务端
  - 合并规则：同 SKU 数量取大 / 累加
```

### 5.6 踩坑

1. **价格存在购物车** → 价格变了购物车还是旧价，结算时实时查价
2. **库存不校验** → 下架/缺货商品还在购物车
3. **购物车无限增长** → 不限制商品数，内存爆炸
4. **合并逻辑错** → 登录合并重复商品处理错
5. **选中状态丢失** → 刷新后选中状态没了

## 6. 监控指标体系

> 各业务系统上线后必须监控的指标，监控不是上线后补，而是设计时就规划好。

### 6.1 支付系统监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 支付成功率 | 成功支付/总支付尝试 | < 95% 告警 |
| 支付 RT（P99） | 支付接口响应时间 | > 3s 告警 |
| 回调延迟 | 支付完成到回调间隔 | > 30s 告警 |
| 对账差异率 | 对账差异笔数/总笔数 | > 0.1% 告警 |
| 退款时效 | 退款申请到成功 | > 24h 告警 |
| 重复支付率 | 重复支付/总支付 | > 0.01% 告警 |
| 第三方渠道可用性 | 渠道接口成功率 | < 99% 告警 |

### 6.2 订单系统监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 下单成功率 | 成功下单/总下单尝试 | < 98% 告警 |
| 下单 RT（P99） | 下单接口响应时间 | > 1s 告警 |
| 支付转化率 | 支付/下单 | 下降 > 10% 告警 |
| 超时取消率 | 超时取消/总下单 | > 30% 告警 |
| 订单状态异常数 | 卡在中间状态的订单 | > 0 告警 |
| 订单查询 RT | 查询响应时间 | > 200ms 告警 |

### 6.3 库存系统监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 库存准确率 | (1-对账差异/总量) | < 99.9% 告警 |
| 超卖次数 | 库存 < 0 的次数 | > 0 立即告警 |
| 预占释放延迟 | 订单取消到库存释放 | > 5min 告警 |
| 缺货率 | 缺货 SKU/总 SKU | > 20% 告警 |
| Redis vs DB 差异 | 缓存与数据库差异 | > 0 告警 |
| 安全库存预警 | 库存低于安全线 | 触发即预警 |

### 6.4 商品系统监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 商品缓存命中率 | 缓存命中/总查询 | < 90% 告警 |
| 搜索到 ES 延迟 | MySQL 变更到 ES 同步 | > 10s 告警 |
| 上下架同步延迟 | 上下架到搜索生效 | > 30s 告警 |
| 商品详情 RT | 详情页响应 | > 200ms 告警 |

### 6.5 购物车监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 购物车操作成功率 | 加车/改量成功率 | < 99% 告警 |
| 结算转化率 | 结算/购物车访问 | 下降 > 10% 告警 |
| 购物车 RT | 操作响应时间 | > 100ms 告警 |
