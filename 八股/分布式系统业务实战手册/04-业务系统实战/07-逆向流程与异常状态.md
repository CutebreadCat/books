# 逆向流程与异常状态

> 真实业务中，正向流程（下单→支付→发货→收货）只占 30%，剩下 70% 是各种异常分支：退款、退货、换货、拒收、部分发货、部分退款……如果面试只画了一条直线状态机，会被追问得答不上来。本篇讲订单的"逆向流程"和"状态交叉"。

## 1. 为什么正向流程不够

```
面试常画的状态机：
  PENDING_PAY → PAID → SHIPPED → DELIVERED → COMPLETED

用户问：
  "我付款后不想要了，怎么退款？"
  "商家发了 3 件，我只收到 2 件，怎么退？"
  "我收到货了，但质量有问题，退货退款？"
  "我拒收了，快递退回去了，怎么退款？"
  "我买了 5 件，只退了 2 件，剩下 3 件还要的"

→ 只画一条直线，说明没有做过真实电商业务

真实电商的复杂度：
  1. 逆向流程有独立的入口（发货前、发货后、收货后）
  2. 逆向流程与正向流程会产生状态交叉（如"已发货但已退款"）
  3. 部分操作（部分退款、部分退货）需要拆分明细
```

## 2. 状态机全景图

```
完整状态机（正逆向合并）：

正向流程：
  PENDING_PAY（待付款）
    ↓ 支付成功
  PAID（已付款/待发货）
    ↓ 发货
  SHIPPED（已发货）
    ↓ 签收
  DELIVERED（已签收）
    ↓ 确认收货/自动完成
  COMPLETED（已完成）

逆向流程入口：
  【入口1：PENDING_PAY 阶段】
    PENDING_PAY → CANCELLED（用户取消，未支付）

  【入口2：PAID 阶段（发货前）】
    PAID → REFUND_APPLIED（申请退款）
      ↓ 商家同意
    REFUNDING（退款中）
      ↓ 退款成功
    REFUNDED（已退款）
      ↓ 商家拒绝
    REFUND_REJECTED（退款拒绝）→ 回到 PAID

  【入口3：SHIPPED 阶段（发货后，未签收）】
    SHIPPED → REJECTED（拒收）→ 快递退回 → 商家收货后退款
    SHIPPED → REFUND_APPLIED（未签收但申请退款）→ 拦截快递 → 退款

  【入口4：DELIVERED 阶段（已签收）】
    DELIVERED → RETURN_APPLIED（申请退货）
      ↓ 商家同意
    RETURNING（退货中，用户寄回）
      ↓ 商家确认收货
    RETURN_REFUNDED（退货退款完成）
    或：
    DELIVERED → EXCHANGE_APPLIED（申请换货）
      ↓ 商家同意
    EXCHANGING（换货中）
      ↓ 商家重新发货
    EXCHANGED（换货完成）

部分操作（贯穿多个阶段）：
  PARTIAL_SHIPPED（部分发货）→ 部分 SHIPPED + 部分 PAID
  PARTIAL_REFUNDED（部分退款）→ 部分 REFUNDED + 部分 PAID/SHIPPED

关键设计：
  1. 正向状态不可逆（PAID 不能回到 PENDING_PAY）
  2. 逆向状态是独立分支，不影响正向状态的历史记录
  3. 部分操作时，用子订单/明细级状态区分
```

## 3. 退款流程

### 3.1 发货前退款（最简单）

```
场景：用户付款后，商家还没发货，用户不想要了

流程：
  1. 用户申请退款（点击"申请退款"）
     → 订单状态：PAID → REFUND_APPLIED
     → 创建退款单（refund_order）
  2. 商家审核（或系统自动审核，如果是未发货）
     → 同意：状态 → REFUNDING
     → 拒绝：状态 → REFUND_REJECTED，用户可再次申请或申诉
  3. 系统调用支付退款接口
     → 原路退回（微信/支付宝/银行卡）
     → 退款成功：状态 → REFUNDED
  4. 释放库存（预占库存回补）
  5. 退还优惠券（如果用了优惠券，退回用户券包）

注意：
  发货前退款通常自动同意（平台规则），不需要商家审核
  退款金额 = 订单实付金额（包含分摊后的优惠金额）
```

### 3.2 发货后退款（拦截快递）

```
场景：商家已发货，用户申请退款

流程：
  1. 用户申请退款
     → 订单状态：SHIPPED → REFUND_APPLIED
  2. 商家审核
     → 同意：商家通知快递拦截（或让用户拒收）
     → 快递拦截成功 → 商家确认收货 → 退款
     → 快递拦截失败 → 等用户拒收或签收后退货
  3. 退款
     → 退款成功：状态 → REFUNDED

难点：
  1. 快递拦截不是 100% 成功（快递可能已经到驿站了）
  2. 如果拦截失败，需要走"拒收"或"退货"流程
  3. 拦截期间订单状态怎么处理？→ 保持 REFUND_APPLIED，等物流结果
```

### 3.3 退款金额计算

```
退款金额 = 订单实付金额 - 已发货商品的实付金额（如果部分发货）

示例：
  订单有 3 件商品：
    A：100 元（未发货）
    B：80 元（已发货）
    C：70 元（未发货）
  订单优惠：满 200 减 50（分摊到 A=20, B=16, C=14）
  订单实付：200 元

  用户申请整单退款：
    退款 = 200 元（全部退）
    但 B 已发货，需要拦截或等拒收

  用户只退 A 和 C：
    A 实付 = 100 - 20 = 80
    C 实付 = 70 - 14 = 56
    退款 = 80 + 56 = 136 元
    B 继续发货

  用户只退 A：
    退款 = 80 元
    但满减门槛变了：剩余商品 B+C = 150 < 200，不满足满减
    → 退 A 时，不能只退 80 元，还要扣除优惠失效的部分？
    → 这非常复杂，通常平台规则：部分退款不追回已享受的优惠（为了用户体验）
```

## 4. 退货退款流程

```
场景：用户已签收，但商品质量有问题，要退货

流程：
  1. 用户申请退货
     → 选择退货原因：质量问题 / 不喜欢 / 描述不符
     → 上传凭证图片（质量问题需要）
     → 订单状态：DELIVERED → RETURN_APPLIED
  2. 商家审核
     → 同意：提供退货地址，状态 → RETURNING
     → 拒绝：用户可申诉（平台介入）
  3. 用户寄回商品
     → 用户填写物流单号
     → 系统跟踪物流状态（快递是否签收）
  4. 商家确认收货
     → 检查商品是否完好
     → 确认：状态 → RETURN_REFUNDED，触发退款
     → 拒绝（商品损坏）：用户可申诉
  5. 退款
     → 原路退回

退款时效：
  商家确认收货后，平台通常要求 48 小时内退款
  如果商家不操作，系统自动退款（保护用户）

运费处理：
  质量问题：商家承担运费（退运费给用户）
  无理由退货：用户承担运费（用户自付寄回）
  运费险：如果用户买了运费险，系统自动理赔运费
```

## 5. 换货流程

```
场景：用户签收后，想换尺码/颜色，不要退款

流程：
  1. 用户申请换货
     → 选择新规格（如从 L 换成 XL）
     → 状态：DELIVERED → EXCHANGE_APPLIED
  2. 商家审核
     → 同意：提供退货地址，状态 → EXCHANGING
     → 拒绝：库存不足或不支持换货
  3. 用户寄回原商品
  4. 商家确认收货
     → 检查商品完好
     → 从库存中扣减新规格商品
     → 重新发货（新规格）
     → 状态 → EXCHANGED

换货的特殊性：
  1. 不涉及资金流转（不创建退款单）
  2. 需要重新生成物流单号（换货发货）
  3. 换货后的新商品也有售后保障（重新计算售后时效）
  4. 如果新规格库存不足，只能退款

表设计：
  换货单（exchange_order）：
    exchange_id, original_order_id, original_sku_id, new_sku_id,
    status, logistics_no, create_time
```

## 6. 拒收流程

```
场景：快递送到驿站/门口，用户直接拒收（不拆包裹，让快递退回）

流程：
  1. 用户拒收（通知快递或驿站）
  2. 快递退回商家
  3. 商家收到退货（快递签收）
  4. 商家确认：商品完好，同意退款
  5. 退款

与退货的区别：
  ┌──────────┬──────────┬──────────┐
  │ 维度      │ 拒收     │ 退货     │
  ├──────────┼──────────┼──────────┤
  │ 触发时机  │ 未签收   │ 已签收   │
  │ 用户操作  │ 不取件   │ 寄回商品  │
  │ 物流     │ 快递自动退回 │ 用户自行寄回 │
  │ 运费     │ 通常商家承担 │ 看原因（质量/无理由）│
  │ 退款时效  │ 快（商家收到后退款）│ 慢（用户寄回+商家确认）│
  └──────────┴──────────┴──────────┘

拒收的关键：
  1. 用户不需要在系统里操作（快递退回即可）
  2. 但系统需要识别"快递已退回"状态
  3. 商家收到退回包裹后，必须在系统里确认，否则用户收不到退款
  4. 如果商家不确认，平台通常设定自动退款（如 7 天）
```

## 7. 部分发货与部分退款

### 7.1 部分发货

```
场景：订单有 5 件商品，商家先发了 3 件，还有 2 件没货

状态设计：
  主订单状态：PARTIAL_SHIPPED（部分发货）
  子订单/明细状态：
    item1：SHIPPED（已发货）
    item2：SHIPPED（已发货）
    item3：SHIPPED（已发货）
    item4：PAID（未发货）
    item5：PAID（未发货）

用户视角：
  订单列表显示："部分发货，3 件已发出，2 件待发货"
  用户可以对未发货的商品申请退款（部分退款）
  已发货的商品，需要等收货后退货

物流：
  已发货的 3 件：有物流单号，可跟踪
  未发货的 2 件：无物流单号，可退款
```

### 7.2 部分退款

```
场景：订单有 5 件商品，用户只想退其中 2 件

判断：哪 2 件？
  1. 如果这 2 件都未发货 → 直接退款（不发货）
  2. 如果这 2 件已发货但未签收 → 拦截快递或拒收
  3. 如果这 2 件已签收 → 退货退款
  4. 如果 1 件未发货 + 1 件已签收 → 分别处理（未发货的直接退，已签收的退货退）

退款金额：
  按商品分摊后的实付价退款
  示例：
    item1：实付 80（原价 100，分摊优惠 20）
    item2：实付 64（原价 80，分摊优惠 16）
    退款 = 80 + 64 = 144 元

优惠处理：
  部分退款后，剩余商品是否还满足优惠门槛？
  → 通常不追回已享受的优惠（用户体验优先）
  → 但如果退的是"主商品"（满减触发商品），可能需要重新计算

状态更新：
  主订单状态保持为 PAID（因为还有商品没退完）
  已退款的明细标记为 REFUNDED
  退款完成后，如果所有明细都退完了，主订单状态 → REFUNDED
```

### 7.3 子订单/明细状态设计

```sql
CREATE TABLE order_item (
    item_id VARCHAR(64) PRIMARY KEY,
    order_id VARCHAR(64),
    sku_id BIGINT,
    quantity INT,
    unit_price DECIMAL(18,2),
    discount_amount DECIMAL(18,2),  -- 分摊到的优惠金额
    pay_amount DECIMAL(18,2),       -- 实付金额
    status ENUM(
        'PENDING_PAY', 'PAID', 'SHIPPED', 'DELIVERED',
        'REFUND_APPLIED', 'REFUNDED', 'RETURN_APPLIED', 'RETURNING',
        'RETURN_REFUNDED', 'EXCHANGE_APPLIED', 'EXCHANGING', 'EXCHANGED'
    ),
    logistics_no VARCHAR(64),       -- 物流单号（发货后有）
    create_time DATETIME
);

-- 退款单（与明细关联）
CREATE TABLE refund_order (
    refund_id VARCHAR(64) PRIMARY KEY,
    order_id VARCHAR(64),
    item_ids TEXT,                  -- 逗号分隔的 item_id（部分退款）
    refund_amount DECIMAL(18,2),
    reason VARCHAR(256),
    status ENUM('PENDING', 'APPROVED', 'REJECTED', 'PROCESSING', 'SUCCESS'),
    create_time DATETIME
);
```

## 8. 踩坑与反模式

### 坑 1：状态机缺少逆向分支
```
问题：只设计了 PENDING_PAY → PAID → SHIPPED → DELIVERED
  → 用户申请退款时，没有对应的状态
解决：状态机必须包含完整的逆向分支，以及逆向与正向的交叉状态
```

### 坑 2：部分退款后优惠金额算不清
```
问题：用户部分退款，退的是原价还是实付价？
  → 退原价：用户多占便宜（优惠没退）
  → 退实付价：需要查分摊表，否则金额对不上
解决：退款必须查分摊表，按实付价退。分摊表在下单时就必须生成。
```

### 坑 3：退款后优惠券没退回
```
问题：用户退款了，但优惠券不退回，用户投诉
  → 或者优惠券退回了，但用户已经用了，导致重复用券
解决：退款成功时，如果优惠券未过期，退回用户券包；如果已过期，不退
```

### 坑 4：换货不重新计算售后时效
```
问题：用户换货后，新商品签收，但售后入口已关闭（因为原订单已超时）
  → 新商品有问题，用户无法申请售后
解决：换货后，新商品的售后时效从换货签收时间重新计算
```

### 坑 5：拒收后商家不确认
```
问题：用户拒收了，快递退回商家，但商家不操作退款
  → 用户钱被扣了，商品也退回去了
解决：商家收到退货后，设定自动确认时限（如 7 天），超时自动退款
```

### 坑 6：部分发货时退款金额超付
```
问题：用户部分退款，系统退了商品原价，但用户实付时享受了优惠
  → 退款金额 > 实付金额，平台亏钱
解决：退款金额不能大于商品实付价，系统必须校验
```

## 9. 面试追问应对

**Q1：订单状态机怎么设计？**  
> 我设计的是主订单 + 明细订单双层状态机。正向流程：PENDING_PAY → PAID → SHIPPED → DELIVERED → COMPLETED。逆向流程有多个入口：发货前退款（PAID → REFUNDED）、发货后拒收（SHIPPED → REJECTED → REFUNDED）、签收后退货（DELIVERED → RETURNING → RETURN_REFUNDED）、签收后换货（DELIVERED → EXCHANGING → EXCHANGED）。对于部分操作，用子订单状态区分，比如部分发货时，主订单是 PARTIAL_SHIPPED，各明细分别标记为 SHIPPED 或 PAID。退款时按明细实付价退，查分摊表。

**Q2：部分退款怎么设计？**  
> 部分退款的核心是按明细退款。首先判断要退的商品状态：未发货的直接退款；已发货未签收的拦截或拒收；已签收的退货退款。退款金额按分摊表中的实付价退，不能退原价。如果部分退款后剩余商品不满足优惠门槛，通常不追回已享优惠（用户体验优先）。退款完成后，如果所有明细都退完了，主订单状态变为 REFUNDED；否则保持原状态。

**Q3：退货和拒收有什么区别？**  
> 核心区别在触发时机和物流方向。拒收是用户未签收时直接不取件，快递自动退回商家，通常商家承担运费，退款快。退货是用户已签收后，自行打包寄回商家，运费看原因（质量问题商家承担，无理由用户承担），退款慢（要等商家确认收货）。系统设计上，拒收不需要用户在系统里操作，但快递退回后商家必须确认才能退款；退货需要用户申请，填物流单号，商家收货后确认。

**Q4：换货为什么不直接退款再下单？**  
> 换货和退款+重新下单的区别：1. 换货不涉及资金流转，用户体验更好（不用等退款到账再付款）；2. 换货可以保留原订单的优惠（如满减、优惠券），如果退款重下，优惠可能已过期；3. 换货有售后保障时效重新计算，退款重下是新订单。技术上，换货需要独立的换货单（exchange_order），关联原订单和新 SKU，重新发货，但不走支付流程。

**Q5：退款后优惠券怎么处理？**  
> 退款成功时，如果用户使用了优惠券，且优惠券未过期，退回用户券包。如果优惠券已过期，不退回。但要注意幂等：同一订单多次退款（部分退款），优惠券只退回一次。退回的优惠券如果用户再次使用，需要校验是否还能用（有些券是一次性的，用了就不能再退）。
