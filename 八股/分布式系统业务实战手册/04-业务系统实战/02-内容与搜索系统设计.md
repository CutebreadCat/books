## 1. 评论与 UGC 系统

UGC（用户生成内容）是社区型产品的核心，评论是最常见的 UGC。

### 1.1 业务模型

```
评论（comment）
  - comment_id：评论 ID
  - target_id：被评论对象 ID（商品/文章/视频）
  - target_type：对象类型
  - user_id：评论人
  - content：内容
  - parent_id：父评论 ID（楼中楼）
  - root_id：根评论 ID（便于查整个评论树）
  - like_count：点赞数
  - reply_count：回复数
  - status：状态（正常/审核中/删除/屏蔽）
  - create_time
```

### 1.2 评论树设计

```
三种评论结构：

1. 平铺式（微博）
  - 只有评论，无回复
  - 简单，适合短评

2. 两级式（淘宝）
  - 一级评论 + 二级回复
  - 二级回复挂在一级下
  - 主流方案

3. 树形（知乎/B站）
  - 评论 + 回复 + 回复回复（无限嵌套）
  - 用 parent_id + root_id 维护

存储设计（两级式为例）：
  - 评论表：一级评论（parent_id=0）
  - 回复表：二级回复（parent_id=评论ID）
  
  查询：
    一级评论：SELECT * FROM comment WHERE target_id=? AND parent_id=0 ORDER BY ...
    二级回复：SELECT * FROM comment WHERE parent_id=? ORDER BY ...
```

### 1.3 排序策略

```
排序方式：
  - 时间倒序：最新优先
  - 热度排序：点赞数 + 时间衰减
  - 置顶：管理员置顶优先

热度公式（参考）：
  score = like_count * 2 + reply_count * 3 + 时间衰减因子

  时间衰减：
    score *= e^(-Δt/decay)
    Δt = 距发布时间（小时）
    decay = 24（24小时衰减一个周期）

实现：
  - Redis ZSet 维护热度排序
  - ZADD comment:hot:<target_id> <score> <comment_id>
  - 定时任务更新 score
```

### 1.4 敏感词过滤

```
UGC 必须过滤敏感词：

过滤方式：
  1. 关键词匹配
     - 维护敏感词库
     - DFA 算法（确定有限自动机）高效匹配
     - 命中 → 替换为 *** 或审核

  2. AI 审核
     - 文本分类模型
     - 识别色情、暴力、政治敏感
     - 适用于复杂语义

  3. 人工审核
     - 高风险内容人工复核
     - 用户举报触发审核

审核流程：
  发布 → 敏感词检查 → 
    无敏感词 → 直接发布
    有敏感词 → 进审核队列 → 人工处理
```

### 1.5 点赞设计

```
点赞（详见 [06 场景2](./04-高频业务场景面试题.md)）：

  Redis：
    like:count:<target_id> → 点赞数
    like:users:<target_id> → 点赞用户（ZSet，score=时间）

  MySQL：
    like_record(user_id, target_id, target_type, create_time)
    唯一索引：(user_id, target_id, target_type)

防重复：
  - Redis SISMEMBER 判断
  - MySQL 唯一索引兜底
```

### 1.6 分页查询

```
评论分页（深分页问题）：

  方案 1：游标分页
    SELECT * FROM comment 
    WHERE target_id=? AND id < ? 
    ORDER BY id DESC LIMIT 20

  方案 2：Redis ZSet
    ZREVRANGEBYSCORE comment:hot:<target_id> <last_score> -INF LIMIT 0 20

  避免深分页：详见 [02 §8.3](../01-基础存储与中间件/01-MySQL业务场景实战.md#8-sql-优化实战)
```

### 1.7 踩坑

1. **评论树无限嵌套** → 查询性能差，限制层级（2-3 级）
2. **敏感词漏过** → 只用关键词匹配不够，需 AI 辅助
3. **点赞防重缺失** → 重复点赞
4. **热点评论性能** → 每次查 DB，需缓存
5. **评论删除影响** → 删评论需同步删回复、减点赞数

## 2. 搜索系统设计

搜索是电商/内容产品的核心入口，直接影响转化率。

### 2.1 搜索架构

```
完整搜索架构（详见 [05 §3](../01-基础存储与中间件/03-中间件组合架构实战.md#3-mysql--elasticsearch-搜索架构)）：

  MySQL（主存储）→ Canal → ES（搜索）
  
  用户搜索 → ES 查询 → 返回 ID → 查详情（Redis/MySQL）→ 组装
```

### 2.2 搜索召回

```
召回阶段（从海量数据中找候选）：

1. 文本召回
   - 全文匹配（ES match）
   - 分词后匹配
   
2. 模糊召回
   - 拼音搜索（pinyin analyzer）
   - 错别字纠正（suggest）
   - 前缀匹配（completion）

3. 语义召回
   - 向量搜索（embedding 相似度）
   - 适用于"语义相近但关键词不同"的场景

4. 个性化召回
   - 基于用户画像
   - 基于行为历史
```

### 2.3 搜索排序

```
排序因素：
  - 文本相关性（TF-IDF / BM25）
  - 销量（热门商品优先）
  - 评分（好评优先）
  - 时间（新品优先）
  - 个性化（用户偏好）
  - 商业化（广告/赞助）

综合排序公式：
  score = text_score * w1 + sales_score * w2 + rating_score * w3 + ...
  
  权重通过 A/B 测试调优

ES 实现：
  function_score 查询，自定义评分函数
```

### 2.4 搜索建议

```
搜索建议（搜索框联想）：

  实现：ES completion 类型 + 前缀查询
  
  1. 维护搜索词索引（completion 字段）
  2. 用户输入 → 前缀匹配 → 返回建议
  3. 按搜索热度排序

  数据来源：
  - 热搜词
  - 用户历史搜索
  - 商品名/品牌名

  Redis 优化：
  - 热搜词缓存 Redis ZSet
  - ZREVRANGE hot_search 0 9
```

### 2.5 搜索热度

```
热搜榜：
  - 统计搜索词频率
  - Redis ZSet：ZINCRBY hot_search 1 <keyword>
  - 定时清理（保留近 1 小时/24 小时）

  分类：
  - 全局热搜
  - 分类热搜
  - 个性化推荐搜索

  降级：
  - 无热搜时返回默认推荐词
```

### 2.6 搜索结果缓存

```
缓存策略：
  - 热门搜索结果缓存
  - key: search:<query_hash>
  - TTL: 5 分钟

  注意：
  - 个性化搜索不缓存（因人而异）
  - 分页结果缓存（同查询不同页）
  - 缓存失效：商品变更触发（精确失效难，可全量失效或接受短暂不一致）
```

### 2.7 搜索性能优化

```
ES 优化：
  1. 索引设计
     - 字段类型合理（keyword vs text）
     - 不需要搜索的字段关闭索引
     - 用 nested 处理规格属性

  2. 查询优化
     - 避免通配符开头（如 *手机）
     - 用 filter 替代 query（filter 缓存）
     - 分页用 search_after 替代 from+size

  3. 集群优化
     - 合理分片数
     - 副本数
     - 冷热分离（热数据 SSD）
```

### 2.8 踩坑

1. **分词不当** → 搜不到（如"iPhone"被分成"i"和"Phone"）
2. **深分页** → from+size 性能差，用 search_after
3. **索引膨胀** → 不必要的字段也索引了
4. **同步延迟** → MySQL 改了 ES 没同步，用户搜不到
5. **无降级** → ES 故障搜索全挂，需降级到 MySQL like

## 3. 文件与附件系统

文件上传是通用功能，但大文件、断点续传、秒传有技术含量。

### 3.1 存储选型

```
存储选型（不脱离业务讲）：

本地存储：
  - 简单，但不适合分布式（多节点文件不同步）
  - 仅开发测试

对象存储（推荐）：
  - 阿里 OSS / 腾讯 COS / AWS S3
  - 高可用、弹性、CDN 加速
  - 生产标配

自建分布式文件系统：
  - FastDFS / MinIO
  - 适合私有化部署
  - 运维成本高
```

### 3.2 上传方式

```
方式 1：服务端转发上传
  客户端 → 业务服务 → 对象存储
  - 优点：可控（鉴权、处理）
  - 缺点：占带宽、慢

方式 2：客户端直传（推荐）
  客户端 → 预签名 URL → 对象存储
  - 业务服务只生成预签名 URL
  - 客户端直传对象存储
  - 省带宽、快

方式 3：分片上传
  大文件分片 → 并行上传 → 服务端合并
  - 适合大文件（> 100MB）
  - 支持断点续传
```

### 3.3 断点续传

```
断点续传流程：
  1. 客户端计算文件 MD5
  2. 查询服务端已上传分片
     - 服务端记录已上传分片（Redis）
  3. 只上传缺失分片
  4. 全部分片上传完成 → 通知合并
  5. 服务端合并分片 → 完整文件

实现：
  - 分片大小：1-5MB
  - 分片状态：Redis 记录（file:upload:<md5> → Set of 分片号）
  - 合并：对象存储的 merge 接口
```

### 3.4 秒传

```
秒传：相同文件不重复上传

流程：
  1. 客户端计算文件 MD5
  2. 查询服务端是否已存在
     - DB 记录：file_hash → file_url
  3. 已存在 → 直接返回 URL（秒传成功）
  4. 不存在 → 正常上传

注意：
  - MD5 冲突概率极低但非零
  - 重要文件可用 MD5 + 文件大小双重校验
  - 不同用户上传同文件 → 共享存储，权限独立
```

### 3.5 图片处理

```
图片处理需求：
  - 缩略图
  - 水印
  - 格式转换（WebP）
  - 裁剪

方案：
  1. 对象存储自带处理
     - 阿里 OSS 图片处理
     - URL 参数控制（?x-oss-process=image/resize,w_100）
  
  2. CDN 边缘处理
     - CDN 节点处理图片
  
  3. 自建图片服务
     - libvips / ImageMagick
     - 处理后缓存

最佳实践：
  - 原图存对象存储
  - 处理通过 URL 参数动态生成
  - CDN 缓存处理结果
```

### 3.6 CDN 加速

```
CDN 加速静态资源：

适用：
  - 图片、视频、JS、CSS
  - 文件下载

配置：
  - 源站：对象存储
  - 缓存策略：按文件类型 TTL
  - 回源策略：缓存未命中回源

优化：
  - 热点资源预热
  - 缓存命中率监控
  - 大文件分片缓存
```

### 3.7 权限控制

```
文件权限：
  - 公开：URL 直接访问
  - 私有：需鉴权
  - 临时授权：预签名 URL（限时访问）

实现：
  - 公开：对象存储设为 public-read
  - 私有：对象存储设为 private + 预签名 URL
  - 预签名 URL 有过期时间（如 1 小时）
```

### 3.8 踩坑

1. **大文件上传超时** → 用分片上传
2. **无秒传** → 重复上传浪费带宽
3. **图片不压缩** → 移动端流量浪费，用 WebP
4. **CDN 不预热** → 首次访问慢
5. **权限缺失** → 私有文件被公开访问
6. **文件名冲突** → 用 UUID/Hash 命名，不用原文件名

## 4. 监控指标体系

### 4.1 评论 UGC 监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 评论发布成功率 | 发布成功/尝试 | < 99% 告警 |
| 敏感词拦截率 | 拦截/总发布 | 监控趋势 |
| 审核积压量 | 待审核评论数 | > 1000 告警 |
| 评论查询 RT | 评论列表响应 | > 200ms 告警 |
| 点赞防重失败 | 重复点赞数 | > 0 告警 |

### 4.2 搜索系统监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 搜索成功率 | 搜索成功/尝试 | < 99% 告警 |
| 搜索 RT（P99） | 搜索响应时间 | > 500ms 告警 |
| ES 同步延迟 | MySQL→ES 延迟 | > 10s 告警 |
| 搜索零结果率 | 无结果搜索/总搜索 | > 30% 评估 |
| 缓存命中率 | 热搜缓存命中 | < 50% 评估 |
| ES 集群健康 | 节点状态 | yellow/red 告警 |

### 4.3 文件附件监控

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 上传成功率 | 上传成功/尝试 | < 99% 告警 |
| 上传 RT（大文件） | 大文件上传耗时 | 超预期告警 |
| 秒传命中率 | 秒传成功/尝试上传 | 监控趋势 |
| CDN 命中率 | CDN 缓存命中 | < 90% 告警 |
| 存储用量 | 对象存储使用量 | > 80% 预警 |
