## 1. B+ Tree 原理

> MySQL/PostgreSQL 用 B+ Tree。已在 [02 §13.2](./01-MySQL业务场景实战.md) 详述，这里简述。

```
B+ Tree 特点：
  - 非叶子节点只存索引 → 扇出大
  - 叶子节点存数据 + 有序链表 → 范围查询高效
  - 树高 3-4 层 → 千万级数据 3-4 次 IO

InnoDB：
  - 聚簇索引：叶子存完整行
  - 二级索引：叶子存主键（需回表）
  - 页 16KB

写：就地更新（in-place）
  - 找到页 → 修改 → 写回
  - 可能触发页分裂
  - 随机写
```

## 2. LSM Tree 原理

> HBase/RocksDB/ClickHouse/Cassandra 用 LSM Tree（Log-Structured Merge Tree）。

### 2.1 核心思想

```
B+ Tree：就地更新（随机写慢）
LSM Tree：追加写（顺序写快）

原理：
  写 → 先写 WAL（顺序写）→ 写内存 MemTable
  MemTable 满 → 刷盘成 SSTable（顺序写）
  多个 SSTable → 后台 Compaction 合并

读：
  MemTable → SSTable（Level 0）→ Level 1 → ...
  → 越新的数据越早找到
```

### 2.2 结构

```
        ┌─────────┐
  写 →  │  WAL    │ （崩溃恢复）
        └────┬────┘
             ↓
        ┌─────────┐
        │ MemTable │ （内存，有序）
        └────┬────┘
             ↓ 满
        ┌─────────┐
        │ SSTable  │ Level 0（磁盘）
        └────┬────┘
             ↓ Compaction
        ┌─────────┐
        │ SSTable  │ Level 1
        └────┬────┘
             ↓ Compaction
        ┌─────────┐
        │ SSTable  │ Level 2
        └─────────┘
```

### 2.3 Compaction

```
为什么需要 Compaction：
  - SSTable 越来越多 → 读要扫多个
  - 同一 key 可能在多个 SSTable → 旧值需清理
  - 删除标记需清理

Compaction 策略：

  Size-Tiered（HBase 默认）：
    - 相似大小的 SSTable 合并
    - 写放大低
    - 读放大高（Level 多）
    - 适合写多读少

  Leveled（RocksDB 默认）：
    - Level N 的 SSTable 总大小有上限
    - 合并到 Level N+1
    - 读放大低
    - 写放大高
    - 适合读多写少

  Hybrid：
    - 混合策略
    - 折中
```

### 2.4 布隆过滤器优化

```
LSM 读要查多层，如何加速？

布隆过滤器：
  - 每个 SSTable 配一个布隆过滤器
  - 读前先查布隆
  - 不存在 → 跳过该 SSTable
  → 大幅减少 IO
```

## 3. 写放大/读放大/空间放大

> 这是 B+ Tree vs LSM Tree 的核心 trade-off。

### 3.1 写放大（Write Amplification）

```
写放大 = 实际写磁盘量 / 用户写入量

B+ Tree：
  - 就地更新，1 次写
  - 但可能页分裂
  - 写放大 ~1-2

LSM Tree：
  - WAL 写 1 次
  - MemTable → SSTable 写 1 次
  - Compaction 多次重写
  - 写放大 ~10-30（Leveled）

→ B+ Tree 写放大低
```

### 3.2 读放大（Read Amplification）

```
读放大 = 读磁盘次数

B+ Tree：
  - 树高 3-4 层
  - 3-4 次 IO
  - 读放大 ~3-4

LSM Tree：
  - MemTable + 多层 SSTable
  - 最坏要查所有层
  - 读放大 ~几十（无布隆）
  - 有布隆 → ~几次

→ B+ Tree 读放大低
```

### 3.3 空间放大（Space Amplification）

```
空间放大 = 实际占用空间 / 有效数据量

B+ Tree：
  - 就地更新
  - 页分裂碎片
  - 空间放大 ~1.5

LSM Tree：
  - 多版本（旧值未清理）
  - Compaction 前 2-10 倍
  - Compaction 后 ~1.1

→ LSM 空间放大高（Compaction 前）
```

### 3.4 对比总结

| 维度 | B+ Tree | LSM Tree |
|------|---------|----------|
| 写放大 | 低（1-2） | 高（10-30） |
| 读放大 | 低（3-4） | 中-高（有布隆优化） |
| 空间放大 | 低（1.5） | 中（Compaction 前 2-10） |
| 随机写 | 慢 | 快（转顺序写） |
| 范围读 | 快 | 快 |
| 点查 | 快 | 中 |

## 4. 对比与选型

### 4.1 核心差异

```
B+ Tree：
  - 读优（就地读，低延迟）
  - 随机写差（页分裂）
  - 适合读多写少

LSM Tree：
  - 写优（顺序写）
  - 读需多查（Compaction 优化）
  - 适合写多读少
```

### 4.2 选型维度

```
读写比：
  读多写少 → B+ Tree（MySQL）
  写多读少 → LSM Tree（HBase）

延迟要求：
  低延迟读 → B+ Tree
  可接受稍慢 → LSM Tree

数据量：
  千万级 → B+ Tree
  十亿+ → LSM Tree

范围查询：
  多 → 两者都支持
  少 → B+ Tree 点查更快
```

## 5. 业务场景选型

| 场景 | 推荐 | 理由 |
|------|------|------|
| 电商订单 | MySQL（B+Tree） | 事务、读多、强一致 |
| 用户资料 | MySQL | 读多写少、强一致 |
| 日志存储 | HBase/ClickHouse（LSM） | 写多、海量 |
| 时序数据 | TDengine/InfluxDB（LSM） | 写密集、时序 |
| 消息历史 | HBase | 写多、海量、少查 |
| 配置/字典 | MySQL | 读多、小数据 |
| 监控数据 | ClickHouse（LSM） | 写密集、聚合分析 |
| 搜索 | ES（Lucene，类LSM） | 全文搜索 |
| 图数据 | Neo4j（定制） | 关系查询 |

### 5.1 MySQL vs HBase 选型

```
选 MySQL：
  - 需要事务
  - 强一致
  - 复杂查询（JOIN、聚合）
  - 数据量 < 亿级

选 HBase：
  - 海量数据（亿+）
  - 写密集
  - 简单 KV 查询
  - 无需复杂查询
  - 可容忍最终一致
```

### 5.2 MySQL vs ClickHouse

```
选 MySQL：
  - OLTP（联机事务）
  - 点查、简单查询
  - 事务

选 ClickHouse：
  - OLAP（联机分析）
  - 聚合统计
  - 海量数据扫描
  - 无事务需求
```
