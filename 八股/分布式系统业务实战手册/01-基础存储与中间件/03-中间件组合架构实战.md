## 1. Redis + MySQL 缓存架构

### 1.1 三层缓存架构

```
请求 → 本地缓存(Caffeine/Guava) → Redis → MySQL

L1: 本地缓存
  - 超热点数据（如配置、字典）
  - 容量小（MB 级）
  - 无网络开销
  - 各节点可能不一致 → 适合变化少的配置数据

L2: Redis
  - 通用缓存层
  - 容量大（GB 级）
  - 集群一致
  - 适合大部分业务数据

L3: MySQL
  - 最终数据源
  - 缓存全部失效时的兜底
```

### 1.2 缓存更新策略

| 策略 | 说明 | 适用 |
|------|------|------|
| Cache Aside | 读时填充缓存，写时删缓存 | 通用推荐 |
| Read Through | 缓存层自动读 DB 填充 | 缓存层封装 |
| Write Through | 缓存层自动同步写 DB | 写多且一致性要求高 |
| Write Behind | 缓存层异步批量写 DB | 写密集、容忍延迟 |

### 1.3 Cache Aside 实战

```
读流程：
  value = redis.get(key)
  if value == null:
      value = mysql.query(key)
      if value != null:
          redis.set(key, value, ttl)
      else:
          redis.set(key, NULL_VALUE, short_ttl)  # 防穿透
  return value

写流程：
  mysql.update(key, value)
  redis.del(key)
  # 可选：延迟双删
  # sleep(500ms)
  # redis.del(key)
```

### 1.4 热 KEY 处理

```
问题：某个 key 被海量请求同时访问 → 打到单个 Redis 节点

方案 1：本地缓存
  热 key 在应用层也缓存一份 → 减少 Redis 访问

方案 2：热 key 分片
  把热 key 复制到多个 Redis 节点
  → 请求随机路由到不同节点
  → key_hot_1, key_hot_2, key_hot_3 ...
  → 读时随机选一个，写时全写

方案 3：热 key 发现
  监控 Redis QPS → 发现异常高访问 key → 自动触发本地缓存

方案 4：读多写少的场景
  使用 Redis 主从 + 读写分离
  读分散到多个从节点
```

### 1.5 大 KEY 处理

```
问题：单个 key 的 value 过大（如 Hash 100 万字段）
  - 单次操作阻塞 Redis（Redis 单线程）
  - 网络传输慢
  - 集群下数据倾斜

方案 1：拆分
  - 大 Hash 拆成多个小 Hash
  - 按字段前缀分片：user:info:1, user:info:2

方案 2：压缩
  - value 用压缩算法（如 gzip）

方案 3：分批次操作
  - HSCAN 代替 HGETALL
  - SSCAN 代替 SMEMBERS

方案 4：迁移到其他存储
  - 大 value 用 MongoDB/MySQL 存储
```

## 2. Redis + MQ 异步架构

### 2.1 异步解耦模式

```
同步流程（慢）：
  用户下单 → 创建订单 → 扣库存 → 发短信 → 加积分 → 返回

异步流程（快）：
  用户下单 → 创建订单 → 发 MQ 消息 → 返回（< 50ms）
  ↓ 消费者异步处理：
  → 扣库存
  → 发短信
  → 加积分

优点：
  - 主流程响应快
  - 各步骤独立失败/重试
  - 步骤可独立扩缩容
```

### 2.2 保证最终一致性

```
关键设计：
  1. 主流程写入 MySQL + 发 MQ → 必须同时成功
     → 本地消息表方案保证

  2. 消费者幂等 → 每步操作可重复执行

  3. 消费者失败重试 → MQ 重投 + 死信队列兜底

  4. 超时未消费 → 定时任务扫描补偿
```

### 2.3 削峰填谷

```
问题：秒杀场景瞬时 10 万 QPS，订单系统承受不了

方案：
  1. Redis 预扣减库存（扛瞬时流量）
  2. 预扣减成功的请求发 MQ
  3. 订单系统按自身能力消费 MQ（如 1000 QPS）
  4. 用户感知：下单成功，订单稍后生成

效果：
  - 10 万 QPS → MQ → 1000 QPS 平滑消费
  - 保护下游系统
  - 用户体验：稍等几秒看到订单
```

### 2.4 异步通知

```
场景：订单状态变更通知用户

方案：MQ 异步通知
  1. 订单状态变更 → 发 MQ 消息
  2. 通知服务消费消息
  3. 根据 user 偏好选择通知方式（短信/Push/邮件）
  4. 失败重试 + 降级（如 Push 失败发短信）

优点：
  - 不阻塞主流程
  - 通知方式可灵活扩展
  - 失败可重试
```

## 3. MySQL + Elasticsearch 搜索架构

### 3.1 双写架构

```
写入流程：
  1. 写 MySQL（主存储）
  2. 写 ES（搜索存储）

问题：双写一致性
  - MySQL 成功 ES 失败 → 数据不一致
  - 并发写入顺序不一致 → ES 数据顺序错

解决：
  方案 1：MQ 异步同步
    写 MySQL 成功 → 发 MQ → ES 消费写入
    → 保证顺序性（MQ 分区有序）

  方案 2：Binlog 同步（Canal）
    MySQL Binlog → Canal → ES
    → 完全解耦业务代码
    → 数据最终一致
    → 延迟通常 < 1 秒

  方案 3：双写 + 定期校验
    业务双写 + 定时对账
    → 发现不一致时从 MySQL 重建 ES
```

### 3.2 搜索架构设计原则

```
1. MySQL 负责：
  - 数据完整性（事务保障）
  - 关键业务查询（按 ID、按精确条件）

2. ES 负责：
  - 全文搜索
  - 复杂条件组合查询
  - 聚合统计
  - 近实时搜索

3. 不建议：
  - ES 做唯一数据源（无事务、数据可能丢失）
  - MySQL 做复杂搜索（性能差、无法全文搜索）
```

### 3.3 Canal 同步 ES 实战

```
架构：
  MySQL → Canal（订阅 Binlog）→ Kafka → Logstash/自定义消费者 → ES

配置要点：
  1. Canal 配置 MySQL 连接 + Binlog 位点
  2. Kafka 分区策略：按表主键分区，保证同记录顺序
  3. 消费者幂等：ES _id 用 MySQL 主键
  4. 异常处理：失败重试 + 死信队列
  5. 全量重建：定期从 MySQL 全量同步 ES（兜底）
```

### 3.4 搜索场景设计

```
电商商品搜索：
  1. 用户搜索 → ES 查询（关键词 + 过滤 + 排序 + 聚合）
  2. ES 返回商品 ID 列表
  3. 按 ID 从 Redis/MySQL 查商品详情
  4. 组装结果返回

优化：
  - ES 只存搜索字段 + 商品 ID（节省空间）
  - 详情从 Redis 缓存读取
  - 热门搜索结果缓存
```

## 4. Redis + ZooKeeper 协调架构

### 4.1 各自定位

```
Redis：
  - 缓存
  - 简单分布式锁（低一致性要求）
  - 计数器
  - 排行榜

ZooKeeper：
  - 强一致分布式锁
  - 选主
  - 配置管理
  - 服务注册发现
  - 分布式协调
```

### 4.2 典型组合场景

**场景 1：选主 + 缓存**

```
多实例服务，需要选主执行定时任务：
  ZK 临时节点选主 → 主实例执行任务 → 主实例宕机自动切换

主实例执行任务时：
  → 数据写入 MySQL
  → 更新 Redis 缓存
```

**场景 2：配置 + 缓存**

```
ZK 存配置 → 各节点监听配置变更
配置变更 → ZK 通知所有节点 → 节点更新本地配置 → 清理 Redis 缓存
```

**场景 3：分布式锁分层**

```
强一致操作 → ZK 锁（如订单状态变更）
高并发操作 → Redis 锁（如防重复提交）
```

## 5. 多级缓存架构

### 5.1 架构

```
请求 → CDN → Nginx 缓存 → 本地缓存(Caffeine) → Redis → MySQL

各层职责：
  CDN：静态资源缓存（图片、JS、CSS）
  Nginx：页面级缓存、代理缓存
  本地缓存：超热点数据
  Redis：通用业务缓存
  MySQL：数据源
```

### 5.2 多级缓存一致性

```
问题：多级缓存如何同步更新？

方案 1：TTL 短 + 容忍不一致
  - 各级缓存 TTL 不同（本地 10s，Redis 60s）
  - 接受短暂不一致

方案 2：消息广播
  - 数据变更 → 发 MQ 消息
  - 各节点收到消息 → 清理本地缓存
  - Redis 缓存删除

方案 3：版本号控制
  - 数据带版本号
  - 缓存 key 包含版本号
  - 版本变更 → 旧缓存自然失效
```

### 5.3 缓存预热

```
场景：系统启动 / 大促前

方案：
  1. 启动时从 MySQL 加载热点数据到 Redis
  2. 定时任务预加载即将到期的缓存
  3. 大促前提前加载商品数据

注意：
  - 预热不要影响正常业务
  - 预热数据量评估
  - 预热失败告警
```

## 6. 消息队列选型与组合

### 6.1 MQ 选型对比

| 特性 | Kafka | RocketMQ | RabbitMQ | Redis Stream |
|------|-------|----------|----------|--------------|
| 吞吐量 | 极高（百万级） | 高（十万级） | 中（万级） | 中 |
| 延迟 | 毫秒级 | 毫秒级 | 微秒级 | 毫秒级 |
| 可靠性 | 高（多副本） | 高（同步刷盘） | 高（ACK） | 中 |
| 事务消息 | 不支持 | 支持 | 不支持 | 不支持 |
| 顺序消息 | 分区有序 | 支持 | 不支持 | 支持 |
| 延时消息 | 不支持 | 支持 | 插件 | 不支持 |
| 死信队列 | 不支持 | 支持 | 支持 | 需自实现 |
| 适用场景 | 日志/大数据 | 业务消息 | 复杂路由 | 轻量级 |

### 6.2 选型决策

| 场景 | 推荐 | 原因 |
|------|------|------|
| 日志收集 | Kafka | 高吞吐 |
| 业务消息（订单、支付） | RocketMQ | 事务消息、可靠 |
| 复杂路由 | RabbitMQ | 丰富路由模型 |
| 轻量级内部异步 | Redis Stream | 无需额外组件 |
| 大数据分析 | Kafka | 生态完善 |
| 实时流处理 | Kafka | 与 Flink/Spark 集成 |

### 6.3 组合使用

```
典型组合：
  - Kafka：日志、埋点数据
  - RocketMQ：业务消息（订单、支付）
  - Redis Stream：轻量级异步任务
  - RabbitMQ：与外部系统对接（如老系统）

原则：
  - 尽量统一 MQ，减少运维成本
  - 不同场景选不同 MQ 时，明确边界
  - 数据流转要清晰
```

## 7. 完整业务系统架构

### 7.1 电商系统架构示例

```
接入层：
  - CDN：静态资源
  - Nginx：反向代理、负载均衡、限流

网关层：
  - Spring Cloud Gateway
  - 鉴权、限流、路由

服务层：
  - 用户服务
  - 商品服务
  - 订单服务
  - 支付服务
  - 库存服务
  - 优惠券服务
  - 通知服务

中间件层：
  - Redis：缓存、分布式锁、限流
  - MySQL：核心业务数据
  - Elasticsearch：商品搜索
  - RocketMQ：业务消息
  - Kafka：日志、埋点

基础设施层：
  - ZooKeeper/Nacos：注册中心、配置中心
  - Sentinel：限流降级
  - SkyWalking：链路追踪
  - Prometheus + Grafana：监控
```

### 7.2 关键业务流程

**下单流程**：

```
1. 用户下单 → 网关 → 订单服务
2. 订单服务：
   a. 检查用户状态（Redis 缓存）
   b. 检查商品库存（Redis 预扣减）
   c. 创建订单（MySQL）
   d. 发 MQ 消息（订单创建事件）
   e. 返回订单号

3. 异步处理：
   a. 库存服务消费 MQ → 扣减库存（MySQL）
   b. 优惠券服务消费 MQ → 标记优惠券已用
   c. 通知服务消费 MQ → 发送下单通知

4. 支付：
   a. 用户支付 → 支付服务
   b. 支付成功 → 发 MQ（支付成功事件）
   c. 订单服务消费 MQ → 更新订单状态
   d. 积分服务消费 MQ → 增加积分

5. 超时处理：
   a. 订单创建时 → 写延迟队列（30 分钟）
   b. 30 分钟后未支付 → 取消订单 + 释放库存 + 释放优惠券
```

### 7.3 数据流转

```
写数据：
  业务操作 → MySQL（主） → Binlog → Canal → ES/统计/缓存

读数据：
  请求 → 本地缓存 → Redis → MySQL（兜底）

缓存更新：
  MySQL 更新 → Binlog → Canal → 删除 Redis 缓存

数据同步：
  MySQL → ES（搜索）
  MySQL → Redis（缓存）
  MySQL → 数据仓库（分析）
```

## 8. 高可用架构设计

### 8.1 高可用目标

```
可用性指标：
  - 99.9%：年停机 < 8.76 小时
  - 99.99%：年停机 < 52.6 分钟
  - 99.999%：年停机 < 5.26 分钟

业务影响：
  - 电商：每分钟停机 = 数万元损失
  - 金融：停机 = 客户信任丢失
```

### 8.2 高可用设计要点

```
1. 冗余
  - 多机房部署
  - 多可用区
  - 主从/集群

2. 故障转移
  - 自动检测故障
  - 自动切换
  - 自动恢复

3. 限流降级
  - 保护核心功能
  - 牺牲非核心功能

4. 监控告警
  - 实时监控
  - 提前预警
  - 快速定位

5. 容灾
  - 同城双活
  - 异地多活
  - 备份恢复
```

### 8.3 各组件高可用

| 组件 | 高可用方案 |
|------|------------|
| MySQL | 主从 + MHA/MGR + 半同步 |
| Redis | 主从 + 哨兵/Cluster |
| MQ | 集群多副本 |
| ES | 集群 + 副本分片 |
| 服务 | 多实例 + 负载均衡 |
| 网关 | 多实例 + VIP |

### 8.4 容灾架构

```
同城双活：
  - 两个机房同城部署
  - 数据实时同步
  - 一机房故障切换到另一机房

异地多活：
  - 多地部署
  - 数据按用户分片
  - 各地独立服务本地用户
  - 数据异地备份

冷备：
  - 数据定期备份到异地
  - 故障时恢复
  - RTO/RPO 较大
```

## 9. 数据迁移实战

数据迁移是分布式系统的高危操作：订单数据迁移、分库分表切换、MySQL→TiDB、ES 重建……做不好就是生产事故。本节讲迁移方法论。

### 9.1 迁移场景分类

| 场景 | 风险 | 典型例子 |
|------|------|----------|
| 同构迁移 | 中 | MySQL 5.7 → 8.0、单库 → 分库分表 |
| 异构迁移 | 高 | MySQL → TiDB、MongoDB → MySQL |
| 双写迁移 | 高 | 老系统 → 新系统数据平滑切换 |
| 数据重建 | 低 | ES 重建、缓存重建 |

### 9.2 迁移方案对比

| 方案 | 停机 | 复杂度 | 风险 | 适用 |
|------|------|--------|------|------|
| 停机迁移 | 长 | 低 | 低 | 小数据量、可停机 |
| 双写迁移 | 无 | 高 | 中 | 大数据量、不可停机 |
| Binlog 同步 | 无 | 中 | 中 | 异构/同构实时同步 |
| 灰度迁移 | 无 | 高 | 低 | 高风险、需验证 |
| 影子库 | 无 | 高 | 低 | 压测验证 |

### 9.3 双写迁移详解（最常用）

```
场景：单库 → 分库分表

阶段一：双写（读老库）
  写：同时写老库 + 新库（老库为主，新库失败容错）
  读：读老库
  → 持续一段时间，确保双写稳定

阶段二：数据对账
  定时对账：老库 vs 新库
  修复差异：以老库为准补齐新库
  → 直到无差异

阶段三：读切到新库（灰度）
  读：1% 流量读新库 → 10% → 50% → 100%
  写：继续双写
  → 观察读新库的正确性、性能

阶段四：写切到新库（灰度）
  写：1% 流量只写新库 → 10% → 50% → 100%
  读：读新库
  → 老库停止写入

阶段五：下线老库
  观察一段时间 → 确认无问题 → 下线老库
```

**双写的坑**：
1. **写顺序**：先写老库还是新库？→ 一般先写老库（保数据），新库失败重试
2. **事务问题**：双写跨库无法事务 → 用本地消息表 / 容错重试
3. **数据回滚**：新库写错怎么办？→ 老库为准，定时对账修复
4. **灰度维度**：按用户 ID 灰度（同用户要么全老要么全新）

### 9.4 Binlog 同步迁移

```
场景：MySQL → TiDB / ES / 新分库分表

架构：
  老库 → Canal/Debezium 订阅 Binlog → MQ → 消费者 → 新库

流程：
  1. 全量同步：mysqldump 老库 → 导入新库（记录 Binlog 位点）
  2. 增量同步：从位点开始消费 Binlog → 实时同步到新库
  3. 数据对账：确认增量追上
  4. 切换读写到新库

优点：
  - 对业务无侵入（不改代码）
  - 实时同步，停机时间短
  - 异构系统友好

缺点：
  - Binlog 顺序保证（单表有序，跨表需注意）
  - 消费者需幂等
  - DDL 同步复杂
```

### 9.5 停机迁移（简单粗暴）

```
适用：小数据量、可接受停机

流程：
  1. 挂维护公告
  2. 停止写入（应用只读模式）
  3. 数据导出（mysqldump / 导出脚本）
  4. 数据导入新库
  5. 数据校验
  6. 切换应用配置
  7. 恢复服务

优点：简单、风险低
缺点：停机时间长（数据量越大越长）
```

### 9.6 迁移验证

迁移后必须验证，不能想当然：

```
1. 数据量校验
   - 老库 COUNT vs 新库 COUNT
   - 分表后总和 vs 原表

2. 数据内容校验
   - 抽样比对关键字段
   - 全量比对（数据量小）

3. 业务功能验证
   - 核心流程回归测试
   - 灰度用户真实使用

4. 性能验证
   - 响应时间对比
   - 压测验证容量

5. 回滚验证
   - 回滚流程演练
   - 确认能快速回滚
```

### 9.7 迁移 Checklist

- [ ] 迁移方案评审（双写/Binlog/停机）
- [ ] 数据备份（迁移前全量备份）
- [ ] 回滚方案明确（能快速回滚）
- [ ] 灰度计划（逐步切换）
- [ ] 对账机制（数据一致性校验）
- [ ] 监控告警（迁移过程监控）
- [ ] 应急预案（出问题怎么办）
- [ ] 通知相关方（业务方、运维）
- [ ] 选择低峰时间窗口
- [ ] 迁移后观察期（至少 1 周）
