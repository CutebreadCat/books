## 1. 为什么需要分库分表

```
单库瓶颈：
  连接数瓶颈：MySQL 默认 max_connections 151，实际支撑 200~500（受线程模型限制）
  CPU 瓶颈：复杂查询、大量并发事务导致 CPU 打满
  I/O 瓶颈：磁盘随机读写达上限（尤其是机械盘，SSD 也有上限）
  存储瓶颈：单实例数据量过大（TB 级），备份/恢复时间不可接受
  锁竞争：并发写入同一表，行锁/页锁冲突加剧

分库分表的本质：
  把数据分散到多个数据库实例、多张表中，降低单节点压力
  与"读写分离"的区别：读写分离是复制（同一份数据），分库分表是拆分（不同数据）

触发条件（经验值）：
  单表数据量 > 5000 万 ~ 1 亿（取决于查询复杂度）
  单库 QPS > 5000 ~ 10000（取决于硬件和查询复杂度）
  单库存储 > 500GB ~ 1TB
  磁盘 I/O 利用率 > 80% 持续
  慢查询比例 > 5% 且无法通过索引优化解决

注意：
  不要盲目拆分，先优化索引、SQL、缓存、读写分离
  分库分表引入复杂度（跨分片 JOIN、排序、事务），是最后手段
```

## 2. 拆分方式：垂直 vs 水平

### 2.1 垂直拆分（Vertical Sharding）

```
垂直分库：按业务拆分，不同业务的数据放到不同数据库
  例：用户库 user_db、订单库 order_db、商品库 product_db
  特点：业务边界清晰，微服务天然匹配（一个服务一个库）
  代价：跨库 JOIN 不可用，需通过 RPC/HTTP 组装数据

垂直分表：把一张表的字段拆到多张表（按字段访问频率）
  例：订单表拆分为 订单基础表 + 订单详情表 + 订单扩展表
  基础表：order_id, user_id, status, amount（高频查询）
  详情表：order_id, address, receiver, phone（低频查询）
  特点：减少单行宽度，减少 I/O（高频查询只读需要的字段）
  代价：查询需要 JOIN 或多次查询

适用场景：
  垂直分库：微服务架构，业务边界清晰
  垂直分表：表字段过多（> 50 个），部分字段很大（TEXT/BLOB）且访问频率低
```

### 2.2 水平拆分（Horizontal Sharding）

```
水平分库：把同一业务的数据按规则分散到多个数据库实例
  例：订单数据分散到 order_db_0, order_db_1, order_db_2
  user_id % 3 = 0 → order_db_0，= 1 → order_db_1，= 2 → order_db_2

水平分表：把同一表的数据按规则分散到多张表（可在同一库或不同库）
  例：订单表拆分为 order_0, order_1, ..., order_127
  128 张表，可分布在 4 个数据库实例（每库 32 张表）

适用场景：
  单表数据量过大，水平分表减少单表数据量
  单库连接数/CPU/I/O 打满，水平分库分散实例压力

水平分片的维度选择：
  按用户 ID：用户相关查询天然路由到单分片（如查询我的订单）
  按订单 ID：写入均匀分布，但按用户查询需扫描所有分片
  按时间：适合日志/流水类数据，但新数据集中在最新分片
  按地区：适合地理分布的业务，但热点地区可能数据不均

关键原则：
  选择查询频率最高的维度作为分片键（Sharding Key）
  80% 的查询应能命中单分片，避免跨分片扫描
```

## 3. 分片策略与路由

### 3.1 常用分片算法

```
取模分片（Hash）：
  规则：shard = hash(sharding_key) % shard_count
  例：user_id = 10001 → hash(10001) % 4 = 1 → db_1/table_1
  优点：数据分布均匀，路由简单，单点查询效率高
  缺点：扩容时需要迁移数据（模数变化，几乎所有数据要重新分布）
  适用：分片数短期不变，或能接受重建数据

范围分片（Range）：
  规则：按时间或 ID 范围划分
  例：user_id [0, 1000万) → db_0，[1000万, 2000万) → db_1
  例：2024-01 数据 → table_202401，2024-02 → table_202402
  优点：扩容简单（新增范围即可），无需迁移历史数据
  缺点：新数据集中在最新分片（热点），数据分布不均匀
  适用：时间序列数据、日志、流水

枚举/标签分片（Tag）：
  规则：按离散值直接映射
  例：region = "north" → db_0，"south" → db_1，"east" → db_2
  例：tenant_id = 特定租户 → 专属分片（多租户 SaaS）
  优点：灵活，可人工控制数据分布
  缺点：需要维护映射表，扩展性差
  适用：多租户、按地区/业务线拆分

一致性 Hash（Consistent Hashing）：
  规则：将节点和数据都映射到哈希环，顺时针找最近的节点
  优点：扩容时只影响相邻节点，迁移数据量小（理论上 1/n）
  缺点：数据分布可能不均匀（需虚拟节点解决）
  适用：缓存分片（Redis 分片），数据库较少使用（因需要精确路由）

混合分片：
  先按时间/范围分库，再按 Hash 分表
  例：订单按月份分库（2024-01 库、2024-02 库），每月库内按 user_id % 128 分表
  优点：结合两者优势，时间维度方便归档，Hash 维度保证单库内均匀
```

### 3.2 路由实现方式

```
应用层路由（Sharding SDK / Client-side）：
  在应用代码中实现分片逻辑
  优点：灵活，无中间件依赖，性能最好（无额外网络跳转）
  缺点：业务侵入，每个应用都要实现路由逻辑，语言绑定
  Go 生态：可使用应用层框架或自己封装路由层
  Rust 生态：应用层实现 Sharding 路由，配合连接池（如 sqlx + 自定义路由）

代理层路由（Proxy / Middleware）：
  独立中间件接收 SQL，解析后路由到对应分片
  优点：对应用透明，语言无关，方便集中管理
  缺点：额外网络跳转，性能损耗，中间件成为单点
  代表：MySQL-Proxy, ShardingSphere-Proxy, Vitess, MyCat

Sidecar 路由（Service Mesh 模式）：
  与代理层类似，但部署为独立进程（Sidecar），随应用实例部署
  优点：语言无关，无单点（每个应用实例自带 Sidecar）
  缺点：运维复杂，资源消耗

推荐方案：
  早期：应用层路由，简单可控
  中期：Proxy 路由（统一中间件），减少业务侵入
  云原生：TiDB / CockroachDB（分布式数据库，透明分片，无需手动分库分表）
```

## 4. 全局唯一 ID

```
分库分表后，自增 ID 不可用（不同分片 ID 冲突），需要全局唯一 ID。

方案对比：

  UUID：
    优点：本地生成，无依赖，全球唯一
    缺点：无序（插入导致页分裂，性能差），长（36 字符），存储大，索引慢
    改良：UUIDv7（时间有序，接近雪花算法）

  数据库号段：
    优点：趋势递增，性能好，简单易实现
    缺点：依赖数据库，需考虑号段缓存和高可用
    实现：从中心数据库批量获取号段（如 1-1000），本地用完再取

  雪花算法（Snowflake）：
    优点：趋势递增，本地生成（无网络依赖），64 位 Long 类型
    缺点：依赖时钟，时钟回拨会重复；需分配 worker_id
    结构：1 位符号 + 41 位时间戳 + 10 位 worker_id + 12 位序列号
    Go 实现：sony/sonyflake, bwmarrin/snowflake
    Rust 实现：自定义实现（简单，基于时间戳 + 节点 ID + 序列号）

  数据库多主自增：
    设置不同起始值和步长：
      db_0: auto_increment = 1, step = 4 → 1, 5, 9, ...
      db_1: auto_increment = 2, step = 4 → 2, 6, 10, ...
    缺点：扩容困难（需改步长），ID 不连续，长距离跳跃

推荐：
  高性能场景：雪花算法（Snowflake）或 UUIDv7
  简单场景：数据库号段（Leaf 方案）
  详细原理见 [01-基础存储与中间件/06-分布式ID生成方案](06-分布式ID生成方案.md)
```

## 5. 跨分片问题与解决方案

### 5.1 跨分片查询

```
问题：WHERE 条件不包含分片键，或分片键在 IN 中命中多个分片
  例：SELECT * FROM orders WHERE status = 'paid'
  状态字段不是分片键，需扫描所有分片

解决方案：
  1. 避免跨分片查询（设计层面）：
     80% 查询都带分片键（如用户查自己的订单，天然带 user_id）
     非分片键查询走异构索引表（见下文）

  2. 异构索引表（映射表）：
     按 order_id 分片的订单表，需要按 status 查询时：
     建立索引表：按 status 分片，存储 order_id → user_id 的映射
     查询时：先查索引表获取 order_id 列表，再回主表按 order_id 查询
     代价：双写维护，数据一致性挑战（最终一致）

  3. 广播表（小表复制）：
     配置表、地区表等数据量小的表，每个分片都存一份全量数据
     查询时本地 JOIN，无需跨分片
     代价：更新时需广播到所有分片

  4. 聚合层计算：
     中间件/应用层并行查询各分片，内存中汇总结果
     仅适合小数据量汇总，大数据量性能差

  5. 搜索引擎（ES）兜底：
     复杂查询（多条件、模糊、排序）走 ES，而非直接查分片数据库
     见 [01-基础存储与中间件/05-缓存与搜索协同设计](05-缓存与搜索协同设计.md)
```

### 5.2 跨分片排序与分页

```
问题：ORDER BY create_time LIMIT 100000, 10
  每个分片取 Top 100010，汇总后排序，取全局 100010-100020
  分片数越多，内存消耗和计算量越大（与 ES 深度分页问题类似）

解决方案：
  1. 禁止深分页：
     产品层面限制（如最多翻 100 页）
     或用上次查询的最后一条数据作为游标（cursor-based pagination）

  2. 排序字段包含分片键：
     如果按 user_id 分片，且查询范围限定某 user_id，则天然单分片，无跨分片问题

  3. 汇总表/宽表：
     需要全局排序的数据，汇总到单表或 ES
     例：订单统计表，定时从各分片聚合，按时间排序

  4. 分片内排序：
     如果查询条件能限定单分片，则排序和分页天然无跨分片问题
     关键：设计查询时尽量带分片键
```

### 5.3 跨分片 JOIN

```
问题：用户表和订单表分片键不同，JOIN 需要跨分片
  例：SELECT u.name, o.amount FROM user u JOIN order o ON u.id = o.user_id
  user 按 user_id 分片，order 按 order_id 分片，同一 user 的数据不在同一分片

解决方案：
  1. 避免跨分片 JOIN（推荐）：
     微服务架构下，用户和订单是不同服务，不同库，本身就不 JOIN
     应用层组装：先查用户，再查订单，代码中合并（多次 RPC）

  2. 绑定分片（Co-location）：
     用户和订单使用相同分片键（都按 user_id 分片）
     同一 user 的数据路由到同一分片，JOIN 变成单分片 JOIN
     注意：需要设计阶段统一分片策略，后期难改

  3. 全局表（广播表）：
     用户表数据量小，每个分片都存一份全量用户表
     订单 JOIN 用户 → 本分片 JOIN 本地用户表

  4. 宽表/反规范化：
     在订单表冗余用户名称（user_name），避免 JOIN
     代价：用户改名时需更新所有订单，数据一致性挑战（最终一致）
```

## 6. 分片中间件与生态

```
语言无关方案（Proxy 层）：
  Vitess（YouTube 开源）：
    基于 MySQL，支持水平分片、读写分离、自动故障转移
    K8s 原生，云友好，YouTube 生产验证
    适合：MySQL 分片，云原生部署

  ShardingSphere（Java 生态为主）：
    支持 JDBC（SDK）和 Proxy 两种模式
    分片、读写分离、加密、影子库、弹性扩缩容
    适合：Java 生态（Go/Rust 需走 Proxy 模式）

  MyCat / DBLE：
    MySQL 代理中间件，老牌方案
    适合：存量 MySQL 分片

透明分布式数据库（推荐，减少手动分片）：
  TiDB（PingCAP，Go 实现）：
    兼容 MySQL 协议，分布式事务（Percolator 模型），水平扩展透明
    内置 TiKV（分布式 KV 存储），自动分片（Region）
    适合：不想手动分片，希望透明扩展的场景
    Go 生态友好（MySQL 协议，Go 的 database/sql 直接连）

  CockroachDB（Rust 实现）：
    兼容 PostgreSQL 协议，分布式 SQL，强一致性
    基于 Raft 的分布式事务，自动分片和重平衡
    适合：需要强一致性、SQL 兼容的分布式数据库
    Rust 开发者可深入学习其源码（Rust 实现）

  OceanBase（阿里）：
    分布式数据库，兼容 MySQL/Oracle
    生产级，金融场景验证

Go/Rust 应用层方案：
  Go：
    使用 database/sql + 自定义路由层（根据分片键计算 db/table）
    连接池：每个分片一个连接池（如 sql.DB 池组）
    库：gorm 可配合 sharding 插件（但生态不如 Java 成熟）
  Rust：
    sqlx + 自定义路由层
    或直接使用分布式数据库（TiDB/CockroachDB），避免手动分片

选型建议：
  新项目：优先考虑 TiDB / CockroachDB（透明分布式，减少分片复杂度）
  存量 MySQL：Vitess 或 ShardingSphere-Proxy
  超大规模/特殊需求：自研应用层路由（如按 user_id 分片，代码层实现）
```

## 7. 扩容与数据迁移

### 7.1 平滑扩容方案

```
取模分片的扩容难点：
  从 4 分片扩到 8 分片，模数变化，50% 数据需要迁移

平滑扩容方案：
  1. 双写阶段：
     同时写入旧分片和新分片
     读流量仍走旧分片

  2. 数据迁移：
     用脚本/工具将旧分片数据迁移到新分片
     校验数据一致性（checksum 比对）

  3. 切换读流量：
     逐步将读流量切到新分片
     观察错误率、延迟

  4. 停止双写：
     确认新分片稳定后，停止写入旧分片
     旧分片保留一段时间（回滚用）

  5. 清理：
     删除旧分片数据，释放资源

减少迁移量的方案：
  一致性 Hash：扩容时只迁移相邻节点数据（1/n 数据量）
  范围分片：新增范围即可，历史数据无需迁移
  但一致性 Hash 在数据库场景较少使用（需精确路由，虚拟节点增加复杂度）

推荐：
  如果预计会扩容，初期用范围分片或较多分片（如 128 分片部署在 4 节点，后期加节点）
  或直接用 TiDB/CockroachDB（自动重平衡，无需手动迁移）
```

### 7.2 数据归档

```
冷热分离：
  热数据：近 3 个月订单，存在高性能分片（SSD）
  温数据：3~12 个月订单，存在普通分片（SATA SSD）
  冷数据：1 年以上订单，归档到低成本存储（对象存储、HBase、ES）

归档方式：
  定时任务：每天将过期数据迁移到归档库，删除热库数据
  双写：写入时同时写热库和归档库（热库设 TTL 自动过期）
  查询路由：按时间路由，新数据查热库，老数据查归档库（对应用透明）

TiDB 方案：
  TiDB 支持 Placement Rules，将冷数据 Region 自动迁移到廉价存储节点
  无需应用层改动
```

## 8. 分布式事务

```
分库分表后，单库事务（ACID）变成跨库事务，传统方案失效。

跨分片事务方案：

  2PC（两阶段提交）：
    准备阶段：各分片执行本地事务，锁定资源，返回 OK/NO
    提交阶段：协调者收到所有 OK 后，发送提交指令；任一 NO 则回滚
    缺点：同步阻塞，协调者单点，性能差，不适合高并发
    代表：XA 协议（MySQL XA，但性能差，互联网很少用）

  3PC（三阶段提交）：
    增加预提交阶段，减少协调者故障导致的阻塞
    但网络分区时仍有问题，互联网几乎不用

  TCC（Try-Confirm-Cancel）：
    Try：预留资源（如冻结库存）
    Confirm：确认执行（扣减库存）
    Cancel：取消释放（解冻库存）
    缺点：业务侵入大，需实现三个接口，Confirm/Cancel 需幂等
    适合：金融交易、库存扣减（对一致性要求高的场景）

  Saga（长事务）：
    将大事务拆分为多个本地事务，每个本地事务有对应的补偿操作
    正向执行：T1 → T2 → T3
    失败回滚：T3 补偿 → T2 补偿 → T1 补偿
    两种模式：
      编排式（Choreography）：各服务监听事件，自主执行/补偿
      协调式（Orchestration）：中央协调器指挥执行/补偿
    缺点：补偿逻辑复杂，中间状态可见（最终一致，非强一致）
    适合：业务流程长，可容忍中间状态的互联网业务

  本地消息表（可靠消息）：
    将分布式事务转为本地事务 + 消息投递
    1. 业务表和消息表在同一个本地事务中写入
    2. 定时任务扫描消息表，发送消息到 MQ
    3. 消费者消费消息，执行下游操作
    4. 失败重试，成功删除消息
    适合：异步场景，最终一致（如订单创建后异步发券）

  最大努力通知：
    上游服务执行后，通过 MQ 通知下游，失败则重试直到成功
    适合：对一致性要求低，可容忍延迟的场景（如通知第三方）

Go/Rust 生态事务方案：
  Go：
    无成熟 Seata 替代品，通常用 Saga 或本地消息表实现
    或直接使用 TiDB（支持分布式事务，Percolator 模型，对应用透明）
  Rust：
    同样推荐分布式数据库（CockroachDB 支持 Serializable 隔离级别）
    或 Saga 模式实现（如用消息队列 + 状态机）

详细见 [02-数据一致性/04-分布式事务与Seata](../02-数据一致性/04-分布式事务与Seata.md)
```

## 9. 生产踩坑

### 坑 1：分片键设计不当导致热点
```
问题：按时间分片，新数据集中在最新分片，该分片写入压力过大
解决：Hash 分片 + 时间分库，或按用户 ID 分片分散写入
```

### 坑 2：分片数太少导致后期无法平滑扩容
```
问题：初期只分 2 个库，数据增长后想扩到 8 个，取模分片需迁移 75% 数据
解决：初期多分片（如 128 张表分布在 4 个库），扩容时加库，表不迁移（或只迁移部分）
      或直接用 TiDB/CockroachDB 避免手动分片
```

### 坑 3：不带分片键的查询拖垮集群
```
问题：后台运营查询 SELECT * FROM orders WHERE status='pending' 扫描所有分片
解决：异构索引表（按 status 分片），或 ES 索引，或限制后台查询必须带时间/用户范围
```

### 坑 4：跨分片事务导致性能雪崩
```
问题：应用层开启跨库 2PC，事务持有锁时间长，连接池耗尽
解决：避免跨分片事务，用 Saga / 本地消息表，最终一致
```

### 坑 5：分片后全局唯一 ID 冲突
```
问题：不同分片使用自增 ID，数据合并时 ID 冲突
解决：使用雪花算法（Snowflake）或数据库号段，禁止各分片独立自增
```

### 坑 6：分片后分页查询越往后越慢
```
问题：LIMIT 100000, 10 需要每个分片取 100010 条，汇总排序
解决：限制最大分页（如最多 1000 页），或使用游标分页（last_id）
```

### 坑 7：扩容时双写数据不一致
```
问题：双写阶段，旧分片和新分片数据不一致，切换后查询结果异常
解决：双写时校验一致性（checksum），切换前全量对比，保留旧分片一段时间可回滚
```

## 10. 面试追问应对

**Q1：分库和分表的区别？先分库还是先分表？**
> 分库是实例级别的拆分，解决连接数、CPU、I/O 瓶颈；分表是库内的拆分，解决单表数据量过大。通常是先分库（连接数打满）再分表（单表过大）。但如果单表数据量已很大但连接数够，可以先分表。微服务架构下，按业务垂直分库天然发生，然后再根据单表大小水平分表。

**Q2：分片键怎么选？**
> 选择查询频率最高、 Cardinality 适中、分布均匀的字段。最常用的是用户 ID（user_id），因为 80% 查询是"查我的订单"、"我的消息"，天然命中单分片。避免用状态、性别等低基数字段（数据倾斜）。避免用时间（热点）。如果是多租户 SaaS，用租户 ID 做分片键，可隔离租户数据。

**Q3：Hash 分片和 Range 分片的区别，什么时候用哪个？**
> Hash 分片数据分布均匀，适合写入密集、分片键分布均匀的场景；缺点是扩容需迁移数据。Range 分片扩容简单（新增范围即可），适合时间序列数据；缺点是热点集中在新分片。生产环境常混合使用：先按时间/范围分库（解决扩容），库内按 Hash 分表（解决均匀分布）。

**Q4：分库分表后怎么解决跨分片 JOIN？**
> 最佳方案是避免跨分片 JOIN：微服务架构下不同服务不同库，应用层组装（多次查询后代码合并）。如果必须 JOIN，可用绑定分片（用户和订单都用 user_id 分片，同一用户数据在同一分片），或全局表（小表复制到每个分片），或宽表反规范化（冗余字段）。不推荐中间件做跨分片 JOIN，性能差。

**Q5：分库分表后怎么保证全局唯一 ID？**
> 禁用自增 ID。推荐雪花算法（Snowflake）：64 位 Long，趋势递增，本地生成，无网络依赖。Go 可用 sonyflake 或 bwmarrin/snowflake，Rust 可自实现（简单）。或数据库号段（Leaf 方案），从中心库批量获取号段，本地分配。UUID 不推荐（无序、存储大）。详细见分布式 ID 专题。

**Q6：分库分表后怎么分页查询？**
> 深分页（LIMIT 100000, 10）是灾难，每个分片要取 100010 条，汇总后排序。解决方案：1）产品限制最大页码（如 100 页）；2）游标分页（用上一页最后一条的 ID 或排序值作为下一页起点，只取 > cursor 的数据）；3）如果必须深分页，用 ES 或汇总表。关键是尽量让查询命中单分片（带分片键）。

**Q7：分库分表后怎么扩容？**
> 取模分片扩容最难，需迁移数据。平滑方案：1）双写新旧分片；2）全量迁移数据；3）校验一致性；4）切换读流量；5）停止旧分片写入。更简单的方案是：初期多分片（如 128 张表），扩容时加库，表迁移量小。终极方案是用 TiDB / CockroachDB，自动重平衡，无需手动扩容。

**Q8：分库分表和分布式数据库（TiDB/CockroachDB）怎么选？**
> 如果团队有运维能力，业务复杂，需要精细控制分片策略：手动分库分表 + 中间件。如果希望减少运维复杂度，业务以标准 SQL 为主，追求透明扩展：直接用 TiDB（兼容 MySQL，Go 友好）或 CockroachDB（兼容 PostgreSQL，Rust 实现）。趋势是向分布式数据库迁移，手动分片是过渡方案。

**Q9：Go/Rust 做分库分表有什么推荐方案？**
> Go：应用层可按 user_id 取模路由，配合 database/sql 连接池组；或用 TiDB（MySQL 协议，gorm/database-sql 直接连）。Rust：sqlx + 自定义路由层；或 CockroachDB（PostgreSQL 协议，sqlx 支持）。不建议 Go/Rust 项目用 ShardingSphere-JDBC（Java 绑定），可用 ShardingSphere-Proxy 或 Vitess 作为独立中间件。
