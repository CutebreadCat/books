## 二、各方案落地 Checklist

### 缓存方案上线 Checklist

- [ ] 缓存 key 命名规范（业务:实体:ID）
- [ ] TTL 设置合理（避免雪崩：随机 TTL）
- [ ] 缓存穿透防护（布隆过滤器 / 空值缓存）
- [ ] 缓存击穿防护（热 key 互斥锁）
- [ ] 缓存雪崩防护（多级缓存 / 永不过期）
- [ ] 缓存一致性方案明确（延迟双删 / Binlog）
- [ ] 大 key 检测与处理
- [ ] 热 key 监控与处理
- [ ] 内存上限设置 + 淘汰策略
- [ ] 持久化策略确认
- [ ] 监控告警（命中率、内存、QPS）

### 分库分表上线 Checklist

- [ ] 分片键选择合理（高频查询条件）
- [ ] 分片数预留扩容空间
- [ ] 全局 ID 方案确定
- [ ] 非分片键查询方案（映射表 / ES / 基因法）
- [ ] 跨分片 JOIN 处理方案
- [ ] 跨分片聚合方案
- [ ] 分布式事务方案
- [ ] 数据迁移方案（双写 / 灰度）
- [ ] 回滚方案
- [ ] 监控（各分片数据量、QPS）

### 消息队列上线 Checklist

- [ ] MQ 选型明确（Kafka/RocketMQ/RabbitMQ）
- [ ] 生产端确认机制开启
- [ ] 消息持久化配置
- [ ] 消费端幂等保证
- [ ] 消费失败重试策略
- [ ] 死信队列处理流程
- [ ] 消息堆积监控告警
- [ ] 消费顺序保证（如需要）
- [ ] 消息轨迹可查
- [ ] 容量评估（峰值 QPS）

### 分布式锁上线 Checklist

- [ ] 锁方案选型（Redisson / ZK）
- [ ] 锁超时时间合理
- [ ] 看门狗续约配置（如 Redisson）
- [ ] 释放锁校验 unique_value
- [ ] 锁等待超时设置
- [ ] 获取锁失败降级方案
- [ ] 监控（锁持有时间、等待时间）

### 分布式事务上线 Checklist

- [ ] 事务方案选型（AT/TCC/Saga/本地消息表）
- [ ] 幂等性保证
- [ ] 补偿机制设计
- [ ] 事务日志保留策略
- [ ] 超时处理
- [ ] 异常告警
- [ ] 性能压测（事务开销）
- [ ] 回滚 / 冲正流程

### 定时任务上线 Checklist

## 三、典型故障案例复盘索引

> 以下故障模式来自业界公开案例的总结，建议对照自己的系统排查类似风险。

### 3.1 缓存类故障

| 故障 | 原因 | 教训 |
|------|------|------|
| **Redis 缓存雪崩** | 大量 key 同一 TTL 同时失效 | TTL 加随机、多级缓存 |
| **热 key 打挂单节点** | 明星绯闻导致某 key QPS 暴增 | 热 key 检测 + 本地缓存 |
| **大 key 阻塞 Redis** | 单 key 100MB Hash | 拆分 / 监控大 key |
| **布隆过滤器误判暴增** | 元素数超预期 | 监控元素量、定期重建 |

### 3.2 数据库类故障

| 故障 | 原因 | 教训 |
|------|------|------|
| **主从延迟导致超卖** | 异步复制 + 读写分离 | 关键操作读主 / 半同步 |
| **大事务拖垮 DB** | 事务内调用外部接口 | 事务内不调用外部接口 |
| **慢查询雪崩** | 无索引 SQL 打满连接池 | 慢查询监控 + 索引审查 |
| **DDL 锁表** | 大表 ALTER 直接锁 | pt-online-schema-change |

### 3.3 消息队列类故障

| 故障 | 原因 | 教训 |
|------|------|------|
| **消息重复消费** | ACK 失败重投 | 消费端必须幂等 |
| **消息堆积** | 消费速度跟不上生产 | 监控堆积 + 扩容 |
| **消息丢失** | 生产未确认 / 消费自动 ACK | 三层保障 |

### 3.4 分布式事务类故障

| 故障 | 原因 | 教训 |
|------|------|------|
| **TCC 悬挂** | Cancel 先于 Try | 悬挂控制 |
| **Saga 补偿失败** | 补偿操作本身失败 | 补偿必须重试到成功 |
| **本地消息表堆积** | 发送失败未重试 | 监控消息表 + 告警 |

### 3.5 容量类故障

| 故障 | 原因 | 教训 |
|------|------|------|
| **连接数耗尽** | 连接池配置不当 | 连接池监控 |
| **磁盘满** | 日志未清理 | 磁盘监控 + 自动清理 |
| **带宽打满** | 大促未评估带宽 | 容量评估 + 弹性扩容 |

### 3.6 复盘方法论

每次故障后写复盘报告，包含：

```
1. 故障概述：时间、影响、损失
2. 时间线：发现 → 响应 → 定位 → 恢复
3. 根因分析：直接原因 + 根本原因
4. 改进措施：
   - 立即修复
   - 短期优化
   - 长期治理
5. 责任划分：不追责个人，追流程
6. 经验沉淀：更新文档、补监控、改流程
```

#### 案例 1：缓存雪崩导致全站不可用 2 小时

**故障概述**：
- 时间：某电商大促日 10:00
- 影响：全站商品页不可访问，持续 2 小时
- 损失：估算 GMV 损失千万级

**时间线**：
```
10:00  大促开始，流量激增
10:01  商品页响应变慢，P99 从 100ms 飙升到 5s
10:02  商品页大量超时，错误率 80%
10:03  运维收到 P0 告警
10:05  排查发现 Redis 大量 key 同时过期
10:10  确认缓存雪崩，MySQL 被击穿，连接池耗尽
10:15  尝试重启 MySQL，缓解连接压力
10:20  缓存重建，但 Redis 再次被打满
10:30  紧急扩容 Redis 集群
11:00  Redis 扩容完成，缓存逐步重建
11:30  MySQL 连接池恢复，错误率下降
12:00  全站恢复
```

**根因分析**：
```
直接原因：
  大促前批量预热商品缓存，所有 key 设置相同 TTL（1 小时）
  10:00 大促开始 → 11:00 缓存批量过期 → 雪崩

根本原因：
  1. 缓存 TTL 未加随机值
  2. 无多级缓存兜底
  3. 缓存重建无限流，Redis 被打满
  4. MySQL 连接池无保护，被击穿
```

**改进措施**：
```
立即修复：
  - 缓存 TTL 加随机（base + random(0, 300s)）
  - Redis 扩容

短期优化：
  - 加本地缓存（Caffeine）作为二级缓存
  - 缓存重建限流（令牌桶）
  - MySQL 连接池保护（最大连接数限制 + 排队）

长期治理：
  - 缓存预热分批 + TTL 分散
  - 熔断降级：缓存不可用时返回兜底数据
  - 大促前全链路压测验证
  - 缓存监控告警（命中率突降告警）
```

**经验沉淀**：
- 缓存 TTL 必须加随机 → 已补到 [01 §13](../01-基础存储与中间件/00-Redis业务场景实战.md#13-缓存三大问题--穿透雪崩击穿)
- 多级缓存是标配 → 已补到 [05 §5](../01-基础存储与中间件/03-中间件组合架构实战.md#5-多级缓存架构)
- 缓存重建必须限流 → 已补到 [01 §13.3](../01-基础存储与中间件/00-Redis业务场景实战.md)

#### 案例 2：秒杀超卖事故，多卖了 50 件

**故障概述**：
- 时间：某商品秒杀活动
- 影响：库存 100 件，实际卖出 150 件
- 损失：50 件需赔偿 + 品牌信誉

**时间线**：
```
14:00  秒杀开始，10 万人抢 100 件
14:01  Redis 预扣减正常，100 件库存扣完
14:02  MQ 异步下单，订单服务消费
14:05  订单服务处理到第 80 个时，实例重启（GC）
14:06  重启后继续消费，但部分消息重复投递
14:10  发现已创建 150 个订单，超卖 50 件
```

**根因分析**：
```
直接原因：
  订单服务重启后，MQ 消息重复投递
  订单创建未做幂等 → 重复消费创建重复订单
  Redis 预扣减已扣完，但重复消息绕过预扣直接创建订单

根本原因：
  1. 订单创建未做幂等（无唯一约束/去重）
  2. MQ 消费者重启后 ACK 丢失，消息重投
  3. Redis 预扣减与订单创建未强关联
  4. MySQL 乐观锁兜底缺失
```

**改进措施**：
```
立即修复：
  - 订单表加唯一索引（user_id + activity_id）
  - 重复订单标记取消，赔偿用户

短期优化：
  - 消费者幂等（唯一键 + 去重表）
  - MySQL 乐观锁兜底（WHERE stock > 0）
  - Redis 预扣减与订单创建关联（预扣记录）

长期治理：
  - MQ 消费幂等规范 → 已补到 [02 §4](../01-基础存储与中间件/01-MySQL业务场景实战.md#4-幂等设计--防重复处理)
  - 库存扣减多层防线 → 已补到 [01 §9](../01-基础存储与中间件/00-Redis业务场景实战.md#9-库存扣减--高并发抢购)
  - 消费者优雅关闭（处理完再退出）
```

**经验沉淀**：
- MQ 消费必须幂等 → 已补到 [数据一致性/05](../02-数据一致性/05-消息与主从一致性.md)
- 库存扣减必须多层防线 → 已补到 [09 §3](../04-业务系统实战/00-电商交易核心系统设计.md#3-库存系统设计)
- 消费者必须优雅关闭

#### 案例 3：消息重复消费导致重复扣款 23 万元

**故障概述**：
- 时间：某支付系统
- 影响：37 笔订单重复扣款，累计 23 万元
- 损失：需全额退款 + 用户投诉

**时间线**：
```
凌晨 02:00  MQ 集群升级，消费者重连
02:05       部分消息重投（ACK 未到达）
02:10       消费者重新处理，重复扣款
02:30       对账发现扣款记录异常
03:00       确认 37 笔重复扣款
03:30       紧急冻结，启动退款
```

**根因分析**：
```
直接原因：
  MQ 升级导致消费者重连，部分已处理消息 ACK 丢失
  MQ 重投 → 消费者重复处理 → 重复扣款

根本原因：
  1. 扣款操作未做幂等（无支付单号去重）
  2. 消费者处理成功但 ACK 未到达时消息重投
  3. 无实时对账，2 小时后才发现
```

**改进措施**：
```
立即修复：
  - 扣款接口加幂等（支付单号唯一约束）
  - 37 笔重复扣款全额退款

短期优化：
  - 所有资金操作幂等 → 已补到 [18 §6](../05-稳定性与可观测/04-错误处理与异常体系.md)
  - 实时对账（支付成功后实时核对）
  - ACK 机制优化（处理成功立即 ACK）

长期治理：
  - 资金类操作必须幂等（Code Review 强制检查）
  - 实时对账体系 → 已补到 [数据一致性/06](../02-数据一致性/06-对账补偿与最终一致.md)
  - MQ 升级流程规范（低峰期 + 灰度）
```

**经验沉淀**：
- 资金操作幂等是红线 → 已补到 [09 §1](../04-业务系统实战/00-电商交易核心系统设计.md#1-支付系统设计)
- 实时对账是兜底 → 已补到 [数据一致性/06](../02-数据一致性/06-对账补偿与最终一致.md)
- MQ 运维变更需谨慎

#### 案例 4：分布式事务补偿失败，资金不一致 3 天

**故障概述**：
- 时间：某转账系统
- 影响：12 笔转账，扣款成功但收款失败，资金不一致 3 天
- 损失：用户投诉 + 合规风险

**时间线**：
```
Day1 10:00  用户发起转账
10:01        TCC Try 成功（扣款冻结）
10:02        TCC Confirm 调用收款服务
10:03        收款服务超时，Confirm 失败
10:04        事务协调者重试 Confirm
10:05        收款服务仍超时（服务挂了）
...          重试多次失败
Day1 18:00   重试耗尽，事务挂起
Day3         对账发现 12 笔不一致
```

**根因分析**：
```
直接原因：
  收款服务故障，TCC Confirm 一直失败
  重试耗尽后事务挂起，未触发 Cancel

根本原因：
  1. Confirm 失败后未自动转 Cancel
  2. 重试策略不合理（次数耗尽就放弃）
  3. 无事务超时自动补偿
  4. 对账延迟 3 天才发现
```

**改进措施**：
```
立即修复：
  - 12 笔手动处理（补 Cancel 或补 Confirm）
  - 用户安抚

短期优化：
  - Confirm 失败 N 次后自动转 Cancel
  - 事务超时自动补偿（定时扫描挂起事务）
  - 重试改为指数退避 + 无限重试（直到成功）

长期治理：
  - TCC 事务日志监控 → 已补到 [数据一致性/04](../02-数据一致性/04-分布式事务与Seata.md)
  - 实时对账 → 已补到 [数据一致性/06](../02-数据一致性/06-对账补偿与最终一致.md)
  - Confirm/Cancel 必须最终成功 → 已补到 [数据一致性/04 §4.1](../02-数据一致性/04-分布式事务与Seata.md)
```

**经验沉淀**：
- TCC 的 Confirm/Cancel 必须最终成功 → 已补到 [数据一致性/04](../02-数据一致性/04-分布式事务与Seata.md)
- 事务必须有超时补偿机制
- 资金类必须有实时对账

#### 案例 5：大促容量不足，服务雪崩 1 小时

**故障概述**：
- 时间：某电商双 11
- 影响：下单服务不可用 1 小时
- 损失：估算 GMV 损失数百万

**时间线**：
```
00:00  双 11 开始，流量激增
00:05  下单 QPS 从 1000 飙升到 5 万
00:08  下单服务 CPU 100%，响应变慢
00:10  下游库存服务被拖垮
00:12  网关限流，但下单服务已雪崩
00:15  紧急扩容，但镜像拉取慢
00:30  扩容 10 倍，服务逐步恢复
01:00  全部恢复
```

**根因分析**：
```
直接原因：
  容量评估不足，预估 1 万 QPS，实际 5 万 QPS
  下单服务 5 个实例扛不住，CPU 打满

根本原因：
  1. 容量评估偏保守（未考虑突发）
  2. 未做全链路压测验证
  3. 自动扩容未配置（手动扩容慢）
  4. 限流配置不合理（网关限流太晚）
  5. 服务无降级（CPU 满了直接挂）
```

**改进措施**：
```
立即修复：
  - 扩容到 50 个实例
  - 调整限流阈值

短期优化：
  - 自动扩容（HPA，CPU > 70% 自动扩）
  - 服务降级（CPU > 90% 自动降级非核心）
  - 限流前置（网关层限流）

长期治理：
  - 全链路压测 → 已补到 [19 §4](../05-稳定性与可观测/05-SRE与混沌工程.md)
  - 容量评估方法论 → 已补到 [04 §13](../03-分布式架构与设计/00-分布式系统核心设计模式.md)
  - 弹性扩缩容
  - 大促前混沌演练 → 已补到 [19 §3](../05-稳定性与可观测/05-SRE与混沌工程.md)
```

**经验沉淀**：
- 容量评估要留余量（预估 × 2-3 倍）
- 全链路压测是大促必做 → 已补到 [19 §4](../05-稳定性与可观测/05-SRE与混沌工程.md)
- 自动扩容 + 自动降级是标配
- 限流要前置（网关层）

## 四、面试速记卡

面试前 10 分钟快速过一遍的高频要点。

### 缓存三连问

```
穿透（查不存在）→ 布隆过滤器 + 空值缓存
雪崩（批量失效）→ TTL 随机 + 多级缓存
击穿（热 key 失效）→ 互斥锁 + 永不过期
```

### 缓存一致性

```
方案：先更新 DB 再删缓存（+ 延迟双删）
进阶：Binlog 订阅（Canal）
兜底：TTL + 定期对账
```

### 分布式事务四方案

```
2PC：强一致，性能差，少用
TCC：最终一致，资金类，复杂
Saga：长流程，补偿
本地消息表：通用、简单、最实用
```

### 分布式锁

```
Redisson：看门狗续约 + Lua 释放
ZK：强一致，性能低
RedLock：有争议，慎用
```

### 消息不丢不重

```
不丢：生产确认 + 持久化 + 手动 ACK
不重：消费端幂等（唯一 ID / 状态机）
顺序：单分区单消费者
```

### 分库分表

```
何时：单表 > 500 万 / 2GB
分片键：高频查询条件 + 均匀分布
非分片键：映射表 / ES / 基因法
全局 ID：Snowflake / 号段
```

### 限流算法

```
令牌桶：允许突发，API 限流
漏桶：匀速，流量整形
滑动窗口：精确，防爬虫
```

### CAP 取舍

```
资金/配置 → CP（ZK/etcd）
电商/社交 → AP（Redis/Cassandra）
注册中心 → AP（Eureka/Nacos）
```
