## 1. 为什么需要链路追踪

```
单体应用：一个请求在一个进程内完成，日志一查就到。

微服务架构：
  请求 → 网关 → 用户服务 → 订单服务 → 库存服务 → 支付服务 → 通知服务
  → 6 个服务、6 台机器、6 份日志
  → 一个请求失败，去哪台机器查哪份日志？

没有链路追踪：
  - 用户报错 → 运维逐台机器 grep 日志 → 耗时数小时
  - 性能慢 → 不知道卡在哪个服务
  - 跨团队推诿 → "不是我的服务的问题"

有链路追踪：
  - 一个 TraceID 串联全链路
  - 可视化看到每个服务的耗时和状态
  - 分钟级定位根因
```

## 2. 核心概念

### 2.1 Trace / Span / 上下文

```
Trace（追踪）：
  - 一个完整请求的全链路
  - 全局唯一 TraceID
  - 包含多个 Span

Span（跨度）：
  - 一次服务调用或一次操作
  - 有唯一 SpanID
  - 有 ParentSpanID（构建调用树）
  - 记录：开始时间、结束时间、操作名、标签、日志、状态

上下文（Context）：
  - TraceID + SpanID + 其他元数据
  - 跨服务传播
  - 是串联全链路的纽带
```

### 2.2 Span 的结构

```
Span {
  traceId: "abc123"
  spanId: "span_001"
  parentSpanId: "span_000"     // 父 Span
  operationName: "orderService.createOrder"
  startTime: 1690000000000
  duration: 150ms
  tags: {                       // 结构化标签
    "http.method": "POST",
    "http.url": "/api/orders",
    "http.status": 200,
    "db.statement": "INSERT...",
    "error": false
  }
  logs: [                       // 时间点事件
    { timestamp: ..., event: "cache_miss" },
    { timestamp: ..., event: "db_query_start" }
  ]
  status: OK / ERROR
}
```

### 2.3 调用树示例

```
Trace: 用户下单
├── Span A: 网关 (50ms)
│   ├── Span B: 用户服务.鉴权 (10ms)
│   └── Span C: 订单服务.创建订单 (200ms)
│       ├── Span D: 库存服务.扣减 (80ms)
│       │   └── Span F: MySQL.INSERT (30ms)
│       ├── Span E: 优惠券服务.核销 (50ms)
│       └── Span G: MQ.发送 (20ms)
└── 总耗时: 250ms

可视化：树形时间轴，一眼看出
  - 哪个 Span 最慢（库存服务 80ms）
  - 哪个 Span 报错（标红）
  - 并发 vs 串行（时间轴重叠）
```

## 3. 上下文传播

### 3.1 为什么需要传播

```
请求从 A 服务调到 B 服务，B 怎么知道自己是 A 的下游？

答案：A 在请求头里带上上下文，B 解析并继承
```

### 3.2 HTTP 传播

```
A → B 的 HTTP 请求头：
  trace-id: abc123
  parent-span-id: span_001
  span-id: span_002

B 解析后：
  - 继承 trace-id
  - 把 parent-span-id 设为 A 的 span-id
  - 生成自己的 span-id
  - 继续向下游传播
```

### 3.3 跨进程传播场景

```
HTTP/RPC：请求头注入
MQ：消息属性（properties/headers）注入
异步线程：ThreadLocal + 显式传递
线程池：需要装饰器传递（否则丢失上下文）
```

### 3.4 常见坑

1. **线程池丢失上下文** → 用装饰器/TransmittableThreadLocal
2. **MQ 消费丢失** → 生产者注入，消费者解析
3. **第三方调用丢失** → 需要手动注入请求头
4. **跨语言** → 用标准格式（W3C TraceContext）

## 4. 采样策略

### 4.1 为什么采样

```
全量采集：
  - 10000 QPS × 6 服务 = 60000 Span/秒
  - 存储和带宽成本极高
  - 大部分正常请求无需细查

采样：
  - 只采部分请求
  - 异常请求必采
  - 降低成本
```

### 4.2 采样策略

| 策略 | 说明 | 适用 |
|------|------|------|
| 概率采样 | 固定比例（如 1%） | 通用 |
| 限流采样 | 每秒最多 N 条 | 防洪 |
| 尾部采样 | 看结果决定（错误必采） | 精准排障 |
| 强制采样 | 特定条件必采 | 关键接口 |

### 4.3 尾部采样（推荐）

```
原理：请求完成后再决定是否采样

  请求开始 → 暂存 Span
  请求完成 → 检查结果
    - 错误 → 100% 采
    - 慢请求（> P99）→ 100% 采
    - 正常 → 1% 采

优点：
  - 错误和慢请求全保留
  - 正常请求少采
  - 成本和排障能力平衡

缺点：
  - 需暂存 Span（内存）
  - 实现复杂
```

## 5. OpenTelemetry 统一标准

### 5.1 为什么需要统一

```
历史问题：
  - Jaeger 用 Jaeger 客户端
  - Zipkin 用 Zipkin 客户端
  - SkyWalking 用自家 SDK
  → 换工具 = 改代码

OpenTelemetry（OTel）：
  - CNCF 标准
  - 统一 API + SDK
  - 后端可切换（Jaeger/Zipkin/SkyWalking）
  → 一次接入，后端随意换
```

### 5.2 OTel 架构

```
应用代码 → OTel API → OTel SDK → Exporter → 后端(Jaeger/Zipkin/...)

  API：标准化接口
  SDK：具体实现（采样、批处理、导出）
  Exporter：导出到后端（OTLP/Jaeger/Zipkin 格式）
```

### 5.3 三大信号

```
OTel 统一三大可观测性信号：
  1. Traces（链路追踪）
  2. Metrics（指标监控）
  3. Logs（日志）

→ 一个 SDK 采集三种数据
→ 通过 TraceID 关联
```

## 6. 工具选型深度对比

| 维度 | SkyWalking | Jaeger | Zipkin | Pinpoint |
|------|-----------|--------|--------|----------|
| 语言 | Java | Go | Java | Java |
| 侵入性 | 字节码增强（低） | SDK | SDK | 字节码增强（无） |
| APM 全功能 | ✅ | 仅 Trace | 仅 Trace | ✅ |
| 国产 | ✅ | ❌ | ❌ | ❌ |
| UI | 好 | 中 | 一般 | 好 |
| 性能影响 | 低 | 低 | 低 | 中 |
| K8s 友好 | 中 | ✅ | 中 | 低 |
| 监控告警 | ✅ | ❌ | ❌ | ✅ |
| 服务拓扑 | ✅ 自动 | 手动 | 手动 | ✅ |
| 社区活跃 | 高 | 高 | 中 | 中 |

### 6.1 选型建议

```
国内 Java 项目 → SkyWalking（推荐）
  - 国产、中文文档好
  - 字节码增强低侵入
  - APM 全功能（监控+告警+拓扑）

云原生/K8s → Jaeger
  - CNCF 原生
  - OTel 集成好
  - 与 Prometheus 配合

老项目/简单需求 → Zipkin
  - 轻量
  - 接入简单

无侵入 Java → Pinpoint
  - 完全无侵入
  - 但性能开销稍大
```

## 7. 排查实战

### 7.1 慢请求排查

```
场景：用户反馈下单慢（5秒）

步骤：
1. 拿到用户 ID + 大概时间
2. 查 Trace 系统，按用户筛选
3. 找到慢的 Trace（5s）
4. 展开调用树，找最慢的 Span
   → 发现库存服务 4.5s
5. 展开库存服务 Span
   → 发现 MySQL 查询 4s
6. 看Span 的 tags/logs
   → SQL: SELECT * FROM stock WHERE sku_id IN (1000个ID)
7. 根因：IN 查询过多 → 优化为批量查询
```

### 7.2 错误排查

```
场景：支付失败率升高

步骤：
1. Trace 系统筛选 ERROR 状态的 Span
2. 按服务聚合，发现支付服务错误最多
3. 展开错误 Span，看 error tag/log
   → "连接超时"
4. 看下游 Span
   → 第三方支付接口 Span 标红
5. 根因：第三方接口超时 → 降级/重试
```

### 7.3 调用链异常

```
场景：偶发性整体变慢

步骤：
1. 看服务拓扑图 → 发现某服务调用异常
2. 看该服务的 Span 分布
   → 发现偶发长尾（P99 高但 P50 正常）
3. 对比正常 vs 异常 Trace
   → 异常 Trace 多了一个重试 Span
4. 根因：下游偶发超时触发重试 → 优化超时策略
```

## 8. 落地 Checklist

- [ ] 选定链路追踪工具（推荐 SkyWalking/Jaeger）
- [ ] 所有服务接入 SDK/Agent
- [ ] HTTP/RPC 上下文传播验证
- [ ] MQ 消息上下文传播验证
- [ ] 线程池上下文传递（装饰器）
- [ ] 采样策略配置（尾部采样推荐）
- [ ] TraceID 透传到日志（MDC）
- [ ] 关键接口打 Tag（用户ID、订单号等）
- [ ] 服务拓扑图自动生成
- [ ] 慢请求/错误请求告警
- [ ] 存储容量评估（ES/Cassandra）
- [ ] 数据保留策略（如 7 天）
