# 分布式定时任务与调度

> 定时任务是业务系统的高频需求：对账、报表、数据同步、状态扫描、缓存预热……本篇梳理分布式环境下的定时任务方案选型与实战。
>
> **为什么单独成篇**：定时任务看似简单，但在分布式环境下涉及"谁来执行、怎么避免重复、怎么分片、失败怎么办"等复杂问题，是面试和实战的高频考点。

## 1. 业务场景

定时任务几乎渗透到每个业务系统：

| 场景类型 | 典型例子 | 频率 |
|----------|----------|------|
| **对账类** | 订单与支付对账、跨系统数据核对 | 每日 |
| **报表类** | 日报、周报、月报生成 | 每日/周/月 |
| **状态扫描** | 订单超时取消、过期清理、催收 | 每分钟 |
| **数据同步** | MySQL → ES、跨库同步、Binlog 补偿 | 每 5-10 分钟 |
| **缓存预热** | 热点数据预加载、字典刷新 | 每小时 |
| **推送通知** | 定时提醒、营销推送、生日祝福 | 定时 |
| **清理类** | 日志清理、临时文件清理、过期数据归档 | 每日 |
| **监控类** | 健康检查、阈值检测、容量统计 | 每分钟 |

## 2. 为什么单机定时任务不够

### 2.1 单机方案的问题

**Linux Crontab / Spring @Scheduled**：

```
问题 1：单点故障
  机器宕机 → 任务不执行 → 订单不取消、对账缺失

问题 2：无法水平扩展
  数据量增大 → 单机跑不完 → 任务积压

问题 3：重复执行
  多实例部署 → 同一任务被多个节点执行 → 数据重复

问题 4：监控缺失
  任务失败不知道、执行时长不知道、下次执行不知道

问题 5：任务管理困难
  修改频率要改代码重启、无法手动触发、无法查看历史
```

### 2.2 分布式定时任务的核心诉求

```
1. 高可用：任一节点宕机，任务仍能执行
2. 不重复：同一任务同一时间只执行一次
3. 可扩展：数据量大时可分片并行
4. 可监控：执行情况、失败告警一目了然
5. 可管理：在线修改、手动触发、查看历史
6. 可编排：任务间有依赖关系时能编排执行顺序
```

## 3. 方案对比与选型

> 选型不脱离业务讲，先看你要解决什么问题，再选工具。

### 3.1 方案全景

| 方案 | 一致性 | 分片 | 编排 | 运维 | 适用 |
|------|--------|------|------|------|------|
| 单机 Crontab/@Scheduled | 无 | 无 | 无 | 极简 | 单机、非关键 |
| Quartz 集群 | DB 锁 | 弱 | 无 | 中 | 传统集群 |
| XXL-Job | DB 锁 | 强 | 弱(子任务) | 简单 | 中小规模通用 |
| ElasticJob | ZK | 强 | 无 | 中 | 强一致需求 |
| PowerJob | DB/Redis | 强 | 强(DAG) | 中 | 复杂编排 |
| Saturn(唯品会) | ZK | 强 | 弱 | 中 | 大规模 |
| 自研 + MQ | 可控 | 自定义 | 自定义 | 高 | 特殊需求 |

### 3.2 按业务场景选型

**场景 A：小团队、任务不多、求简单**
→ **XXL-Job**
- 部署简单（一个 admin + 执行器）
- UI 友好，开箱即用
- 文档中文，社区活跃

**场景 B：强一致、不能漏执行也不能重复执行**
→ **ElasticJob**
- 基于 ZK，强一致
- 任务分片完善
- 适合金融类对准确性要求高的场景

**场景 C：任务有复杂依赖关系（A 完成才执行 B，B 失败跳过 C）**
→ **PowerJob**
- 支持 DAG 任务编排
- 支持 MapReduce 模型
- 适合 ETL、数据处理流水线

**场景 D：单机/小项目，任务简单**
→ **Spring @Scheduled + 分布式锁**
- 引入 Redis 分布式锁防重复
- 适合任务少、改动不频繁的场景

**场景 E：超大规模、特殊需求**
→ **自研 + MQ**
- 任务发到 MQ，多消费者分片消费
- 完全可控但运维成本高

### 3.3 XXL-Job vs ElasticJob 深度对比

这是面试高频问题，也是选型时最纠结的两个：

| 维度 | XXL-Job | ElasticJob |
|------|---------|------------|
| **一致性** | DB 锁（轮流抢占） | ZK 临时节点（主从选举） |
| **分片** | 静态分片（手动配置） | 动态分片（自动均衡） |
| **失败转移** | 弱（任务级） | 强（分片级） |
| **运维** | 简单（无 ZK） | 中（需 ZK） |
| **UI** | 完善 | 一般 |
| **文档** | 中文好 | 一般 |
| **社区** | 活跃（国内主流） | 一般（Apache 顶级项目但维护慢） |
| **适用规模** | 中小（百级任务） | 中大（千级任务） |

**选型口诀**：
- 求简单、任务不太多 → XXL-Job
- 求强一致、任务多、要动态分片 → ElasticJob
- 求任务编排 → PowerJob

## 4. XXL-Job 实战

### 4.1 架构

```
XXL-Job-Admin（调度中心）
   ↓  调度任务
XXL-Job-Executor（执行器，业务服务）
   ↑  回调结果
```

- **调度中心**：负责任务调度、管理、监控（集群部署，DB 锁保证只有一个调度）
- **执行器**：业务服务集成 SDK，注册到调度中心，接收并执行任务

### 4.2 任务示例

```java
@Component
public class OrderTimeoutJob {

    // 简单任务：超时订单扫描
    @XxlJob("orderTimeoutHandler")
    public void orderTimeout() {
        XxlJobHelper.log("开始扫描超时订单...");
        
        int batchSize = 1000;
        int processed = 0;
        Long lastId = 0L;
        
        while (true) {
            List<Order> orders = orderMapper.selectTimeoutOrders(lastId, batchSize);
            if (orders.isEmpty()) break;
            
            for (Order order : orders) {
                try {
                    cancelOrder(order);  // 取消订单 + 释放库存
                    processed++;
                } catch (Exception e) {
                    XxlJobHelper.log("订单 {} 取消失败: {}", order.getId(), e.getMessage());
                }
            }
            lastId = orders.get(orders.size() - 1).getId();
        }
        
        XxlJobHelper.log("扫描完成，共处理 {} 个订单", processed);
    }
    
    // 分片任务：按分片并行处理
    @XxlJob("orderShardingHandler")
    public void orderSharding() {
        int shardIndex = XxlJobHelper.getShardIndex();   // 当前分片号
        int shardTotal = XxlJobHelper.getShardTotal();   // 总分片数
        
        XxlJobHelper.log("分片 {}/{} 开始处理", shardIndex, shardTotal);
        
        // 按 user_id % shardTotal == shardIndex 过滤
        List<Order> orders = orderMapper.selectByShard(shardIndex, shardTotal);
        for (Order order : orders) {
            processOrder(order);
        }
    }
}
```

### 4.3 分片广播

XXL-Job 支持"分片广播"模式：一次调度，所有执行器都收到，各自处理自己负责的分片。

```
配置：路由策略 = 分片广播
3 个执行器节点：
  节点 0：处理 user_id % 3 == 0 的数据
  节点 1：处理 user_id % 3 == 1 的数据
  节点 2：处理 user_id % 3 == 2 的数据

优点：
  - 并行处理，速度快
  - 数据按分片隔离，无冲突
```

## 5. ElasticJob 实战

### 5.1 架构

```
ElasticJob（无中心化）
  ├── ZK：注册中心（选举、分片）
  └── 执行器：业务服务（内置调度能力）
```

与 XXL-Job 不同，ElasticJob **没有独立的调度中心**，调度逻辑在执行器内部，通过 ZK 协调。

### 5.2 任务示例

```java
// 简单任务
@ElasticSimpleJob(jobName = "dataSyncJob", cron = "0 0/10 * * * ?", shardingTotalCount = 3)
public class DataSyncJob implements SimpleJob {
    @Override
    public void execute(ShardingContext ctx) {
        int shard = ctx.getShardingItem();
        int total = ctx.getShardingTotalCount();
        
        // 按 shard 处理数据
        syncData(shard, total);
    }
}

// 数据流任务（分批处理）
@ElasticDataflowJob(jobName = "orderProcessJob", cron = "0/30 * * * * ?", 
                    shardingTotalCount = 3, streamingProcess = true)
public class OrderProcessJob implements DataflowJob<Order> {
    
    @Override
    public List<Order> fetchData(ShardingContext ctx) {
        return orderMapper.fetchPending(ctx.getShardingItem(), ctx.getShardingTotalCount(), 100);
    }
    
    @Override
    public void processData(ShardingContext ctx, List<Order> data) {
        for (Order order : data) {
            processOrder(order);
        }
    }
}
```

### 5.3 动态分片

ElasticJob 的核心优势：**节点增减时自动重新分片**。

```
当前 3 个节点：
  Node A：分片 0, 1
  Node B：分片 2, 3
  Node C：分片 4

Node B 宕机：
  ZK 检测到 B 下线 → 触发重新分片
  Node A：分片 0, 1, 2
  Node C：分片 3, 4

→ 自动故障转移，任务不中断
```

## 6. PowerJob 进阶

### 6.1 适用场景

当任务有复杂依赖关系时，XXL-Job/ElasticJob 都力不从心，PowerJob 的 DAG 编排就派上用场了。

### 6.2 DAG 任务编排示例

```
场景：每日数据报表生成

任务依赖：
  数据抽取(ETL) → 数据清洗 → 数据聚合
                          ↓
                    报表生成 → 报表推送
                          ↓
                    归档旧数据

PowerJob DAG 定义：
  A(数据抽取) → B(数据清洗) → C(数据聚合)
                          ↓
                    D(报表生成) → E(推送) 
                              ↓
                          F(归档)

  C 失败 → 跳过 D、E → 仍执行 F（归档昨日数据）
```

### 6.3 MapReduce 模型

PowerJob 支持 MapReduce 模型，适合大数据量处理：

```java
// 场景：处理 1000 万订单
// 处理器把任务分片 → 多 worker 并行处理 → 汇总结果

public class OrderBatchProcessor extends MapReduceProcessor {
    
    @Override
    public ProcessResult process(TaskSlice slice) {
        // 每个 slice 处理一批订单
        List<Order> orders = fetchOrders(slice.getStartId(), slice.getEndId());
        for (Order order : orders) {
            processOrder(order);
        }
        return new ProcessResult(true);
    }
    
    @Override
    public ProcessResult reduce(List<Task> results) {
        // 汇总各分片结果
        int total = results.stream().mapToInt(r -> r.getResult()).sum();
        return new ProcessResult(true, "共处理 " + total + " 个订单");
    }
}
```

## 7. 任务分片设计

分片是分布式定时任务的核心能力，分片策略直接影响性能和正确性。

### 7.1 分片策略对比

| 策略 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **按 ID 取模** | `id % shardTotal` | 简单均匀 | 扩容要重分布 |
| **按范围** | `id between x and y` | 扩容友好 | 可能不均 |
| **按时间** | `create_time between` | 天然分散 | 冷热不均 |
| **按业务维度** | `user_id % n` | 业务内聚 | 分片键与业务耦合 |
| **一致性哈希** | `hash(key) % 2^32` | 扩容迁移少 | 实现复杂 |

### 7.2 分片键选择

```
选择原则：
1. 与查询条件对齐 → 分片键 = 查询条件，避免广播
2. 数据均匀 → 避免某分片过载
3. 业务关联 → 相关数据同一分片

常见分片键：
  - user_id → 用户维度任务（订单处理、积分计算）
  - order_id → 订单维度任务
  - create_time → 时间维度任务（日志清理、归档）
  - region → 地域维度任务（区域统计）
```

### 7.3 分片数选择

```
经验值：
  分片数 = 执行器节点数 × (1-3)
  
  例：3 个执行器节点 → 6-9 个分片
  → 每个节点处理 2-3 个分片
  → 节点宕机时，分片自动转移到其他节点

注意：
  - 分片数不宜过多（调度开销大）
  - 分片数不宜过少（无法发挥并行优势）
  - 分片数固定后扩容受限（需重新规划）
```

## 8. 失败重试与告警

### 8.1 失败重试策略

```
立即重试：
  - 失败后立即重试
  - 适用于网络抖动等瞬时故障
  - 限制重试次数（3 次）

延迟重试：
  - 失败后等待一段时间再重试
  - 适用于资源不足等持续故障
  - 指数退避（1min, 5min, 30min）

跳过 + 告警：
  - 重试 N 次仍失败 → 跳过本次 → 告警
  - 下次调度继续执行
  - 避免任务积压

死信处理：
  - 多次失败的任务进入死信
  - 人工介入处理
```

### 8.2 告警设计

```
告警触发条件：
  - 任务执行失败
  - 任务执行超时（超过预期时长）
  - 任务未按时执行（错过调度）
  - 任务执行结果异常（处理数为 0、错误率高等）

告警方式：
  - 钉钉/企业微信机器人
  - 邮件
  - 短信（重要任务）
  - 电话（核心任务）

告警分级：
  P0（电话）：核心对账、资金相关
  P1（短信+IM）：重要业务任务
  P2（IM）：一般任务
  P3（邮件）：监控类任务
```

### 8.3 幂等性保障

定时任务可能重复执行（重试、故障转移），必须幂等：

```
方案 1：状态机
  UPDATE order SET status='CANCELLED' WHERE id=? AND status='PENDING'
  → 已取消的不会重复处理

方案 2：去重表
  每次执行记录 (task_id, batch_id)
  执行前检查是否已处理

方案 3：唯一键
  批处理结果写入带唯一键的表
  重复执行时唯一键冲突跳过
```

## 9. 任务编排（DAG）

### 9.1 为什么需要编排

```
复杂业务场景：
  报表生成 = 数据抽取 + 数据清洗 + 数据聚合 + 报表生成 + 报表推送

如果用简单定时任务：
  5 个独立任务，分别调度
  → 时序难以保证
  → 一个失败影响后续
  → 人工协调成本高

DAG 编排：
  定义依赖关系
  → 自动按序执行
  → 失败处理策略明确
  → 全流程可视化
```

### 9.2 编排模式

```
串行：A → B → C
并行：A, B, C 同时执行 → 全部完成后 D
分支：A → (B 成功 | C 失败) → D
循环：A → B → (条件成立? B : C)
```

### 9.3 工具选型

| 工具 | 编排能力 | 适用 |
|------|----------|------|
| PowerJob | DAG 原生支持 | 复杂编排 |
| Apache Airflow | DAG（Python 定义） | 数据工程 |
| DolphinScheduler | DAG（可视化） | 数据中台 |
| XXL-Job 子任务 | 弱（子任务链） | 简单依赖 |
| 自研 + MQ | 完全自定义 | 特殊需求 |

## 10. 与延迟队列的区别

定时任务和延迟队列都涉及"延迟执行"，但场景不同：

| 维度 | 定时任务 | 延迟队列 |
|------|----------|----------|
| **触发方式** | 时间驱动（cron） | 事件驱动（消息入队） |
| **数据来源** | 主动扫描数据库 | 消息队列 |
| **执行时机** | 固定时间点 | 相对时间（如 30 分钟后） |
| **数据量** | 批量 | 单条 |
| **典型场景** | 每日对账、定时报表 | 订单超时、预约提醒 |
| **工具** | XXL-Job/ElasticJob | RocketMQ/Redis ZSet |

**组合使用**：
- 订单超时取消：延迟队列（精准 30 分钟） + 定时任务（兜底扫描）
- 详见 [01-Redis §6 延迟队列](../01-基础存储与中间件/00-Redis业务场景实战.md#6-延迟队列--超时处理)

## 11. 常见踩坑

1. **任务执行时间长导致下次调度错过**
   - 解决：串行/丢弃策略配置（XXL-Job 的 BlockStrategy）
   - 优化：分片并行处理

2. **任务幂等性没做**
   - 问题：重试/故障转移导致重复处理
   - 解决：状态机/去重表/唯一键

3. **分片不均**
   - 问题：某分片数据多，拖慢整体
   - 解决：监控各分片执行时长，调整分片策略

4. **任务间无依赖却串行调度**
   - 问题：本可并行的任务被串行执行，浪费时间
   - 解决：用 DAG 编排

5. **大事务**
   - 问题：整个任务一个事务，数据量大锁表
   - 解决：分批处理，每批独立事务

6. **日志缺失**
   - 问题：任务失败无法排查
   - 解决：关键步骤打日志，集成到调度中心

7. **任务被遗忘**
   - 问题：临时任务上线后忘记下线
   - 解决：任务清单治理，定期 review

## 12. 落地 Checklist

上线前逐项确认：

- [ ] **高可用**：调度中心/执行器多节点部署
- [ ] **幂等**：任务可重复执行无副作用
- [ ] **分片**：大数据量已分片，分片键合理
- [ ] **重试**：失败重试策略明确（次数/间隔）
- [ ] **告警**：失败/超时/漏执行告警已配置
- [ ] **监控**：执行时长、成功率、数据量监控
- [ ] **日志**：关键步骤日志完整，可追溯
- [ ] **超时**：任务超时时间设置合理
- [ ] **阻塞策略**：串行/丢弃策略明确
- [ ] **依赖隔离**：任务间无隐式依赖
- [ ] **回滚**：任务异常时的回滚/补偿方案
