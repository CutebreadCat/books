## 1. 日志分级

```
DEBUG：开发调试，生产关闭
  - 详细流程、变量值

INFO：关键业务节点，生产开启
  - 下单成功、支付成功
  - 状态变更
  - 定时任务执行

WARN：可预期异常，需关注
  - 重试成功
  - 降级触发
  - 限流触发

ERROR：不可预期错误，需处理
  - 异常堆栈
  - 外部依赖失败
  - 业务逻辑错误

FATAL：系统级故障，立即处理
  - 无法启动
  - 数据库连接断开
```

### 1.1 分级原则

```
1. 生产默认 INFO
2. 线上临时调 DEBUG → 动态日志级别
3. ERROR 不滥用 → 不是所有异常都 ERROR
4. FATAL 慎用 → 真正的致命错误

反模式：
  ✗ try-catch 里全打 ERROR → ERROR 泛滥
  ✗ 循环里打 INFO → 日志爆炸
  ✗ 异常不打堆栈 → 无法排查
```

## 2. 结构化日志

### 2.1 为什么结构化

```
非结构化（传统）：
  "用户 12345 下单成功，订单号 67890"
  → 难搜索、难解析、难聚合

结构化（JSON）：
  {
    "timestamp": "2024-01-01T10:00:00Z",
    "level": "INFO",
    "service": "order-service",
    "traceId": "abc123",
    "userId": 12345,
    "orderId": 67890,
    "message": "下单成功",
    "amount": 99.9
  }
  → 可搜索（userId=12345）
  → 可聚合（按 service 统计）
  → 可关联（traceId 关联链路）
```

### 2.2 推荐字段

```
必填：
  timestamp、level、service、traceId、message

业务：
  userId、orderId、action、amount

技术：
  host、pid、thread、duration、error

扩展：
  自定义业务标签
```

## 3. 日志规范

### 3.1 打日志原则

```
1. 关键路径必打
   - 下单、支付、退款
   - 状态变更

2. 异常必打堆栈
   log.error("下单失败", e)  // 带堆栈
   不是 log.error("下单失败: " + e.getMessage())  // 丢堆栈

3. 不打敏感信息
   - 密码、身份证、银行卡
   - 脱敏处理

4. 不打大对象
   - 大 JSON、大 List
   - 摘要即可

5. 不在循环里打
   - 循环 1 万次 = 1 万条日志

6. 条件打日志
   if (log.isDebugEnabled()) {
       log.debug("详情: {}", detail);
   }
```

### 3.2 日志内容规范

```
好的日志：
  - 时间：何时
  - 主体：谁（userId）
  - 动作：做了什么（下单）
  - 结果：成功/失败
  - 上下文：订单号、金额

示例：
  log.info("用户{}下单成功, 订单={}, 金额={}", userId, orderId, amount);

差的日志：
  log.info("success");  // 谁成功？什么成功？
  log.info("error");    // 什么错误？
```

## 4. 日志采集架构

### 4.1 ELK 架构

```
应用 → 日志文件 → Filebeat → Kafka → Logstash → Elasticsearch → Kibana

  Filebeat：轻量采集
  Kafka：缓冲削峰
  Logstash：解析过滤
  ES：存储+搜索
  Kibana：可视化

优点：功能全、生态成熟
缺点：重、成本高
```

### 4.2 Loki 架构（轻量替代）

```
应用 → 日志文件 → Promtail → Loki → Grafana

  Promtail：采集
  Loki：存储（只索引标签，不索引内容）
  Grafana：可视化

优点：轻量、成本低、与 Grafana 集成
缺点：全文搜索不如 ES
```

### 4.3 选型

```
日志量小（< 10GB/天）→ Loki
日志量大 + 全文搜索需求 → ELK
云上 → 云日志服务（SLS/CLS）
```

## 5. 日志与链路追踪关联

### 5.1 为什么要关联

```
链路追踪告诉你"哪个服务慢"
日志告诉你"为什么慢"

关联后：
  拿到 TraceID → 在日志系统搜 TraceID → 看到该请求所有日志
  → 完整还原请求全过程
```

### 5.2 实现方式

```
MDC（Mapped Diagnostic Context）：
  1. 请求入口生成 TraceID
  2. 存入 MDC（ThreadLocal）
  3. 日志框架自动输出 TraceID
  4. 跨服务通过请求头传播

配置（Logback 示例）：
  <pattern>%d [%X{traceId}] %-5level %logger - %msg%n</pattern>

  → 输出：2024-01-01 [abc123] INFO OrderService - 下单成功
```

### 5.3 跨服务日志关联

```
网关生成 TraceID → 透传到所有服务 → 所有服务日志带同一 TraceID

排查：
  1. 拿到 TraceID
  2. 在 Kibana/Loki 搜 traceId:abc123
  3. 看到全链路所有日志
  4. 按时间排序还原流程
```

## 6. 日志成本控制

### 6.1 成本来源

```
存储：ES/Loki 存储
网络：采集传输
计算：索引、解析
```

### 6.2 控制策略

```
1. 分级保留
   - ERROR：30 天
   - WARN：15 天
   - INFO：7 天
   - DEBUG：不存

2. 采样
   - 高频日志采样（如 10%）
   - 关键日志全量

3. 压缩
   - 历史日志压缩存储
   - 冷热分离

4. 字段裁剪
   - 不必要字段不打
   - 大字段摘要

5. 采样率动态调整
   - 平时低采样
   - 故障时高采样
```
