## 1. 监控分层体系

```
从上到下四层：

1. 业务监控（最重要）
   - 下单量、支付成功率、转化率
   - 异常订单数、投诉量
   - 业务指标异常 = 直接影响用户

2. 应用监控
   - QPS、响应时间、错误率
   - JVM、线程池、连接池
   - 接口级指标

3. 中间件监控
   - MySQL：慢查询、连接数、复制延迟
   - Redis：内存、命中率、大 key
   - MQ：堆积量、消费速度
   - ES：集群健康、查询延迟

4. 基础设施监控
   - CPU、内存、磁盘、网络
   - 容器指标
   - 机房网络
```

## 2. 指标设计方法论

### 2.1 USE 原则（资源类）

```
U - Utilization（使用率）：CPU 80%
S - Saturation（饱和度）：等待队列长度
E - Errors（错误）：丢包数、错误率

适用：CPU/内存/磁盘/网络等资源
```

### 2.2 RED 原则（服务类）

```
R - Rate（请求率）：QPS
E - Errors（错误率）：错误请求占比
D - Duration（耗时）：P50/P90/P99

适用：HTTP 接口、RPC 服务
```

### 2.3 Four Golden Signals（Google SRE）

```
Latency（延迟）：请求耗时
Traffic（流量）：QPS
Errors（错误）：错误率
Saturation（饱和度）：资源使用率

= RED + Saturation
```

## 3. SLI/SLO/Error Budget

### 3.1 概念

```
SLI（Service Level Indicator）：服务等级指标
  - 具体测量值，如 "P99 延迟"
  - 如：P99 < 200ms

SLO（Service Level Objective）：服务等级目标
  - 目标值，如 "99.9% 请求 P99 < 200ms"
  - 30 天滚动窗口

SLA（Service Level Agreement）：服务等级协议
  - 对外承诺，违约赔偿
  - 比 SLO 更宽松

Error Budget（错误预算）
  - 100% - SLO = 错误预算
  - SLO 99.9% → 错误预算 0.1%
  - 预算用完 → 停止发新功能，专注稳定性
```

### 3.2 实战示例

```
订单服务 SLO：
  - 可用性：99.9%（每月停机 < 43 分钟）
  - 延迟：P99 < 500ms
  - 成功率：> 99.5%

Error Budget 计算：
  - 月 30 天 = 43200 分钟
  - 99.9% 可用 → 预算 43 分钟
  - 已停机 20 分钟 → 剩余 23 分钟
  → 预算耗尽前可继续上线
  → 预算耗尽 → 冻结发布
```

## 4. 告警设计

### 4.1 告警分级

```
P0（电话+IM）：核心业务中断、数据丢失
  → 5 分钟内响应

P1（IM+短信）：重要功能异常、错误率突增
  → 15 分钟内响应

P2（IM）：次要功能异常、性能下降
  → 1 小时内响应

P3（邮件）：仅记录，不主动通知
```

### 4.2 告警原则

```
1. 可执行：告警来了能知道做什么
2. 低噪音：避免频繁误报（告警疲劳）
3. 有上下文：附带排查线索（TraceID、服务名）
4. 可收敛：问题解决后告警自动恢复
5. 分级合理：不是所有异常都 P0

反模式：
  ✗ 一个指标配 10 个告警规则 → 告警风暴
  ✗ 无分级全发短信 → 告警疲劳
  ✗ 告警无上下文 → 还要手动查
```

### 4.3 告警规则设计

```
阈值告警：
  - CPU > 80% 持续 5 分钟
  - 错误率 > 1% 持续 1 分钟
  - 慎用瞬时阈值（抖动误报）

趋势告警：
  - QPS 环比下降 50%
  - 错误率同比上升 3 倍
  - 适合业务指标

异常检测告警：
  - 基于历史基线
  - AI 识别异常
  - 适合无明显阈值的指标

组合告警：
  - CPU 高 + 响应慢 → P0
  - CPU 高 + 响应正常 → P2
  → 多条件组合减少误报
```

### 4.4 告警降噪

```
1. 聚合：同类告警合并
2. 抑制：高级别告警抑制低级别
3. 静默：维护期间静默
4. 频率限制：同一告警 N 分钟内只发一次
5. 恢复通知：问题恢复后通知
```

## 5. 监控大盘设计

### 5.1 大盘分层

```
全局大盘（CTO/总监看）：
  - 核心业务指标（下单量、GMV）
  - 全局可用性
  - Error Budget 消耗

服务大盘（架构师/TL 看）：
  - 各服务 RED 指标
  - 服务拓扑
  - 错误率 TOP

实例大盘（开发看）：
  - 单实例资源使用
  - JVM 详情
  - 慢查询
```

### 5.2 大盘设计原则

```
1. 一屏看清：核心指标一屏展示
2. 层次分明：总览 → 下钻
3. 颜色语义：红=异常，绿=正常，黄=警告
4. 时间范围：默认近 1 小时，可切换
5. 告警标注：在图上标注告警点
```

## 6. 工具栈选型

```
指标监控：Prometheus + Grafana
  - Prometheus：采集+存储
  - Grafana：可视化
  - 业界标准

日志：ELK / Loki
  - ELK：功能全，重
  - Loki：轻量，与 Grafana 集成

链路：SkyWalking / Jaeger
  - 详见 [15-链路追踪](./01-链路追踪深度实战.md)

告警：Alertmanager
  - 与 Prometheus 配合
  - 分级路由

统一可观测性：
  - Datadog（商业，全功能）
  - 阿里云 ARMS（云上）
  - OpenTelemetry（开源标准）
```
