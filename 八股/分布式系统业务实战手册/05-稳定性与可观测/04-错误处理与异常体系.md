# 错误处理与异常体系

## 1. 错误分类

```
按可恢复性：

1. 瞬时错误（可重试）
   - 网络抖动
   - 超时
   - 限流
   → 重试可恢复

2. 永久错误（不可重试）
   - 参数错误
   - 业务规则不满足
   - 权限不足
   → 重试无用，需修正请求

3. 系统错误（需人工）
   - DB 连接断开
   - 磁盘满
   - 依赖服务宕机
   → 需运维介入

按层次：

1. 用户输入错误（400）
   - 参数缺失、格式错
   - 返回明确提示

2. 业务逻辑错误（400/422）
   - 余额不足、库存不够
   - 返回业务原因

3. 认证授权错误（401/403）
   - 未登录、无权限

4. 系统错误（500）
   - 空指针、DB 异常
   - 记录堆栈，内部排查

5. 依赖错误（502/503/504）
   - 下游超时、熔断
   - 降级处理
```

## 2. 异常处理原则

### 2.1 核心原则

```
1. 快速失败（Fail Fast）
   - 发现问题立即报错
   - 不要吞异常继续执行
   - 越早失败越容易定位

2. 明确处理
   - 捕获具体异常，不要 catch Exception
   - 针对不同异常不同处理

3. 不吞异常
   - catch 后必须处理（日志/抛出/降级）
   - 不能空 catch

4. 保留上下文
   - 抛新异常时保留原异常（cause）
   - 日志带堆栈

5. 幂等友好
   - 异常后重试要安全
   - 业务设计支持幂等
```

### 2.2 分层处理

```
Controller 层：
  - 参数校验
  - 统一异常处理（@ControllerAdvice）
  - 统一错误响应

Service 层：
  - 业务异常抛出（自定义业务异常）
  - 事务回滚

DAO 层：
  - 数据异常转换（DuplicateKey 等）
  - 不暴露 SQL 细节

外部调用层：
  - 超时控制
  - 重试
  - 降级
```

## 3. 错误码体系

### 3.1 为什么需要错误码

```
没有错误码：
  { "message": "余额不足" }
  → 前端无法精确处理（是余额不足还是其他？）

有错误码：
  { "code": "INSUFFICIENT_BALANCE", "message": "余额不足" }
  → 前端按 code 精确处理
  → 国际化按 code 翻译
  → 监控按 code 聚合
```

### 3.2 错误码设计

```
方案 1：数字码
  10001 = 参数错误
  20001 = 余额不足
  → 紧凑但不直观

方案 2：字符串码（推荐）
  PARAM_INVALID
  INSUFFICIENT_BALANCE
  ORDER_NOT_FOUND
  → 直观可读

方案 3：HTTP + 业务码
  HTTP 400 + { "code": "INSUFFICIENT_BALANCE" }
  → 符合 HTTP 语义 + 业务精确
```

### 3.3 统一响应格式

```json
{
  "success": false,
  "code": "INSUFFICIENT_BALANCE",
  "message": "余额不足",
  "traceId": "abc123",
  "timestamp": 1690000000000,
  "data": null
}
```

### 3.4 错误码管理

```
1. 集中定义
   - 错误码枚举/常量
   - 团队共享

2. 文档化
   - 错误码字典
   - 含义、HTTP 状态、处理建议

3. 版本化
   - 错误码不删除，只废弃
   - 兼容旧版本

4. 监控
   - 按错误码聚合统计
   - 异常错误码告警
```

## 4. 重试与降级

### 4.1 重试策略

```
可重试的错误：
  - 网络超时
  - 连接拒绝
  - 503 服务不可用

不可重试的错误：
  - 400 参数错误
  - 401 未授权
  - 业务逻辑错误

重试参数：
  - 最大重试次数：3
  - 重试间隔：指数退避（1s, 2s, 4s）
  - 抖动：避免重试风暴（加随机）
```

### 4.2 重试的坑

```
1. 重试风暴
   - A 重试 B，B 重试 C
   - 放大效应
   → 限制重试次数 + 退避

2. 非幂等重试
   - 转账重试 → 重复扣款
   → 确保幂等再重试

3. 重试超总时限
   - 重试 3 次每次 5s = 15s
   → 设总超时
```

### 4.3 降级策略

```
降级触发：
  - 重试耗尽
  - 熔断触发
  - 依赖不可用

降级方式：
  1. 返回默认值
  2. 返回缓存
  3. 返回简化结果
  4. 异步化（写队列稍后处理）
  5. 功能降级（关闭非核心）

详见 [04 §4 服务降级与熔断](../03-分布式架构与设计/00-分布式系统核心设计模式.md)
```

## 5. 错误恢复

### 5.1 自动恢复

```
1. 重试
2. 熔断恢复（半开探测）
3. 故障转移（主备切换）
4. 重新选举（Raft Leader）
```

### 5.2 手动恢复

```
1. 数据修复
   - 对账发现差异 → 手动修正

2. 补偿
   - 业务冲正
   - 详见 [数据一致性/06](../02-数据一致性/06-对账补偿与最终一致.md)

3. 回滚
   - 版本回滚
   - 数据回滚（备份恢复）
```

## 6. 错误处理反模式

```
1. 吞异常
   catch (Exception e) {}
   → 问题被隐藏，日后暴雷

2. 日志代替处理
   catch (Exception e) { log.error(e); }
   → 记了但没处理，用户仍受影响

3. 过宽捕获
   catch (Exception e) → 连 NPE 也捕获
   → 应捕获具体异常

4. 异常控制流程
   用异常做 if-else
   → 性能差、可读性差

5. 不带堆栈
   log.error("失败: " + e.getMessage())
   → 丢堆栈，无法排查

6. 循环内重试不退出
   while(true) { retry(); }
   → 死循环

7. 错误码不一致
   同一错误不同接口不同码
   → 前端难处理
```
