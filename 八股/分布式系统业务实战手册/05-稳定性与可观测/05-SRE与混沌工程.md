## 1. SRE 方法论

```
SRE（Site Reliability Engineering）= 用工程方法解决运维问题

核心理念：
  1. 追求稳定性 ≠ 追求 100% 可用
     - 100% 成本极高
     - 用户感知不到（99.9% vs 99.99%）
     - 用 Error Budget 平衡稳定和迭代

  2. 运维工程化
     - 自动化 > 手工
     - 代码 > 文档
     - 监控 > 巡检

  3. 数据驱动
     - SLI/SLO 度量
     - Error Budget 决策
     - 故障复盘改进

SRE vs 传统运维：
  传统运维：人肉救火、被动响应
  SRE：工程化、数据驱动、主动预防
```

## 2. Error Budget 实战

### 2.1 概念

```
SLO 99.9% → Error Budget 0.1%
  月 43200 分钟 → 预算 43 分钟

预算用途：
  - 允许的停机时间
  - 允许的错误请求
  - 允许的慢请求

预算消耗场景：
  - 发布导致的短暂错误
  - 故障恢复时间
  - 压测影响
```

### 2.2 预算管理

```
预算充足（剩余 > 50%）：
  → 鼓励创新、快速迭代
  → 可接受一定风险

预算紧张（剩余 20-50%）：
  → 谨慎发布
  → 增加测试

预算耗尽（剩余 < 20%）：
  → 冻结新功能发布
  → 专注稳定性
  → 偿还技术债

预算超额（超支）：
  → 必须事故复盘
  → 改进措施
```

### 2.3 实施步骤

```
1. 定义 SLI（测量什么）
   - 可用性：成功请求数/总请求数
   - 延迟：P99 < 500ms 的占比
   - 质量：非降级响应占比

2. 设定 SLO（目标值）
   - 可用性 99.9%
   - 延迟 P99 < 500ms（99% 请求）
   - 与业务对齐

3. 监控 Error Budget
   - 实时消耗
   - 趋势预测
   - 超支告警

4. 预算驱动决策
   - 发布审批
   - 稳定性投入
```

## 3. 混沌工程

### 3.1 什么是混沌工程

```
混沌工程 = 主动注入故障，验证系统韧性

理念：
  - 故障一定会发生（墨菲定律）
  - 与其等故障来，不如主动演练
  - 在非高峰期发现问题，避免高峰期暴雷

Netflix 创立 Chaos Monkey：
  - 随机杀生产环境实例
  - 验证自动恢复
  → 倒逼系统高可用
```

### 3.2 混沌实验设计

```
1. 假设稳态
   - 定义正常指标（如 P99 < 200ms）

2. 注入故障
   - 杖器：杀进程、CPU 打满、网络延迟
   - 中间件：Redis 宕机、MQ 堆积
   - 依赖：下游超时、DNS 故障

3. 观察系统
   - 稳态是否保持？
   - 自动恢复是否生效？
   - 告警是否触发？

4. 改进
   - 发现弱点 → 修复
   - 自动化恢复 → 完善
```

### 3.3 故障注入类型

```
基础设施层：
  - 杀容器/进程
  - CPU/内存/磁盘打满
  - 网络延迟/丢包/分区

中间件层：
  - DB 连接断开
  - Redis 不可用
  - MQ 堆积

应用层：
  - 依赖服务超时
  - 返回错误
  - 慢响应

外部依赖：
  - 第三方接口不可用
  - DNS 故障
```

### 3.4 混沌工程工具

```
Chaos Mesh（CNCF）：K8s 原生
Chaos Monkey：Netflix
ChaosBlade（阿里）：国内
Litmus：K8s
```

### 3.5 落地建议

```
1. 从小规模开始
   - 非核心服务
   - 非高峰时段

2. 逐步扩大
   - 核心服务
   - 跨服务
   - 跨机房

3. 必须有应急预案
   - 实验失控能快速恢复
   - 有"终止实验"按钮

4. 文化建设
   - 不追责
   - 鼓励发现问题
```

## 4. 全链路压测

### 4.1 为什么需要

```
单接口压测：
  - 只测单点
  - 看不到全链路瓶颈

全链路压测：
  - 模拟真实用户行为
  - 全链路压测
  - 发现系统真实瓶颈
  → 大促前必备
```

### 4.2 压测架构

```
1. 流量录制
   - 录制生产真实请求
   - 脱敏处理

2. 影子库
   - 压测数据写影子库（不污染生产）
   - 或压测标记 + 数据隔离

3. 压测执行
   - 分布式压测机
   - 逐步加压

4. 全链路监控
   - 压测流量标红
   - 各服务指标
   - 瓶颈定位
```

### 4.3 压测流程

```
1. 准备
   - 确定压测目标（如 10万 QPS）
   - 准备压测数据
   - 影子库就绪

2. 预热
   - 小流量预热
   - 检查链路

3. 阶梯加压
   - 1万 → 5万 → 10万
   - 每阶段观察

4. 瓶颈定位
   - CPU/内存/IO 瓶颈
   - DB 慢查询
   - 连接池耗尽
   - 网络带宽

5. 优化验证
   - 优化后重新压测
   - 确认提升

6. 报告
   - 容量评估
   - 瓶颈点
   - 优化建议
```

### 4.4 压测工具

```
JMeter：通用、GUI
wrk：轻量高性能
Gatling：高性能、脚本化
自研全链路压测平台：大厂必备
```

## 5. 故障应急

### 5.1 故障响应流程

```
1. 发现（0-5 分钟）
   - 告警触发
   - 用户反馈
   - 监控异常

2. 响应（5-15 分钟）
   - Oncall 介入
   - 初步判断影响范围
   - 通知相关方

3. 定位（15-30 分钟）
   - 查链路追踪
   - 查监控
   - 查日志
   - 详见 [15-链路追踪](./01-链路追踪深度实战.md)

4. 恢复（30-60 分钟）
   - 降级/回滚/扩容
   - 优先恢复服务

5. 复盘（1-3 天）
   - 写故障报告
   - 根因分析
   - 改进措施
```

### 5.2 故障恢复手段

```
按速度从快到慢：

1. 降级（秒级）
   - 关闭非核心功能
   - 返回缓存/默认值

2. 限流（秒级）
   - 限制流量保护系统

3. 回滚（分钟级）
   - 版本回滚
   - 配置回滚

4. 扩容（分钟级）
   - 增加实例

5. 切换（分钟级）
   - 主从切换
   - 机房切换

6. 修复（小时级）
   - 热修复
   - 数据修复
```

### 5.3 故障复盘

```
复盘原则：
  - 对事不对人
  - 不追责个人，追流程
  - 关注改进而非追责

复盘内容：
  1. 故障概述：时间、影响、损失
  2. 时间线：发现→响应→定位→恢复
  3. 根因：直接原因 + 根本原因
  4. 改进：立即修复 + 短期 + 长期
  5. 责任：流程问题非个人

改进跟踪：
  - 改进项有 owner 和 deadline
  - 定期 review 进度
```
